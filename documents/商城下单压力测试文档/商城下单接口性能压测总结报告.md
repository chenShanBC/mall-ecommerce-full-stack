# 商城下单接口性能压测总结报告

## 一、压测概述

本次针对商城下单核心接口完成四轮梯度压力测试，压测时长统一为 40s，覆盖 50 并发、60 并发（两轮）、100 并发场景。测试过程中持续观察接口响应耗时、成功率、吞吐量、性能抖动、慢请求峰值等核心指标，并结合多轮代码优化结果，完成系统性能迭代验证与上线能力评估。

本次压测目标并非单纯追求极限并发数字，而是重点验证下单链路在真实电商业务场景中的稳定性、响应速度和容量边界。商城下单接口属于交易核心链路，涉及用户身份校验、商品快照读取、订单聚合创建、库存预占、订单明细持久化、订单超时消息发布等多个环节，对一致性、幂等性和性能稳定性要求较高。

测试目的如下：

- 验证商城下单接口在不同并发量级下的稳定性和响应速度；
- 排查高并发场景下的性能瓶颈、慢请求来源和系统抖动原因；
- 验证优化前后系统性能变化，判断优化方案是否有效；
- 评估当前系统是否满足中型商城日常、高峰及小型活动流量需求；
- 为后续上线交付、容量规划和性能优化提供依据。

## 二、行业通用性能评级标准（电商下单接口）

### 2.1 及格标准（线上可用）

- 错误率：0%，无业务报错、接口超时或系统异常；
- 平均响应耗时：≤300ms；
- P95 耗时：≤500ms；
- 无大量 1s 以上严重慢请求；
- 系统无持续卡顿、连接池耗尽、线程阻塞或服务崩溃现象。

### 2.2 优秀标准（线上优质体验）

- 错误率：0%，全程稳定无异常；
- 平均响应耗时：≤100ms；
- P95 耗时：≤300ms；
- 大多数请求稳定在 100ms 以内；
- 仅允许极少量瞬时长尾波动，不应出现持续性上千毫秒尖峰。

### 2.3 真实电商并发量级对照

结合普通 B2C 商城交易接口真实访问特征，可按以下量级进行参考：

- 商城日常平峰：20-40 并发；
- 商城晚间高峰：50-70 并发；
- 商城小型活动/直播促销高峰：100 并发；
- 100 并发以上：通常已属于中小型商城下单接口的高压测试或活动级容量验证场景。

需要说明的是，行业中常见的“百万并发”多指全站连接数、长连接、静态资源访问、网关层流量或多个业务接口合并统计，并不等同于真实下单接口的瞬时并发。下单接口涉及库存、订单、支付前置校验等核心交易逻辑，其并发口径应以真实交易请求为准。

## 三、四轮压测全量数据对比与分析

### 3.1 第一轮：50 并发 / 40s（初始版本）

- 成功率：100%；
- 核心表现：主流耗时约 30-550ms，最小耗时 28ms，最大耗时 1620ms；
- QPS：约 220-240；
- 存在问题：响应两极分化明显，存在大量慢请求；压测中后期性能抖动明显，出现多次超大延迟尖峰；
- 综合评级：及格。

第一轮 50 并发下，系统整体吞吐量表现尚可，成功率达到 100%，说明接口主流程可用，库存与订单链路没有出现明显业务错误。但从响应耗时看，系统存在明显不稳定问题：一部分请求可在几十毫秒内完成，另一部分请求却出现数百毫秒甚至 1.6s 的长尾。

该轮结果说明系统在较低并发下具备基本承载能力，但链路中存在潜在阻塞点或资源竞争点。一旦并发继续升高，慢请求可能被进一步放大，导致整体吞吐下降。

### 3.2 第二轮：60 并发 / 40s（未优化版本）

- 成功率：100%；
- 核心表现：正常请求约 13-200ms，但批量出现 1300-3320ms 超大延迟；
- 峰值耗时：3320ms；
- QPS：约 120-140；
- 存在问题：并发提升后瓶颈彻底暴露，中后期出现集体卡顿，资源争抢和线程阻塞严重，吞吐量相比 50 并发明显下降；
- 综合评级：不及格。

第二轮将并发从 50 提升到 60 后，系统性能出现明显退化。理论上并发小幅提升不应导致 QPS 大幅下降，但实际 QPS 从第一轮约 220-240 降至 120-140，说明系统出现了明显的排队、阻塞或热点资源竞争。

该轮最核心的问题不是成功率，而是响应稳定性。虽然没有业务失败，但大量请求耗时超过 1s，峰值达到 3320ms，已经不满足交易接口线上体验要求。对于下单接口而言，3s 级别响应会直接影响用户支付转化率，也会增加重复点击、重复下单、重复请求等风险。

### 3.3 第三轮：60 并发 / 40s（优化后版本）

- 成功率：100%；
- 核心表现：90% 以上请求稳定在 30-80ms；
- 慢请求情况：仅零星请求约 200-300ms，无 500ms 以上慢请求；
- QPS：约 185-190；
- 优化效果：系统卡顿和超大延迟问题基本消除，全程性能表现平稳，无明显抖动；
- 综合评级：优秀。

第三轮仍然保持 60 并发，但经过优化后，系统表现发生明显改善。相比第二轮，QPS 从 120-140 提升到 185-190，同时 1s 以上慢请求完全消失，主流耗时稳定在 30-80ms。

该轮结果说明前期瓶颈并非硬件容量不足，而主要来自代码链路中的同步开销、重复查询、日志/消息/数据库写入竞争、幂等处理不充分等工程实现问题。优化后系统能够稳定覆盖商城日常和晚间高峰流量。

### 3.4 第四轮：100 并发 / 40s（极限承压版本）

- 成功率：100%，无任何报错、超时或业务异常；
- 核心表现：绝大多数请求稳定在 30-90ms；
- 长尾情况：少量请求约 100-270ms，仅极个别瞬时波动达到 350-600ms；
- 慢请求情况：无 1s 以上恶性慢请求；
- 承压能力：100 并发高压场景下，系统无崩盘、无持续卡顿，仅存在正常瞬时网络、数据库或 JVM 调度波动；
- 综合评级：优秀偏上。

第四轮将并发提升到 100 后，系统仍保持 100% 成功率，且绝大多数请求稳定在 100ms 以内。相比第一轮和第二轮，最大变化在于系统没有出现持续性卡顿和秒级慢请求，说明核心链路已经具备较强的抗压能力。

少量 350-600ms 的瞬时波动属于可接受范围，尤其是在本地开发环境、数据库连接池、JVM GC、RabbitMQ/Redis 调用和操作系统调度共同参与的情况下，极个别长尾无法完全避免。关键在于该类波动没有形成持续堆积，也没有引发错误率上升。

## 四、四轮压测综合排名

综合成功率、平均耗时、慢请求数量、峰值耗时、吞吐稳定性和高并发承压能力，四轮压测表现排名如下：

**100 并发（最新版本） > 优化后 60 并发 > 初始 50 并发 > 未优化 60 并发**

从数据趋势看，系统优化后并没有因为并发提升而出现明显性能衰减，反而在 100 并发下表现优于初始 50 并发和未优化 60 并发，说明优化方向正确，系统瓶颈已被有效缓解。

## 五、核心性能瓶颈分析

结合四轮压测数据和下单链路代码实现，前期性能瓶颈主要集中在以下几个方面。

### 5.1 数据库往返次数偏多

下单接口涉及订单、订单明细、商品快照、库存记录等多类数据。如果每个订单项都单独查询商品快照、单独插入订单明细、订单保存后再反查订单聚合，会导致一次下单请求产生多次数据库往返。

在低并发下该问题不明显，但在 60 并发后，数据库连接池、网络往返和 SQL 执行时间会被放大，形成请求排队和长尾延迟。

### 5.2 订单列表和后台查询存在 N+1 风险

订单分页查询如果先查询订单主表，再对每条订单逐个查询订单明细，一页 10 条订单就可能产生 11 次 SQL。该问题对列表接口影响更明显，也会间接影响数据库整体负载，从而影响下单链路稳定性。

### 5.3 库存链路存在重复查询和重复事件问题

库存预占链路需要校验库存状态、初始化 Redis 库存缓存、执行 Lua 原子扣减、保存库存锁记录、记录库存日志并发布 MQ 同步事件。前期实现中存在多 SKU 场景下重复加载库存、重复判断缓存、幂等请求仍可能重复发布 MQ 的问题。

这些问题在高并发下会增加 Redis、DB 和 MQ 的综合压力，导致系统中后期出现抖动。

### 5.4 同步日志和高频 info 日志影响长尾

下单链路属于高频接口，如果成功路径中存在大量同步日志、库存操作日志、超时判断日志等，会增加 IO 压力。日志本身可能不是平均耗时的主要来源，但非常容易放大 P95、P99 和最大响应耗时。

### 5.5 幂等处理依赖代码判断，存在并发竞态

库存操作日志原先采用“先查是否存在，再插入”的方式。在并发请求或 MQ 重复消费场景下，两个线程可能同时判断不存在，然后同时插入，最终依赖唯一索引报错兜底。这类实现会增加异常成本，也会影响高并发下的稳定性。

## 六、性能优化核心方案与正确性说明

### 6.1 最优优化方案（已落地见效）

本次优化遵循“核心交易强一致、外围操作最终一致、减少同步阻塞、保持领域边界清晰”的原则，在不破坏现有业务功能的前提下完成了以下改造。

#### 6.1.1 商品快照批量查询

下单创建订单聚合时，将原先的逐个 SKU 查询商品快照，优化为按 SKU 集合批量查询，并在内存中构建 Map 进行匹配。

优化前：

- 一个订单包含 N 个 SKU，可能触发 N 次商品快照查询；
- 同一 SKU 重复出现时，存在重复查询和重复匹配；
- 构建订单项时多次 stream filter，存在额外 CPU 开销。

优化后：

- 对请求中的 SKU 进行聚合；
- 一次性批量查询商品快照；
- 使用 Map 快速构建订单项；
- 保留订单聚合在领域服务中创建，符合非贫血 DDD 思路。

#### 6.1.2 订单明细批量插入

订单持久化时，将原先逐条插入订单明细，优化为批量插入。

优化前：

- 插入订单主表 1 次；
- 插入订单明细 N 次；
- 保存后再次反查订单和明细。

优化后：

- 插入订单主表 1 次；
- 批量插入订单明细 1 次；
- 直接使用内存中的订单数据组装返回结果，避免无意义反查。

当订单包含多个商品时，该优化可显著减少数据库往返次数，对下单接口平均耗时和 P95 均有直接收益。

#### 6.1.3 订单查询消除 N+1

针对订单列表、后台订单搜索、超时订单扫描等场景，将逐个订单加载明细的方式改为批量加载。

优化前：

- 查询订单分页 1 次；
- 每条订单再查询一次明细；
- 一页 10 条订单约 11 次 SQL。

优化后：

- 查询订单分页 1 次；
- 按 order_id in (...) 批量查询明细 1 次；
- 内存分组后组装订单聚合；
- 一页 10 条订单约 2 次 SQL。

该优化降低了数据库连接池压力，也减少了后台查询对交易链路的干扰。

#### 6.1.4 库存预占批量加载与请求聚合

库存预占链路采用方案 A：批量预加载库存 + 请求项聚合 + 单 SKU Lua 原子预占。

优化后流程如下：

1. 对请求中的库存项按 SKU 聚合，避免同一 SKU 重复预占；
2. 批量加载所有 SKU 的库存领域对象；
3. 统一校验 SKU 是否存在；
4. 保持每个 SKU 使用 Redis Lua 脚本完成原子预占；
5. 保留失败回滚逻辑，已预占成功的 SKU 仍可按原逻辑释放；
6. 保留库存锁记录、库存操作日志和 MQ 异步同步语义。

该方案没有引入分布式锁，也没有改变 Redis 作为实时库存准入层、DB 通过 MQ 最终同步的整体架构，因此风险较低，且能有效减少多 SKU 下单时的重复 DB 查询。

#### 6.1.5 支付确认状态条件更新

支付确认从普通更新优化为状态条件更新，即只有订单仍处于 `PENDING_PAYMENT` 状态时，才允许更新为 `PAID`。

该优化解决了重复支付确认、重复回调或并发调用可能导致的重复加销量问题。库存确认本身保持幂等，订单状态更新也通过数据库条件保证幂等，整体更符合交易系统设计要求。

#### 6.1.6 库存操作日志唯一索引幂等插入

库存操作日志从“先查后插”优化为“直接插入，唯一键冲突则认为幂等成功”。

优化收益：

- 减少一次 exists 查询；
- 避免并发下先查后插的竞态问题；
- 降低重复 MQ 消费或重复请求导致的异常成本；
- 更符合以数据库唯一约束保障幂等的工程实践。

#### 6.1.7 重复库存操作不再重复发布 MQ

Redis Lua 脚本已经能够区分“本次新执行”和“之前已经执行过”。优化后仅当 Lua 返回新执行成功时，才发布库存同步 MQ。

这可以减少无效 MQ 消息，降低库存 DB 同步压力，避免重复事件造成日志、对账和消费端额外负担。

#### 6.1.8 高频成功日志降级

将下单成功、订单超时判断、发布超时消息等高频成功路径日志从 `info` 降为 `debug`。

该优化不会影响异常排查能力，但能减少压测和生产高峰期的日志 IO 压力，降低 P99 长尾概率。

### 6.2 正确性说明

本次优化没有破坏核心交易一致性，原因如下：

- 库存扣减仍由 Redis Lua 脚本完成，单 SKU 维度具备原子性；
- 库存预占失败后，仍会对已预占成功的库存项执行回滚释放；
- 库存锁记录保留唯一业务键，支持业务维度幂等；
- 库存 DB 同步仍通过 MQ 异步完成，并保留对账和修复能力；
- 订单支付确认通过数据库状态条件更新，避免重复支付确认造成重复副作用；
- 订单、库存、商品快照仍通过领域服务和仓储接口组织，没有将业务规则下沉到 Controller 或 SQL 中；
- 改动主要集中在应用服务编排优化、仓储实现优化和基础设施层 SQL 优化，符合 DDD 分层边界。

### 6.3 关键注意事项

虽然当前压测结果已经达到优秀水平，但生产环境仍需关注以下事项：

- Redis 是实时库存准入层，必须保证可用性和数据恢复能力；
- RabbitMQ 负责库存 DB 最终同步和订单超时关闭，需监控消息堆积、死信和消费失败；
- 库存对账任务需要常态化开启，及时发现 Redis、DB、库存锁记录之间的不一致；
- 数据库连接池、Redis 连接池、RabbitMQ 连接参数需要根据生产机器配置合理设置；
- 压测环境与生产环境硬件、网络、数据库配置不同，最终上线前仍建议在准生产环境复测；
- 订单接口不建议无限制提高并发，超过 100 并发后应引入限流、排队、熔断或活动专用库存策略；
- 下单接口应保持核心路径精简，非核心操作如埋点、复杂日志、消息通知等应尽量异步化。

## 七、最终验收结论与上线建议

### 7.1 验收结论

经过多轮迭代优化，商城下单接口性能从“及格但不稳定”提升至“优秀偏上水平”。

综合四轮压测数据，可以得出以下结论：

- 接口成功率始终保持 100%，说明业务主链路稳定；
- 未优化版本在 60 并发下出现严重长尾和吞吐下降，说明前期确实存在性能瓶颈；
- 优化后 60 并发下 90% 以上请求稳定在 30-80ms，无 500ms 以上慢请求；
- 最新版本在 100 并发高压场景下仍保持绝大多数请求 30-90ms，无 1s 以上恶性慢请求；
- 当前系统可稳定覆盖中型商城日常平峰、晚间高峰和小型直播/促销活动场景；
- 系统性能瓶颈已从明显阻塞型瓶颈转变为偶发性系统调度、网络或数据库瞬时波动，整体已达到可交付水平。

### 7.2 上线建议

基于当前压测结果，商城下单接口可以进入上线交付阶段。

建议如下：

- 当前性能完全满足线上生产环境使用，可作为当前版本交付；
- 不建议继续盲目提高并发压测数值，100 并发已经覆盖当前业务定位下的主要高峰场景；
- 上线前建议在准生产环境使用相同脚本复测一次，确认数据库、Redis、RabbitMQ 配置与本地压测结果一致；
- 生产环境需要开启基础监控，包括接口耗时、错误率、QPS、数据库连接池、Redis 延迟、MQ 堆积、库存对账异常数；
- 后续如有大型促销活动，应单独设计活动库存策略、接口限流和排队机制，而不是直接复用普通下单链路硬抗无限流量；

# B2C电商五期平台化与智能化能力开发主文档

> 本文档用于指导 `mallFei` 在完成 MVP、一阶段交易闭环、一阶段增强可靠性、二期运营履约、三期平台经营、四期生产治理之后，进入五期平台化与智能化能力建设阶段。  
> 五期的核心目标不是继续补齐单点业务功能，而是让系统具备多端、多渠道、多主体、多场景扩展能力，并在推荐、客服、风控、运营、选品、数据分析等方向逐步引入智能化辅助能力。

---

# 1 项目概述

## 1.1 阶段定位

五期阶段定位为：**平台化与智能化能力演进阶段**。

在前置阶段中：

- MVP 阶段验证最小交易闭环；
- 一期基础与一期增强阶段建设可靠交易闭环；
- 二期阶段补齐运营履约能力；
- 三期阶段建设会员、积分、客服、BI、风控等平台经营能力；
- 四期阶段补齐测试、CI/CD、监控、日志、链路、压测、安全、备份等生产治理能力。

五期开始，平台已经具备业务完整性、经营能力和生产治理能力。后续重点应从“单一自营 B2C 系统”进一步演进为“具备平台化扩展和智能化运营能力的企业级电商平台”。

五期核心建设方向包括：

- 多端适配；
- App、小程序、H5、PC、Admin 多端统一能力；
- 多渠道流量接入；
- 渠道来源、渠道归因、渠道投放效果；
- 多商户模型预留；
- 店铺模型预留；
- 平台结算预留；
- 多租户与数据隔离预留；
- 智能推荐；
- 智能客服；
- 智能风控；
- 智能选品；
- 自动补货建议；
- 自动运营策略；
- AI 商品文案；
- AI 客服辅助；
- AI 数据分析；
- 智能运营工作台。

五期完成后，平台应从“可生产运行的企业经营平台”升级为“可平台化、多端化、多渠道化、智能化扩展的企业级电商平台”。

## 1.2 阶段设计目标

### 1.2.1 业务目标

五期业务目标包括：

1. 平台可以支持 H5、PC、App、小程序等多端能力预留；
2. 不同终端可以共享商品、用户、订单、支付、营销等核心能力；
3. 平台可以记录渠道来源，并对注册、访问、下单、支付等行为做渠道归因；
4. 平台可以为后续多商户、店铺、平台结算保留清晰模型；
5. 商品、订单、库存、售后、财务等核心数据具备商户/店铺维度扩展预留；
6. 推荐能力从规则推荐升级为基于用户行为、商品特征、标签、转化效果的智能推荐；
7. 客服能力从工单处理升级为 FAQ 自动匹配、客服辅助回复、工单摘要和智能分流；
8. 风控能力从规则事件升级为风险评分、风险画像、风险策略建议和自动拦截预留；
9. 运营人员可以使用 AI 辅助生成商品文案、活动文案、客服话术、运营分析摘要；
10. 平台可以基于销售、库存、浏览、转化、退货等数据给出智能选品和补货建议；
11. 管理者可以通过 AI 数据分析辅助理解经营趋势、异常原因和增长机会；
12. 平台具备向五期后续的多租户、多商户、智能化运营平台演进的基础。

### 1.2.2 技术目标

五期技术目标包括：

1. 在不推翻原有模块化单体架构的前提下，为多端、多渠道、多商户模型预留扩展字段和边界；
2. 建立终端、渠道、商户、店铺等平台化基础模型；
3. 建立统一开放接口层或 API 聚合层，适配不同终端展示差异；
4. 建立用户行为、商品特征、订单转化、推荐效果等智能化数据基础；
5. 建立推荐候选、召回、排序、曝光、点击、转化的基础链路；
6. 建立 AI 能力接入适配层，避免业务模块直接依赖具体大模型或第三方服务；
7. 智能客服、智能风控、AI 文案等能力必须可配置、可审核、可回退；
8. 平台化扩展不得破坏原有自营 B2C 的交易闭环；
9. 智能化能力不得直接替代关键交易、退款、风控、财务决策，必须保留人工确认和审计；
10. 为后续微服务化、多商户化、智能运营平台化演进预留接口与数据边界。

## 1.3 五期建设原则

1. **先预留平台模型，后全面多商户化**：五期重点做商户、店铺、渠道、终端模型预留，不急于完全改造成开放平台。
2. **先辅助智能，后自动决策**：AI 能力先做建议、摘要、辅助，不直接替代人做高风险决策。
3. **先可解释推荐，后复杂算法**：推荐先建立召回、排序、曝光、点击、转化链路，再逐步引入模型优化。
4. **先统一接入，后多端差异化**：优先统一 API、鉴权、响应规范，再做各端体验优化。
5. **平台化不破坏自营主线**：多商户、店铺、渠道预留不能影响当前自营 B2C 的稳定运行。
6. **智能化必须可审计**：AI 生成内容、智能风控建议、智能客服回复都必须记录输入、输出、操作人和使用结果。
7. **数据权限前置**：多渠道、多端、多商户预留必须同步考虑数据隔离、权限边界和审计要求。

# 2 系统架构设计

## 2.1 五期整体架构演进

五期建议在现有架构上增加平台化接入层、智能能力层和平台基础模型层。

```text
多端入口
  -> H5 / PC / App / 小程序 / Admin / OpenAPI
  -> API 聚合层 / BFF 层 / 网关层
  -> 业务模块应用服务
  -> 平台基础模型层
  -> 智能能力适配层
  -> 数据与治理能力
```

五期新增横向能力：

```text
平台化能力：
  -> 终端模型
  -> 渠道模型
  -> 商户模型预留
  -> 店铺模型预留
  -> 平台结算预留
  -> 开放接口预留
  -> 数据隔离预留

智能化能力：
  -> 智能推荐
  -> 智能客服
  -> 智能风控
  -> 智能选品
  -> 自动补货建议
  -> AI 文案
  -> AI 数据分析
```

## 2.2 五期新增与增强领域模块

| 模块 | 模块定位 | 五期职责 |
|---|---|---|
| mall-channel | 渠道模块 | 渠道来源、渠道归因、投放效果、渠道统计 |
| mall-terminal | 多端模块 | 终端类型、版本、能力开关、多端配置 |
| mall-merchant | 商户模型预留 | 商户主体、商户资质、商户状态、商户边界预留 |
| mall-shop | 店铺模型预留 | 店铺信息、店铺商品关系、店铺展示预留 |
| mall-settlement | 平台结算预留 | 商户账单、平台佣金、结算单、结算规则预留 |
| mall-openapi | 开放接口预留 | API 网关、签名鉴权、开放接口、调用日志 |
| mall-ai-gateway | AI 能力网关 | 大模型接入、Prompt 模板、调用审计、降级策略 |
| mall-intelligent-recommend | 智能推荐 | 用户召回、商品召回、排序、推荐效果优化 |
| mall-intelligent-service | 智能客服 | FAQ 匹配、辅助回复、工单摘要、智能分流 |
| mall-intelligent-risk | 智能风控 | 风险评分、风险画像、策略建议、自动拦截预留 |
| mall-intelligent-operation | 智能运营 | AI 文案、活动建议、选品建议、补货建议、数据解读 |
| mall-data-intelligence | 数据智能 | 特征加工、模型结果、智能分析、经营洞察 |

## 2.3 五期模块依赖原则

- 多端模块负责端能力识别，不承载商品、订单、支付主逻辑；
- 渠道模块负责来源归因，不直接修改订单金额；
- 商户和店铺模块五期以模型预留为主，不全面改变自营交易闭环；
- 平台结算模块读取订单、支付、退款、对账数据，不直接执行支付退款；
- AI 网关统一封装大模型、推荐服务、智能客服服务，业务模块不得直接依赖第三方 AI SDK；
- 智能推荐读取用户画像、行为、商品、订单数据，输出推荐结果，不修改商品主数据；
- 智能客服给出辅助建议，客服人员确认后才能对外回复；
- 智能风控给出评分和建议，高风险自动拦截必须有明确规则、开关和审计；
- AI 运营建议只作为辅助，不直接发布商品、活动或财务结论。

# 4 五期核心业务详细设计

## 4.1 多端适配域业务设计

### 4.1.1 核心业务流程

```text
用户从不同终端访问
  -> 网关识别终端类型、版本、设备信息
  -> 加载终端能力配置
  -> API 聚合层适配展示字段
  -> 业务模块返回统一核心数据
  -> 终端按自身交互展示
```

### 4.1.2 业务边界

#### 核心职责

多端模块负责终端识别和终端差异配置。

- 终端类型；
- 终端版本；
- 设备信息；
- 端能力开关；
- 多端首页配置适配；
- 多端接口字段裁剪；
- 多端灰度能力；
- 终端访问统计。

#### 对内边界

- 多端模块只处理端差异，不复制业务规则；
- 不同端共用交易核心逻辑；
- 终端差异应通过配置或 BFF 适配，不应在领域层写大量 if-else；
- 端能力开关需要记录版本、生效范围和操作日志。

#### 对外边界

- 对 C 端：提供不同终端适配能力；
- 对商品、订单、营销模块：只传递终端上下文，不修改业务语义；
- 对渠道模块：提供终端来源数据；
- 对监控模块：输出终端维度访问指标。

#### 禁止范围

- 禁止为每个终端复制一套订单逻辑；
- 禁止终端配置直接修改商品价格；
- 禁止终端差异破坏支付、退款、库存状态机；
- 禁止未授权终端访问后台或开放接口。

## 4.2 多渠道归因域业务设计

### 4.2.1 核心业务流程

```text
用户通过渠道链接访问
  -> 记录渠道参数
  -> 绑定访问会话
  -> 用户注册 / 加购 / 下单 / 支付
  -> 渠道模块记录转化链路
  -> 后台查看渠道效果
```

### 4.2.2 业务边界

#### 核心职责

渠道模块负责流量来源、转化归因和渠道效果分析。

- 渠道定义；
- 渠道链接；
- 渠道参数解析；
- 访问归因；
- 注册归因；
- 下单归因；
- 支付归因；
- 渠道转化漏斗；
- 渠道 ROI 预留；
- 渠道效果报表。

#### 对内边界

- 渠道归因应记录首次来源、最近来源和成交来源；
- 渠道参数应防篡改或签名预留；
- 渠道效果统计允许异步计算；
- 渠道归因不能改变订单真实金额；
- 渠道数据需要与用户隐私保护兼容。

#### 对外边界

- 对用户模块：记录注册来源；
- 对订单模块：记录订单来源渠道；
- 对营销模块：支持渠道专属活动预留；
- 对 BI 模块：输出渠道转化和效果数据；
- 对 Admin：提供渠道配置和报表。

#### 禁止范围

- 禁止渠道模块创建订单；
- 禁止渠道模块直接发券；
- 禁止渠道来源覆盖支付事实；
- 禁止渠道参数未校验直接进入核心业务；
- 禁止用渠道数据绕过用户授权和隐私规则。

## 4.3 商户、店铺与平台结算预留设计

### 4.3.1 核心业务流程

```text
平台创建商户主体
  -> 创建店铺
  -> 关联商品、订单、售后、结算维度预留
  -> 订单产生商户/店铺归属
  -> 支付、退款、对账沉淀结算基础数据
  -> 后续生成商户账单和平台结算单
```

五期不要求全面上线开放平台招商入驻，但应完成模型预留和边界设计。

### 4.3.2 业务边界

#### 核心职责

商户、店铺、结算预留模块负责平台化主体模型建设。

- 商户主体；
- 商户资质；
- 商户状态；
- 店铺信息；
- 店铺状态；
- 商品归属预留；
- 订单归属预留；
- 售后归属预留；
- 平台佣金规则预留；
- 商户账单预留；
- 结算单预留。

#### 对内边界

- 五期默认仍以自营 B2C 为主，多商户为预留能力；
- 商品、订单、售后、财务表可增加商户/店铺字段预留，但不能破坏现有数据；
- 平台结算只做基础数据沉淀和模型设计，不强制做完整资金清分；
- 商户状态变化必须记录操作日志；
- 商户数据隔离需要与权限体系联动预留。

#### 对外边界

- 对商品模块：预留商品所属商户和店铺；
- 对订单模块：预留订单商户和店铺归属；
- 对售后模块：预留售后责任主体；
- 对财务模块：预留商户账单、佣金、结算数据；
- 对权限模块：预留商户后台和数据权限。

#### 禁止范围

- 禁止五期直接推翻自营商品模型；
- 禁止未设计数据隔离就开放商户后台；
- 禁止平台结算模块直接修改支付流水；
- 禁止商户模型影响当前自营订单闭环；
- 禁止无审核启用商户主体。

## 4.4 开放接口与 API 网关预留设计

### 4.4.1 核心业务流程

```text
外部系统申请接入
  -> 平台分配 appKey / secret
  -> 外部系统按签名规则调用 API
  -> 网关校验签名、权限、限流
  -> 路由到内部服务
  -> 记录调用日志
```

### 4.4.2 业务边界

#### 核心职责

开放接口模块负责未来外部合作方、渠道方、商户系统接入预留。

- 应用接入；
- appKey / secret；
- 签名鉴权；
- 接口权限；
- 接口限流；
- 调用日志；
- 回调通知预留；
- API 文档预留；
- 接入方状态管理。

#### 对内边界

- 开放接口必须通过网关，不允许直接访问内部 Controller；
- 开放 API 返回字段应经过脱敏和权限控制；
- 外部调用必须有调用日志和 traceId；
- 接口限流应按应用、接口、IP 维度配置；
- 高风险写接口默认不开放或需要额外审批。

#### 对外边界

- 对渠道方：提供渠道数据、商品展示、订单状态等受控接口；
- 对商户系统：未来预留商品、订单、售后、结算接口；
- 对内部模块：通过 Facade 调用，不暴露数据库结构；
- 对安全模块：提供签名、权限、风险日志。

#### 禁止范围

- 禁止开放接口绕过登录鉴权；
- 禁止外部系统直接访问数据库；
- 禁止开放敏感用户信息；
- 禁止无调用日志的开放接口；
- 禁止无版本管理变更开放 API。

## 4.5 AI 能力网关设计

### 4.5.1 核心业务流程

```text
业务模块发起 AI 请求
  -> AI 网关匹配能力类型和 Prompt 模板
  -> 脱敏输入数据
  -> 调用模型或第三方 AI 服务
  -> 记录输入摘要、输出结果、耗时、成本
  -> 返回建议结果
  -> 人工审核或业务侧使用
```

### 4.5.2 业务边界

#### 核心职责

AI 网关负责统一管理智能化能力接入。

- 模型服务接入；
- Prompt 模板；
- 输入脱敏；
- 输出审核；
- 调用限流；
- 成本统计；
- 调用日志；
- 失败降级；
- 多模型路由预留；
- AI 结果缓存。

#### 对内边界

- AI 输入必须按场景脱敏；
- AI 输出必须区分建议、草稿、可发布内容；
- 高风险场景必须人工确认；
- Prompt 模板变更需要版本记录；
- AI 调用失败不能阻塞核心交易；
- AI 调用成本需要统计和限制。

#### 对外边界

- 对智能客服：提供回复建议和工单摘要；
- 对智能运营：提供文案生成、活动建议、数据解读；
- 对智能风控：提供风险解释和策略建议；
- 对数据智能：提供自然语言分析摘要；
- 对安全合规：提供调用审计和敏感数据保护。

#### 禁止范围

- 禁止业务模块直接调用第三方 AI SDK；
- 禁止向 AI 服务发送明文敏感隐私数据；
- 禁止 AI 结果未经审核直接执行高风险操作；
- 禁止 AI 失败影响下单、支付、退款主链路；
- 禁止无日志、无成本统计调用 AI。

## 4.6 智能推荐域业务设计

### 4.6.1 核心业务流程

```text
采集用户行为和商品特征
  -> 构建用户特征、商品特征、场景特征
  -> 召回候选商品
  -> 排序模型或规则排序
  -> C端展示推荐结果
  -> 记录曝光、点击、加购、支付
  -> 反馈优化推荐策略
```

### 4.6.2 业务边界

#### 核心职责

智能推荐模块负责从规则推荐升级为可持续优化的推荐链路。

- 用户特征；
- 商品特征；
- 召回策略；
- 排序策略；
- 推荐场景；
- 推荐实验预留；
- 推荐曝光；
- 推荐点击；
- 推荐转化；
- 推荐效果评估。

#### 对内边界

- 推荐结果必须经过商品可售校验；
- 推荐策略需要支持版本管理；
- 推荐效果应按场景统计；
- 用户不感兴趣行为应进入反馈；
- 推荐服务异常时应降级为热销、上新或运营配置；
- 推荐不得泄露用户隐私偏好。

#### 对外边界

- 对商品模块：读取商品公开信息和可售状态；
- 对用户画像：读取用户偏好标签；
- 对 BI：输出推荐效果指标；
- 对 C 端：提供首页、详情页、购物车、搜索无结果推荐；
- 对运营后台：提供策略配置和效果分析。

#### 禁止范围

- 禁止推荐模块修改商品价格和库存；
- 禁止推荐结果绕过商品上下架；
- 禁止推荐服务异常阻断商品详情页；
- 禁止不可解释推荐直接用于高风险决策；
- 禁止使用未授权隐私数据做推荐。

## 4.7 智能客服域业务设计

### 4.7.1 核心业务流程

```text
用户发起咨询
  -> 智能客服识别问题意图
  -> 匹配 FAQ、订单、售后、物流上下文
  -> 生成回复建议
  -> 简单问题自动回复或客服确认后回复
  -> 复杂问题转人工工单
  -> 记录服务效果
```

### 4.7.2 业务边界

#### 核心职责

智能客服模块负责客服效率提升。

- 意图识别；
- FAQ 自动匹配；
- 订单问题摘要；
- 售后问题摘要；
- 物流问题摘要；
- 客服辅助回复；
- 工单自动摘要；
- 工单智能分流；
- 相似问题推荐；
- 服务质量分析。

#### 对内边界

- 自动回复范围应可配置；
- 涉及退款、赔付、投诉、高风险问题必须转人工；
- AI 回复需要保留原始问题、上下文摘要和回复内容；
- 客服采纳、修改、拒绝 AI 建议需要记录；
- 智能客服不可直接关闭工单。

#### 对外边界

- 对客服工单模块：提供摘要、分流、回复建议；
- 对订单模块：只读取订单摘要，不修改订单；
- 对售后模块：只读取售后摘要，不审批售后；
- 对消息模块：发送用户回复通知；
- 对 BI：输出智能客服解决率、转人工率、满意度。

#### 禁止范围

- 禁止 AI 自动执行退款；
- 禁止 AI 自动承诺赔付；
- 禁止 AI 修改订单、售后、支付状态；
- 禁止 AI 对高风险投诉直接关闭；
- 禁止未记录上下文的自动回复。

## 4.8 智能风控域业务设计

### 4.8.1 核心业务流程

```text
用户登录 / 下单 / 领券 / 支付 / 退款 / 售后
  -> 风控采集行为和交易特征
  -> 计算风险评分
  -> 生成风险解释和处置建议
  -> 低风险放行
  -> 中风险人工审核
  -> 高风险按策略限制或拦截
  -> 记录处置结果反馈模型
```

### 4.8.2 业务边界

#### 核心职责

智能风控模块负责从规则风控升级为评分和策略辅助。

- 风险特征；
- 风险评分；
- 风险画像；
- 风险解释；
- 策略建议；
- 自动拦截预留；
- 人工审核；
- 处置反馈；
- 模型效果评估；
- 风险看板。

#### 对内边界

- 风险评分必须可追溯到特征和规则/模型版本；
- 自动拦截必须有明确开关和适用范围；
- 人工审核结果应反馈给风控系统；
- 风控模型或规则变更需要灰度和回滚；
- 风控误杀需要申诉和恢复机制预留。

#### 对外边界

- 对订单模块：提供下单风险评分；
- 对营销模块：提供领券、活动参与风险；
- 对售后模块：提供退款、退货风险；
- 对用户模块：提供风险标签；
- 对客服模块：风险争议可转人工工单；
- 对审计模块：提供风险处置留痕。

#### 禁止范围

- 禁止无解释自动拒绝高价值交易；
- 禁止风控直接删除订单；
- 禁止风控直接扣减用户资产；
- 禁止模型结果无版本记录；
- 禁止无人工复核通道处理争议风险。

## 4.9 智能运营与数据智能域业务设计

### 4.9.1 核心业务流程

```text
平台沉淀经营数据
  -> 智能运营模块分析商品、用户、库存、活动、转化
  -> 生成选品建议、补货建议、活动建议、文案草稿
  -> 运营人员审核和调整
  -> 发布内容或活动
  -> 追踪效果并反馈优化
```

### 4.9.2 业务边界

#### 核心职责

智能运营模块负责提升运营决策效率。

- 智能选品；
- 自动补货建议；
- 活动策略建议；
- AI 商品标题；
- AI 商品卖点；
- AI 活动文案；
- AI 客服话术；
- AI 数据分析摘要；
- 异常指标解释；
- 增长机会提示。

#### 对内边界

- AI 生成内容默认是草稿；
- 运营发布前必须人工确认；
- 选品和补货建议必须展示依据；
- 数据分析结论需标明统计口径和时间范围；
- AI 文案生成需要保存版本和采纳记录；
- 建议效果需要后续追踪。

#### 对外边界

- 对商品模块：提供标题、卖点、选品建议，不直接修改商品；
- 对库存模块：提供补货建议，不直接创建采购或入库；
- 对营销模块：提供活动建议，不直接发布活动；
- 对 BI 模块：读取经营指标并生成分析摘要；
- 对内容模块：生成专题、活动、商品内容草稿。

#### 禁止范围

- 禁止 AI 自动上架商品；
- 禁止 AI 自动修改价格；
- 禁止 AI 自动创建营销活动并生效；
- 禁止 AI 自动补货入库；
- 禁止 AI 数据分析替代财务正式报表；
- 禁止无审核发布 AI 文案。

# 11 验收标准

五期完成后，应满足以下验收标准：

1. 系统具备终端类型和终端能力配置；
2. H5、PC、App、小程序等终端能力具备接口或模型预留；
3. 核心业务接口可以接收并记录终端上下文；
4. 平台可以创建渠道并生成渠道链接；
5. 用户访问、注册、下单、支付可以记录渠道归因；
6. 后台可以查看渠道转化漏斗和效果统计；
7. 系统具备商户、店铺、平台结算基础模型预留；
8. 商品、订单、售后、财务等关键模型具备商户/店铺维度扩展预留；
9. 开放接口具备 appKey、签名、权限、限流、调用日志等基础能力；
10. AI 能力统一通过 AI 网关调用；
11. AI 调用具备脱敏、日志、成本统计和失败降级；
12. 推荐能力具备用户特征、商品特征、召回、排序、曝光、点击、转化链路；
13. 推荐服务异常时可以降级为默认推荐；
14. 智能客服可以识别常见问题并给出回复建议；
15. 高风险客服场景可以转人工；
16. 工单摘要和客服辅助回复具备采纳记录；
17. 智能风控可以生成风险评分和处置建议；
18. 风控自动拦截具备开关、规则、日志和人工复核预留；
19. AI 商品文案、活动文案、客服话术可以生成草稿；
20. AI 生成内容发布前需要人工确认；
21. 系统可以生成选品建议和补货建议；
22. AI 数据分析可以基于 BI 指标生成经营摘要；
23. 五期新增能力不破坏原有自营 B2C 交易闭环；
24. 平台化和智能化关键操作具备审计日志；
25. 敏感数据在 AI 调用、开放接口、多渠道分析中得到脱敏和权限控制。

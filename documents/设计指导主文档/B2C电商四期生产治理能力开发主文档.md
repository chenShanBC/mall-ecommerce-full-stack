# B2C电商四期生产治理能力开发主文档

> 本文档用于指导 `mallFei` 在完成 MVP、一阶段交易闭环、一阶段增强可靠性、二期运营履约能力、三期平台经营能力之后，进入四期生产治理能力建设阶段。  
> 四期的核心目标不是继续堆业务功能，而是让系统具备真实生产环境所需的稳定性、可观测性、可恢复性、安全性、可测试性、可发布性和容量治理能力。

---

# 1 项目概述

## 1.1 阶段定位

四期阶段定位为：**生产治理能力完善阶段**。

在前置阶段中：

- MVP 阶段验证最小交易链路；
- 一期基础阶段补齐订单、支付、库存、售后、退款、对账、权限审计等企业级交易闭环；
- 一期增强阶段强化库存防超卖、支付异常、线上人工对账、Dashboard、认证风控等交易可靠性；
- 二期阶段补齐物流履约、营销优惠、搜索、消息、评价、首页装修等运营履约体验；
- 三期阶段建设会员、积分、用户画像、推荐、客服工单、BI、财务经营、风控等经营能力。

四期开始，平台已经具备较完整的业务能力，但如果要进入真实企业级生产运行，还必须补齐工程治理与生产治理能力。

四期核心建设方向包括：

- 自动化测试体系；
- 单元测试、集成测试、接口测试、E2E 测试；
- CI/CD 自动化流水线；
- 自动构建、自动部署、灰度发布、回滚机制；
- 运行时监控指标；
- 日志聚合；
- 链路追踪；
- 告警中心；
- 错误追踪；
- 性能压测；
- 容量评估；
- 限流、熔断、降级；
- 缓存治理；
- 数据库治理；
- 数据备份与恢复；
- 安全扫描；
- 权限巡检；
- 数据脱敏；
- 审计合规；
- 搜索索引监控；
- 消息可靠投递；
- 文件安全、CDN、防盗链、分片上传；
- 财务审计、账务合规留档。

四期完成后，平台应从“功能完整的企业经营平台”升级为“可稳定、安全、可观测、可恢复地生产运行的平台”。

## 1.2 阶段设计目标

### 1.2.1 业务目标

四期业务目标包括：

1. 系统核心交易链路具备自动化测试保护；
2. 后端、前端、数据库变更可以通过标准流水线构建、测试、部署；
3. 发布失败时可以快速回滚；
4. 核心业务指标、系统指标、接口指标可以被监控；
5. 异常、慢接口、错误日志可以被快速定位；
6. 用户请求可以通过链路追踪定位到具体模块、接口和数据库操作；
7. 系统异常可以触发告警并通知技术团队；
8. 下单、支付、退款、库存、营销、搜索等高风险链路经过性能压测；
9. 系统具备基础容量评估能力；
10. 高流量场景可以通过限流、熔断、降级保护核心链路；
11. 数据库慢 SQL、索引、表容量、归档策略得到治理；
12. 数据库和关键文件具备备份恢复方案；
13. 后台权限、数据权限、高危操作可以定期巡检；
14. 用户隐私数据、手机号、地址、身份证号、支付信息等敏感数据具备脱敏能力；
15. 财务、对账、发票、退款等关键数据具备审计留档；
16. 系统具备安全扫描、依赖漏洞检查和基础合规治理能力。

### 1.2.2 技术目标

四期技术目标包括：

1. 建立覆盖核心领域的测试分层体系；
2. 建立后端、前端、数据库脚本、配置文件的 CI/CD 流程；
3. 建立统一日志规范，包括 traceId、userId、orderNo、bizType、errorCode 等关键字段；
4. 建立统一监控指标体系，包括系统指标、JVM 指标、数据库指标、业务指标；
5. 建立链路追踪能力，支持跨模块请求定位；
6. 建立告警规则、告警等级、告警通知和告警处理闭环；
7. 建立压测脚本、压测报告、容量评估和瓶颈分析机制；
8. 建立限流、熔断、降级、超时、重试、幂等治理标准；
9. 建立数据库慢 SQL、索引、分表预留、归档、备份恢复治理规范；
10. 建立安全扫描、权限巡检、数据脱敏、审计合规治理机制；
11. 保证四期治理能力不破坏已有业务功能；
12. 为五期平台化、智能化、微服务化演进预留工程基础。

## 1.3 四期建设原则

1. **先核心链路，后全量覆盖**：优先治理注册登录、商品浏览、购物车、下单、支付、库存、退款、对账、发货、营销等核心链路。
2. **先可观测，后自动化治理**：先让问题看得见、定位得到，再逐步自动化扩缩容、自愈、智能告警。
3. **先标准化，后平台化**：先建立日志、指标、告警、测试、发布、安全规范，再建设内部平台。
4. **生产治理不侵入业务语义**：限流、熔断、监控、日志、测试、发布不能改变业务状态机和业务事实。
5. **所有异常可定位**：每个核心请求必须能通过 traceId、日志、指标和错误码定位问题。
6. **所有发布可回滚**：发布方案必须考虑失败回滚、配置回退、数据库脚本回退或补偿。
7. **所有关键数据可恢复**：数据库、文件、对账、发票、财务、订单等关键数据必须具备备份和恢复策略。
8. **安全合规前置**：敏感数据脱敏、权限巡检、高危操作审计、安全扫描应进入常态化。

# 2 系统架构设计

## 2.1 四期治理架构总览

四期不改变前几期形成的业务模块结构，而是在现有模块化单体 + DDD 四层架构之上增加生产治理层。

```text
C端 / Admin端
  -> 网关与安全治理
  -> 业务模块 interfaces
  -> 业务模块 application
  -> 业务模块 domain
  -> 业务模块 infrastructure
  -> 数据库 / 缓存 / 文件 / 搜索 / 消息

横向治理能力：
  -> 测试体系
  -> CI/CD
  -> 日志聚合
  -> 指标监控
  -> 链路追踪
  -> 告警中心
  -> 压测与容量评估
  -> 限流熔断降级
  -> 数据库治理
  -> 备份恢复
  -> 安全合规
```

四期治理能力是横向能力，不应反向破坏业务模块边界。

## 2.2 四期新增与增强治理模块

| 模块 | 模块定位 | 四期职责 |
|---|---|---|
| mall-test | 测试体系 | 单元测试、集成测试、接口测试、E2E、测试数据管理 |
| mall-devops | DevOps 治理 | CI/CD、构建、部署、灰度、回滚、环境管理 |
| mall-observability | 可观测性 | 指标监控、日志聚合、链路追踪、错误追踪 |
| mall-alert | 告警中心 | 告警规则、告警通知、告警处理、告警升级 |
| mall-performance | 性能治理 | 压测脚本、压测报告、容量评估、性能基线 |
| mall-resilience | 稳定性治理 | 限流、熔断、降级、超时、重试、幂等治理 |
| mall-db-governance | 数据库治理 | 慢 SQL、索引、归档、备份恢复、容量监控 |
| mall-security | 安全合规 | 安全扫描、敏感数据脱敏、权限巡检、安全审计 |
| mall-audit | 审计合规 | 操作审计、财务审计、合规留档、审计报表 |
| mall-file-governance | 文件治理 | CDN、防盗链、私有访问、分片上传、文件审计 |
| mall-message-governance | 消息治理 | 可靠投递、失败补偿、死信、消息监控 |
| mall-search-governance | 搜索治理 | 索引监控、索引重建、搜索性能、同步失败处理 |

以上模块可以是独立工程模块，也可以是现有模块中的治理 package，但逻辑边界必须清晰。

## 2.3 四期治理能力依赖关系

四期治理模块依赖原则：

- 测试体系依赖业务模块接口和领域服务，但不改变业务代码语义；
- CI/CD 依赖代码仓库、构建脚本、部署环境、测试结果；
- 日志、指标、链路追踪通过统一拦截器、AOP、Filter、SDK 或中间件采集；
- 告警中心消费监控指标、日志错误、链路异常和业务异常事件；
- 性能治理依赖压测环境、压测数据、监控指标和压测报告；
- 稳定性治理作用于接口、外部依赖和高风险链路；
- 数据库治理依赖 SQL 日志、慢查询、表容量、索引统计和备份任务；
- 安全合规依赖权限、日志、敏感字段、依赖漏洞扫描结果；
- 审计合规读取操作日志、财务记录、对账记录和高危操作记录。

# 3 四期核心治理对象与状态定义

## 3.1 流水线状态定义

| 状态 | 含义 |
|---|---|
| CREATED | 已创建 |
| RUNNING | 运行中 |
| SUCCESS | 成功 |
| FAILED | 失败 |
| CANCELED | 已取消 |
| ROLLBACKING | 回滚中 |
| ROLLBACK_SUCCESS | 回滚成功 |
| ROLLBACK_FAILED | 回滚失败 |

## 3.2 告警等级定义

| 等级 | 含义 | 示例 |
|---|---|---|
| P0 | 严重故障 | 下单、支付、登录大面积不可用 |

| P2 | 中优先级 | 慢接口升高、局部功能异常 |
| P3 | 低优先级 | 非核心任务失败、统计延迟 |

| CLOSED | 已关闭 |

## 3.4 压测任务状态定义

| 状态 | 含义 |

## 3.6 安全扫描风险等级

| 等级 | 含义 |

# 4 四期核心治理能力详细设计

## 4.1 自动化测试体系设计

### 4.1.1 核心流程

```text
开发提交代码
  -> 触发单元测试
  -> 触发集成测试
  -> 触发接口测试
  -> 关键分支触发 E2E 测试
  -> 生成测试报告
  -> 测试失败阻断合并或发布
```

四期测试范围应优先覆盖：

- 用户注册登录；
- 商品查询；
- 购物车；
- 订单创建；
- 库存预占、扣减、释放；
- 支付回调；
- 退款；
- 对账；
- 发货；
- 优惠券锁定、核销、释放；
- 积分冻结、消费、解冻；
- 售后退款；
- 权限校验。

### 4.1.2 业务边界

#### 核心职责

测试体系负责保障核心业务在持续迭代中的正确性。

- 单元测试；
- 集成测试；
- 接口测试；
- E2E 测试；
- Mock 外部依赖；
- 测试数据准备；
- 测试报告；
- 覆盖率统计；
- 回归测试集。

#### 对内边界

- 单元测试重点覆盖领域服务和核心计算规则；
- 集成测试重点覆盖数据库、事务、跨模块应用服务；
- 接口测试重点覆盖 Controller 入参、鉴权、响应结构；
- E2E 测试重点覆盖 C 端和 Admin 端核心流程；
- 测试数据应与生产数据隔离；
- 测试失败必须可定位到用例、接口、模块和错误原因。

#### 对外边界

- 对 CI/CD：提供测试执行结果，失败时阻断流水线；
- 对业务模块：验证业务行为，不改变业务实现；
- 对质量报告：输出覆盖率、通过率、失败原因；
- 对开发流程：提供回归测试保护。

#### 禁止范围

- 禁止测试用例依赖生产数据；
- 禁止测试代码修改生产逻辑；
- 禁止为了通过测试降低业务校验；
- 禁止把偶发失败用例长期忽略；
- 禁止核心链路无测试直接发布。

## 4.2 CI/CD 与发布治理设计

### 4.2.1 核心流程

```text
代码提交
  -> 静态检查
  -> 单元测试
  -> 构建后端与前端产物
  -> 集成测试 / 接口测试
  -> 构建镜像或部署包
  -> 部署测试环境
  -> 人工确认或自动部署预发
  -> 灰度发布生产
  -> 监控发布指标
  -> 全量发布或回滚
```

### 4.2.2 业务边界

#### 核心职责

CI/CD 模块负责让构建、测试、部署、回滚标准化。

- 代码拉取；
- 静态检查；
- 自动测试；
- 前端构建；
- 后端构建；
- 镜像或部署包生成；
- 环境变量管理；
- 数据库脚本执行控制；
- 自动部署；
- 灰度发布；
- 发布审批；
- 回滚机制；
- 发布记录。

#### 对内边界

- 每次发布必须关联代码版本、构建产物和发布人；
- 配置变更必须可追溯；
- 数据库脚本应具备执行顺序、幂等性和回滚预案；
- 灰度发布需要定义流量比例、目标用户或目标实例；
- 回滚应优先回滚应用版本，数据库变更需使用补偿脚本谨慎处理；
- 发布过程中必须监控错误率、响应时间、核心业务成功率。

#### 对外边界

- 对测试体系：依赖测试结果决定是否继续发布；
- 对监控告警：发布期间重点观察指标；
- 对配置中心或环境配置：读取部署配置；
- 对开发团队：提供发布记录、失败日志和回滚入口。

#### 禁止范围

- 禁止跳过测试直接发布核心链路；
- 禁止无版本记录发布；
- 禁止生产环境手工随意覆盖文件；
- 禁止无回滚方案进行高风险发布；
- 禁止数据库破坏性脚本无备份执行；
- 禁止将密钥、密码写入代码仓库。

## 4.3 日志聚合与链路追踪设计

### 4.3.1 核心流程

```text
用户请求进入系统
  -> 生成 traceId
  -> 经过网关 / Controller / Application / Repository
  -> 日志记录 traceId、userId、bizNo、耗时、状态
  -> 链路追踪采集调用关系
  -> 日志平台按 traceId 查询完整上下文
```

### 4.3.2 业务边界

#### 核心职责

日志与链路模块负责问题定位。

- 统一 traceId；
- 请求日志；
- 错误日志；
- 慢接口日志；
- SQL 日志摘要；
- 业务关键日志；
- 链路调用关系；
- 异常堆栈聚合；
- 日志检索；
- 日志留存策略。

#### 对内边界

- 核心链路日志必须包含 traceId；
- 订单、支付、退款、对账、发货等业务日志必须包含业务单号；
- 错误日志必须包含错误码和异常摘要；
- 日志中敏感数据必须脱敏；
- 日志级别应规范使用 DEBUG、INFO、WARN、ERROR；
- 链路追踪不应明显拖慢核心接口。

#### 对外边界

- 对告警中心：输出错误率、异常数量、慢接口等信号；
- 对客服和运维：支持按订单号、用户 ID、traceId 定位问题；
- 对安全合规：提供审计所需日志留存；
- 对开发团队：提供异常分析依据。

#### 禁止范围

- 禁止日志记录明文密码、token、身份证、完整手机号、完整地址等敏感信息；
- 禁止用日志替代业务数据；
- 禁止核心异常只打印字符串不打印上下文；
- 禁止吞异常不记录；
- 禁止链路追踪修改业务结果。

## 4.4 指标监控与告警中心设计

### 4.4.1 核心流程

```text
系统采集指标
  -> 指标平台聚合展示
  -> 告警规则匹配阈值
  -> 触发告警
  -> 通知责任人
  -> 责任人确认并处理
  -> 故障恢复后关闭告警
```

### 4.4.2 业务边界

#### 核心职责

监控告警模块负责及时发现异常。

- 系统指标监控；
- JVM 指标监控；
- 数据库指标监控；
- Redis 指标监控；
- 接口 QPS、RT、错误率；
- 业务成功率；
- 支付回调成功率；
- 库存扣减失败率；
- 订单创建失败率；
- 告警规则；
- 告警通知；
- 告警确认；
- 告警升级；
- 告警记录。

#### 对内边界

- P0/P1 告警必须有明确负责人；
- 告警规则必须有阈值、持续时间、恢复条件；
- 告警应避免频繁抖动和重复轰炸；
- 业务指标和系统指标应分层展示；
- 告警处理过程应记录确认人、处理人、处理结果。

#### 对外边界

- 对日志链路：告警后可跳转日志和 traceId；
- 对 CI/CD：发布期间观察告警作为是否回滚依据；
- 对消息模块：可通过站内信、邮件、企业微信等渠道通知；
- 对管理者：输出系统健康度和故障统计。

#### 禁止范围

- 禁止只有监控没有告警；
- 禁止核心链路无业务成功率监控；
- 禁止告警无人负责；
- 禁止告警触发后无处理记录；
- 禁止用单一 CPU 指标判断业务是否健康。

## 4.5 性能压测与容量评估设计

### 4.5.1 核心流程

```text
选择压测场景
  -> 准备压测数据
  -> 编写压测脚本
  -> 执行阶梯压测 / 稳定性压测
  -> 观察 QPS、RT、错误率、CPU、内存、DB、Redis
  -> 分析瓶颈
  -> 输出容量评估和优化建议
```

重点压测场景：

- 商品列表；
- 商品详情；
- 购物车列表；
- 订单创建；
- 库存扣减；
- 支付回调；
- 优惠券领取；
- 秒杀基础活动；
- 搜索接口；
- 首页配置查询；
- 推荐接口。

### 4.5.2 业务边界

#### 核心职责

性能治理模块负责识别系统容量和性能瓶颈。

- 压测场景管理；
- 压测脚本；
- 压测数据；
- 压测环境；
- 压测执行；
- 压测报告；
- 性能基线；
- 容量评估；
- 瓶颈分析；
- 优化建议。

#### 对内边界

- 压测环境应与生产隔离；
- 压测数据不得污染生产数据；
- 压测报告应包含 QPS、TP90、TP95、TP99、错误率、资源使用；
- 容量评估应明确单机能力、集群能力、瓶颈点和扩容建议；
- 性能优化后应重新压测验证。

#### 对外边界

- 对监控系统：依赖监控观察资源瓶颈；
- 对数据库治理：输出慢 SQL、索引问题；
- 对稳定性治理：提供限流阈值、降级策略参考；
- 对业务团队：说明当前系统可承载业务规模。

#### 禁止范围

- 禁止直接压测生产环境核心写接口；
- 禁止无监控执行压测；
- 禁止只看平均响应时间；
- 禁止压测后不输出报告；
- 禁止压测数据影响真实用户。

## 4.6 限流熔断降级与稳定性治理设计

### 4.6.1 核心流程

```text
请求进入系统
  -> 判断接口限流规则
  -> 调用外部依赖时设置超时和重试
  -> 依赖异常达到阈值触发熔断
  -> 非核心能力降级
  -> 核心链路保留最小可用能力
  -> 记录限流、熔断、降级日志和指标
```

### 4.6.2 业务边界

#### 核心职责

稳定性治理模块负责保护系统在异常流量和外部依赖故障下仍可维持核心能力。

- 接口限流；
- 用户维度限流；
- IP 维度限流；
- 活动维度限流；
- 外部依赖超时；
- 重试策略；
- 熔断策略；
- 降级开关；
- 幂等治理；
- 防重复提交；
- 失败补偿任务。

#### 对内边界

- 核心写接口必须具备幂等能力；
- 重试必须避免重复扣库存、重复支付、重复退款；
- 限流返回应给出明确错误码；
- 降级能力只能关闭非核心体验，不得破坏交易事实；
- 熔断恢复需要半开探测或人工确认；
- 稳定性策略变更需要记录操作日志。

#### 对外边界

- 对订单模块：保护下单接口、防重复提交；
- 对支付模块：保护回调幂等和支付查询重试；
- 对营销模块：保护领券、秒杀、活动接口；
- 对搜索推荐：异常时可降级为热销或默认推荐；
- 对消息模块：异常时可异步补偿。

#### 禁止范围

- 禁止无幂等重试核心写操作；
- 禁止降级后产生错误订单金额；
- 禁止熔断支付回调核心处理；
- 禁止限流策略写死不可配置；
- 禁止把降级逻辑混入领域核心规则；
- 禁止失败补偿无日志记录。

## 4.7 数据库治理与备份恢复设计

### 4.7.1 核心流程

```text
采集慢 SQL 和表容量
  -> 分析索引和查询计划
  -> 优化 SQL / 索引 / 分页策略
  -> 制定归档策略
  -> 定期备份数据库
  -> 定期执行恢复演练
  -> 输出数据库健康报告
```

### 4.7.2 业务边界

#### 核心职责

数据库治理模块负责数据库性能、容量和安全恢复。

- 慢 SQL 监控；
- 索引治理；
- 大表容量监控；
- 数据归档；
- 数据清理；
- 数据库备份；
- 恢复演练；
- 数据库变更审核；
- 连接池监控；
- 事务耗时监控。

#### 对内边界

- 核心表变更必须评估影响；
- 大字段、无索引模糊查询、深分页需要治理；
- 历史日志、消息、行为数据应制定归档策略；
- 备份必须定期验证可恢复；
- 数据清理必须区分业务事实数据和可归档日志数据；
- 财务、订单、支付、退款、对账数据不得随意物理删除。

#### 对外边界

- 对业务模块：提供数据库性能优化建议；
- 对 CI/CD：数据库脚本纳入发布治理；
- 对安全合规：提供备份、留存、脱敏策略；
- 对 BI：提供历史数据归档和分析数据同步建议。

#### 禁止范围

- 禁止无备份执行高风险 DDL；
- 禁止随意删除交易、支付、退款、对账、发票事实数据；
- 禁止生产大表无条件全表扫描；
- 禁止业务代码依赖不可控慢 SQL；
- 禁止备份只创建不验证恢复；
- 禁止数据库账号权限过大。

## 4.8 安全合规与权限巡检设计

### 4.8.1 核心流程

```text
定期安全扫描
  -> 检查依赖漏洞、接口风险、敏感信息泄露
  -> 执行权限巡检
  -> 检查高危操作日志
  -> 检查敏感数据脱敏
  -> 输出安全问题清单
  -> 整改并复查
```

### 4.8.2 业务边界

#### 核心职责

安全合规模块负责基础安全治理和合规留痕。

- 依赖漏洞扫描；
- 代码安全扫描；
- 接口鉴权检查；
- 越权风险检查；
- 敏感数据识别；
- 数据脱敏；
- 权限巡检；
- 高危操作审计；
- 登录安全审计；
- 管理员账号安全；
- 安全问题整改跟踪。

#### 对内边界

- 敏感字段需要定义清单；
- 日志、报表、导出文件必须脱敏；
- 高危操作需要二次确认或审批预留；
- 管理员权限应遵循最小权限原则；
- 离职、禁用、长期未登录账号需要巡检；
- 安全问题应有等级、负责人、整改期限。

#### 对外边界

- 对权限模块：读取角色、菜单、按钮、数据权限；
- 对日志模块：读取登录日志、操作日志、高危操作日志；
- 对文件模块：检查私有文件访问和防盗链；
- 对 CI/CD：安全扫描结果可作为发布门禁；
- 对审计模块：输出合规审计材料。

#### 禁止范围

- 禁止接口无鉴权暴露敏感数据；
- 禁止日志明文输出敏感信息；
- 禁止管理员共用账号；
- 禁止超级管理员长期用于日常操作；
- 禁止导出文件无权限校验；
- 禁止安全问题无跟踪关闭。

## 4.9 审计合规与财务留档设计

### 4.9.1 核心流程

```text
关键业务操作发生
  -> 写入操作日志或审计日志
  -> 财务、对账、发票、退款记录归档
  -> 生成审计报表
  -> 支持按时间、操作人、业务单号追溯
```

### 4.9.2 业务边界

#### 核心职责

审计合规模块负责关键行为和关键业务数据可追责。

- 操作审计；
- 登录审计；
- 高危操作审计；
- 权限变更审计；
- 订单关键操作审计；
- 退款操作审计；
- 对账差异处理审计；
- 发票操作审计；
- 财务报表留档；
- 审计报表。

#### 对内边界

- 审计日志应包含操作人、操作时间、操作对象、操作前后摘要、IP、traceId；
- 审计日志不应允许普通后台用户修改；
- 财务、对账、退款、发票留档应保留业务单号关联；
- 审计报表应支持导出，但导出需要权限控制和脱敏；
- 审计数据应有留存周期。

#### 对外边界

- 对权限模块：审计权限变更和高危授权；
- 对订单模块：审计订单关闭、备注、异常处理；
- 对支付退款：审计退款、异常修复；
- 对对账模块：审计差异处理、挂账、归档；
- 对财务模块：审计发票、日报、月结报表。

#### 禁止范围

- 禁止关键后台操作无审计；
- 禁止普通业务人员修改审计日志；
- 禁止审计日志记录敏感明文；
- 禁止审计报表无权限导出；
- 禁止财务留档被业务统计覆盖。

## 4.10 文件、消息、搜索专项治理设计

### 4.10.1 文件治理

核心职责：

- 私有文件访问；
- 文件访问权限；
- 分片上传；
- 文件审计；

- 禁止索引重建影响线上查询；
- 禁止搜索异常阻断下单核心链路。

# 5 四期标准治理闭环流程

## 5.1 代码提交到生产发布闭环

```text
开发提交代码
  -> 静态检查
  -> 自动化测试
  -> 构建产物
  -> 部署测试环境
  -> 接口回归
  -> 预发验证
  -> 灰度发布
  -> 监控指标观察
  -> 全量发布或回滚
  -> 发布记录归档
```

## 5.2 线上故障发现与处理闭环

```text
指标异常 / 日志错误 / 用户反馈
  -> 告警触发
  -> 责任人确认
  -> 通过 traceId 和日志定位
  -> 判断是否回滚、限流、降级或修复
  -> 故障恢复
  -> 告警关闭
  -> 复盘与改进
```

## 5.3 性能压测与容量规划闭环

```text
选择核心接口
  -> 准备压测数据
  -> 执行压测
  -> 采集系统和业务指标
  -> 分析瓶颈
  -> 优化代码、SQL、缓存、配置
  -> 复测验证
  -> 输出容量规划
```

## 5.4 数据备份恢复闭环

```text
定义备份策略
  -> 定时备份数据库和文件
  -> 校验备份完整性
  -> 定期恢复演练
  -> 记录恢复耗时和结果
  -> 优化备份恢复方案
```

## 5.5 安全巡检整改闭环

```text
定期安全扫描
  -> 发现漏洞、越权、敏感信息风险
  -> 生成问题清单
  -> 分配责任人
  -> 修复问题
  -> 复查确认
  -> 安全报告归档
```

# 11 验收标准

四期完成后，应满足以下验收标准：

1. 核心领域服务具备单元测试；
2. 核心交易链路具备接口测试或集成测试；
3. C 端核心购物流程具备 E2E 测试；
4. 代码提交后可以自动执行构建和测试；
5. 测试失败可以阻断核心分支合并或发布；
6. 后端和前端可以通过标准流水线部署；
7. 发布记录可以追溯代码版本、构建产物、发布人和发布时间；
8. 发布失败时具备回滚方案；
9. 系统日志包含 traceId，核心业务日志包含业务单号；
10. 可以通过 traceId 查询一次请求的完整链路；
11. 核心接口具备 QPS、RT、错误率监控；
12. 下单、支付、退款、库存、营销等核心业务成功率可监控；
13. P0/P1 告警可以通知责任人并记录处理过程；
14. 慢接口和异常日志可以被检索和定位；
15. 下单、支付回调、优惠券领取、搜索等接口完成压测并输出报告；
16. 系统具备基础容量评估报告；
17. 核心写接口具备幂等保护；
18. 高频接口具备限流配置；
19. 外部依赖调用具备超时、重试或降级策略；
20. 慢 SQL 可以被发现和治理；
21. 核心表具备备份策略；
22. 备份数据经过恢复演练验证；
23. 敏感字段具备脱敏规则；
24. 后台权限具备定期巡检能力；
25. 高危操作具备审计日志；
26. 安全扫描可以发现依赖漏洞或基础风险；
27. 文件访问具备基础安全控制；
28. 消息失败具备重试或补偿记录；
29. 搜索索引同步异常可监控；
30. 四期治理能力不破坏已有业务闭环。

# 13 四期完成后的平台能力变化

四期完成后，`mallFei` 应具备以下能力：

- 从“功能完整”升级为“可生产运行”；
- 从“问题靠人工排查”升级为“日志、链路、指标可定位”；
- 从“手工发布”升级为“流水线发布、灰度、回滚”；
- 从“无容量认知”升级为“压测报告和容量评估”；
- 从“异常后被动处理”升级为“监控告警主动发现”；
- 从“数据风险不可控”升级为“备份恢复、脱敏、审计可追溯”；
- 从“业务系统”升级为“具备工程治理能力的企业级平台”。

四期是平台进入真实生产环境前最关键的治理阶段，也是五期平台化、多端化、智能化、微服务化、自动化运营演进的基础。

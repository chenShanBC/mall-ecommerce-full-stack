# B2C电商二期运营履约能力指导主文档

> 本文档用于指导 `mallFei` 在完成 MVP、一期基础交易闭环、一期增强交易可靠性之后，进入二期运营履约能力建设阶段。  
> 二期不再只关注“订单能不能正确交易”，而是重点补齐真实电商平台中用户可感知、运营可配置、履约可追踪、售后可联动的运营履约能力。

---

# 1 项目概述

## 1.1 阶段定位

二期阶段定位为：**运营履约能力补齐阶段**。

在 MVP 阶段，系统完成了用户、商品、购物车、订单、库存、支付的最小交易闭环；在一期基础与一期增强阶段，系统进一步补齐了订单超时关闭、库存预占与释放、支付状态同步、退款、售后、基础对账、线上人工对账、权限审计、Dashboard、支付异常核验等企业级交易可靠性能力。

二期的核心目标是在已有交易闭环之上，补齐电商平台正式运营所必需的能力，包括：

- 订单发货与物流履约；
- C 端物流轨迹查询；
- 售后退货物流联动；
- 优惠券与基础营销活动；
- 下单优惠计算与金额分摊；
- 商品搜索、筛选、排序；
- 首页装修、推荐位、运营位配置；
- 商品评价与评价聚合；
- 站内消息、订单消息、售后消息；
- 购物车失效商品、价格变动、优惠提示；
- 后台运营配置、查询、统计与审计。

二期完成后，平台应从“交易系统”升级为“具备基础运营与履约体验的电商系统”。

## 1.2 阶段设计目标

### 1.2.1 业务目标

二期阶段的业务目标包括：

1. 用户支付成功后，后台运营人员可以对订单进行发货；
2. 用户可以在订单详情中查看物流单号、物流公司、物流轨迹与签收状态；
3. 订单履约状态能够与订单主状态协同，不破坏一期订单、支付、库存、售后状态机；
4. 售后退货退款场景中，用户可以填写退货物流，后台可以确认退货签收；
5. 平台可以配置优惠券、满减等基础营销能力，并在购物车、结算、下单时正确计算优惠；
6. 用户可以通过关键词、分类、价格区间、销量、上架时间等条件搜索商品；
7. 运营人员可以配置首页 Banner、频道、推荐商品、活动入口等首页装修内容；
8. 用户在订单完成后可以评价商品，商品详情页展示评价摘要与评价列表；
9. 系统能够向用户发送订单、支付、发货、签收、售后等站内消息；
10. 后台能够对物流、营销、搜索、评价、消息等二期能力进行管理、查询、统计和审计。

### 1.2.2 技术目标

二期阶段的技术目标包括：

1. 在现有 DDD 模块化单体基础上继续保持领域边界清晰；
2. 新增物流、营销、搜索、消息、评价、内容运营等模块时，不直接侵入订单、支付、库存、售后核心领域；
3. 所有跨模块协作优先通过应用服务、领域事件或明确的防腐接口完成；
4. 优惠计算必须形成可追溯的优惠明细与金额分摊结果，避免退款时金额无法还原；
5. 物流状态、订单状态、售后状态需要松耦合联动，避免互相直接修改核心状态；
6. 搜索能力初期可基于数据库查询实现，但应预留后续接入 Elasticsearch / OpenSearch 的索引模型；
7. 消息能力初期可使用站内消息表落地，但应预留短信、邮件、微信、App Push 等多渠道扩展；
8. 后台关键运营配置必须具备权限控制、操作日志与状态流转记录；
9. 二期新增接口必须补齐基础参数校验、权限校验、幂等控制和异常返回；
10. 保持前后端接口风格、响应结构、鉴权方式、错误码规范与一期一致。

## 1.3 二期建设原则

1. **不破坏一期交易闭环**：二期所有能力都应围绕已有订单、支付、库存、售后体系扩展，不能重写一期核心流程。
2. **履约状态独立建模**：物流、发货、签收属于履约状态，不应简单等同于订单支付状态。
3. **优惠金额可追溯**：优惠券、满减、活动价必须保存计算快照与分摊结果，支撑售后退款。
4. **运营配置可上下线**：Banner、推荐位、活动、优惠券等均应具备启用、禁用、生效时间、失效时间。
5. **用户体验闭环**：搜索、首页、优惠、物流、评价、消息都应在 C 端形成完整可感知体验。
6. **后台操作可审计**：发货、物流修正、营销配置、评价处理、消息补发等操作必须可追踪。
7. **先基础能力，后复杂增长**：二期只做基础营销、基础搜索、基础消息、基础评价，不提前引入复杂推荐、会员、积分、BI、风控。

# 2 系统架构设计

## 2.1 核心架构设计原则

二期继续沿用一期的模块化单体 + DDD 分层架构。整体仍然以“业务模块独立、模块内部四层分明、跨模块通过应用服务或防腐接口协作”为核心原则。

二期新增物流、营销、搜索、消息、评价、内容运营等模块后，不能把所有逻辑堆在 Controller 或 Service 中，而应继续保持清晰的四层结构：

```text
interfaces 接口层
  -> application 应用服务层
  -> domain 领域层
  -> infrastructure 基础设施层
```

每个业务模块内部保持四层结构，不允许 Controller 直接访问 Repository，不允许跨模块直接操作数据库。

### 2.1.1 二期整体模块目录建议

二期是在现有 MVP、一期基础、一期增强模块之上继续扩展，不建议推翻原有模块结构。推荐整体目录如下：

```text
mallFei/
├── backend/
│   ├── mall-user          // 用户域：用户账号、地址、登录态、用户资料
│   ├── mall-product       // 商品域：分类、SPU、SKU、商品上下架、商品展示
│   ├── mall-cart          // 购物车域：购物车、勾选、结算预览、二期优惠提示增强
│   ├── mall-order         // 订单域：订单创建、订单状态、履约状态、确认收货
│   ├── mall-stock         // 库存域：库存预占、扣减、释放、补偿
│   ├── mall-pay           // 支付域：支付单、支付回调、退款单、支付异常处理
│   ├── mall-aftersale     // 售后域：售后申请、审核、退款、二期退货物流增强
│   ├── mall-reconciliation // 对账域：支付对账、退款对账、线上人工对账
│   ├── mall-logistics     // 二期新增：物流公司、发货单、物流轨迹、退货物流
│   ├── mall-marketing     // 二期新增：优惠券、满减、活动、优惠计算、优惠分摊
│   ├── mall-search        // 二期新增：商品搜索、筛选、排序、热搜词
│   ├── mall-message       // 二期新增：站内消息、消息模板、消息读取状态
│   ├── mall-review        // 二期新增：商品评价、评分、晒单、评价审核、评价聚合
│   ├── mall-content       // 二期新增：首页装修、Banner、频道、推荐位、活动入口
│   ├── mall-admin         // Admin 后台：运营配置、订单管理、营销管理、物流管理
│   ├── mall-auth          // 认证鉴权：登录认证、接口权限、角色权限、操作审计支撑
│   ├── mall-common        // 公共能力：统一响应、异常、枚举、工具、基础 DTO
│   └── mall-api           // 可选：跨模块 API、Facade 接口、事件定义聚合
└── frontend/
    ├── mall-h5            // C 端商城：首页、搜索、购物车、结算、订单、消息、评价
    └── mall-admin         // Admin 前端：物流、营销、内容、评价、消息等后台管理
```

如果当前项目仍是一个后端工程内按 package 划分模块，也应保持上述逻辑边界，避免二期功能直接散落在已有订单或商品模块中。

### 2.1.2 模块内部四层结构（DDD 四层架构）

二期每个新增模块均应遵循“接口层 -> 应用服务层 -> 领域层 -> 基础设施层”的结构。以 `mall-logistics`、`mall-marketing`、`mall-review` 等模块为例，推荐结构如下：

```text
xxx-module/
├── interfaces
│   ├── controller          // 接口层：C端接口、Admin接口、内部回调接口
│   ├── request             // 入参对象：创建、查询、审核、发货等请求对象
│   ├── response            // 出参对象：列表、详情、状态、统计等响应对象
│   └── assembler           // 接口模型转换：Request/Response 与应用层 DTO 转换
├── application
│   ├── service             // 应用服务：编排业务流程、事务控制、权限校验、幂等控制
│   ├── command             // 命令对象：创建优惠券、订单发货、提交评价等写操作入参
│   ├── query               // 查询对象：分页查询、详情查询、筛选查询等读操作入参
│   ├── dto                 // 应用层 DTO：跨层传输对象，不承载核心业务规则
│   └── event               // 应用事件：发货成功、优惠券核销、评价通过等事件
├── domain
│   ├── model               // 领域实体/值对象：物流单、优惠券、用户券、评价、消息等
│   ├── service             // 领域服务：优惠计算、优惠分摊、评价资格校验等核心规则
│   ├── repository          // 仓储接口：只定义领域需要的持久化能力
│   ├── event               // 领域事件：物流签收、优惠券锁定、评价已发布等
│   └── enums               // 领域枚举：状态、类型、来源、渠道等
├── infrastructure
│   ├── persistence         // 持久化实现：DO、Mapper、RepositoryImpl
│   ├── mapper              // MyBatis Mapper 或 JPA Repository
│   ├── integration         // 外部系统适配：快递 API、短信网关、搜索引擎等
│   ├── config              // 模块配置：第三方配置、线程池、定时任务配置
│   └── convert             // 基础设施对象转换：DO 与领域对象转换
├── facade                  // 对其他模块暴露的防腐接口，避免跨模块直接访问内部实现
└── XxxApplication.java     // 模块启动类或模块配置入口，按实际工程结构决定是否需要
```

### 2.1.3 四层职责说明

| 层级 | 职责 | 禁止事项 |
|---|---|---|
| interfaces 接口层 | 接收 HTTP 请求、参数校验、登录态解析、返回统一响应、适配 C 端/Admin 端接口 | 不写核心业务规则；不直接访问数据库；不直接调用其他模块 Repository |
| application 应用服务层 | 编排完整用例、控制事务、调用领域服务、调用仓储接口、发布事件、做幂等与权限校验 | 不堆积复杂业务规则；不直接拼 SQL；不绕过领域对象修改状态 |
| domain 领域层 | 承载核心业务概念、状态流转、领域规则、领域服务、仓储接口定义 | 不依赖 Web 框架；不依赖 MyBatis；不依赖第三方 API 细节 |
| infrastructure 基础设施层 | 实现数据库访问、缓存访问、消息发送、第三方物流 API、文件服务、搜索引擎适配 | 不反向调用 Controller；不承载业务决策；不让 DO 对象向上层泄漏 |
| facade 防腐接口 | 对其他模块提供稳定能力入口，屏蔽模块内部实现细节 | 不暴露内部表结构；不让其他模块绕过应用服务直接改数据 |

### 2.1.5 二期跨模块协作方式

二期跨模块协作应优先使用 Facade、防腐接口或应用事件，避免直接依赖对方的 Controller、Mapper、DO 对象。

推荐协作方式如下：

| 场景 | 推荐方式 | 说明 |
|---|---|---|
| 订单创建时计算优惠 | 订单应用服务调用 `MarketingFacade.calculateDiscount` | 订单只关心优惠结果和分摊快照，不关心优惠券内部表结构 |
| 订单支付成功后核销优惠券 | 支付/订单发布支付成功事件，营销模块消费后核销 | 避免支付模块直接修改用户券状态 |
| 后台订单发货 | Admin 调用物流应用服务，物流模块通过 `OrderFulfillmentFacade` 同步履约状态 | 发货属于物流域，订单只接收履约结果 |
| 物流签收 | 物流模块发布签收事件，订单模块更新履约状态，消息模块生成通知 | 签收、订单、消息解耦 |
| 售后退货签收 | 物流模块通知售后模块退货已签收 | 售后模块决定是否进入验收退款 |
| 订单完成后评价 | 订单模块发布订单完成事件，评价模块开放评价资格 | 评价模块不直接扫描订单表 |
| 业务消息生成 | 各业务模块发布业务事件，消息模块生成站内消息 | 消息模块只负责通知，不反向控制业务 |

## 2.3 模块依赖关系

二期模块依赖应遵循以下原则：

```text
C端 / Admin端
  -> 各模块 interfaces
  -> 各模块 application
  -> 各模块 domain
  -> 各模块 infrastructure
```

跨模块关系建议如下：

- 订单模块可以调用物流模块创建发货单，但不直接写物流表；
- 物流模块可以回调订单模块同步履约结果，但不直接修改订单支付状态；
- 营销模块负责优惠计算，订单模块只保存优惠快照和最终金额；
- 售后模块退款时读取订单优惠分摊结果，不重新计算历史优惠；
- 搜索模块读取商品公开信息，不直接管理商品上下架状态；
- 内容模块配置推荐商品时只引用商品 ID，不复制商品主数据；
- 消息模块接收订单、支付、物流、售后等事件后生成消息，不反向控制业务流程；
- 评价模块依赖订单完成状态校验评价资格，不直接改变订单状态。

# 3 二期核心单据与状态定义

## 3.1 物流单状态定义

物流单是二期履约域核心单据，用于记录订单发货、快递公司、物流单号、物流状态、轨迹信息。

| 状态 | 含义 | 说明 |
|---|---|---|
| CREATED | 已创建 | 已生成物流单，但尚未真正发货 |
| SHIPPED | 已发货 | 后台已填写物流公司和物流单号 |
| COLLECTED | 已揽收 | 快递公司已揽收 |
| IN_TRANSIT | 运输中 | 包裹运输中 |
| DELIVERING | 派送中 | 快递员派送中 |
| SIGNED | 已签收 | 用户或代收点签收 |
| EXCEPTION | 物流异常 | 丢件、拒收、超时、轨迹异常等 |
| CANCELED | 已取消 | 发货单取消或作废 |

## 3.2 订单履约状态定义

订单履约状态用于补充一期订单状态机，不替代订单主状态。

| 状态 | 含义 |
|---|---|
| NOT_SHIPPED | 未发货 |
| PART_SHIPPED | 部分发货，二期可预留，初期可不实现 |
| SHIPPED | 已发货 |
| SIGNED | 已签收 |
| CONFIRMED | 已确认收货 |
| FULFILLMENT_EXCEPTION | 履约异常 |

## 3.3 优惠券状态定义

| 状态 | 含义 |
|---|---|
| DRAFT | 草稿 |
| ENABLED | 已启用 |
| DISABLED | 已停用 |
| EXPIRED | 已过期 |

用户券状态：

| 状态 | 含义 |
|---|---|
| UNUSED | 未使用 |
| LOCKED | 已锁定，订单待支付中 |
| USED | 已使用 |
| RETURNED | 已退回 |
| EXPIRED | 已过期 |
| INVALID | 已作废 |

## 3.4 营销活动状态定义

| 状态 | 含义 |
|---|---|
| DRAFT | 草稿 |
| SCHEDULED | 待生效 |
| ACTIVE | 生效中 |
| PAUSED | 已暂停 |
| ENDED | 已结束 |
| CANCELED | 已取消 |

## 3.5 评价状态定义

| 状态 | 含义 |
|---|---|
| PENDING_REVIEW | 待审核 |
| APPROVED | 已通过 |
| REJECTED | 已驳回 |
| HIDDEN | 已隐藏 |

## 3.6 消息状态定义

| 状态 | 含义 |
|---|---|

# 4 二期核心业务详细设计

## 4.1 物流履约域业务设计

### 4.1.1 核心业务流程

#### 订单发货流程

```text
订单支付成功
  -> 后台进入待发货订单列表
  -> 选择物流公司
  -> 填写物流单号 / 发货备注
  -> 创建物流单
  -> 订单履约状态更新为已发货
  -> 生成发货站内消息
  -> C端订单详情展示物流信息
```

发货时必须校验：

- 订单必须已支付；
- 订单不能已取消；
- 订单不能已全额退款；
- 订单不能已关闭；
- 当前管理员必须具备发货权限；
- 物流公司必须启用；
- 物流单号不能为空且格式合法；
- 同一订单不能重复创建有效物流单。

#### 物流轨迹查询流程

```text
用户打开订单详情
  -> 查询订单物流单
  -> 查询本地物流轨迹缓存
  -> 如需刷新，则调用物流轨迹同步能力
  -> 返回物流公司、物流单号、当前状态、轨迹节点
```

二期初期可使用模拟物流轨迹或手动维护轨迹，后续再接入真实快递 API。

#### 物流签收流程

```text
物流轨迹同步为已签收
  -> 物流单状态更新为 SIGNED
  -> 订单履约状态更新为 SIGNED
  -> 生成签收消息
  -> 用户可确认收货
  -> 订单进入可评价状态
```

#### 退货物流流程

```text
售后审核通过
  -> 系统展示退货地址
  -> 用户填写退货物流公司和单号
  -> 生成退货物流记录
  -> 后台查看退货轨迹
  -> 后台确认退货签收
  -> 售后进入验收 / 退款处理流程
```

### 4.1.2 业务边界

#### 核心职责

物流模块是二期履约能力的核心模块，负责承接订单支付成功后的实物履约过程，重点管理“发货、物流单、物流轨迹、签收、退货物流”相关业务。

物流模块核心职责包括：

- 物流公司基础资料管理；
- 订单发货单创建与维护；
- 发货物流单号、物流公司、发货备注记录；
- 物流轨迹节点记录与查询；
- 物流状态流转；
- 物流签收状态确认；
- 退货物流信息记录；
- 退货签收确认；
- 物流异常标记与备注；
- 发货、签收、异常等履约事件发布；
- 物流相关后台操作日志记录。

#### 对内边界

物流模块内部应重点保证物流单据自身的一致性：

- 物流单只能围绕已支付、待发货或已发货订单创建；
- 同一个订单二期默认只允许存在一个有效发货物流单，部分发货能力只做状态预留；
- 物流单状态必须按照合法状态机流转，不允许从已取消直接变为已签收；
- 物流轨迹只能归属于明确的物流单或退货物流单；
- 退货物流与发货物流需要分开建模，不能混用同一条物流记录；
- 物流公司被停用后，不允许被新的发货单引用，但历史物流单仍可查询；
- 物流异常只表示履约异常，不直接代表订单关闭或售后成立。

#### 对外边界

物流模块对外只提供履约结果，不接管其他模块的业务决策：

- 对订单模块：提供发货成功、签收成功、履约异常等结果，由订单模块决定订单履约状态如何更新；
- 对售后模块：提供退货物流提交、退货轨迹查询、退货签收结果，由售后模块决定是否进入验收与退款；
- 对消息模块：发布发货、签收、物流异常事件，由消息模块生成站内消息；
- 对 Admin 模块：提供发货、轨迹维护、退货签收等后台操作能力；
- 对 C 端：提供订单物流详情、物流轨迹、退货物流提交入口。

#### 禁止范围

物流模块禁止承担以下职责：

- 禁止修改订单支付状态；
- 禁止直接关闭订单；
- 禁止直接扣减或释放库存；
- 禁止决定售后是否审核通过；
- 禁止直接执行退款；
- 禁止重新计算订单金额；
- 禁止维护商品上下架状态；
- 禁止维护用户收货地址主数据；
- 禁止让订单、售后等模块直接写物流表；
- 禁止把第三方快递 API 返回结构直接暴露给上层业务。

## 4.2 营销优惠域业务设计

### 4.2.1 核心业务流程

#### 优惠券配置流程

```text
运营人员创建优惠券
  -> 配置券名称、类型、面额、门槛、有效期、发放数量、适用范围
  -> 提交保存为草稿
  -> 审核或直接启用
  -> 用户领取 / 后台发放
  -> 用户结算时使用
```

优惠券类型二期建议支持：

- 满减券：满 N 减 M；
- 无门槛券：直接减 M；
- 商品券：指定商品可用；
- 分类券：指定分类可用；
- 全场券：全平台可用。

#### 下单优惠计算流程

```text
购物车勾选商品
  -> 进入结算页
  -> 查询可用优惠券
  -> 用户选择优惠券
  -> 调用营销模块计算优惠
  -> 返回商品金额、运费、优惠金额、实付金额、优惠分摊明细
  -> 创建订单时保存优惠快照
```

优惠计算必须保存：

- 使用的优惠券 ID；
- 优惠券名称；
- 优惠类型；
- 优惠门槛；
- 优惠金额；
- 订单维度分摊金额；
- 订单项维度分摊金额；
- 计算时间；
- 计算规则版本。

#### 支付超时释放优惠券流程

```text
订单创建成功
  -> 用户券状态变为 LOCKED
  -> 订单支付成功，用户券状态变为 USED
  -> 订单超时关闭，用户券状态恢复 UNUSED 或按规则作废
```

#### 售后退款优惠处理流程

```text
用户发起售后
  -> 售后读取订单优惠分摊快照
  -> 按订单项实际支付金额计算可退金额
  -> 不重新计算优惠规则
  -> 退款完成后记录优惠退款明细
```

### 4.2.2 业务边界

#### 核心职责

营销模块负责二期基础促销能力，重点解决“优惠从哪里来、用户能不能用、优惠怎么算、优惠如何锁定、优惠如何核销、退款时如何还原优惠分摊”的问题。

营销模块核心职责包括：

- 优惠券模板定义；
- 优惠券启用、停用、过期；
- 用户领券；
- 后台发券；
- 用户券状态管理；
- 优惠券适用范围判断；
- 满减、无门槛、商品券、分类券、全场券等基础优惠规则计算；
- 结算页优惠试算；
- 订单创建时优惠锁定；
- 支付成功后优惠核销；
- 订单关闭后优惠释放或作废；
- 订单级与订单项级优惠分摊；
- 售后退款时提供历史优惠分摊快照；
- 优惠券使用记录与操作日志。

#### 对内边界

营销模块内部应重点保证优惠规则、用户券状态、优惠分摊结果的一致性：

- 优惠券模板一旦被用户领取或被订单使用，关键计算字段不得随意物理修改；
- 优惠规则变更必须通过版本、复制或新建模板处理，避免影响历史订单；
- 用户券只能按照 UNUSED、LOCKED、USED、RETURNED、EXPIRED、INVALID 等状态合法流转；
- 结算试算结果只代表当前时刻可用优惠，不等同于最终锁定结果；
- 订单创建时必须重新校验优惠券可用性，并保存优惠快照；
- 优惠分摊结果必须保存到订单项维度，支撑后续部分退款；
- 售后退款不能重新套用当前优惠规则，只能读取下单时的历史分摊快照；
- 优惠券库存、领取次数、使用次数需要具备并发控制，避免超发超用。

#### 对外边界

营销模块对外提供优惠能力，但不接管交易主流程：

- 对购物车模块：提供可用优惠提示、凑单提示、结算试算结果；
- 对订单模块：提供最终优惠计算、优惠锁定、优惠快照与分摊明细；
- 对支付模块：接收支付成功事件后核销用户券；
- 对订单关闭流程：接收订单关闭事件后释放或作废用户券；
- 对售后模块：提供历史优惠分摊结果，辅助计算可退金额；
- 对消息模块：发布优惠券到账、即将过期、使用成功等事件；
- 对 Admin 模块：提供优惠券模板、发券、活动配置、使用记录查询能力。

#### 禁止范围

营销模块禁止承担以下职责：

- 禁止创建订单主单；
- 禁止直接修改订单状态；
- 禁止直接扣减商品库存；
- 禁止直接发起支付；
- 禁止直接执行退款；
- 禁止修改商品销售价主数据；
- 禁止在售后阶段重新计算历史订单优惠；
- 禁止让订单模块直接修改用户券状态；
- 禁止把优惠试算结果当作最终成交金额；
- 禁止绕过优惠快照进行退款金额计算。

## 4.3 商品搜索域业务设计

### 4.3.1 核心业务流程

```text
用户输入关键词
  -> 搜索模块解析关键词
  -> 组合分类、品牌、价格区间、销量、上架时间、综合排序等条件
  -> 查询可售商品
  -> 返回商品列表、筛选项、排序结果
  -> 记录搜索关键字与搜索次数
```

二期搜索范围包括：

- 商品名称；
- 商品副标题；
- 商品分类；
- 商品价格区间；
- 商品上下架状态；
- 商品销量；
- 商品创建时间；
- 商品推荐标识。

二期排序方式包括：

- 综合排序；
- 销量优先；
- 价格升序；
- 价格降序；
- 上新优先。

### 4.3.2 业务边界

#### 核心职责

搜索模块负责商品发现能力，重点解决“用户如何更快找到商品”的问题。二期搜索以基础检索、筛选、排序和热搜词为主，不引入复杂个性化推荐。

搜索模块核心职责包括：

- 商品关键词检索；
- 分类、价格、销量、上架时间等条件筛选；
- 综合、销量、价格、上新等排序；
- 搜索关键词记录；
- 热搜词统计；
- 热搜词后台配置；
- 搜索结果商品摘要封装；
- 商品搜索快照预留；
- 后续接入搜索引擎的索引模型预留。

#### 对内边界

搜索模块内部应保证搜索结果只面向“可展示、可售卖”的商品集合：

- 搜索结果必须过滤已下架、已删除、不可售商品；
- 搜索结果展示的价格、图片、标题应来源于商品公开信息或搜索快照；
- 搜索关键词日志不能影响真实商品排序，只用于统计分析和热词沉淀；
- 热搜词配置只影响搜索入口展示，不等同于个性化推荐；
- 如果后续引入搜索引擎，数据库商品主数据仍应作为最终可信来源；
- 搜索索引同步失败时，应记录异常，不应影响商品主流程下单支付。

#### 对外边界

搜索模块对外提供商品检索结果，不管理商品生命周期：

- 对商品模块：读取商品公开信息、分类、价格、上下架状态，不反向修改商品；
- 对内容运营模块：可提供搜索关键词或热搜词数据，用于运营配置参考；
- 对 C 端：提供搜索列表、筛选项、排序结果、热搜词；
- 对 Admin 模块：提供热搜词管理、搜索日志查询、搜索统计能力；
- 对后续推荐模块：可提供搜索行为数据，但二期不直接承接推荐算法。

#### 禁止范围

搜索模块禁止承担以下职责：

- 禁止维护商品主数据；
- 禁止审批商品上下架；
- 禁止修改商品价格；
- 禁止扣减库存；
- 禁止创建订单；
- 禁止直接做个性化推荐算法；
- 禁止把热搜词配置等同于广告投放系统；
- 禁止让搜索索引成为商品主数据的唯一来源。

## 4.4 首页装修与内容运营域业务设计

### 4.4.2 业务边界

内容运营模块负责 C 端首页与运营位配置，重点解决“首页展示什么、入口跳到哪里、运营资源如何上下线”的问题。

内容运营模块核心职责包括：

- 图片、标题、跳转链接维护；
- 展示时间、生效时间、失效时间管理；
- 排序权重管理；
- 内容启用、停用、下线；
- 首页配置查询与缓存预留；
- 内容运营操作日志。

#### 对内边界

内容运营模块内部应重点保证运营配置的可控性和可回溯性：

- 所有运营位必须具备明确的状态，不能只有物理删除；
- Banner、频道、推荐位必须具备排序字段；
- 运营配置应支持生效时间和失效时间，避免过期活动继续展示；
- 推荐商品只保存商品引用关系，不复制完整商品主数据；
- 跳转链接需要限制合法类型，例如商品详情、分类页、活动页、优惠券页；
- 已下架商品、失效活动、停用优惠券不应继续在首页有效展示；
- 首页配置变更需要记录操作人、操作时间和变更内容。

#### 对外边界

内容运营模块对外只提供展示配置，不接管交易与搜索逻辑：

- 对商品模块：引用商品 ID 并读取商品展示信息，不维护商品主数据；
- 对营销模块：引用优惠券或活动入口，不修改优惠规则；
- 对搜索模块：可配置搜索入口、热词入口，但不影响搜索排序算法；
- 对文件服务：使用图片上传结果，不直接管理文件存储底层；
- 对 C 端：首页拉取当前生效的运营配置；
- 对 Admin 模块：提供 Banner、频道、推荐位、活动入口管理能力。

#### 禁止范围

内容运营模块禁止承担以下职责：

- 禁止修改商品价格；
- 禁止维护商品库存；
- 禁止创建订单；
- 禁止计算优惠金额；
- 禁止直接发放优惠券；
- 禁止控制搜索排序算法；
- 禁止把推荐位当作个性化推荐系统；
- 禁止直接操作文件存储底层实现；
- 禁止在首页展示已失效、已下架或无权限访问的资源。

## 4.5 商品评价域业务设计

### 4.5.1 核心业务流程

```text
订单确认收货 / 自动完成
  -> 用户进入待评价列表
  -> 用户提交评分、文字、图片
  -> 系统校验评价资格
  -> 保存评价
  -> 后台审核或自动通过
  -> 商品详情页展示评价与评分聚合
```

评价内容包括：

- 商品评分；
- 物流评分；
- 服务评分；
- 文字评价；
- 图片评价；
- 是否匿名；
- 订单项关联；
- SKU 快照；
- 评价时间。

### 4.5.2 业务边界

#### 核心职责

评价模块负责订单完成后的商品评价能力，重点解决“谁能评价、评价什么、评价如何展示、评价如何审核、评分如何聚合”的问题。

评价模块核心职责包括：

- 评价资格校验；
- 待评价列表查询；
- 商品评分提交；
- 物流评分、服务评分提交；
- 文字评价提交；
- 图片评价关联；
- 匿名评价处理；
- 评价审核；
- 评价隐藏或驳回；
- 商品详情评价列表展示；
- 商品评分聚合；
- 用户我的评价查询；
- 评价操作日志。

#### 对内边界

评价模块内部应重点保证评价资格、评价内容和评价展示状态的一致性：

- 用户只能评价自己已完成且具备评价资格的订单项；
- 同一订单项二期默认只允许评价一次，追评能力可后续预留；
- 评价必须绑定订单项、商品 ID、SKU 快照和用户 ID；
- 商品评分聚合应基于已通过或可展示评价计算；
- 被隐藏、驳回、删除的评价不应进入前台展示和评分聚合；
- 评价图片应通过文件服务上传，评价模块只保存文件引用；
- 评价审核状态变化需要记录操作人、操作时间和原因。

#### 对外边界

评价模块对外提供评价能力，不接管订单履约和客服处理：

- 对订单模块：依赖订单完成或确认收货结果判断评价资格，但不修改订单状态；
- 对商品模块：提供商品评分、评价数量、评价列表，不修改商品主数据；
- 对文件服务：保存评价图片引用，不管理文件存储；
- 对消息模块：评价审核通过或驳回时可发布消息事件；
- 对 Admin 模块：提供评价审核、隐藏、查询能力；
- 对 C 端：提供待评价、提交评价、商品评价列表、我的评价能力。

#### 禁止范围

评价模块禁止承担以下职责：

- 禁止创建订单；
- 禁止修改订单支付状态；
- 禁止修改订单履约状态；
- 禁止确认物流签收；
- 禁止维护商品主数据；
- 禁止处理客服工单；
- 禁止直接执行赔付或退款；
- 禁止绕过订单完成状态允许评价；
- 禁止将未审核通过的评价计入前台评分聚合。

## 4.6 消息通知域业务设计

### 4.6.1 核心业务流程

```text
业务事件产生
  -> 消息模块接收事件
  -> 匹配消息模板
  -> 生成站内消息
  -> 用户消息中心查看
  -> 用户标记已读 / 删除
```

二期消息类型包括：

- 订单创建提醒；
- 支付成功提醒；
- 订单关闭提醒；
- 发货提醒；
- 签收提醒；
- 售后审核结果；
- 退款成功提醒；
- 优惠券到账提醒；
- 优惠券即将过期提醒。

### 4.6.2 业务边界

#### 核心职责

消息模块负责二期基础站内通知能力，重点解决“业务事件如何通知用户、用户如何查看、消息是否已读”的问题。

消息模块核心职责包括：

- 消息模板管理；
- 订单、支付、物流、售后、优惠券等业务消息生成；
- 用户站内消息列表；
- 消息详情；
- 未读消息数量；
- 标记已读；
- 批量已读；
- 删除消息；
- 消息发送记录；
- 消息补发；
- 后续短信、邮件、Push 等渠道扩展预留。

#### 对内边界

消息模块内部应重点保证消息生成、读取状态和模板版本的稳定性：

- 消息必须有明确接收用户、消息类型、标题、内容、业务来源和业务单号；
- 消息模板变更不能影响历史已发送消息内容；
- 站内消息生成应具备基础幂等，避免同一业务事件重复生成多条相同消息；
- 用户只能读取自己的消息；
- 删除消息二期建议为用户侧逻辑删除，不影响后台发送记录；
- 消息发送失败应记录失败原因，支持后台查询或补发；
- 多渠道通知能力只做扩展预留，二期以站内消息为主。

#### 对外边界

消息模块对外接收业务事件并生成通知，不反向控制业务流程：

- 对订单模块：接收订单创建、订单关闭、确认收货等事件；
- 对支付模块：接收支付成功、退款成功等事件；
- 对物流模块：接收发货、签收、物流异常等事件；
- 对售后模块：接收售后审核、退货签收、退款完成等事件；
- 对营销模块：接收优惠券到账、优惠券即将过期等事件；
- 对 C 端：提供消息列表、详情、未读数、已读、删除能力；
- 对 Admin 模块：提供模板管理、发送记录、补发能力。

#### 禁止范围

消息模块禁止承担以下职责：

- 禁止修改订单状态；
- 禁止修改支付状态；
- 禁止修改售后状态；
- 禁止修改物流状态；
- 禁止核销优惠券；
- 禁止执行退款；
- 禁止直接触发复杂营销活动；
- 禁止把消息发送成功作为业务成功的前置条件；
- 禁止让业务模块直接写用户消息表。

## 4.7 购物车与结算增强设计

### 4.7.1 核心业务流程

```text
用户进入购物车
  -> 校验商品上下架状态
  -> 校验 SKU 是否可售
  -> 校验库存是否充足
  -> 检查价格是否变动
  -> 查询可用优惠
  -> 展示失效商品、价格变动、优惠提示
```

购物车二期增强包括：

- 失效商品提示；
- 商品下架提示；
- 库存不足提示；
- 价格变动提示；
- 可用优惠券提示；
- 凑单提示；
- 结算页优惠预览。

### 4.7.2 业务边界

#### 核心职责

购物车与结算增强属于已有购物车模块的二期扩展，重点解决“购物车商品当前还能不能买、价格有没有变化、有没有可用优惠、结算前用户能看到什么”的问题。

购物车模块二期核心职责包括：

- 购物车商品有效性校验；
- 商品上下架状态提示；
- SKU 可售状态提示；
- 库存不足提示；
- 商品价格变动提示；
- 可用优惠券提示；
- 凑单提示；
- 结算页优惠试算入口；
- 结算金额预览；
- 失效商品清理。

#### 对内边界

购物车模块内部应重点保证展示结果的及时性和用户选择的一致性：

- 购物车保存的是用户选购意向，不代表库存已锁定；
- 购物车金额仅作为展示和试算，不代表最终订单金额；
- 商品失效、下架、库存不足时，购物车项应保留但禁止结算；
- 价格变动应提示用户，最终价格以结算和下单时商品价格为准；
- 优惠提示只表示可能可用，最终优惠以订单创建时营销模块校验结果为准；
- 结算页应重新校验商品、库存、价格和优惠，不应完全信任购物车缓存。

#### 对外边界

购物车模块对外聚合商品、库存、营销等信息，但不接管这些模块的主职责：

- 对商品模块：读取商品名称、图片、SKU、上下架状态、当前价格；
- 对库存模块：读取可售库存或库存是否充足；
- 对营销模块：查询可用优惠、优惠提示和结算试算；
- 对订单模块：提交用户选择的结算项，由订单模块进行最终下单校验；
- 对 C 端：展示购物车列表、失效提示、优惠提示、结算预览。

#### 禁止范围

购物车模块禁止承担以下职责：

- 禁止锁定库存；
- 禁止扣减库存；
- 禁止创建订单主单；
- 禁止核销优惠券；
- 禁止执行支付；
- 禁止保存最终成交金额；
- 禁止直接修改商品价格；
- 禁止把优惠试算结果作为最终订单优惠；
- 禁止跳过订单创建阶段的最终校验。

# 5 二期标准业务闭环流程

## 5.1 正常下单优惠履约闭环

```text
用户浏览首页运营位
  -> 搜索商品
  -> 加入购物车
  -> 领取优惠券
  -> 进入结算页
  -> 选择优惠券
  -> 营销模块计算优惠
  -> 创建订单并保存优惠快照
  -> 锁定优惠券
  -> 支付成功
  -> 优惠券变为已使用
  -> 后台发货
  -> 用户查看物流
  -> 物流签收
  -> 用户确认收货
  -> 用户评价商品
```

## 5.2 支付超时优惠释放闭环

```text
用户创建订单并锁定优惠券
  -> 用户未支付
  -> 订单超时关闭
  -> 库存释放
  -> 优惠券解锁或作废
  -> 生成订单关闭消息
```

## 5.3 发货物流闭环

```text
订单支付成功
  -> 进入待发货列表
  -> 后台选择物流公司并填写物流单号
  -> 物流单创建
  -> 订单履约状态变为已发货
  -> 用户收到发货消息
  -> 用户查看物流轨迹
  -> 物流签收
  -> 用户确认收货
```

## 5.4 退货物流售后闭环

```text
用户申请退货退款
  -> 后台审核通过
  -> 用户填写退货物流
  -> 后台查看退货轨迹
  -> 后台确认退货签收
  -> 售后验收
  -> 按优惠分摊金额计算退款
  -> 支付退款
  -> 对账归档
```

## 5.5 评价闭环

```text
订单完成
  -> 用户获得评价资格
  -> 用户提交评价
  -> 后台审核评价
  -> 商品详情展示评价
  -> 商品评分聚合更新
```

# 11 验收标准

二期完成后，应满足以下验收标准：

1. 用户可以在首页看到后台配置的 Banner、频道、推荐商品和活动入口；
2. 用户可以通过关键词搜索商品，并进行筛选和排序；
3. 用户可以领取优惠券；
4. 结算页可以展示可用优惠券、不可用原因、优惠金额和实付金额；
5. 创建订单时可以保存优惠快照和订单项优惠分摊；
6. 支付超时关闭订单时，优惠券能够正确释放或按规则作废；
7. 支付成功后，优惠券能够正确核销为已使用；
8. 售后退款时，可以基于历史优惠分摊计算可退金额；
9. 后台可以对已支付订单进行发货；
10. 用户可以在订单详情查看物流公司、物流单号和物流轨迹；
11. 物流签收后，订单履约状态可以正确更新；
12. 用户可以确认收货；
13. 订单完成后，用户可以提交商品评价；
14. 商品详情页可以展示评价列表和评分聚合；
15. 后台可以审核、隐藏或查看评价；
16. 售后退货退款场景中，用户可以提交退货物流；
17. 后台可以确认退货签收，并推动售后进入后续退款流程；
18. 系统可以生成订单、支付、发货、签收、售后、优惠券等站内消息；
19. 用户可以查看消息、标记已读和删除消息；
20. 后台物流、营销、内容、评价、消息等关键操作均有权限控制和操作日志；
21. 二期新增能力不破坏一期订单、支付、库存、售后、对账闭环；
22. 二期新增接口具备统一鉴权、参数校验、错误返回和基础幂等控制。

  -> 筛选排序
  -> 热搜词

# 13 二期完成后的平台能力变化

二期完成后，`mallFei` 应具备以下能力：

- 从“能下单支付”升级为“能运营、能发货、能追踪、能评价”；
- 从“基础商品展示”升级为“可搜索、可装修、可推荐”；
- 从“订单金额固定”升级为“支持优惠计算与优惠分摊”；
- 从“售后退款闭环”升级为“退货物流与退款协同闭环”；
- 从“后台基础管理”升级为“运营后台可配置、可审计”；
- 从“交易可靠性”进一步走向“真实电商运营体验”。

二期是平台从交易系统走向运营系统的关键阶段，也是三期会员、积分、客服、BI、推荐、风控等经营能力建设的前置基础。

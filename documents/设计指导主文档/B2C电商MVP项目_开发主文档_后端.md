# B2C 电商 MVP 项目开发主文档（融合版）

> 本文档作为 `mallFei` MVP 阶段后端开发、联调、验收的主指导文档。
>
> MVP 阶段只聚焦“最小可用交易闭环”，不提前引入售后、退款、真实支付、对账、物流、营销、评价、会员、客服、风控、BI、生产治理等企业级增强能力。

---

# 1 项目概述

## 1.1 项目定位

本项目为 B2C 自营电商平台 MVP（Minimum Viable Product，最小可用版本），目标是以最小范围跑通 C 端用户从商品浏览、加入购物车、提交订单、模拟支付到查看订单的基础交易闭环，同时提供一个极简 PC 运营后台，用于基础商品、库存、订单、用户数据运维。

MVP 阶段不是完整商用电商系统，而是“可演示、可联调、可扩展”的基础版本，重点验证：

- 用户能注册、登录；
- 用户能浏览商品；
- 用户能选择 SKU 加入购物车；
- 用户能选择收货地址提交订单；
- 系统能校验商品、SKU、库存；
- 系统能生成订单和订单项快照；
- 系统能创建支付单；
- 用户能通过 Mock 支付完成支付；
- 支付成功后订单状态能同步变化；
- 用户能查看自己的订单；
- 后台管理员能维护基础商品、库存并查看基础数据。

MVP 阶段终端范围：

| 终端 | 范围 | 说明 |
|---|---|---|
| C 端 H5 | 必须 | 用户注册登录、商品浏览、购物车、下单、Mock 支付、订单查询 |
| PC Admin 后台 | 必须 | 管理员登录、商品管理、库存管理、订单查看、用户查看、基础看板 |
| App | 不包含 | 后续阶段预留 |
| 小程序 | 不包含 | 后续阶段预留 |
| 开放 API | 不包含 | 五期平台化阶段预留 |

## 1.2 设计目标

### 1.2.1 业务目标

MVP 业务目标是跑通最小交易演示闭环：

```text
用户注册 / 登录
  -> 浏览商品
  -> 查看商品详情
  -> 加入购物车
  -> 购物车结算预览
  -> 选择收货地址
  -> 创建订单
  -> 基础库存处理
  -> 创建支付单
  -> Mock 支付成功
  -> 订单变为已支付
  -> 用户查看订单
  -> 后台查看基础运营数据
```

### 1.2.2 技术目标

MVP 技术目标包括：

1. 采用模块化单体架构，按业务域拆分模块；
2. 模块内部采用 DDD 四层结构；
3. 使用统一响应、统一异常、统一登录态；
4. 使用 MySQL 保存核心业务数据；
5. 使用 Redis 作为基础缓存或库存查询优化预留；
6. 使用 Sa-Token 实现用户与管理员基础认证；
7. 保证核心链路代码清晰、可维护、可扩展；
8. 为一期基础交易闭环、一增强可靠性、二期履约运营等后续阶段预留扩展边界。

### 1.2.3 求职展示目标

MVP 阶段需要体现以下能力：

- 电商核心业务拆解能力；
- 商品、购物车、订单、库存、支付主链路设计能力；
- DDD 模块化单体落地能力；
- 基础数据建模能力；
- 接口设计与前后端联调能力；
- 基础权限认证与后台管理能力。

## 1.3 技术栈选型

### 1.3.1 后端技术栈

| 技术 | 用途 |
|---|---|
| JDK 17 / 21 | Java 运行环境 |
| Spring Boot 3 | 后端应用框架 |
| MyBatis-Plus | ORM 与基础 CRUD |
| MySQL 8.0 | 核心业务数据库 |
| Redis | 基础缓存、库存查询优化预留 |
| Sa-Token | 用户与管理员登录认证 |
| Lombok | 简化实体、DTO、VO 编写 |
| Swagger / Knife4j | 接口文档与联调 |

### 1.3.2 前端技术栈

| 终端 | 技术 |
|---|---|
| C 端 H5 | Vue3 + Vant |
| PC Admin 后台 | Vue3 + Element Plus |

### 1.3.3 部署架构

MVP 阶段采用轻量化部署：

```text
Nginx
  -> C端H5静态资源
  -> Admin静态资源
  -> Spring Boot 单体后端
  -> MySQL 单实例
  -> Redis 单实例
```

MVP 不要求高可用、集群化、自动化部署、灰度发布、监控告警，相关能力由四期生产治理阶段建设。

## 1.4 MVP 建设原则

1. **最小闭环优先**：优先保证用户能买、订单能生成、支付能模拟成功。
2. **业务边界清晰**：商品管静态商品数据，库存管库存，订单管交易单据，支付管支付单。
3. **不提前企业化**：不在 MVP 阶段提前做售后、退款、对账、物流、营销等复杂能力。
4. **状态显性化**：用户、商品、库存、订单、支付都必须有清晰状态。
5. **快照优先**：订单必须保存商品快照和收货地址快照。
6. **模块化演进**：MVP 可以是单体，但模块边界必须为后续扩展预留。
7. **基础安全可用**：必须区分公开接口、用户接口、管理员接口。

# 2 系统架构设计

## 2.1 架构设计原则

MVP 阶段采用 **模块化单体 + DDD 四层架构**。

整体原则：

- 高内聚、低耦合；
- 模块边界清晰；
- Controller 不直接访问 Mapper；
- 跨模块不直接操作对方数据库表；
- 跨模块通过应用服务、Facade 或防腐接口协作；
- 领域规则尽量沉淀在领域层；
- 基础设施细节不向上层泄漏。

## 2.2 顶级模块结构

```text
mallFei/
├── mall-common          // 公共模块：统一响应、异常、枚举、工具类、基础DTO
├── mall-auth            // 认证鉴权：Sa-Token登录、路由鉴权、会话管理
├── mall-user            // 用户域：注册、登录、资料、地址、用户基础查询
├── mall-product         // 商品域：分类、SPU、SKU、商品列表、商品详情
├── mall-cart            // 购物车域：加购、改数量、勾选、删除、结算预览
├── mall-order           // 订单域：订单创建、订单项、订单查询、订单基础状态
├── mall-stock           // 库存域：SKU库存、库存校验、基础扣减、后台调整
├── mall-pay             // 支付域：支付单、Mock支付、支付结果同步
├── mall-file            // 文件域：商品图片、头像等基础文件上传访问
├── mall-admin           // 后台域：管理员登录、商品/订单/库存/用户基础管理
└── mall-start           // 启动模块：Spring Boot启动入口和模块装配
```

如果实际工程不是多 Maven Module，也应在 package 层面保持上述逻辑边界。

## 2.3 模块内部四层结构

每个业务模块内部建议采用：

```text
xxx-module/
├── interfaces
│   ├── controller       // 接口层：HTTP接口、参数校验、统一返回
│   ├── request          // 请求对象
│   ├── response         // 响应对象
│   └── assembler        // 接口模型转换
├── application
│   ├── service          // 应用服务：流程编排、事务控制
│   ├── command          // 写操作命令对象
│   ├── query            // 查询对象
│   └── dto              // 应用层DTO
├── domain
│   ├── model            // 实体、值对象、聚合根
│   ├── service          // 领域服务：核心业务规则
│   ├── repository       // 仓储接口
│   └── enums            // 状态、类型枚举
├── infrastructure
│   ├── persistence      // DO、Repository实现
│   ├── mapper           // MyBatis Mapper
│   ├── config           // 模块配置
│   └── convert          // DO与领域对象转换
└── facade               // 对其他模块暴露的防腐接口
```

## 2.4 MVP 模块依赖关系

推荐依赖关系：

```text
C端 / Admin端
  -> interfaces
  -> application
  -> domain
  -> infrastructure
```

跨模块依赖方向：

- `mall-cart` 读取 `mall-product`、`mall-stock` 的商品和库存信息；
- `mall-order` 读取 `mall-user`、`mall-product`、`mall-cart`、`mall-stock` 信息创建订单；
- `mall-stock` 提供库存校验与基础扣减能力；
- `mall-pay` 根据订单创建支付单，支付成功后通知订单更新状态；
- `mall-admin` 调用各业务模块应用服务或 Facade 进行后台管理；
- 所有需要登录的模块依赖 `mall-auth` 获取当前用户或管理员身份。

禁止依赖：

- 禁止购物车直接写商品表；
- 禁止订单直接改商品价格；
- 禁止支付模块直接扣库存；
- 禁止库存模块创建订单；
- 禁止 Admin 直接绕过应用服务操作业务表；
- 禁止 Controller 直接调用 Mapper。

# 4 核心业务详细设计与边界

## 4.1 用户与认证域

### 4.1.1 核心业务流程

#### 用户注册流程

```text
用户提交手机号和密码
  -> 校验手机号格式
  -> 校验手机号是否已存在
  -> 校验密码强度
  -> 密码加密
  -> 创建用户
  -> 返回注册结果
```

#### 用户登录流程

```text
用户提交手机号和密码
  -> 查询用户
  -> 校验用户状态
  -> 校验密码
  -> 创建 Sa-Token 登录态
  -> 返回 token 和用户信息
```

#### 地址管理流程

```text
用户新增/编辑地址
  -> 校验登录态
  -> 校验收货人、手机号、省市区、详细地址
  -> 保存地址
  -> 如设置默认地址，则取消其他默认地址
```

### 4.1.2 业务边界

#### 核心职责

用户与认证域负责平台基础身份和收货信息：

- 用户注册；
- 用户密码登录；
- 用户退出；
- 当前用户信息查询；
- 用户资料维护；
- 用户收货地址管理；
- 管理员登录；
- 基础路由鉴权；
- 当前用户/管理员身份识别。

#### 对内边界

- 用户模块只维护用户账号、资料、地址；
- 认证模块只负责登录态和鉴权；
- 订单模块只能读取用户和地址信息，不能修改用户资料；
- 购物车、订单、支付接口必须依赖认证模块获取当前用户；
- Admin 后台接口必须依赖认证模块获取当前管理员。

#### 对外边界

- C 端开放注册、登录、退出、用户资料、地址接口；
- Admin 端开放管理员登录和用户只读查询；
- 商品浏览接口可免登录；
- 购物车、订单、支付、地址接口必须登录。

#### 禁止范围

- 禁止 MVP 实现短信登录；
- 禁止 MVP 实现第三方登录；
- 禁止 MVP 实现手机号换绑；
- 禁止 MVP 实现黑名单、登录风控、异常登录告警；
- 禁止 Admin 在 MVP 阶段随意禁用 C 端用户；
- 禁止业务模块自行解析 token 绕过认证模块。

## 4.2 商品域

### 4.2.1 核心业务流程

#### 后台商品维护流程

```text
管理员登录后台
  -> 创建/编辑分类
  -> 创建/编辑 SPU
  -> 创建/编辑 SKU
  -> 配置价格、图片、描述、上下架状态
  -> C端商品列表和详情可见
```

#### C 端商品浏览流程

```text
用户进入商城
  -> 查询分类
  -> 分页查询上架 SPU
  -> 查看商品详情
  -> 查询 SKU、价格、图片、库存
  -> 用户选择 SKU 加购
```

### 4.2.2 业务边界

#### 核心职责

商品域负责商品静态主数据：

- 商品分类；
- SPU 主商品；
- SKU 规格；
- 商品名称、图片、描述；
- SKU 价格；
- 商品上下架；
- C 端商品列表；
- C 端商品详情；
- 后台商品维护。

#### 对内边界

- 商品域只维护商品静态数据，不维护库存数量；
- 商品域可以提供 SKU 当前销售价，但最终订单必须保存价格快照；
- 商品上下架会影响加购和下单；
- 商品域不负责订单金额汇总；
- 商品域不负责营销价格和优惠计算。

#### 对外边界

- 对购物车：提供 SKU 名称、图片、价格、状态；
- 对订单：提供下单时商品快照数据；
- 对库存：提供 SKU 基础信息；
- 对 C 端：提供商品列表、详情；
- 对 Admin：提供分类、SPU、SKU 管理。

#### 禁止范围

- 禁止商品域扣减库存；
- 禁止商品域创建订单；
- 禁止商品域处理支付；
- 禁止 MVP 做商品销量统计、商品运营事件、热销滞销识别；
- 禁止购物车或订单直接修改商品主数据。

## 4.3 购物车域

### 4.3.1 核心业务流程

```text
用户选择 SKU
  -> 加入购物车
  -> 同用户同 SKU 自动合并数量
  -> 用户修改数量 / 勾选 / 删除
  -> 查询购物车列表
  -> 结算预览
  -> 下单前校验商品和库存
```

### 4.3.2 业务边界

#### 核心职责

购物车域负责用户临时选购意向：

- 加入购物车；
- 修改数量；
- 勾选/取消勾选；
- 删除购物车项；
- 清空购物车；
- 查询购物车列表；
- 结算预览；
- 下单前基础校验。

#### 对内边界

- 购物车保存的是用户选购意向，不代表库存已锁定；
- 购物车金额只是预估金额，不代表最终订单金额；
- 购物车结算预览必须读取商品当前价格和状态；
- 下单时订单模块仍需重新校验商品和库存；
- 购物车不保存商品完整快照，只保存用户 ID、SKU ID、数量和勾选状态。

#### 对外边界

- 对商品模块：读取 SKU 展示信息；
- 对库存模块：读取可售库存；
- 对订单模块：提供已勾选购物车项作为下单候选；
- 对 C 端：提供购物车操作接口；
- 不提供 Admin 管理接口。

#### 禁止范围

- 禁止购物车锁定库存；
- 禁止购物车扣减库存；
- 禁止购物车创建订单主单；
- 禁止购物车核销优惠；
- 禁止 MVP 做未登录购物车、跨端同步、优惠提示、凑单推荐。

## 4.4 订单域

### 4.4.1 核心业务流程

#### 创建订单流程

```text
用户提交结算请求
  -> 校验登录态
  -> 校验收货地址
  -> 查询勾选购物车项或提交商品项
  -> 校验商品、SKU 状态
  -> 校验库存充足
  -> 计算订单金额
  -> 保存订单主单
  -> 保存订单项快照
  -> 调用库存模块处理基础库存
  -> 创建支付单
  -> 返回订单和支付信息
```

#### 订单查询流程

```text
用户进入订单列表
  -> 查询当前用户订单
  -> 查询订单项
  -> 返回订单摘要
```

### 4.4.2 业务边界

#### 核心职责

订单域负责交易单据：

- 订单创建；
- 订单号生成；
- 订单金额计算；
- 商品快照保存；
- 收货地址快照保存；
- 订单状态流转；
- 用户订单列表；
- 用户订单详情；
- 用户取消待支付订单；
- 用户逻辑删除历史订单；
- 后台订单查看。

#### 对内边界

- 订单是交易事实主单；
- 订单创建时必须保存商品快照和地址快照；
- 订单状态只能按合法状态流转；
- 订单金额以创建订单时计算结果为准；
- 支付成功后由支付模块通知订单更新为已支付；
- 库存扣减或占用必须通过库存模块完成。

#### 对外边界

- 对用户模块：读取用户和地址；
- 对商品模块：读取商品快照；
- 对库存模块：调用库存校验和处理；
- 对支付模块：提供订单号、金额、用户信息创建支付单；
- 对 C 端：提供订单创建、列表、详情、取消、删除；
- 对 Admin：提供订单查询和详情。

#### 禁止范围

- 禁止订单模块直接修改商品价格；
- 禁止订单模块绕过库存模块扣库存；
- 禁止订单模块执行支付渠道逻辑；
- 禁止 MVP 做发货、物流、售后、退款、超时自动取消、自动确认收货；
- 禁止 Admin 在 MVP 阶段随意改订单状态。

## 4.5 库存域

### 4.5.1 核心业务流程

```text
后台初始化 SKU 库存
  -> C端商品详情展示可售库存
  -> 下单时校验可售库存
  -> 库存充足则基础扣减或占用
  -> 库存不足则下单失败
```

### 4.5.2 业务边界

#### 核心职责

库存域负责 SKU 库存基础数据：

- SKU 库存初始化；
- 库存查询；
- 可售库存校验；
- 下单基础库存处理；
- 取消订单库存释放预留；
- 后台库存调整。

#### 对内边界

- 库存表以 SKU 为维度；
- MVP 可采用数据库事务保证基础一致性；
- 库存可先做简单扣减，不要求完整预占锁单；
- 库存数量不能由商品模块直接维护；
- 库存调整必须校验数量合法性。

#### 对外边界

- 对商品模块：提供展示库存；
- 对购物车：提供库存充足性；
- 对订单：提供下单校验和基础库存处理；
- 对 Admin：提供库存查询和调整。

#### 禁止范围

- 禁止库存模块创建订单；
- 禁止库存模块处理支付；
- 禁止 MVP 做库存锁单、库存流水、库存对账、防超卖补偿、Redis 高并发扣减；
- 禁止 C 端直接调整库存。

## 4.6 支付域

### 4.6.1 核心业务流程

```text
订单创建成功
  -> 创建支付单
  -> 用户进入支付页
  -> 用户触发 Mock 支付成功
  -> 更新支付单状态
  -> 通知订单更新为已支付
  -> 返回支付结果
```

### 4.6.2 业务边界

#### 核心职责

支付域负责 MVP 模拟支付闭环：

- 支付单创建；
- 支付单查询；
- Mock 支付成功；
- 支付状态更新；
- 支付成功后同步订单状态；
- 保存支付单基础信息。

#### 对内边界

- 支付单金额必须等于订单实付金额；
- 同一订单原则上只有一个有效待支付支付单；
- 支付成功后不能重复支付；
- 支付模块只更新支付状态，并通过订单接口更新订单支付结果；
- MVP 可不做真实渠道回调。

#### 对外边界

- 对订单模块：接收订单信息创建支付单，支付成功后同步订单；
- 对 C 端：提供支付单查询、Mock 支付；
- Admin MVP 阶段不需要支付运维后台。

#### 禁止范围

- 禁止 MVP 接入真实支付生产能力；
- 禁止 MVP 做退款；
- 禁止 MVP 做对账；
- 禁止支付模块扣库存；
- 禁止支付模块修改商品、购物车数据；
- 禁止无订单创建支付单。

## 4.7 文件服务域

### 4.7.1 核心业务流程

```text
管理员上传商品图片
  -> 校验文件类型和大小
  -> 保存文件
  -> 生成访问 URL
  -> 保存文件记录
  -> 商品配置引用 URL
```

### 4.7.2 业务边界

#### 核心职责

文件服务负责基础静态资源管理：

- 图片上传；
- 文件类型校验；
- 文件大小校验；
- 本地存储或简单对象存储预留；
- 访问 URL 生成；
- 文件记录保存。

#### 对内边界

- 文件服务只保存文件元信息和访问地址；
- 商品、用户头像只引用文件 URL；
- 文件服务不理解商品、订单业务规则；
- 文件删除和生命周期治理后续阶段增强。

#### 对外边界

- 对商品模块：提供商品图片上传；
- 对用户模块：提供头像上传预留；
- 对 Admin：提供后台素材上传。

#### 禁止范围

- 禁止匿名无权限上传；
- 禁止 MVP 做 CDN、防盗链、分片上传、私有访问；
- 禁止文件服务直接修改商品主数据。

## 4.8 Admin 后台域

### 4.8.1 核心业务流程

```text
管理员登录
  -> 进入后台首页
  -> 查看基础 Dashboard
  -> 管理分类、SPU、SKU
  -> 查询用户、订单、库存
  -> 调整基础库存
```

### 4.8.2 业务边界

#### 核心职责

Admin 后台负责 MVP 基础运营管理：

- 管理员登录；
- 商品分类管理；
- SPU 管理；
- SKU 管理；
- 用户基础查看；
- 订单基础查看；
- 库存查询与调整；
- 基础 Dashboard。

#### 对内边界

- Admin 后台应调用业务模块应用服务；
- Admin 不直接操作数据库；
- MVP 后台以基础运维为主；
- 细粒度权限、操作审计后续增强。

#### 对外边界

- 只向管理员开放；
- 与 C 端接口严格隔离；
- 后台操作必须进行管理员登录校验。

#### 禁止范围

- 禁止 MVP 做多角色细粒度权限；
- 禁止 MVP 做高危操作审批；
- 禁止 MVP 做售后审核、退款处理、对账处理；
- 禁止后台绕过业务规则直接改订单支付状态。

# 5 MVP 核心状态设计

## 5.1 用户状态

| 状态 | 含义 |
|---|---|
| ENABLED | 正常启用 |
| DISABLED | 禁用，MVP 可预留 |

## 5.2 商品状态

| 状态 | 含义 |
|---|---|

| 状态 | 含义 | MVP 说明 |

| 状态 | 含义 |
|---|---|
| PAY_PENDING | 待支付 |
| PAY_SUCCESS | 支付成功 |
| PAY_FAILED | 支付失败 |
| CLOSED | 已关闭 |

## 5.5 库存状态

| 状态 | 含义 |
|---|---|
| ACTIVE | 正常可用 |
| DISABLED | 不可用 |

# 8 MVP 数据一致性设计

## 8.1 下单一致性

创建订单涉及：

1. 校验用户登录态；
2. 校验地址；
3. 校验商品和 SKU；
4. 校验库存；
5. 创建订单；
6. 创建订单项；
7. 处理基础库存；
8. 创建支付单。

MVP 阶段建议使用数据库事务保证以上步骤一致性。任一环节失败，整个下单流程回滚。

## 8.2 支付一致性

Mock 支付成功时需要：

1. 校验支付单存在；
2. 校验支付单状态为待支付；
3. 更新支付单为支付成功；
4. 更新订单为已支付；
5. 写入支付时间。

MVP 可使用同一事务保证支付单和订单状态一致。

## 8.3 库存一致性

MVP 库存可以使用基础数据库事务和乐观锁控制：

- 下单时校验 `available_stock >= quantity`；
- 库存不足直接失败；
- 库存充足则扣减可售库存；
- 后续一期再引入库存锁单、超时释放、补偿和对账。

# 9 MVP 异常场景设计

| 异常场景 | MVP 处理策略 |
|---|---|
| 用户未登录访问购物车/订单/支付 | 返回未登录 |
| 管理员未登录访问后台 | 返回未登录或无权限 |
| 手机号已注册 | 注册失败 |
| 用户被禁用 | 禁止登录 |
| 商品已下架 | 禁止加购和下单 |
| SKU 不存在或下架 | 禁止加购和下单 |
| 库存不足 | 下单失败，提示库存不足 |
| 地址不存在或不属于当前用户 | 下单失败 |
| 购物车无勾选商品 | 不允许结算 |
| 订单不存在 | 返回订单不存在 |
| 用户查看他人订单 | 返回无权限 |
| 订单不是待支付 | 不允许取消或支付 |
| 支付单不存在 | 返回支付单不存在 |
| 支付单已成功 | 不允许重复支付 |
| 文件类型非法 | 上传失败 |
| 文件过大 | 上传失败 |

# 13 MVP 验收标准

MVP 完成后，应满足：

1. 用户可以注册并登录；
2. 用户可以维护个人资料；
3. 用户可以新增、编辑、删除、设置默认地址；
4. 后台管理员可以登录；
5. 后台可以维护分类、SPU、SKU；
6. 后台可以维护 SKU 库存；
7. C 端可以查看分类、商品列表、商品详情；
8. C 端可以选择 SKU 加入购物车；
9. 用户可以修改购物车数量和勾选状态；
10. 用户可以进行结算预览；
11. 用户可以选择地址提交订单；
12. 系统可以生成订单和订单项；
13. 订单项保存商品名称、图片、价格等快照；
14. 订单保存收货地址快照；
15. 下单时可以完成基础库存校验和处理；
16. 系统可以创建支付单；
17. 用户可以通过 Mock 支付完成支付；
18. 支付成功后支付单状态变为成功；
19. 支付成功后订单状态变为已支付；
20. 用户可以查看订单列表和订单详情；
21. 用户不能查看他人订单；
22. 后台可以查看订单、用户、库存；
23. C 端和 Admin 端核心流程可以完整联调；
24. 接口文档覆盖 MVP 核心接口；
25. MVP 主链路无阻断性异常。

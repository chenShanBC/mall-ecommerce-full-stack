# B2C电商三期平台经营能力开发主文档

> 本文档用于指导 `mallFei` 在完成 MVP、一阶段交易闭环、一阶段增强可靠性、二期运营履约能力之后，进入三期平台经营能力建设阶段。  
> 三期的核心目标不是继续补基础交易能力，而是从“能交易、能履约、能运营配置”进一步升级为“能做会员经营、用户分层、客服协同、内容增长、推荐转化、经营分析、风险识别”的企业级经营平台。

---

# 1 项目概述

## 1.1 阶段定位

三期阶段定位为：**平台经营能力增强阶段**。

在前置阶段中：

- MVP 阶段完成最小商城交易闭环；
- 一期基础阶段完成订单、库存、支付、售后、退款、对账、权限审计等基础企业级交易闭环；
- 一期增强阶段强化库存防超卖、支付异常核验、线上人工对账、Dashboard、认证风控等交易可靠性；
- 二期阶段补齐物流履约、退货物流、优惠券、基础营销、搜索、消息、评价、首页装修等运营履约体验。

三期阶段开始，平台建设重点从“流程可用”转向“经营增长”。系统需要具备对用户、商品、营销、客服、内容、推荐、数据和风险的持续运营能力。

三期核心建设方向包括：

- 会员等级；
- 成长值体系；
- 积分账户；
- 积分获取、消费、冻结、过期；
- 会员权益；
- 用户标签；
- 用户画像基础；
- 商品收藏；
- 浏览记录；
- 个性化推荐基础；
- 商品详情推荐、购物车推荐、猜你喜欢；
- 客服工单；
- 售后协商；
- FAQ 常见问题；
- 用户满意度评价；
- 内容运营增强；
- 专题页、活动页、商品问答；
- 营销增强，如新人优惠、秒杀、拼团基础模型；
- 经营 BI 报表；
- 用户、商品、营销、售后、搜索、会员分析；
- 财务清结算、发票、资金日报基础；
- 交易风控、营销风控、售后风控、风险事件。

三期完成后，平台应从“具备基础运营履约体验的电商系统”升级为“具备用户经营、服务协同、数据决策和风险识别能力的企业经营平台”。

## 1.2 阶段设计目标

### 1.2.1 业务目标

三期业务目标包括：

1. 用户完成注册、下单、评价、活动参与后，可以沉淀成长值、积分、标签和画像数据；
2. 平台可以按照用户消费金额、订单次数、成长值等维度划分会员等级；
3. 用户可以查看积分账户、积分明细、成长值明细和会员权益；
4. 平台可以配置会员等级、成长值规则、积分规则和基础会员权益；
5. 用户可以收藏商品、查看浏览记录，平台可以基于用户行为做基础推荐；
6. 商品详情页、购物车页、首页可以展示基础个性化或半个性化推荐结果；
7. 客服人员可以创建、处理、转交、关闭工单；
8. 售后争议场景可以引入客服协商记录和客服介入；
9. 平台可以维护 FAQ 常见问题，降低重复咨询成本；
10. 用户可以对客服服务和售后处理进行满意度评价；
11. 运营人员可以创建专题页、活动页、商品问答等内容运营资源；
12. 平台可以开展新人优惠、秒杀、拼团等基础增长型营销活动；
13. 管理者可以查看用户、商品、订单、营销、售后、搜索、会员等经营分析报表；
14. 财务人员可以查看资金日报、发票记录和基础清结算数据；
15. 系统能够识别交易、营销、售后中的基础风险事件，并沉淀风险日志。

### 1.2.2 技术目标

三期技术目标包括：

1. 保持二期已经确立的模块化单体 + DDD 四层架构；
2. 新增会员、积分、用户画像、客服工单、推荐、BI、风控、内容增强等模块时，继续保持清晰模块边界；
3. 用户行为数据、积分流水、成长值流水、风险事件、经营指标必须可追溯；
4. 积分、成长值、会员等级变更必须通过流水驱动，禁止直接覆盖结果；
5. 推荐系统初期可以基于规则和行为数据实现，但应预留算法化、模型化、异步计算扩展；
6. BI 报表初期可以基于业务库聚合或轻量统计表实现，但应预留后续数仓和指标体系；
7. 客服工单状态、售后状态、订单状态必须解耦，客服只协同处理，不直接破坏交易状态机；
8. 风控模块初期以规则识别和风险记录为主，不直接做复杂自动拦截；
9. 三期新增后台操作必须接入权限、操作日志、数据权限预留；
10. 三期新增能力必须保证不会破坏一期交易闭环和二期履约运营闭环。

## 1.3 三期建设原则

1. **先经营沉淀，后智能增长**：三期先沉淀会员、积分、标签、行为、工单、报表、风险数据，不急于做复杂智能算法。
2. **流水驱动资产变化**：积分、成长值、权益、风险记录、财务数据都必须有明细流水支撑。
3. **客服协同不越权交易**：客服可以记录协商、推动售后、备注订单，但不能绕过订单、支付、售后状态机。
4. **推荐先规则化，后算法化**：三期推荐以收藏、浏览、购买、分类、热销、相似商品等规则为主。
5. **BI 先轻量可用，后数仓治理**：三期先提供经营看板和统计报表，四期再做完整指标治理和数据仓库。
6. **风控先识别记录，后自动处置**：三期风控以风险事件识别、标记、人工处置为主，复杂规则引擎放到四期。
7. **内容经营与交易解耦**：专题页、问答、内容运营只引导转化，不直接修改订单、库存、支付等核心交易数据。

# 2 系统架构设计

## 2.1 核心架构设计原则

三期继续沿用模块化单体 + DDD 四层架构：

```text
interfaces 接口层
  -> application 应用服务层
  -> domain 领域层
  -> infrastructure 基础设施层
```

每个业务模块内部继续保持：

```text
xxx-module/
├── interfaces           // Controller、Request、Response、Assembler
├── application          // 应用服务、Command、Query、DTO、应用事件
├── domain               // 领域模型、领域服务、仓储接口、领域事件、枚举
├── infrastructure       // 持久化、外部系统适配、配置、转换
└── facade               // 对其他模块暴露的防腐接口
```

三期比二期新增更多经营型模块，因此必须重点控制模块边界，避免会员、推荐、BI、风控等能力反向侵入订单、支付、库存、售后等交易核心。

## 2.2 三期新增与增强领域模块

| 模块 | 模块定位 | 三期职责 |
|---|---|---|
| mall-member | 会员模块 | 会员等级、成长值、会员权益、等级规则 |
| mall-points | 积分模块 | 积分账户、积分流水、积分获取、消费、冻结、过期 |
| mall-user-profile | 用户画像模块 | 用户标签、用户分层、行为聚合、基础画像 |
| mall-favorite | 收藏足迹模块 | 商品收藏、浏览记录、行为记录 |
| mall-recommend | 推荐模块 | 规则推荐、猜你喜欢、相似商品、推荐效果统计 |
| mall-customer-service | 客服工单模块 | 工单、客服介入、协商记录、FAQ、满意度 |
| mall-content | 内容运营增强 | 专题页、活动页、商品问答、内容效果统计 |
| mall-marketing | 营销增强 | 新人优惠、秒杀、拼团基础模型、营销效果分析 |
| mall-bi | 数据分析模块 | 用户、商品、营销、售后、搜索、会员分析报表 |
| mall-finance | 财务经营模块 | 发票、资金日报、基础清结算、月结报表 |
| mall-risk | 风控模块 | 交易风控、营销风控、售后风控、风险事件 |
| mall-admin-auth | 权限增强 | 数据权限、组织岗位、审批流预留 |
| mall-file | 文件服务增强 | 素材中心、图片处理、私有访问预留 |

## 2.3 三期模块依赖关系

三期推荐依赖方向：

```text
C端 / Admin端
  -> 会员 / 积分 / 客服 / 推荐 / BI / 风控 / 内容 / 营销等应用服务
  -> 各自领域模型与领域服务
  -> 基础设施实现
```

关键依赖原则：

- 会员模块可以读取用户、订单、积分、成长值结果，但不直接创建订单；
- 积分模块接收订单完成、评价完成、活动参与等事件生成积分流水，不直接修改订单状态；
- 用户画像模块消费用户行为、订单、收藏、浏览、搜索、评价等数据，不直接控制交易流程；
- 推荐模块读取用户画像、商品、行为数据，输出推荐结果，不修改商品主数据；
- 客服模块可以关联订单、售后、用户，但不直接执行退款、发货、支付；
- BI 模块消费业务数据形成指标，不反向驱动核心业务状态；
- 风控模块识别风险和记录风险事件，三期默认不直接强拦截交易；
- 财务经营模块读取支付、退款、对账、发票数据，不直接执行支付退款。

# 4 三期核心业务详细设计

## 4.1 会员与成长值域业务设计

### 4.1.1 核心业务流程

#### 会员等级配置流程

```text
运营人员创建会员等级
  -> 配置等级名称、等级图标、成长值门槛、权益说明
  -> 启用等级
  -> 系统按成长值计算用户当前等级
  -> 用户会员中心展示等级与权益
```

#### 成长值获取流程

```text
用户完成订单 / 完成评价 / 参与活动
  -> 业务模块发布事件
  -> 会员模块匹配成长值规则
  -> 生成成长值流水
  -> 更新用户成长值余额
  -> 重新计算会员等级
  -> 生成会员变更记录
```

### 4.1.2 业务边界

#### 核心职责

会员模块负责会员等级、成长值和会员权益基础能力。

- 会员等级配置；
- 成长值规则配置；
- 成长值获取、扣减、调整；
- 成长值流水；
- 用户会员等级计算；
- 会员权益展示；
- 会员等级变更记录；
- 会员中心基础信息。

#### 对内边界

- 成长值变化必须通过流水产生；
- 会员等级应由成长值、等级规则计算得出，不应人工随意覆盖；
- 等级规则变更应记录版本和操作日志；
- 用户会员等级变化需要记录变更前后等级；
- 会员权益三期以展示和基础识别为主，不直接承诺复杂权益自动履约。

#### 对外边界

- 对订单模块：接收订单完成事件，计算购物成长值；
- 对评价模块：接收评价完成事件，计算评价成长值；
- 对营销模块：提供会员等级用于活动适用范围判断；
- 对用户画像模块：输出会员等级、成长值区间等标签；
- 对 C 端：提供会员中心、等级权益展示；
- 对 Admin：提供等级、规则、权益配置能力。

#### 禁止范围

- 禁止直接创建订单；
- 禁止直接修改订单金额；
- 禁止直接执行优惠抵扣；
- 禁止绕过成长值流水直接改等级；
- 禁止在会员模块中实现复杂营销活动逻辑；
- 禁止直接执行退款或售后处理。

## 4.2 积分域业务设计

### 4.2.1 核心业务流程

#### 积分获取流程

```text
业务事件发生
  -> 积分模块匹配积分规则
  -> 判断用户是否满足获取条件
  -> 生成积分流水
  -> 更新积分账户可用余额
  -> 用户积分明细可查询
```

#### 积分消费流程

```text
用户在结算页选择积分抵扣
  -> 订单模块调用积分试算
  -> 积分模块返回可用积分和抵扣金额
  -> 创建订单时冻结积分
  -> 支付成功后积分正式消费
  -> 订单关闭则解冻积分
```

### 4.2.2 业务边界

#### 核心职责

积分模块负责用户积分资产管理。

- 积分账户；
- 积分规则；
- 积分获取；
- 积分消费；
- 积分冻结；
- 积分解冻；
- 积分过期；
- 积分调整；
- 积分流水；
- 积分余额查询。

#### 对内边界

- 积分账户余额必须由积分流水汇总或同步更新；
- 积分消费前必须冻结，支付成功后再正式扣减；
- 订单关闭、支付失败、售后取消时应释放冻结积分；
- 积分过期必须生成过期流水；
- 后台调整积分必须记录操作人、原因和审批预留；
- 积分抵扣金额规则必须可配置且可追溯。

#### 对外边界

- 对订单模块：提供积分抵扣试算、冻结、确认消费、解冻能力；
- 对支付模块：接收支付成功或失败结果，确认积分消费或释放；
- 对售后模块：提供积分退回或扣回能力；
- 对会员模块：积分行为可作为会员成长依据；
- 对 C 端：提供积分账户和明细查询；
- 对 Admin：提供积分规则、调整、流水查询能力。

#### 禁止范围

- 禁止创建订单；
- 禁止执行支付；
- 禁止直接退款；
- 禁止直接修改订单实付金额；
- 禁止无流水修改积分余额；
- 禁止积分抵扣规则散落在订单模块中；
- 禁止售后退款时重新计算历史积分规则。

## 4.3 用户标签与画像域业务设计

### 4.3.1 核心业务流程

```text
用户注册 / 浏览 / 搜索 / 收藏 / 下单 / 评价 / 售后
  -> 行为数据进入画像模块
  -> 系统按标签规则计算标签
  -> 生成或更新用户标签
  -> 形成用户基础画像
  -> 推荐、营销、客服、BI 使用画像结果
```

### 4.3.2 业务边界

#### 核心职责

用户画像模块负责用户经营数据沉淀。

- 用户标签；
- 用户分层；
- 行为聚合；
- 消费能力识别；
- 活跃度识别；
- 偏好分类识别；
- 风险标签引用；
- 会员标签引用；
- 用户画像查询。

#### 对内边界

- 标签应区分系统标签、人工标签、行为标签、交易标签、风险标签、会员标签；
- 自动标签应有计算规则和更新时间；
- 人工标签应记录操作人和原因；
- 画像结果应允许延迟更新，不要求所有行为实时计算；
- 标签删除、覆盖、调整需要保留操作日志。

#### 对外边界

- 对推荐模块：提供偏好标签、行为特征；
- 对营销模块：提供用户分层、会员标签、活跃标签；
- 对客服模块：提供用户画像摘要；
- 对风控模块：接收风险标签或输出用户基础信息；
- 对 BI 模块：提供用户标签统计；
- 对 Admin：提供标签管理和用户画像查询。

#### 禁止范围

- 禁止直接修改订单；
- 禁止直接修改会员等级；
- 禁止直接控制推荐结果强制排序；
- 禁止直接执行营销发券；
- 禁止将临时行为数据不加规则地永久写成标签；
- 禁止画像模块承载交易核心规则。

## 4.4 收藏足迹与推荐域业务设计

### 4.4.1 核心业务流程

```text
用户浏览商品 / 收藏商品 / 加购 / 下单
  -> 记录用户行为
  -> 推荐模块基于行为、商品分类、热销、相似商品生成推荐候选
  -> C端首页、商品详情、购物车展示推荐商品
  -> 记录推荐曝光和点击
  -> 形成推荐效果统计
```

### 4.4.2 业务边界

#### 核心职责

推荐相关模块负责用户行为记录和基础推荐结果生成。

- 商品收藏；
- 取消收藏；
- 浏览记录；
- 行为事件记录；
- 相似商品推荐；
- 热销商品推荐；
- 基于分类偏好的推荐；
- 猜你喜欢；
- 推荐曝光记录；
- 推荐点击记录；
- 推荐效果统计。

#### 对内边界

- 浏览记录可以异步写入，不应阻塞商品详情主流程；
- 收藏状态需要保证用户维度唯一；
- 推荐结果三期以规则生成和缓存为主；
- 推荐曝光、点击、转化需要可追踪；
- 推荐规则需要支持启用、停用、排序权重和生效时间。

#### 对外边界

- 对商品模块：读取商品公开信息，不维护商品主数据；
- 对用户画像模块：读取用户偏好标签和行为特征；
- 对搜索模块：可使用热搜、搜索行为作为推荐参考；
- 对 C 端：首页、详情、购物车、订单完成页提供推荐结果；
- 对 BI 模块：输出推荐曝光、点击、转化数据。

#### 禁止范围

- 禁止修改商品价格；
- 禁止修改商品库存；
- 禁止直接创建订单；
- 禁止把推荐结果作为商品可售性的判断依据；
- 禁止在推荐模块中实现复杂搜索查询逻辑；
- 禁止三期引入不可解释的复杂模型强依赖主流程。

## 4.5 客服与工单域业务设计

### 4.5.1 核心业务流程

#### 普通工单流程

```text
用户发起咨询 / 后台创建工单
  -> 系统生成工单
  -> 分配客服
  -> 客服处理并记录沟通内容
  -> 必要时关联订单或售后单
  -> 用户确认或客服关闭
  -> 用户进行满意度评价
```

#### 售后协商流程

```text
用户售后产生争议
  -> 客服介入
  -> 记录协商过程
  -> 给出售后处理建议
  -> 售后模块根据规则继续审核、退款或关闭
  -> 工单记录归档
```

### 4.5.2 业务边界

#### 核心职责

客服工单模块负责用户问题协同处理和服务记录沉淀。

- 工单创建；
- 工单分类；
- 工单优先级；
- 工单分配；
- 工单处理；
- 沟通记录；
- 客服备注；
- 售后协商记录；
- 客服介入；
- 工单 SLA；
- 超时提醒；
- FAQ；
- 满意度评价；
- 客服绩效基础统计。

#### 对内边界

- 工单状态必须按照 CREATED、ASSIGNED、PROCESSING、WAIT_USER、RESOLVED、CLOSED 等状态合法流转；
- 工单处理记录必须保留处理人、处理时间和内容；
- 售后协商只记录过程和建议，不直接覆盖售后审核结果；
- 工单 SLA 只负责提醒和统计，不自动修改售后、订单、退款状态；
- 客服备注需要区分内部可见和用户可见。

#### 对外边界

- 对用户模块：读取用户基础信息和联系方式；
- 对订单模块：关联订单，查看订单摘要，不直接改订单；
- 对售后模块：关联售后单，记录协商建议；
- 对消息模块：发送工单回复、处理完成、超时提醒消息；
- 对 BI 模块：输出工单量、响应时长、解决时长、满意度指标；
- 对 Admin：提供客服工作台和工单管理能力。

#### 禁止范围

- 禁止直接执行退款；
- 禁止直接发货；
- 禁止直接修改支付状态；
- 禁止直接关闭订单；
- 禁止绕过售后模块批准售后；
- 禁止将客服备注作为财务退款依据；
- 禁止工单模块直接修改库存。

## 4.6 内容运营增强域业务设计

### 4.6.1 核心业务流程

```text
运营创建专题页 / 活动页 / 商品问答
  -> 配置内容、商品、图片、跳转链接、生效时间
  -> 审核或直接启用

  -> 记录曝光、点击和转化

```

### 4.6.2 业务边界

#### 核心职责

内容运营模块负责内容化导购与运营转化。

- 专题页；
- 活动页；
- 商品问答；
- 内容素材关联；
- 内容上下线；
- 内容排序；
- 内容曝光；
- 内容点击；
- 内容转化统计。

#### 对内边界

- 内容配置应有草稿、启用、停用、过期状态；
- 内容引用商品 ID，不复制商品主数据；
- 商品问答需要区分用户提问、官方回答、审核状态；
- 内容效果统计允许异步聚合；
- 素材文件应通过文件服务管理，内容模块保存引用关系。

#### 对外边界

- 对商品模块：读取商品公开展示信息；
- 对文件模块：引用素材图片、视频等文件；
- 对推荐模块：可提供运营推荐位；
- 对 BI 模块：输出内容曝光、点击、转化数据；
- 对 C 端：提供专题页、活动页、商品问答展示。

#### 禁止范围

- 禁止修改商品价格；
- 禁止修改库存；
- 禁止创建订单；
- 禁止执行营销优惠核销；
- 禁止直接决定搜索排序；
- 禁止用内容状态替代商品上下架状态。

## 4.7 营销增强域业务设计

### 4.7.1 核心业务流程

```text
运营创建增长活动
  -> 配置活动类型、商品范围、时间、库存、预算、参与限制
  -> 活动预热
  -> 用户参与活动
  -> 营销模块校验资格并计算优惠
  -> 订单保存活动快照
  -> 支付成功后记录活动效果
```

三期营销增强包括：

- 新人优惠；
- 秒杀基础模型；
- 拼团基础模型；
- 活动预算；
- 活动效果分析；
- 营销人群圈选预留。

### 4.7.2 业务边界

#### 核心职责

营销增强模块负责增长型活动规则和效果统计。

- 新人优惠规则；
- 秒杀活动配置；
- 拼团活动基础配置；
- 活动商品范围；
- 活动库存或名额；
- 活动预算；
- 活动资格校验；
- 活动优惠计算；
- 活动快照；
- 活动效果统计。

#### 对内边界

- 活动规则必须有生效时间、失效时间和状态；
- 活动库存或名额需要并发控制；
- 活动优惠必须保存到订单快照；
- 活动预算消耗需要记录明细；
- 秒杀、拼团三期只做基础模型，不做高并发极限优化；
- 活动结束后不能影响历史订单优惠。

#### 对外边界

- 对订单模块：提供活动校验、优惠计算和活动快照；
- 对库存模块：校验活动库存或商品库存；
- 对会员模块：可基于会员等级限制活动资格；
- 对用户画像模块：可基于用户标签筛选参与人群；
- 对 BI 模块：输出活动曝光、参与、转化、成本数据。

#### 禁止范围

- 禁止直接创建订单；
- 禁止直接支付；
- 禁止直接退款；
- 禁止绕过库存模块扣减库存；
- 禁止三期承诺生产级秒杀高并发能力；
- 禁止修改历史订单活动优惠快照。

## 4.8 BI经营分析域业务设计

### 4.8.1 核心业务流程

```text
业务数据产生
  -> 统计任务或事件流采集数据
  -> 聚合生成指标
  -> 写入统计表
  -> Admin经营看板展示
  -> 管理者按时间、渠道、类目、会员等级等维度分析
```

三期 BI 分析范围包括：

- 用户分析；
- 商品分析；
- 订单分析；
- 营销分析；
- 售后分析；
- 搜索分析；
- 会员分析；
- 客服分析；
- 推荐分析；
- 内容分析。

### 4.8.2 业务边界

#### 核心职责

BI 模块负责经营指标聚合、查询和展示。

- 指标定义；
- 统计任务；
- 经营看板；
- 趋势分析；
- 排行榜；
- 漏斗分析基础；
- 用户分层统计；
- 活动效果统计；
- 售后原因统计；
- 客服效率统计。

#### 对内边界

- 三期 BI 可以采用轻量统计表，不强制建设完整数仓；
- 指标口径必须在文档或配置中明确；
- 统计结果允许 T+1 或准实时，不要求所有指标实时；
- 统计任务失败需要记录失败原因；
- 指标修正需要记录操作和原因。

#### 对外边界

- 从订单、商品、用户、营销、售后、搜索、会员、客服等模块读取或接收数据；
- 向 Admin 提供经营看板和报表查询；
- 向运营提供活动效果、用户分层、商品表现等分析；
- 向管理层提供核心经营指标。

#### 禁止范围

- 禁止 BI 模块反向修改业务状态；
- 禁止在 BI 模块中执行退款、发货、核销；
- 禁止使用未定义口径的指标做正式报表；
- 禁止将统计表作为交易事实唯一来源；
- 禁止在核心交易链路中同步阻塞等待 BI 统计完成。

## 4.9 财务经营域业务设计

### 4.9.1 核心业务流程

```text
订单支付 / 退款 / 对账完成
  -> 财务模块聚合资金数据
  -> 生成资金日报
  -> 用户申请发票
  -> 后台审核并开票
  -> 月末生成基础月结报表
```

### 4.9.2 业务边界

#### 核心职责

财务经营模块负责经营层财务数据展示和发票基础能力。

- 资金日报；
- 支付金额统计；
- 退款金额统计；
- 手续费统计预留；
- 发票申请；
- 发票审核；
- 发票开具记录；
- 月结报表基础；
- 清结算数据汇总。

#### 对内边界

- 财务数据应来源于支付、退款、对账和订单事实；
- 发票金额不得超过可开票金额；
- 发票状态变化必须记录操作日志；
- 资金日报允许按日聚合；
- 月结报表三期只做基础汇总，不做完整财务总账。

#### 对外边界

- 对订单模块：读取订单金额、商品明细、用户信息；
- 对支付模块：读取支付流水；
- 对退款模块：读取退款流水；
- 对对账模块：读取对账差异和归档结果；
- 对 C 端：提供发票申请和发票查询；
- 对 Admin：提供发票审核、资金日报、月结报表。

#### 禁止范围

- 禁止直接执行支付；
- 禁止直接执行退款；
- 禁止直接修改对账结果；
- 禁止修改订单金额；
- 禁止以财务统计结果反向覆盖支付事实；
- 禁止三期承诺完整财务总账能力。

## 4.10 风控域业务设计

### 4.10.1 核心业务流程

```text
用户登录 / 下单 / 领券 / 退款 / 售后
```

- 售后风险识别；
- 高频下单识别；
- 高频领券识别；
- 高频退款识别；
- 异常售后识别；
- 风险事件记录；
- 风险处置记录；
- 风控看板基础。

### 4.10.2 业务边界

#### 核心职责

风控模块负责风险识别、记录和人工处置闭环。

- 风控规则配置；
- 风险事件识别；
- 风险等级；
- 风险日志；
- 风险处置；
- 黑名单引用或维护预留；
- 风控看板；
- 风险标签输出。

#### 对内边界

- 三期风控以规则识别和事件记录为主；
- 风险规则需要有启用、停用、阈值和适用场景；
- 风险事件需要记录触发规则、业务对象、用户、时间和风险等级；
- 人工处理需要记录处理结果、处理人和原因；
- 风险标签输出给用户画像模块，但不直接改变会员等级。

#### 对外边界

- 对订单模块：接收下单、取消、异常订单事件；
- 对营销模块：接收领券、活动参与、优惠使用事件；
- 对售后模块：接收退款、退货、售后频次事件；
- 对用户模块：引用用户状态、黑名单等信息；
- 对客服模块：可将风险事件转工单；
- 对 BI 模块：输出风险统计数据。

#### 禁止范围

- 三期默认禁止直接强拦截交易，除非业务明确配置；
- 禁止直接修改订单状态；
- 禁止直接执行退款拒绝；
- 禁止直接冻结用户资产；
- 禁止把风控规则写死在订单或营销模块；
- 禁止没有风险日志的人工处置。

# 5 三期标准业务闭环流程

## 5.1 会员积分成长闭环

```text
用户完成订单
  -> 订单发布完成事件
  -> 积分模块计算购物积分
  -> 会员模块计算成长值
  -> 生成积分流水和成长值流水
  -> 更新积分账户与会员等级
  -> 用户会员中心展示最新权益
```

## 5.2 积分抵扣订单闭环

```text
用户进入结算页
  -> 查询可用积分
  -> 选择积分抵扣
  -> 创建订单时冻结积分
  -> 支付成功后确认消费积分
  -> 订单关闭时解冻积分
  -> 售后退款时按规则退回或扣回积分
```

## 5.3 用户行为推荐闭环

```text
用户浏览 / 收藏 / 搜索 / 加购商品
  -> 记录行为数据
  -> 用户画像更新偏好标签
  -> 推荐模块生成推荐结果
  -> C端展示推荐商品
  -> 记录曝光、点击、转化
  -> BI统计推荐效果
```

## 5.4 客服售后协同闭环

```text
用户发起售后争议
  -> 创建客服工单
  -> 客服记录协商过程
  -> 售后模块根据协商建议继续处理
  -> 工单关闭
  -> 用户评价服务满意度
  -> BI统计客服效率和满意度
```

## 5.5 内容运营转化闭环

```text
运营创建专题页
  -> 配置商品和内容素材
  -> C端展示专题页
  -> 用户点击并进入商品详情
  -> 用户下单支付
  -> 内容模块记录转化
  -> BI分析内容效果
```

## 5.6 风险识别处置闭环

```text
用户高频领券 / 高频退款 / 异常下单
  -> 风控模块匹配规则
  -> 生成风险事件
  -> 后台人工确认
  -> 标记处理结果
  -> 用户画像沉淀风险标签
  -> BI展示风险趋势
```

# 6 C端页面与交互范围

## 6.1 会员中心

- 当前会员等级；

# 11 验收标准

三期完成后，应满足以下验收标准：

1. 后台可以配置会员等级、成长值规则和会员权益；
2. 用户完成订单后可以获得成长值，并自动计算会员等级；
3. 用户可以在会员中心查看等级、成长值、权益和升级进度；
4. 后台可以配置积分规则；
5. 用户可以获得积分、查看积分明细；
6. 订单结算时可以使用积分抵扣，并能正确冻结、消费、解冻；
7. 售后退款时积分能够按规则退回或扣回；
8. 用户可以收藏商品和查看浏览记录；
9. 系统可以基于行为、分类、热销、相似商品生成基础推荐；
10. 首页、商品详情、购物车等场景可以展示推荐商品；
11. 推荐曝光、点击、转化可以统计；
12. 用户标签和基础画像可以沉淀并被查询；
13. 后台可以人工打标和查看用户画像；
14. 用户可以创建客服工单；
15. 客服可以处理、转交、关闭工单；
16. 售后争议可以关联客服协商记录；
17. FAQ 可以在 C 端展示并由后台维护；
18. 用户可以提交客服满意度评价；
19. 后台可以创建专题页、活动页、商品问答；
20. C 端可以展示专题页、活动页、商品问答；
21. 后台可以配置新人优惠、秒杀、拼团基础活动；
22. 活动订单可以保存活动快照和效果数据；
23. Admin 可以查看用户、商品、订单、营销、售后、搜索、会员、客服、推荐、内容分析报表；
24. 财务人员可以查看资金日报、发票申请、发票记录和基础月结报表；
25. 系统可以识别高频领券、高频退款、异常售后等基础风险事件；
26. 风险事件可以被后台人工处理并记录结果；
27. 三期新增能力不破坏一期交易闭环和二期履约运营闭环；
28. 三期关键后台操作具备权限控制和操作日志。

# 13 三期完成后的平台能力变化

三期完成后，`mallFei` 应具备以下能力：

- 从“基础运营履约系统”升级为“企业经营平台”；
- 从“交易用户”升级为“可分层、可画像、可运营的会员用户”；
- 从“简单优惠”升级为“积分、成长值、会员权益、活动增长协同”；
- 从“商品展示”升级为“收藏、足迹、推荐、内容导购”；
- 从“售后处理”升级为“客服工单、协商记录、满意度闭环”；
- 从“后台查询”升级为“经营分析、财务日报、风险事件管理”；
- 从“人工经验运营”升级为“基于数据和标签的运营决策”。

三期是平台从电商交易系统走向企业经营系统的关键阶段，也是四期生产治理、数据仓库、指标治理、监控告警、安全合规和智能化演进的业务基础。

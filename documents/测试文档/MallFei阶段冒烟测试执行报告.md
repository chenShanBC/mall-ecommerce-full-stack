# MallFei 阶段冒烟测试执行报告

## 一、报告概述

本报告用于记录 MallFei 商城项目本次阶段化冒烟测试脚本的编写、修正与执行结果。

本次测试根据《mallFei 项目已实现业务功能阶段性梳理文档》中对业务能力的阶段划分，将后端接口冒烟测试拆分为两个脚本：

- MVP 阶段冒烟测试脚本：`backend/scripts/smoke-test-mvp.ps1`
- 一期及一期增强阶段冒烟测试脚本：`backend/scripts/smoke-test-phase1-enhanced.ps1`

测试目标是验证当前后端服务在本地 `9090` 端口启动后，核心业务接口是否能够正常访问，主链路是否可用，阶段增强能力接口是否具备基本可调用性。

## 四、脚本修正记录

### 4.1 服务端口修正

初始脚本默认使用 `8080` 端口，但当前后端服务实际运行在 `9090` 端口。

已将两个脚本默认地址统一修正为：

```powershell
[string]$BaseUrl = "http://localhost:9090"
```

### 4.2 鉴权请求头修正

初始版本中曾尝试同时传递：

```powershell
Authorization: <token>
satoken: <token>
```

实际执行发现后端当前接口需要：

```powershell
Authorization: Bearer <token>
```

如果同时传递 `Authorization` 和 `satoken`，后端可能取到错误 token，导致已登录接口返回 `AUTH_401`。

已修正为只传递：

```powershell
$headers["Authorization"] = "Bearer $Token"
```

### 4.3 Windows PowerShell 编码修正

由于脚本包含中文输出，Windows PowerShell 5.1 在读取 UTF-8 无 BOM 文件时可能将中文字符串解析错误，导致脚本语法报错。

已将脚本重新保存为 UTF-8 with BOM，保证 Windows PowerShell 能够正常执行。

说明：终端显示仍可能受控制台代码页影响出现中文乱码，但不影响脚本执行。

### 4.4 兜底注册手机号修正

MVP 脚本中，当默认用户登录因验证码策略失败时，会自动注册新用户作为兜底。

原先生成手机号长度不符合 11 位规则，导致地址接口校验手机号格式失败。

已修正为生成 11 位手机号，地址相关接口已通过。

#### 一期及增强脚本 SKIP 说明

一期及一期增强阶段脚本 SKIP 数量较多，主要原因是该阶段大量接口属于状态敏感型接口或依赖特定前置业务数据。

常见跳过原因包括：

| 类型 | 说明 | 是否影响冒烟结论 |
|---|---|---|
| 登录验证码策略 | C 端默认密码登录需要验证码校验，未传验证码时返回业务校验失败 | 不影响 Admin 侧和增强接口测试 |
| 支付宝配置缺失 | 本地未配置支付宝 appId、商户私钥、支付宝公钥等参数 | 不影响 Mock 支付和后台支付管理 |
| 订单状态不匹配 | 支付异常修复、恢复待支付、关闭释放库存等接口需要订单处于特定状态 | 不影响接口可调用性判断 |
| 售后状态不匹配 | 售后审核需要待审核售后单，当前售后单状态不满足时跳过 | 不影响售后列表和详情接口 |
| 退款单缺失 | 退款同步需要传入有效 `RefundNo` | 不影响退款列表查询 |
| 库存操作 DTO 字段不完全匹配 | 部分库存内部操作接口对业务类型、业务单号、金额等字段有强校验 | 需要后续按 DTO 精确补充请求体 |
| 对账差异数据不足 | 线上对账任务未生成有效差异时，差异详情、挂账跟进等需要具体 `DiffId` 的接口跳过 | 不影响任务、账单、报表类接口 |
| 本地账单为空 | 当前任务日期范围内无满足条件的支付/退款流水时，账单生成或自动勾兑会业务失败 | 不影响对账工作台接口可访问性 |

## 八、风险与后续建议

### 7.1 当前风险

1. **C 端密码登录已受验证码策略影响**
   - 默认账号密码登录无法直接冒烟通过，需要验证码校验凭证。
   - 当前 MVP 脚本已通过注册新用户方式兜底。

2. **一期及增强写操作依赖业务状态**
   - 支付异常处理、订单恢复待支付、关闭释放库存、售后审核、退款同步等接口并非任意状态均可执行。
   - 冒烟脚本不能简单按固定顺序强制调用全部写操作。

3. **部分增强能力需要专项数据集**
   - 对账差异、库存差异、退款异常、挂账跟进等能力需要构造特定异常数据。
   - 当前脚本更适合验证接口可访问性和基本入口，不等同于完整业务验收测试。

4. **Windows 控制台中文输出乱码**
   - 脚本编码已修复，但终端显示受代码页影响仍可能乱码。
   - 不影响执行结果，可后续考虑将输出改为英文或执行前设置控制台编码。

### 7.2 后续建议

1. **保留当前两个阶段化冒烟脚本**
   - `smoke-test-mvp.ps1` 作为主链路冒烟脚本。
   - `smoke-test-phase1-enhanced.ps1` 作为增强能力入口检查脚本。

2. **补充专项测试数据构造脚本**
   - 支付异常数据构造。
   - 退款异常数据构造。
   - 库存差异数据构造。
   - 线上对账差异数据构造。
   - 挂账跟进数据构造。

3. **将增强脚本拆分为更小的专项脚本**
   - 支付与退款专项冒烟。
   - 库存与库存对账专项冒烟。
   - 售后退款专项冒烟。
   - 线上人工对账专项冒烟。
   - Dashboard 与权限审计专项冒烟。

4. **在 CI 或部署后验证中优先运行 MVP 脚本**
   - MVP 脚本通过率高，能够快速判断核心交易链路是否可用。
   - 一期增强脚本适合人工验证或预发环境专项验证。

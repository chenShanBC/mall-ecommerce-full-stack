# 阶段范围与验收标准说明

## 1. 文档目的

本文用于在项目启动前定义 `mallFei` 大型企业级电商系统各建设阶段的目标、范围、交付内容和验收标准。

本文属于项目先行设计文档，不描述当前实现进度。后续开发过程中，应以本文作为阶段规划和验收依据，判断某一阶段是否达到可交付标准。

---

## 2. 阶段划分总览

项目整体分为以下阶段：

```text
MVP 阶段
  -> 一期基础企业级交易闭环
  -> 一期增强能力
  -> 二期运营履约能力
  -> 三期平台经营能力
  -> 四期生产治理能力
```

各阶段定位如下：

| 阶段 | 阶段定位 | 核心目标 |
|---|---|---|
| MVP 阶段 | 最小可用商城交易闭环 | 验证用户能浏览商品、加购、下单、支付、查看订单 |
| 一期基础 | 基础企业级交易闭环 | 补齐售后、退款、库存预占、订单超时、基础对账和后台权限 |
| 一期增强 | 交易可靠性增强 | 强化库存防超卖、支付核验、对账闭环、权限审计和 Dashboard |
| 二期 | 运营履约能力补齐 | 补齐物流、营销、搜索、消息、评价等运营能力 |
| 三期 | 平台经营能力增强 | 建设会员、积分、BI、客服、内容、风控等经营能力 |
| 四期 | 生产治理能力完善 | 建设 CI/CD、测试、监控、高可用、安全合规和容量治理 |

---

## 3. MVP 阶段范围

## 3.1 阶段目标

MVP 阶段用于验证最小可用商城交易链路。

核心流程为：

```text
用户注册/登录
  -> 浏览商品
  -> 加入购物车
  -> 结算确认
  -> 创建订单
  -> 完成支付
  -> 查看订单
```

MVP 阶段不追求复杂企业能力，而是确认系统最基本的交易闭环是否成立。

## 3.2 MVP 包含范围

### 用户与认证

- 用户注册；
- 用户登录；
- 用户登录态；
- 用户资料基础维护；
- 用户地址管理；
- 管理员基础登录。

### 商品

- 商品分类；
- 商品 SPU；
- 商品 SKU；
- 商品列表；
- 商品详情；
- 商品基础上下架；
- 后台商品维护。

### 购物车

- 加入购物车；
- 修改数量；
- 删除购物车项；
- 勾选商品；
- 结算预览。

### 订单

- 创建订单；
- 订单列表；
- 订单详情；
- 订单基础状态；
- 商品快照。

### 库存

- SKU 库存记录；
- 下单基础库存处理；
- 取消订单释放库存；
- 后台库存基础调整。

### 支付

- 支付单创建；
- Mock 支付或支付沙箱最小链路；
- 支付结果同步订单；
- 支付返回页。

### C 端商城

- 首页；
- 商品详情；
- 购物车；
- 结算页；
- 订单列表；
- 订单详情；
- 登录页；
- 个人中心；
- 地址管理。

### Admin 后台

- 后台登录；
- 商品基础管理；
- 订单基础管理；
- 库存基础管理；
- 用户基础查看；
- 基础 Dashboard。

## 3.3 MVP 不包含范围

MVP 阶段明确不包含：

- 售后；
- 退款；
- 对账；
- 复杂库存补偿；
- 支付异常核验；
- 物流履约；
- 营销活动；
- 优惠券；
- 搜索；
- 消息通知；
- 商品评价；
- 会员积分；
- 客服工单；
- 风控体系；
- BI 报表；
- 完整 CI/CD；
- 生产级监控告警。

## 3.4 MVP 验收标准

MVP 阶段完成后，应满足以下标准：

1. 用户能够完成注册和登录；
2. 用户能够维护收货地址；
3. 用户能够浏览商品列表和商品详情；
4. 用户能够选择 SKU 并加入购物车；
5. 用户能够在购物车中修改数量和选择结算商品；
6. 用户能够提交订单；
7. 系统能够生成订单和订单项；
8. 系统能够处理基础库存扣减或占用；
9. 用户能够通过 Mock 或沙箱支付完成支付；
10. 支付成功后订单状态能够更新；
11. 用户能够查看订单列表和订单详情；
12. 后台能够维护商品、查看订单、查看用户和调整库存；
13. C 端和 Admin 端均可完成主链路联调。

## 4. 一期基础企业级交易闭环范围

## 4.1 阶段目标

一期基础阶段在 MVP 基础上补齐真实电商交易中最基础的企业级保障能力。

一期重点解决：

- 下单后未支付如何处理；
- 库存如何预占和释放；
- 支付成功后如何同步订单；
- 用户退款如何处理；
- 售后如何审核；
- 订单、支付、退款状态如何基础核验；
- 后台操作如何留痕；
- 后台权限如何管理。

## 4.2 一期包含范围

### 订单增强

- 待支付订单超时取消；
- 自动确认收货；
- 订单退款状态；
- 订单后台异常处理。

### 库存增强

- 库存预占；
- 库存锁定记录；
- 订单取消释放库存；
- 订单超时释放库存；
- 库存操作日志。

### 支付增强

- 第三方支付沙箱接入；
- 支付回调记录；
- 支付状态同步；
- 支付退款；
- 退款结果同步。

### 售后基础

- 用户发起退款申请；
- 售后单创建；
- 后台售后审核；
- 审核通过触发退款；
- 审核驳回；
- 售后状态流转。

### 基础对账

- 订单与支付状态核验；
- 退款状态核验；
- 支付成功但订单未更新的差异识别；
- 订单取消但支付成功的差异识别；
- 后台人工处理入口。

### 后台权限与审计

- 后台账号管理；
- 角色管理；
- 权限管理；
- 操作日志；
- 支付、售后、对账后台管理入口。

## 4.3 一期不包含范围

一期基础阶段暂不包含：

- 完整退货退款物流；
- 换货；
- 客服协商；
- 完整渠道账单自动拉取；
- 财务清结算；
- 多仓库存；
- 复杂营销；
- 搜索引擎；
- 会员体系；
- 完整生产治理。

## 4.4 一期验收标准

一期基础阶段完成后，应满足以下标准：

1. 待支付订单能够按规则超时取消；
2. 订单取消后库存能够释放；
3. 库存预占和释放过程有记录；
4. 支付回调能够更新支付单和订单状态；
5. 用户能够发起基础退款申请；
6. 后台能够审核售后申请；
7. 审核通过后能够发起退款；
8. 退款结果能够同步订单和售后状态；
9. 后台能够进行基础支付/订单/退款核验；
10. 后台账号、角色、权限和操作日志具备基础能力；
11. 支付、售后、对账、库存操作均有后台管理入口。

## 5. 一期增强能力范围

## 5.1 阶段目标

一期增强用于强化交易系统中风险最高的核心链路，重点提升库存、支付、订单、售后和对账的可靠性。

一期增强不是独立大版本，而是一期基础交易闭环上的可靠性加强。

## 5.2 一期增强包含范围

### 库存可靠性增强

- 防超卖策略；
- 库存补偿任务；
- 库存一致性校验；
- 库存对账记录；
- 库存异常后台处理；
- 防超卖压测。

### 支付可靠性增强

- 支付渠道抽象；
- 支付回调幂等；
- 支付金额校验；
- 支付状态主动查询；
- 支付异常识别；
- 退款异常识别。

### 对账闭环增强

- 订单、支付、退款、售后状态交叉核验；
- 线上人工对账工作台；
- 差异类型识别；
- 差异处理建议；
- 挂账跟进；
- 归档报告。

### 认证与后台增强

- 会话并发策略；
- 强制下线；
- 登录事件日志；
- 后台多角色 Dashboard；
- 财务、仓储、商品、运营等视角数据看板。

### 用户体验增强

- 第三方登录；
- 支付结果页体验增强；
- 订单异常状态提示。

## 5.3 一期增强验收标准

一期增强完成后，应满足以下标准：

1. 高并发下库存扣减不出现明显超卖；
2. 库存锁定、释放、补偿过程可追踪；
3. 支付回调重复通知不会导致重复处理；
4. 支付成功但订单未更新等异常能够被识别；
5. 退款成功但订单/售后未同步等异常能够被识别；
6. 后台能够查看并处理对账差异；
7. 对账处理过程具备操作日志和归档记录；
8. 管理端支持更细粒度权限和会话安全控制；
9. Dashboard 能够支撑至少运营、财务、仓储等核心角色视角。

---

## 6. 二期运营履约能力范围

## 6.1 阶段目标

二期目标是从“能交易”升级为“能运营、能履约、能触达用户”。

## 6.2 二期包含范围

- 物流履约；
- 发货管理；
- 物流轨迹；
- 退货物流；
- 优惠券；
- 营销活动；
- 商品搜索；
- 消息通知；
- 商品评价；
- 售后增强；
- 首页装修初版；
- 推荐位配置。

## 6.3 二期验收标准

二期完成后，应满足以下标准：

1. 后台能够对已支付订单进行发货；
2. 用户能够查看物流信息；
3. 用户能够使用优惠券或参与基础营销活动；
4. 商品支持关键词搜索和基础筛选排序；
5. 支付、发货、售后等关键节点能够触发消息通知；
6. 用户能够对完成订单进行评价；
7. 售后支持退货退款等增强流程；
8. 首页和推荐位能够由后台配置。

---

## 7. 三期平台经营能力范围

## 7.1 阶段目标

三期目标是从“业务交易系统”升级为“企业经营平台”。

## 7.2 三期包含范围

- 会员等级；
- 积分体系；
- 成长值；
- 会员权益；
- 数据统计与 BI；
- 用户画像；
- 客服工单；
- 内容装修；
- 风控中心初版；
- 精细化运营。

## 7.3 三期验收标准

三期完成后，应满足以下标准：

1. 用户消费、评价、签到等行为能够沉淀会员成长数据；
2. 系统支持积分获取、使用和记录；
3. 后台能够查看销售、订单、支付、退款、用户、商品等统计数据；
4. 客服人员能够通过工单处理咨询、投诉和售后问题；
5. 运营人员能够配置首页内容和专题页；
6. 系统具备基础风控规则和异常行为识别能力；
7. 运营人员能够基于数据进行精细化运营。

---

## 8. 四期生产治理能力范围

## 8.1 阶段目标

四期目标是让系统具备真实生产环境运行能力。

## 8.2 四期包含范围

- CI/CD；
- Docker / Kubernetes；
- 自动化测试；
- 接口测试；
- E2E 测试；
- 性能压测；
- 链路追踪；
- 监控告警；
- 日志规范；
- 限流；
- 熔断；
- 降级；
- MQ 死信补偿；
- 安全合规；
- 容灾备份；
- 灰度发布；
- 容量评估。

## 8.3 四期验收标准

四期完成后，应满足以下标准：

1. 项目支持自动化构建和部署；
2. 核心链路具备单元测试、集成测试和接口测试；
3. 下单、支付、库存等核心链路具备压测报告；
4. 系统具备基础监控和告警能力；
5. 请求链路具备 TraceId 或链路追踪能力；
6. 关键接口具备限流、防刷和安全校验；
7. 业务消息具备重试、死信和补偿机制；
8. 敏感配置和支付密钥得到隔离管理；
9. 数据具备备份和恢复方案；
10. 生产发布具备回滚或灰度能力。

## 9. 阶段边界原则

## 9.1 MVP 不做复杂企业能力

MVP 阶段只验证最小交易闭环，不应提前引入售后、对账、营销、搜索、会员、风控等复杂能力。

## 9.2 一期开始解决真实交易异常

一期应开始处理交易系统不可避免的异常问题，包括订单超时、库存释放、支付回调、退款、售后和基础对账。

## 9.3 一期增强聚焦高风险链路

一期增强不追求功能数量，而是聚焦库存、支付、订单、售后、对账这些高风险链路的可靠性。

## 9.4 二期补运营体验

二期应补齐用户购买体验和后台运营能力，包括物流、搜索、营销、消息、评价等。

## 9.5 三期补经营平台

三期应建设会员、数据、客服、内容、风控等经营能力，让系统从交易平台升级为运营平台。

## 9.6 四期补生产治理

四期应系统化补齐工程质量、可观测性、高可用、安全合规和容量治理。

---

## 10. 总结

本文定义了 `mallFei` 从 MVP 到四期生产治理的阶段范围和验收标准。

后续开发应严格按照阶段边界推进：

1. 先完成 MVP 最小交易闭环；
2. 再通过一期补齐基础企业级交易保障；
3. 再通过一期增强强化库存、支付、对账等核心风险链路；
4. 再通过二期补齐运营履约能力；
5. 再通过三期建设平台经营能力；
6. 最后通过四期完善生产治理能力。

分期路线文档将进一步把各阶段能力拆分到具体模块中，形成可执行的开发计划。

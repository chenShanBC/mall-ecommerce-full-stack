# 系统整体架构设计

## 1. 文档目的

本文用于在项目启动前描述 `mallFei` 大型企业级电商系统的目标架构，包括前端架构、后端架构、业务模块组织、数据与中间件架构、外部系统集成、部署视角和后续企业级演进方向。

本文属于项目先行设计文档，不描述具体实现进度。各模块的领域模型、状态流转、接口能力和分阶段能力规划，将在后续“模块完整设计文档”中单独说明。

---

## 2. 架构设计目标

`mallFei` 的系统架构设计目标包括：

1. 支撑 C 端商城和 Admin 运营后台双端访问；
2. 支撑用户、商品、购物车、订单、库存、支付、售后、对账等核心电商业务；
3. 支撑从 MVP 到企业级完整电商系统的分阶段演进；
4. 通过模块化设计保持业务边界清晰；
5. 为后续营销、物流、搜索、消息、会员、BI、风控等模块扩展预留空间；
6. 为后续 CI/CD、监控、告警、链路追踪、高可用和安全治理预留演进方向；
7. 支持从模块化单体逐步演进到服务化或微服务架构。

项目初期建议采用“模块化单体 + 前后端分离”的工程形态。该形态适合 MVP 和一期阶段快速建设，也便于通过模块边界为后续服务化演进打基础。

---

## 3. 目标总体架构

系统目标架构如下：

```text
C端用户浏览器 / 移动端 WebView
    |
    | 访问 H5 商城
    v
frontend/mall-h5
    |
    | HTTP API
    v
后端 API 服务
    |
    | 业务模块编排
    v
认证 / 用户 / 商品 / 购物车 / 订单 / 库存 / 支付 / 售后 / 对账 / 文件 / 后台
    |
    | 数据与基础设施
    v
MySQL / Redis / MQ / 对象存储 / 搜索引擎 / 第三方支付 / 短信 / 物流
```

Admin 后台目标访问链路如下：

```text
后台运营人员浏览器
    |
    | 访问 Admin 管理后台
    v
frontend/mall-admin
    |
    | HTTP API
    v
后端 API 服务
    |
    | 后台应用服务编排
    v
运营后台模块 + 各业务中心
    |
    v
数据库 / 缓存 / 对象存储 / 支付渠道 / 物流服务 / 消息服务
```

## 4. 前端架构规划

项目规划两个前端应用：C 端 H5 商城和 Admin 运营后台。

## 4.1 C 端 H5 商城

建议路径：`frontend/mall-h5`

建议技术栈：

- Vue 3；
- Vite；
- Pinia；
- Vue Router；
- Vant；
- Axios。

主要职责：

- 用户登录/注册；
- 首页商品展示；
- 商品列表与商品详情；
- SKU 选择；
- 购物车；
- 结算页；
- 订单列表；
- 订单详情；
- 支付返回；
- 地址管理；
- 个人中心；
- 售后入口；
- 后续搜索、优惠券、消息、评价、会员等能力。

阶段规划：

- MVP：完成基础购物链路；
- 一期：补充支付返回、订单状态、售后入口等；
- 二期：补充搜索、优惠券、消息、评价、物流；
- 三期：补充会员、积分、内容推荐和个性化运营。

## 4.2 Admin 运营后台

## 5. 后端工程架构规划

后端建议采用 Maven 多模块结构，由启动模块统一集成业务模块。

建议模块结构：

```text
backend/
  mall-common
  mall-auth
  mall-user
  mall-product
  mall-cart
  mall-order
  mall-stock
  mall-pay
  mall-aftersale
  mall-reconciliation
  mall-file
  mall-admin
  mall-logistics
  mall-marketing
  mall-coupon
  mall-search
  mall-message
  mall-review
  mall-member
  mall-statistics
  mall-risk
  mall-start
```

其中，并不是所有模块都需要在 MVP 阶段创建。项目可以按阶段逐步增加模块，也可以在初期将低复杂度能力作为子域放在已有模块内，但设计边界应提前明确。

## 5.1 核心模块说明

| 模块 | 职责 | 阶段 |
|---|---|---|
| `mall-common` | 公共响应、异常、枚举、认证对象、通用工具 | MVP 起步 |
| `mall-auth` | 登录态、会话、权限、安全控制 | MVP 起步，一期增强 |
| `mall-user` | 用户账号、资料、地址、第三方登录 | MVP 起步，一期增强 |
| `mall-product` | 分类、SPU、SKU、商品查询和管理 | MVP 起步 |
| `mall-cart` | 购物车、结算预览、下单前校验 | MVP 起步 |
| `mall-order` | 订单创建、订单状态、订单超时、退款关联 | MVP 起步，一期增强 |
| `mall-stock` | 库存预占、释放、补偿、一致性校验 | MVP 起步，一期增强 |
| `mall-pay` | 支付单、支付渠道、回调、退款、支付核验 | MVP 起步，一期增强 |
| `mall-aftersale` | 售后退款申请、审核、状态流转 | 一期起步 |
| `mall-reconciliation` | 订单、支付、退款、库存对账和差异处理 | 一期起步，后续增强 |
| `mall-file` | 文件上传、本地存储、对象存储 | MVP 起步 |
| `mall-admin` | 后台管理、权限、Dashboard、对账工作台 | MVP 起步，持续增强 |
| `mall-start` | 应用启动、配置装配、模块集成 | MVP 起步 |

## 5.2 分层结构建议

业务模块建议采用以下分层：

```text
controller
  -> application service
      -> domain model / domain service / repository interface
          -> infrastructure repository / mapper / dataobject
```

典型职责如下：

| 层级 | 职责 |
|---|---|
| Controller | 接收 HTTP 请求，做参数入口和权限入口 |
| Application Service | 编排业务用例，协调领域服务和外部模块 |
| Domain Model | 表达业务实体、状态和核心规则 |
| Domain Service | 承载跨实体领域规则 |
| Repository Interface | 定义领域持久化接口 |
| Infrastructure Repository | 实现持久化、缓存、第三方调用等基础设施适配 |
| Mapper / DO | 数据库表映射 |
| Facade | 跨模块协作接口 |
| Event / Consumer | 异步事件发布与消费 |

## 6. 后端运行架构规划

## 6.1 MVP 与一期运行形态

MVP 和一期建议采用模块化单体：

```text
mall-start Spring Boot 应用
    |
    +-- mall-common
    +-- mall-auth
    +-- mall-user
    +-- mall-product
    +-- mall-cart
    +-- mall-order
    +-- mall-stock
    +-- mall-pay
    +-- mall-aftersale
    +-- mall-reconciliation
    +-- mall-file
    +-- mall-admin
```

模块化单体的优点：

- 本地开发简单；
- 部署复杂度低；
- 事务边界相对容易控制；
- 适合 MVP 和一期快速迭代；
- 便于保持统一代码仓库和统一调试体验。

模块化单体的风险：

- 模块边界容易被代码层面突破；
- 应用服务可能逐渐膨胀；
- 所有业务共享同一运行进程；
- 无法按模块独立扩缩容；
- 生产级隔离和容灾能力有限。

## 6.2 后续服务化演进

后续可根据复杂度逐步拆分服务，优先候选包括：

- 订单服务；
- 库存服务；
- 支付服务；
- 对账服务；
- 搜索服务；
- 消息服务；
- 营销服务；
- 文件服务。

服务化不应过早进行。项目初期应先保证模块边界、领域模型、事件规范和接口契约清晰，再根据性能、团队协作和部署需求进行拆分。

## 9. 核心链路架构

## 9.1 下单链路

```text
C端用户
  -> H5商城选择商品/SKU
  -> 购物车结算
  -> 订单中心创建订单
  -> 商品中心生成商品快照
  -> 库存中心预占库存
  -> 订单保存
  -> 支付中心创建支付单
  -> 返回支付信息
```

关键设计要求：

- 商品价格和 SKU 信息需要快照化；
- 库存预占需要具备失败回滚能力；
- 订单保存失败需要释放已占库存；
- 支付单和订单需要建立稳定关联。

## 9.2 支付链路

```text
C端用户发起支付
  -> 支付中心选择支付渠道
  -> 第三方支付页面/Mock支付
  -> 支付渠道回调
  -> 支付中心记录回调
  -> 支付状态核验
  -> 更新支付单
  -> 通知订单中心更新订单状态
```

关键设计要求：

- 回调需要验签；
- 回调需要幂等；
- 金额需要校验；
- 支付成功但订单未更新时，需要后续核验和补偿。

## 9.3 订单超时链路

```text
订单创建后处于待支付
  -> 超过支付时限
  -> 订单超时任务扫描
  -> 订单状态变更为已取消
  -> 库存中心释放锁定库存
  -> 支付单关闭或标记无需支付
```

关键设计要求：

- 需要处理支付成功和超时取消并发；
- 库存释放要避免重复释放；
- 超时任务要可重试和可观测。

## 9.4 售后退款链路

```text
用户申请售后
  -> 售后中心创建售后单
  -> 后台审核
  -> 审核通过后请求支付中心退款
  -> 支付中心处理退款结果
  -> 订单中心同步退款状态
  -> 对账中心核验退款一致性
```

关键设计要求：

- 售后审核和支付退款要解耦；
- 退款结果可能异步返回；
- 退款失败需要人工处理；
- 订单、售后、支付三方状态需要一致。

## 9.5 对账链路

```text
后台触发对账
  -> 查询订单、支付单、退款单、库存记录
  -> 必要时查询第三方支付渠道状态
  -> 比较本地状态和渠道状态
  -> 识别差异类型
  -> 生成处理建议
  -> 后台人工处理
  -> 执行业务修复或归档
```

关键设计要求：

- 一期对账定位为业务一致性核验，不直接等同于完整财务清结算；
- 差异处理需要可追踪；
- 对账动作不能绕过订单、支付、库存的领域规则。

认证鉴权体系建议以统一登录态和权限注解为基础，逐步演进为后台 RBAC、菜单权限、按钮权限和数据权限。

规划能力包括：

- 用户登录态；
- 管理员登录态；
- 用户/管理员身份区分；
- 接口权限注解；
- 后台账号；
- 角色；
- 权限；
- 菜单权限；
- 按钮权限；
- 数据权限；
- Redis 会话存储；
- 强制下线；

- 支付密钥隔离；
- 安全审计和告警。

## 11. 事件与异步协作架构

订单、库存、支付、售后、消息、统计等模块之间需要大量异步协作。项目应提前规划事件模型。

典型事件包括：

- 商品库存初始化事件；
- 订单创建事件；
- 订单取消事件；
- 订单支付成功事件；
- 订单超时事件；
- 库存释放事件；
- 售后申请事件；
- 退款申请事件；
- 退款成功事件；
- 支付异常事件；
- 对账差异事件。

事件协作设计要求：

- 事件命名统一；
- 事件 ID 唯一；
- 消息消费幂等；
- 支持重试；
- 支持死信队列；
- 支持失败补偿；
- 记录事件日志；
- 支持消费监控。

MVP 阶段可以先使用同步调用或本地事件，一期增强和二期以后应逐步引入可靠消息机制。

## 12. 部署架构规划

## 12.1 MVP / 一期部署形态

MVP 和一期建议部署形态：

```text
Nginx
  -> H5 前端静态资源
  -> Admin 前端静态资源
  -> Spring Boot 后端应用
  -> MySQL
  -> Redis
  -> 本地文件存储 / MinIO
```

该形态适合：

- 本地开发；
- 测试环境；
- 演示环境；
- 低并发业务验证。

## 12.2 二期/三期部署形态

二期和三期可演进为：

```text
Nginx / Gateway
  -> H5 Frontend
  -> Admin Frontend
  -> Backend API Service
  -> MySQL
  -> Redis
  -> RabbitMQ / RocketMQ
  -> MinIO
  -> Elasticsearch / OpenSearch
  -> Prometheus / Grafana
```

## 12.3 四期生产治理部署形态

四期可进一步演进为：

```text
Kubernetes
  -> Ingress / Gateway
  -> 多实例后端服务
  -> 独立订单服务
  -> 独立支付服务
  -> 独立库存服务
  -> 独立搜索服务
  -> 独立消息服务
  -> MySQL 主从/集群
  -> Redis 集群
  -> MQ 集群
  -> 对象存储
  -> 监控告警体系
```

## 13. 架构演进路线

## 13.1 MVP 阶段架构

- 前后端分离；
- 模块化单体；
- MySQL + Redis；
- Mock 支付或沙箱支付；
- 基础后台管理。

## 13.2 一期架构

- 引入售后、退款、基础对账；
- 强化库存预占和订单超时；
- 增强支付回调和状态核验；
- 增强后台权限和操作日志。

## 13.3 一期增强架构

- 库存补偿和一致性检查；
- 支付异常核验；
- 订单支付售后对账闭环；
- 线上人工对账工作台；
- 多角色 Dashboard；
- 强制下线和会话策略。

## 13.4 二期架构

- 引入物流、营销、优惠券、搜索、消息、评价；
- 引入搜索引擎；
- 引入 MQ 统一异步协作；
- 前端补充更多运营页面。

## 13.5 三期架构

- 引入会员、积分、BI、客服、内容装修、风控；
- 统计能力逐步从后台聚合转向独立统计中心；
- 风控和用户画像开始沉淀规则与数据模型。

## 13.6 四期架构

- 自动化测试；
- CI/CD；
- 容器化；
- 链路追踪；
- 监控告警；
- 高可用治理；
- 安全合规；
- 服务拆分和容量治理。

---

## 14. 架构设计约束

## 14.1 不过早微服务化

项目初期优先保证业务闭环和模块边界，不为了架构形式提前拆分微服务。

## 14.2 关键状态必须显式建模

订单状态、支付状态、退款状态、售后状态、库存锁状态、对账状态必须有明确枚举、状态流转和异常处理规则。

## 14.3 跨模块调用必须经过接口边界

订单、库存、支付、售后、对账之间不得随意跨表修改对方数据，应通过应用服务、Facade 或事件协作。

## 14.4 对账与人工处理必须可追踪

凡是涉及资金、库存和订单状态修复的后台操作，都应记录操作人、操作时间、操作前后状态和处理原因。

## 14.5 生产治理能力分阶段引入

MVP 不追求完整生产治理，但设计上应避免阻碍后续接入日志、监控、链路追踪、告警、安全审计和 CI/CD。

---

## 15. 总结

`mallFei` 目标架构应采用“前后端分离 + 模块化单体起步 + 业务域清晰拆分 + 分阶段演进”的方式建设。

项目初期重点不是追求复杂架构，而是围绕电商主链路建立清晰的业务边界：用户、商品、购物车、订单、库存、支付、后台先形成 MVP 闭环；一期补齐售后、退款、对账、权限和审计；一期增强强化库存、支付和对账等高风险链路；二期、三期、四期再逐步建设运营履约、平台经营和生产治理能力。

后续开发应严格遵循本文定义的架构原则，避免模块职责混乱和核心业务规则散落。

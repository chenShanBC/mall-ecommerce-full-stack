# 核心业务流程总览

## 1. 文档目的

本文用于在项目启动前，从全局角度定义 `mallFei` 大型企业级电商系统的核心业务流程。

本文属于项目先行设计文档，不描述具体实现进度。后续模块完整设计、接口设计、数据库设计、关键专题设计和测试用例设计，均应基于本文定义的业务流程进一步展开。

---

## 2. 总体交易闭环

完整电商业务的核心闭环如下：

```text
用户注册/登录
  -> 浏览商品
  -> 加入购物车
  -> 结算确认
  -> 创建订单
  -> 库存预占
  -> 发起支付
  -> 支付成功
  -> 订单状态更新
  -> 履约/确认收货
  -> 售后退款
  -> 对账核验
  -> 后台运营处理
```

在分阶段建设中，该闭环应逐步拆分实现：

- MVP 阶段：完成用户、商品、购物车、订单、库存基础处理、支付和订单查看；
- 一期阶段：补齐库存预占、订单超时、支付回调、退款、售后和基础对账；
- 一期增强：强化库存防超卖、支付核验、对账闭环和后台审计；
- 二期阶段：补齐物流、营销、搜索、消息和评价；
- 三期阶段：补齐会员、数据、客服、内容和风控；
- 四期阶段：补齐监控、测试、部署、高可用和安全治理。

## 3. 用户注册与登录流程

## 3.1 用户密码登录流程

```text
用户进入登录页
  -> 输入手机号/账号和密码
  -> 前端提交登录请求
  -> 用户中心校验账号状态
  -> 密码组件校验密码
  -> 认证中心创建登录态
  -> 返回用户信息和 token/session
  -> 前端保存登录状态
  -> 用户进入商城
```

阶段规划：MVP。

设计要求：

- 密码必须加密存储；
- 登录失败应返回明确错误；
- 登录态应可被后端统一识别；
- 用户端和管理端身份应区分。

后续增强：

- 登录失败次数限制；
- 设备识别；
- 风控校验；
- 异地登录提醒；
- 用户注销和账号冻结。

## 3.2 短信验证码登录流程

```text
用户输入手机号
  -> 请求发送短信验证码
  -> 用户中心生成验证码
  -> 缓存验证码
  -> 用户输入验证码并提交
  -> 用户中心校验验证码
  -> 认证中心创建登录态
  -> 返回登录结果
```

阶段规划：MVP 或一期。

设计要求：

- 验证码需要有效期；
- 同一手机号发送频率需要限制；
- 验证失败次数需要限制；
- 可通过 Mock 短信支持本地开发。

## 3.3 第三方登录流程

```text
用户点击第三方登录
  -> 后端生成授权地址和 state
  -> 前端跳转第三方授权页
  -> 用户授权后回跳
  -> 前端拿到 auth code
  -> 后端校验 state
  -> 调用第三方平台换取用户信息
  -> 查询第三方绑定关系
  -> 已绑定则登录已有用户
  -> 未绑定则创建或绑定用户
  -> 认证中心创建登录态
```

阶段规划：一期增强。

设计要求：

- state 必须防重放；
- 第三方用户标识必须唯一；
- 需要定义自动注册、绑定和账号合并策略；
- 需要处理授权失败、回调丢失和重复绑定。

## 4. 商品浏览流程

## 4.1 商品首页/列表流程

```text
用户进入商城首页
  -> 前端请求商品列表
  -> 商品中心查询上架商品
  -> 商品中心组装商品卡片信息
  -> 返回商品名称、图片、价格、销量等
  -> 前端渲染商品列表
```

阶段规划：MVP。

设计要求：

- 只展示上架商品；
- 商品价格以 SKU 或默认 SKU 为准；
- 图片应通过文件服务或对象存储访问；
- 商品列表应支持分页。

后续增强：

- 搜索；
- 分类筛选；
- 价格筛选；
- 销量排序；
- 推荐算法；
- 首页装修和推荐位。

## 4.2 商品详情流程

```text
用户点击商品
  -> 前端请求商品详情
  -> 商品中心查询 SPU 信息
  -> 商品中心查询 SKU 信息
  -> 商品中心查询图片、价格、库存展示数据
  -> 前端渲染商品详情和 SKU 选择器
```

阶段规划：MVP。

设计要求：

- 商品详情应包含 SPU 基础信息；
- SKU 应包含规格、价格、库存展示状态；
- 下架商品不可购买；
- 商品详情中展示的库存仅作为参考，最终以下单校验为准。

后续增强：

- 商品评价；
- 促销价格；
- 优惠券展示；
- 物流服务说明；
- 库存区域展示；
- 商品详情内容装修。

## 5. 购物车流程

## 5.1 加入购物车流程

```text
用户在商品详情页选择 SKU 和数量
  -> 点击加入购物车
  -> 前端提交 SKU 和数量
  -> 购物车中心校验用户身份
  -> 商品中心校验商品/SKU 是否有效
  -> 库存中心校验基础可售库存
  -> 购物车中心新增或合并购物车项
  -> 返回购物车数量和操作结果
```

阶段规划：MVP。

设计要求：

- 未登录用户应引导登录或支持临时购物车；
- 商品下架不可加入购物车；
- SKU 不存在或禁用不可加入购物车；
- 加购数量不能超过限购或库存约束。

## 5.2 购物车结算预览流程

```text
用户进入购物车
  -> 勾选商品
  -> 点击结算
  -> 购物车中心查询选中商品
  -> 商品中心校验商品状态和价格
  -> 库存中心校验库存
  -> 计算应付金额
  -> 返回结算预览
```

阶段规划：MVP。

设计要求：

- 结算前必须重新校验商品状态；
- 商品价格应以后端实时价格为准；
- 库存不足时应提示用户；
- 结算预览不代表库存已锁定。

后续增强：

- 优惠券；
- 满减；
- 促销活动；
- 运费计算；
- 地址可配送校验；
- 拆单规则。

---

## 6. 下单流程

## 6.1 MVP 下单流程

```text
用户在结算页确认商品和地址
  -> 提交订单
  -> 订单中心校验用户身份
  -> 订单中心校验地址
  -> 商品中心生成商品快照
  -> 库存中心处理基础库存
  -> 订单中心创建订单和订单项
  -> 支付中心创建支付单
  -> 返回订单号和支付信息
```

阶段规划：MVP。

设计要求：

- 订单需要保存商品快照；
- 下单需要校验商品状态；
- 下单需要校验库存；
- 创建订单后进入待支付状态；
- 支付单应与订单建立唯一关联。

## 6.2 一期下单增强流程

```text
用户提交订单
  -> 订单中心创建订单上下文
  -> 商品中心生成快照
  -> 库存中心预占库存
  -> 生成库存锁定记录
  -> 保存订单和订单项
  -> 记录库存操作日志
  -> 创建支付单
  -> 返回支付入口
```

阶段规划：一期基础 / 一期增强。

设计要求：

- 库存预占成功后，后续失败必须能补偿释放；
- 订单保存失败不能导致库存长期锁定；
- 库存锁定记录用于后续释放、核验和对账；
- 支付单与订单必须建立稳定关联；
- 下单接口应具备防重复提交能力。

---

## 7. 库存预占、释放与补偿流程

## 7.1 库存预占流程

```text
订单提交
  -> 库存中心校验 SKU 库存
  -> 扣减可售库存或增加锁定库存
  -> 生成库存锁定记录
  -> 记录库存操作日志
  -> 返回预占结果
```

阶段规划：一期基础。

设计要求：

- 预占操作需要防止超卖；
- 预占结果需要可追踪；
- 失败时需要回滚已预占库存；
- 库存锁定记录需要与订单关联。

## 7.2 订单取消释放库存流程

```text
用户取消 / 系统超时取消订单
  -> 订单中心变更订单状态为已取消
  -> 库存中心查询库存锁定记录
  -> 释放锁定库存
  -> 更新库存锁状态
  -> 记录库存释放日志
```

阶段规划：一期基础。

设计要求：

- 释放库存需要幂等；
- 不能重复释放；
- 订单支付成功后不能再释放库存；
- 超时取消与支付成功并发时需要明确处理规则。

## 7.3 库存补偿流程

```text
定时任务扫描异常库存锁
  -> 识别订单不存在但库存已锁定
  -> 识别订单已取消但库存未释放
  -> 识别库存锁过期但未处理
  -> 执行补偿释放或生成异常记录
  -> 记录补偿结果
```

阶段规划：一期增强。

设计要求：

- 补偿任务必须幂等；
- 补偿结果必须可追踪；
- 无法自动修复的问题应进入后台异常处理；
- 后续应支持 MQ 失败补偿和告警。

## 7.4 库存一致性核验流程

```text
库存对账任务启动
  -> 查询库存主表
  -> 查询库存锁定记录
  -> 计算理论库存
  -> 对比实际库存
  -> 生成库存一致性快照
  -> 发现差异则生成库存对账记录
  -> 后台处理差异
```

阶段规划：一期增强。

## 8. 支付流程

## 8.1 Mock 支付流程

```text
用户选择支付
  -> 支付中心创建支付单
  -> 选择 Mock 支付渠道
  -> 模拟支付成功
  -> 更新支付单状态
  -> 通知订单中心更新订单状态
  -> 返回支付结果
```

阶段规划：MVP。

用途：

- 本地开发；
- 前后端联调；
- 测试环境验证；
- 避免早期依赖真实第三方支付环境。

## 8.2 第三方支付流程

```text
用户选择第三方支付
  -> 支付中心创建支付单
  -> 支付中心调用第三方支付接口
  -> 返回支付表单/支付跳转信息
  -> 用户在第三方平台完成支付
  -> 第三方平台异步回调后端
  -> 支付中心验签和记录回调
  -> 支付中心更新支付单状态
  -> 订单中心更新订单状态
  -> 前端支付返回页查询支付结果
```

阶段规划：一期基础 / 一期增强。

设计要求：

- 支付单号必须唯一；
- 回调需要记录；
- 回调需要验签；
- 回调需要幂等；
- 支付金额需要校验；
- 支付成功后订单状态需要同步。

## 8.3 支付状态核验流程

```text
后台或系统触发支付核验
  -> 查询订单
  -> 查询支付单
  -> 查询支付回调记录
  -> 调用支付渠道单笔查询
  -> 对比本地状态和渠道状态
  -> 输出核验结论、风险等级和建议动作
```

阶段规划：一期增强。

适用场景：

- 支付成功但订单未更新；
- 订单已取消但渠道支付成功；
- 本地成功但渠道未确认；
- 金额不一致；
- 回调丢失或延迟。

## 9. 订单状态流转流程

## 9.1 基础状态流转

```text
待支付
  -> 支付成功 -> 待发货/待履约
  -> 用户取消/超时取消 -> 已取消

待发货/待履约
  -> 发货 -> 待收货
  -> 申请售后 -> 售后处理中

待收货
  -> 用户确认/自动确认 -> 已完成
  -> 申请售后 -> 售后处理中

售后处理中
  -> 退款成功 -> 已退款/部分退款
  -> 售后驳回 -> 回到原业务状态
```

阶段规划：

- MVP：待支付、已支付、已取消、已完成基础状态；
- 一期：超时取消、退款关联、自动确认；
- 二期：待发货、已发货、待收货、物流履约状态；
- 二期/三期：完整售后状态联动。

## 9.2 订单超时取消流程

```text
订单创建后进入待支付
  -> 超过支付有效期
  -> 定时任务扫描待支付订单
  -> 判断订单是否仍未支付
  -> 取消订单
  -> 释放库存
  -> 记录操作日志
```

阶段规划：一期基础。

设计要求：

- 支付回调和超时取消并发时必须有明确规则；
- 取消订单必须幂等；
- 库存释放必须幂等；
- 支付成功但订单已取消时应进入异常核验或对账流程。

## 9.3 自动确认收货流程

```text
订单进入待确认/已发货状态
  -> 达到自动确认时间
  -> 定时任务扫描订单
  -> 自动确认收货
  -> 订单进入已完成
```

阶段规划：一期基础起步，二期结合物流履约增强。

## 10. 售后退款流程

## 10.1 简单退款售后流程

```text
用户在订单详情发起退款申请
  -> 售后中心创建售后单
  -> 订单中心记录退款申请
  -> 后台运营查看售后单
  -> 后台审核通过或驳回
  -> 审核通过则支付中心发起退款
  -> 支付中心处理退款结果
  -> 售后中心更新售后状态
  -> 订单中心同步退款状态
```

阶段规划：一期基础。

设计要求：

- MVP 阶段不建设售后；
- 一期只做基础退款售后；
- 售后审核和退款执行应解耦；
- 退款结果需要同步订单和售后；
- 退款失败需要后台处理入口。

## 10.2 退款失败处理流程

```text
支付中心发起退款
  -> 渠道返回失败或状态未知
  -> 记录退款失败
  -> 后台支付/售后管理查看异常
  -> 人工重新核验或处理
  -> 必要时进入对账流程
```

阶段规划：一期基础 / 一期增强。

后续增强：

- 退款自动重试；
- 退款失败告警；
- 退款对账；
- 客服工单联动。

---

## 11. 基础业务对账流程

## 11.1 支付对账流程

```text
后台输入订单号或触发对账任务
  -> 查询订单
  -> 查询支付单
  -> 查询渠道支付状态
  -> 对比订单状态、支付单状态、渠道状态、金额
  -> 识别差异类型
  -> 生成支付对账记录
  -> 后台执行处理动作
```

阶段规划：一期基础 / 一期增强。

可识别问题：

- 支付单成功但订单未支付；
- 渠道成功但本地未成功；
- 本地成功但渠道未确认；
- 订单已支付但缺少支付单；
- 金额不一致。

## 11.2 退款对账流程

```text
后台触发退款核验
  -> 查询退款单
  -> 查询本地退款状态
  -> 查询渠道退款状态
  -> 对比退款金额和状态
  -> 生成退款对账记录
  -> 后台处理差异
```

阶段规划：一期基础。

## 11.3 库存对账流程

```text
库存对账任务启动
  -> 查询库存主表
  -> 查询库存锁定记录
  -> 查询库存操作日志
  -> 计算理论库存
  -> 对比实际库存
  -> 生成库存对账记录
  -> 后台处理差异
```

阶段规划：一期增强。

## 11.4 线上人工对账流程

```text
后台创建对账任务
  -> 系统生成本地账单/对账数据
  -> 导入或获取渠道账单数据
  -> 系统识别差异
  -> 后台查看差异明细
  -> 根据差异类型执行建议动作
  -> 标记待处理、已处理、忽略或归档
  -> 生成归档报告
```

阶段规划：一期增强。

设计边界：

- 一期对账是业务一致性核验与处置工作台；
- 完整渠道账单自动拉取、财务账期和清结算放到后续阶段。

## 12. 后台运营流程

## 12.1 商品运营流程

```text
管理员登录后台
  -> 创建或编辑商品
  -> 配置分类、SPU、SKU、图片、价格
  -> 上架商品
  -> 商品展示到 C 端
  -> 根据销售数据调整商品
```

阶段规划：MVP 起步，二期/三期增强。

## 12.2 订单运营流程

```text
管理员进入订单管理
  -> 查询订单列表
  -> 查看订单详情
  -> 查看支付、商品、用户、地址信息
  -> 处理异常订单
  -> 必要时触发支付核验或对账
```

阶段规划：MVP 起步，一期增强。

## 12.3 库存运营流程

```text
管理员进入库存管理
  -> 查看库存列表
  -> 查看库存预警和操作日志
  -> 调整库存
  -> 查看库存一致性结果
  -> 处理库存异常
```

阶段规划：MVP 起步，一期增强。

## 12.4 支付运营流程

```text
管理员进入支付管理
  -> 查询支付单
  -> 查看支付状态和回调记录
  -> 触发支付核验
  -> 查看退款信息
  -> 处理支付异常
```

阶段规划：一期基础 / 一期增强。

## 12.5 售后运营流程

```text
管理员进入售后管理
  -> 查看售后单
  -> 审核退款申请
  -> 驳回或通过
  -> 触发退款
  -> 跟进退款结果
```

阶段规划：一期基础。

## 12.6 对账运营流程

```text
管理员进入对账管理
  -> 创建或查看对账任务
  -> 查看差异明细
  -> 触发重新对账
  -> 执行业务修复动作
  -> 标记处理结果
  -> 归档报告
```

阶段规划：一期增强。

---

## 13. 权限与审计流程

## 13.1 管理员登录流程

```text
管理员输入账号密码
  -> 后台认证服务校验账号
  -> 校验密码
  -> 创建管理员登录态
  -> 返回权限信息
  -> 前端加载后台菜单和页面
```

阶段规划：MVP 起步。

## 13.2 后台权限校验流程

```text
管理员发起后台操作
  -> 接口权限注解校验身份
  -> 校验管理员权限
  -> 通过后执行业务动作
  -> 记录操作日志
```

阶段规划：一期基础 / 一期增强。

## 13.3 强制下线流程

```text
管理员会话被策略判定需要下线
  -> 认证中心使登录态失效
  -> 推送强制下线消息
  -> 前端提示并跳转登录页
  -> 记录会话事件
```

阶段规划：一期增强。

---

## 14. Dashboard 数据流程

```text
管理员进入 Dashboard
  -> 前端请求不同角色视图数据
  -> 后台聚合订单、支付、库存、商品、售后等数据
  -> 返回指标卡片、趋势图、待办和表格数据
  -> 前端通过图表组件展示
```

阶段规划：一期增强起步，三期演进为 BI 能力。

设计要求：

- 一期 Dashboard 可直接聚合业务数据；
- 三期应建设统一统计和 BI 能力；
- 统计口径需要统一定义；
- 数据看板需要考虑角色权限和数据权限。

## 15. 二期业务流程规划

## 15.1 物流履约流程

```text
订单支付成功
  -> 后台发货
  -> 填写物流公司和单号
  -> 同步物流轨迹
  -> 用户查看物流
  -> 用户确认收货
```

## 15.2 优惠券流程

```text
后台创建优惠券
  -> 用户领取优惠券
  -> 结算页选择优惠券
  -> 订单计算优惠金额
  -> 支付成功后核销优惠券
  -> 退款时退还或作废优惠券
```

## 15.3 营销活动流程

```text
后台配置活动
  -> 商品参与活动
  -> C端展示活动价
  -> 结算时计算活动优惠
  -> 订单记录优惠明细
```

## 15.4 商品搜索流程

```text
商品上架/更新
  -> 同步搜索索引
  -> 用户输入关键词
  -> 搜索中心查询索引
  -> 返回筛选和排序结果
```

## 15.5 消息通知流程

```text
订单/支付/发货/售后状态变化
  -> 发布通知事件
  -> 消息中心根据模板生成通知
  -> 发送短信/站内信/邮件
  -> 记录发送结果
```

## 15.6 商品评价流程

```text
订单完成
  -> 用户评价商品
  -> 上传图片
  -> 后台审核
  -> 商品详情展示评价
```

```text
  -> 发放会员权益
```

```text
  -> Dashboard 和 BI 页面展示
```

```text
  -> 用户确认或评价
  -> 工单归档
```

## 16.4 风控流程

```text

  -> 风控中心识别行为特征

```

## 17.1 CI/CD 流程

```text
提交代码
  -> 触发构建
  -> 执行测试
  -> 构建镜像
  -> 部署测试环境
```

## 19. 总结

本文从项目先行设计视角定义了 `mallFei` 的核心业务流程。

后续开发应按照以下顺序推进：

1. MVP 阶段先跑通用户、商品、购物车、订单、库存基础处理、支付和订单查看；
2. 一期阶段补齐售后、退款、订单超时、库存预占、后台权限和基础对账；
3. 一期增强重点强化库存防超卖、第三方支付、支付核验、对账闭环、人工对账和 Dashboard；
4. 二期补齐物流、营销、搜索、消息、评价；
5. 三期补齐会员、BI、客服、内容、风控；
6. 四期补齐 CI/CD、监控、链路追踪、高可用和安全治理。

模块完整设计文档应基于本文中的流程，进一步展开每个业务模块的状态、模型、接口、异常场景和阶段能力。

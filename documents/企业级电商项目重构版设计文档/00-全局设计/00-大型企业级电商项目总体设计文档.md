# 大型企业级电商项目总体设计文档

## 1. 文档目的

本文用于在项目启动前定义 `mallFei` 大型企业级电商系统的总体建设目标、业务范围、系统边界、分期路线和设计原则。

本文属于项目先行设计文档，主要用于指导后续需求拆解、模块设计、数据库设计、接口设计、前后端开发、联调测试和阶段验收。从目标系统出发，说明一个大型企业级电商项目应当如何分阶段建设。

后续所有模块完整设计文档、分期交付文档、关键专题设计文档、接口联调文档和测试验收文档，均以本文定义的项目目标和阶段口径为基础。

---

## 2. 项目总体定位

`mallFei` 规划为一个面向中大型 B2C 电商业务场景的全栈电商系统，目标是覆盖从用户访问、商品浏览、购物车结算、订单创建、库存占用、支付履约、售后退款、后台运营、业务对账、数据分析到生产治理的完整业务闭环。

系统采用前后端分离架构，规划包含：

- C 端 H5 商城；
- Admin 运营管理后台；
- Java 后端业务服务；
- MySQL、Redis、对象存储、消息队列、搜索引擎等基础设施；
- 第三方支付、短信服务、对象存储、物流查询等外部服务集成；
- 面向生产环境的测试、部署、监控、告警、安全和高可用治理能力。

本项目不是单一的商城页面或后台 CRUD 系统，而是一个按照企业级电商业务域拆分、分阶段演进、逐步补齐交易可靠性和运营能力的综合业务系统。

---

## 3. 建设背景

电商系统的核心不只是展示商品和创建订单，而是围绕交易链路建立一整套可运营、可追踪、可核验、可扩展的业务平台。

一个完整企业级电商系统通常需要解决以下问题：

1. 用户如何注册、登录、维护资料和地址；
2. 商品如何建模、展示、上下架、维护图片和规格；
3. 购物车如何处理商品选择、数量变更、价格变化和结算校验；
4. 订单如何创建、支付、取消、完成、退款和售后；
5. 库存如何防止超卖、预占、释放、补偿和核验；
6. 支付如何接入第三方渠道、处理回调、退款、核验和异常状态；
7. 后台如何支撑商品、订单、库存、用户、支付、售后和对账运营；
8. 系统如何发现订单、支付、退款、库存之间的数据不一致；
9. 后续如何补齐物流、营销、搜索、消息、评价、会员和数据分析；
10. 生产环境如何保障测试、部署、监控、告警、安全和高可用。

因此，本项目采用分阶段建设策略：先完成最小商城交易闭环，再逐步引入企业级交易保障能力、运营履约能力、平台经营能力和生产治理能力。

## 4. 建设目标

## 4.1 业务目标

项目最终应支持一套完整的 B2C 电商业务闭环：

```text
用户注册/登录
  -> 浏览商品
  -> 加入购物车
  -> 提交订单
  -> 库存预占
  -> 发起支付
  -> 支付成功
  -> 订单履约
  -> 售后退款
  -> 对账核验
  -> 后台运营
  -> 数据分析
```

围绕该闭环，系统需要逐步建设以下业务能力：

- 用户与认证；
- 商品与分类；
- 购物车与结算；
- 订单与交易状态；
- 库存与防超卖；
- 支付与退款；
- 售后与服务；
- 对账与异常处置；
- 运营后台与权限审计；
- 物流履约；
- 营销优惠；
- 搜索与推荐；
- 消息通知；
- 评价体系；
- 会员成长；
- 数据统计与 BI；
- 风控与安全；
- 生产治理。

## 4.2 技术目标

项目技术建设目标包括：

- 采用前后端分离架构；
- 后端按业务域进行模块化拆分；
- 核心业务模块遵循应用层、领域层、基础设施层的分层思路；
- 使用统一响应、统一异常、统一认证鉴权和统一日志规范；
- 为库存、支付、订单、售后、对账等复杂链路设计清晰的状态和边界；
- 为后续消息队列、搜索引擎、对象存储、数据统计和监控体系预留扩展空间；
- 支持从模块化单体向服务化架构演进。

## 4.3 工程目标

项目工程建设目标包括：

- 文档先行，模块边界清晰；
- 阶段目标明确，每期可验收；
- 接口设计规范，便于前后端联调；
- 关键链路具备测试用例和压测方案；
- 配置区分本地、测试、生产环境；
- 后续支持 Docker、CI/CD、监控告警和自动化测试。

## 5. 系统使用角色

## 5.1 C 端用户

C 端用户是商城的购买者，主要通过 H5 商城完成注册登录、浏览商品、加入购物车、提交订单、支付、查看订单、管理地址、申请售后和查看个人中心等操作。

## 5.2 后台运营人员

后台运营人员负责商品维护、订单处理、用户查看、售后审核、活动配置、内容维护、对账处理和日常运营管理。

## 5.3 仓储/库存人员

仓储或库存人员负责库存查看、库存调整、库存预警、库存异常处理、发货协作和库存核验。

## 5.4 财务/对账人员

财务或对账人员负责核查订单、支付、退款、渠道流水和库存数据之间的一致性，处理长短款、状态不一致和金额不一致等问题。

## 5.5 系统管理员

系统管理员负责后台账号、角色、权限、菜单、系统参数、操作日志、安全策略和基础配置。

## 5.6 运维/技术人员

运维和技术人员负责部署、监控、告警、故障排查、容量评估、安全治理和系统演进。

## 7. 阶段建设策略

项目建设按照以下阶段推进：

```text
MVP 阶段
  -> 一期基础企业级交易闭环
  -> 一期增强能力
  -> 二期运营履约能力
  -> 三期平台经营能力
  -> 四期生产治理能力
```

## 7.1 MVP 阶段：最小商城交易闭环

MVP 阶段用于验证最小可用商城交易链路。

目标流程：

```text
用户登录 -> 浏览商品 -> 加入购物车 -> 提交订单 -> 完成支付 -> 查看订单
```

MVP 阶段包含：

- 用户注册/登录；
- 用户资料和地址；
- 商品分类、SPU、SKU、列表、详情；
- 购物车加购、修改数量、勾选、结算预览；
- 订单创建、订单列表、订单详情；
- 基础库存扣减和释放；
- Mock 支付或第三方支付最小链路；
- C 端 H5 基础购物页面；
- Admin 基础商品、订单、库存、用户管理。

MVP 阶段不包含：

- 售后；
- 退款；
- 对账；
- 营销；
- 物流；
- 搜索；
- 消息；
- 评价；
- 会员；
- 客服；
- 风控；
- 完整生产治理。

## 7.2 一期基础企业级交易闭环

一期在 MVP 基础上开始补齐基础企业级交易保障能力。

一期目标是：

> 让系统不仅能够完成交易，还能够处理基础异常、基础售后、支付退款、库存预占、订单超时和后台运营管理。

一期包含：

- 简单售后退款；
- 支付退款；
- 支付回调记录；
- 订单超时取消；
- 自动确认收货；
- 库存预占与释放；
- 基础业务对账；
- 后台权限与账号体系；
- 操作日志与审计；
- 支付、售后、对账、库存后台管理。

## 7.3 一期增强能力

一期增强用于加强一期交易链路中风险最高、最容易出现数据不一致的环节。

一期增强重点包括：

- 库存防超卖；
- 库存补偿和一致性校验；
- 第三方支付渠道抽象；
- 支付回调幂等和状态核验；
- 支付异常处理；
- 订单、支付、退款、售后的对账闭环；
- 线上人工对账工作台；
- 后台多角色 Dashboard；
- 认证会话策略和强制下线；
- 第三方登录能力。

## 7.4 二期运营履约能力

二期目标是从“能交易”升级到“能运营、能履约、能触达用户”。

二期规划包括：

- 物流履约；
- 优惠券；
- 营销活动；
- 商品搜索；
- 消息通知；
- 商品评价；
- 售后增强；
- 首页装修和推荐位。

## 7.5 三期平台经营能力

三期目标是从“业务交易系统”升级为“企业经营平台”。

三期规划包括：

- 会员等级；
- 积分体系；
- 数据统计与 BI；
- 客服工单；
- 内容装修；
- 用户画像；
- 风控中心；
- 精细化运营。

## 7.6 四期生产治理能力

四期目标是让系统具备真实生产环境运行能力。

四期规划包括：

- CI/CD；
- Docker / Kubernetes 部署；
- 自动化测试；
- 链路追踪；
- 监控告警；
- 限流、熔断、降级；
- MQ 死信和补偿；
- 安全合规；
- 容灾备份；
- 灰度发布；
- 容量评估和压测体系。

## 8. 总体架构原则

## 8.1 模块化优先

项目初期采用模块化单体，按业务域拆分后端模块，保证边界清晰、开发和部署简单。后续根据业务复杂度再逐步演进为服务化架构。

## 8.2 核心链路稳定优先

订单、库存、支付、售后、对账属于电商系统核心高风险链路，设计时应优先考虑状态一致性、幂等、补偿、异常处理和人工兜底。

## 8.3 前后端职责清晰

前端负责页面展示、用户交互和基础表单校验；后端负责业务规则、状态变更、数据一致性、权限校验和审计记录。

## 8.4 阶段边界明确

MVP 阶段只做最小交易闭环，不提前引入过多复杂能力；一期开始补齐基础企业级交易保障；二期补运营履约；三期补经营平台；四期补生产治理。

## 8.5 文档与实现一致

模块设计文档应先描述完整企业级形态，再说明各阶段交付切片。开发过程中应根据分期路线逐步实现，避免模块能力随意膨胀或职责混乱。

## 9. 总体技术选型建议

| 层级 | 技术建议 | 说明 |
|---|---|---|
| C 端前端 | Vue 3 + Vite + Vant | 适合 H5 商城快速开发 |
| Admin 前端 | Vue 3 + Vite + Element Plus + ECharts | 适合企业后台和数据看板 |
| 后端框架 | Spring Boot 3 | 企业 Java 应用基础框架 |
| 权限认证 | Sa-Token | 支持登录态、权限、会话管理 |
| ORM | MyBatis-Plus | 简化数据库访问 |
| 数据库 | MySQL | 核心业务数据存储 |
| 缓存 | Redis | 登录态、验证码、库存、缓存、限流 |
| 文件存储 | Local / MinIO | 本地与对象存储适配 |
| 支付渠道 | Mock Pay / Alipay Sandbox / WeChat Pay | 分阶段接入支付能力 |
| 消息队列 | RabbitMQ 或 RocketMQ | 订单、库存、支付、消息通知异步解耦 |
| 搜索引擎 | Elasticsearch / OpenSearch | 二期商品搜索能力 |
| 部署 | Docker / Docker Compose / Kubernetes | 从演示环境逐步演进到生产环境 |
| 监控 | Prometheus + Grafana | 四期生产治理 |
| 日志追踪 | TraceId + 链路追踪组件 | 四期生产治理 |

## 10. 文档体系规划

项目设计文档建议采用以下结构：

```text
documents/企业级电商项目重构版设计文档/

  00-全局设计/
  01-模块完整设计/
  02-分期路线与交付切片/
  03-关键专题设计/
  04-接口联调测试与压测/
  05-项目复盘与求职展示/
```

其中：

- `00-全局设计`：定义项目目标、业务域、架构、阶段路线和核心流程；
- `01-模块完整设计`：逐模块描述从 MVP 到完整企业级形态的能力设计；
- `02-分期路线与交付切片`：说明每个阶段从各模块中交付哪些能力；
- `03-关键专题设计`：针对库存防超卖、支付回调、对账闭环等复杂问题做专题说明；
- `04-接口联调测试与压测`：沉淀接口测试、联调任务和压测报告；
- `05-项目复盘与求职展示`：用于项目评估、简历表达和面试讲解。

## 11. 总结

`mallFei` 的目标是建设一个按照企业级电商业务域拆分、分阶段演进、最终覆盖完整交易与运营闭环的全栈电商系统。

项目建设应遵循以下路线：

1. MVP 阶段先完成最小商城交易闭环；
2. 一期补齐基础售后、退款、库存预占、订单超时、基础对账和后台权限；
3. 一期增强重点强化库存防超卖、第三方支付、支付回调、对账闭环和后台审计；
4. 二期补齐物流、营销、搜索、消息、评价等运营履约能力；
5. 三期建设会员、数据、客服、内容、风控等平台经营能力；
6. 四期完善 CI/CD、测试、监控、高可用、安全合规等生产治理能力。

后续开发应以本文为总体蓝图，再通过模块完整设计文档明确每个业务域的完整形态，并通过分期路线文档确定各阶段交付范围。

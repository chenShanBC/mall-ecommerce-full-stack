# B2C 电商 MVP 项目开发主文档（后端）

## 三、核心业务设计

## 3.1 MVP 阶段业务范围说明

MVP 阶段的目标是完成 B2C 电商系统最小可用交易闭环，确保用户能够完成从注册登录、浏览商品、加入购物车、提交订单、完成支付到查看订单的基础流程。

MVP 阶段重点验证以下主链路：

```text
用户注册/登录
  -> 浏览商品
  -> 查看商品详情
  -> 加入购物车
  -> 购物车结算
  -> 创建订单
  -> 基础库存处理
  -> 发起支付
  -> 支付成功
  -> 查看订单
```

MVP 阶段只建设支撑主交易链路所需的基础能力，不引入复杂企业级扩展功能。

### 3.1.1 MVP 阶段包含模块

MVP 阶段后端包含以下模块：

| 模块 | 建议模块名 | MVP 职责 |
|---|---|---|
| 公共模块 | `mall-common` | 统一响应、异常、枚举、通用工具、基础认证对象 |
| 认证模块 | `mall-auth` | 用户/管理员基础登录态、接口身份识别 |
| 用户模块 | `mall-user` | 用户注册、登录、资料、地址 |
| 商品模块 | `mall-product` | 分类、SPU、SKU、商品列表、商品详情、后台商品维护 |
| 购物车模块 | `mall-cart` | 加购、改数量、删除、勾选、结算预览 |
| 订单模块 | `mall-order` | 创建订单、订单列表、订单详情、订单基础状态 |
| 库存模块 | `mall-stock` | SKU 库存、下单基础库存处理、取消释放、后台调整 |
| 支付模块 | `mall-pay` | 支付单、Mock 支付或沙箱支付最小链路、支付结果同步 |
| 文件模块 | `mall-file` | 商品图片等基础文件上传和访问 |
| 后台模块 | `mall-admin` | 管理员登录、商品/订单/库存/用户基础管理 |
| 启动模块 | `mall-start` | Spring Boot 启动入口和模块集成 |

### 3.1.2 MVP 阶段不包含模块

MVP 阶段明确不包含以下内容：

- 售后模块；
- 退款流程；
- 支付退款；
- 支付对账；
- 订单支付售后对账；
- 线上人工对账；
- 复杂库存补偿；
- 库存对账；
- 物流履约；
- 优惠券；
- 营销活动；
- 商品搜索；
- 消息通知；
- 商品评价；
- 会员积分；
- 客服工单；
- 风控体系；
- 数据 BI；
- 完整 CI/CD 和生产级监控治理。

以上能力从一期或后续阶段开始逐步建设。

## 3.2 用户与认证业务设计

## 3.2.1 业务目标

用户与认证模块用于支撑 C 端用户进入商城并完成交易，同时支撑 Admin 管理员进入后台进行基础运营管理。

MVP 阶段目标：

- 用户可以注册账号；
- 用户可以登录系统；
- 用户可以维护个人基础资料；
- 用户可以维护收货地址；
- 管理员可以登录后台；
- 后端能够识别当前请求是 C 端用户还是后台管理员。

## 3.2.2 用户注册流程

```text
用户提交注册信息
  -> 校验手机号/账号是否已存在
  -> 校验密码格式
  -> 加密密码
  -> 创建用户账号
  -> 返回注册结果
```

### 关键规则

1. 手机号或账号应唯一；
2. 密码不得明文存储；
3. 注册成功后可以直接登录，也可以要求用户重新登录；
4. 用户初始状态应为正常启用状态。

## 3.2.3 用户登录流程

```text
用户提交账号和密码
  -> 查询用户账号
  -> 校验账号状态
  -> 校验密码
  -> 创建登录态
  -> 返回用户信息和登录凭证
```

### 关键规则

1. 账号不存在或密码错误时返回统一错误提示；
2. 被禁用用户不得登录；
3. 登录成功后后端应生成可识别用户身份的登录态；
4. 后续需要登录的接口应校验登录态。

## 3.2.4 用户资料流程

```text
用户登录
  -> 查看个人资料
  -> 修改昵称、头像、性别等基础信息
  -> 后端校验并保存
  -> 返回最新资料
```

MVP 阶段用户资料只做基础字段，不涉及会员等级、积分、成长值和用户画像。

## 3.2.5 用户地址流程

```text
用户登录
  -> 新增收货地址
  -> 编辑收货地址
  -> 删除收货地址
  -> 设置默认地址
  -> 下单时选择地址
```

### 关键规则

1. 地址必须归属于当前登录用户；
2. 用户只能操作自己的地址；
3. 下单时必须校验地址存在且属于当前用户；
4. 默认地址同一用户只能有一个。

## 3.2.6 管理员登录流程

```text
管理员提交账号密码
  -> 查询管理员账号
  -> 校验密码和账号状态
  -> 创建管理员登录态
  -> 返回后台登录结果
```

MVP 阶段管理员权限可以先做基础身份识别，不要求复杂 RBAC、菜单权限和按钮权限。

## 3.3 商品业务设计

## 3.3.1 业务目标

商品模块用于支撑 C 端商品展示和 Admin 后台商品维护。

MVP 阶段商品模块需要支持：

- 商品分类；
- SPU 商品；
- SKU 规格商品；
- 商品列表；
- 商品详情；
- 商品上下架；
- 后台新增、编辑、查看商品。

## 3.3.2 商品模型

MVP 阶段商品模型采用 SPU + SKU 结构。

### SPU

SPU 表示标准商品，例如“某品牌 T 恤”。

主要字段建议：

- 商品 ID；
- 商品名称；
- 分类 ID；
- 主图；
- 商品描述；
- 商品状态；
- 销量；
- 创建时间；
- 更新时间。

### SKU

SKU 表示具体可售规格，例如“白色 / L 码”。

主要字段建议：

- SKU ID；
- SPU ID；
- SKU 名称；
- 规格属性；
- 售价；
- 划线价；
- SKU 图片；
- SKU 状态；
- 创建时间；
- 更新时间。

### 分类

分类用于组织商品。

主要字段建议：

- 分类 ID；
- 父级分类 ID；
- 分类名称；
- 排序；
- 启用状态。

## 3.3.3 商品列表流程

```text
C端请求商品列表
  -> 商品模块查询上架商品
  -> 按分类、分页条件过滤
  -> 组装商品卡片信息
  -> 返回商品列表
```

### 关键规则

1. C 端只展示上架商品；
2. 商品列表应支持分页；
3. 商品卡片展示价格可以取最低 SKU 价格或默认 SKU 价格；
4. 下架商品不能进入购买流程。

## 3.3.4 商品详情流程

```text
用户点击商品
  -> 查询 SPU 信息
  -> 查询 SKU 列表
  -> 查询分类和图片信息
  -> 返回商品详情
```

### 关键规则

1. 商品详情必须包含可售 SKU；
2. SKU 状态异常时不可购买；
3. 页面展示库存仅作为参考，最终以下单时库存校验为准。

## 3.3.5 后台商品管理流程

```text
管理员登录后台
  -> 新增商品 SPU
  -> 配置 SKU
  -> 上传商品图片
  -> 设置商品价格和状态
  -> 保存商品
  -> 上架或下架商品
```

### 关键规则

1. 商品必须至少有一个 SKU；
2. 商品上架前应校验基础信息完整；
3. 下架商品不可被 C 端购买；
4. 商品价格变更不影响已创建订单，订单中应保存商品快照。

## 3.4 购物车业务设计

## 3.4.1 业务目标

购物车模块用于暂存用户有购买意向的商品，并支撑结算前的商品选择和金额预览。

MVP 阶段购物车需要支持：

- 加入购物车；
- 修改商品数量；
- 删除购物车项；
- 勾选/取消勾选商品；
- 查询购物车列表；
- 结算预览。

## 3.4.2 加入购物车流程

```text
用户选择 SKU 和数量
  -> 提交加购请求
  -> 校验用户登录态
  -> 校验商品和 SKU 是否存在
  -> 校验 SKU 是否可售
  -> 校验库存是否满足基础要求
  -> 如果购物车已有该 SKU，则合并数量
  -> 如果没有，则新增购物车项
  -> 返回加购结果
```

### 关键规则

1. 用户必须登录后才能加入购物车；
2. 已下架商品不可加入购物车；
3. SKU 不存在或不可售时不可加入购物车；
4. 加购数量必须大于 0；
5. 同一用户同一 SKU 在购物车中只保留一条记录。

## 3.4.3 修改数量流程

```text
用户修改购物车商品数量
  -> 校验购物车项归属当前用户
  -> 校验数量是否合法
  -> 校验商品和 SKU 状态
  -> 更新数量
  -> 返回最新购物车信息
```

## 3.4.4 购物车结算预览流程

```text
用户勾选购物车商品
  -> 点击结算
  -> 查询选中购物车项
  -> 校验商品状态
  -> 校验 SKU 状态
  -> 校验基础库存
  -> 计算商品总金额
  -> 返回结算预览
```

### 关键规则

1. 结算预览时必须重新校验商品状态；
2. 商品价格以后端实时查询结果为准；
3. 库存不足时应提示用户调整数量；
4. 结算预览不锁定库存，真正库存处理发生在创建订单时。

## 3.5 订单业务设计

## 3.5.1 业务目标

订单模块是 MVP 阶段的核心模块，用于承接用户提交订单、保存订单信息、维护订单状态，并为支付和后续订单查询提供基础数据。

MVP 阶段订单模块需要支持：

- 创建订单；
- 保存订单项；
- 保存商品快照；
- 订单列表；
- 订单详情；
- 订单基础状态流转；
- 后台订单查看。

## 3.5.2 订单基础状态

MVP 阶段建议订单状态包括：

| 状态 | 含义 |
|---|---|
| 待支付 | 订单已创建，等待用户支付 |
| 已支付 | 支付成功，订单完成 MVP 阶段支付闭环 |
| 已取消 | 用户取消或系统取消 |
| 已完成 | 订单完成，MVP 可由后台或系统简化处理 |

MVP 阶段不引入复杂履约状态，不区分待发货、已发货、待收货、售后中等状态。

## 3.5.3 创建订单流程

```text
用户在结算页提交订单
  -> 校验用户登录态
  -> 校验收货地址
  -> 查询购物车选中商品
  -> 校验商品和 SKU 状态
  -> 校验库存
  -> 生成商品快照
  -> 计算订单金额
  -> 创建订单主表
  -> 创建订单项
  -> 处理基础库存
  -> 创建支付单
  -> 返回订单号和支付信息
```

### 关键规则

1. 用户只能使用自己的地址下单；
2. 下单时必须重新校验商品状态和价格；
3. 订单金额以后端计算为准；
4. 订单项必须保存商品快照，避免商品后续变更影响历史订单；
5. 下单成功后订单进入待支付状态；
6. 创建订单与库存处理需要保证一致性。

## 3.5.4 商品快照规则

订单创建时应保存商品快照，至少包括：

- 商品 ID；
- SKU ID；
- 商品名称；
- SKU 规格；
- 商品图片；
- 下单单价；
- 购买数量。

商品快照用于保证订单历史数据不受商品后续编辑、改价、下架影响。

## 3.5.5 订单列表流程

```text
用户登录
  -> 请求订单列表
  -> 订单模块按用户查询订单
  -> 返回订单摘要、金额、状态、商品概览
```

### 关键规则

1. 用户只能查看自己的订单；
2. 订单列表应支持分页；
3. 后台管理员可以按条件查看全部订单。

## 3.5.6 订单详情流程

```text
用户点击订单
  -> 校验订单归属当前用户
  -> 查询订单主信息
  -> 查询订单项
  -> 查询支付状态
  -> 返回订单详情
```

## 3.6 库存业务设计

## 3.6.1 业务目标

MVP 阶段库存模块用于支撑商品可售校验和下单后的基础库存处理。

MVP 阶段库存模块需要支持：

- SKU 库存记录；
- 查询库存；
- 下单时校验库存；
- 下单时基础扣减或占用库存；
- 订单取消时释放库存；
- 后台调整库存。

MVP 阶段不要求实现复杂库存补偿、防超卖压测、多仓库存、库存对账等能力。

## 3.6.2 库存模型

建议库存以 SKU 为维度管理。

主要字段建议：

- 库存 ID；
- SKU ID；
- 可售库存；
- 锁定库存；
- 已售数量；
- 预警库存；
- 更新时间。

MVP 阶段可以简化为可售库存字段，但设计上应预留锁定库存扩展。

## 3.6.3 下单库存处理流程

```text
订单创建时
  -> 查询 SKU 库存
  -> 判断库存是否充足
  -> 库存充足则扣减或占用库存
  -> 库存不足则下单失败
  -> 返回库存处理结果
```

### 关键规则

1. 库存不足时不允许下单；
2. 同一订单中多个 SKU 应全部库存处理成功后才能创建成功；
3. 库存处理失败时订单不应创建成功；
4. MVP 阶段可以使用数据库事务保证基础一致性。

## 3.6.4 取消订单释放库存流程

```text
订单取消
  -> 查询订单项
  -> 根据订单项 SKU 和数量恢复库存
  -> 更新订单状态
```

MVP 阶段订单取消可以由用户主动取消或后台取消触发，不要求实现复杂超时取消任务。

## 3.6.5 后台库存调整流程

```text
管理员进入库存管理
  -> 查询 SKU 库存
  -> 输入调整数量
  -> 后端校验调整结果不能小于 0
  -> 更新库存
  -> 返回调整结果
```

MVP 阶段库存调整可以只做基础记录，详细库存操作日志和审计可放到一期增强。

## 3.7 支付业务设计

## 3.7.1 业务目标

MVP 阶段支付模块用于验证订单支付闭环，使订单能够从待支付流转到已支付。

MVP 阶段支付模块需要支持：

- 创建支付单；
- Mock 支付或第三方支付沙箱最小链路；
- 支付成功后更新支付单状态；
- 支付成功后更新订单状态；
- C 端支付返回页查询支付结果。

MVP 阶段不要求实现支付退款、支付对账、复杂回调幂等、渠道账单和清结算。

## 3.7.2 支付单模型

支付单用于记录订单支付信息。

主要字段建议：

- 支付单 ID；
- 支付单号；
- 订单 ID；
- 用户 ID；
- 支付金额；
- 支付渠道；
- 支付状态；
- 支付时间；
- 创建时间；
- 更新时间。

## 3.7.3 支付状态

MVP 阶段支付状态建议包括：

| 状态 | 含义 |
|---|---|
| 待支付 | 支付单已创建，等待支付 |
| 支付成功 | 用户支付成功 |
| 支付失败 | 支付失败或取消 |
| 已关闭 | 支付单关闭 |

## 3.7.4 Mock 支付流程

```text
用户点击支付
  -> 查询订单和支付单
  -> 校验订单为待支付
  -> 模拟支付成功
  -> 更新支付单为支付成功
  -> 更新订单为已支付
  -> 返回支付结果
```

Mock 支付用于本地开发和联调阶段，保证不依赖真实支付渠道也能跑通交易闭环。

## 3.7.5 第三方支付沙箱最小流程

```text
用户点击支付
  -> 支付模块创建支付请求
  -> 调用第三方支付沙箱接口
  -> 返回支付跳转信息
  -> 用户完成沙箱支付
  -> 支付渠道回调后端
  -> 后端更新支付单状态
  -> 后端更新订单状态
```

MVP 阶段如果接入第三方支付，只要求跑通最小支付链路，不要求完整生产级验签、对账、清结算能力。

## 3.8 文件服务业务设计

## 3.8.1 业务目标

文件服务模块用于支撑商品图片、用户头像等基础文件上传和访问。

MVP 阶段文件服务需要支持：

- 文件上传；
- 文件记录保存；
- 文件访问 URL 返回；
- 商品图片关联。

## 3.8.2 文件上传流程

```text
管理员选择商品图片
  -> 前端上传文件
  -> 文件服务校验文件类型和大小
  -> 保存文件到本地或对象存储
  -> 保存文件记录
  -> 返回文件访问地址
  -> 商品模块保存图片地址
```

### 关键规则

1. 文件大小应有限制；
2. 文件类型应有白名单；
3. 文件访问地址应可被前端展示；
4. MVP 阶段可以先使用本地存储，后续扩展对象存储。

## 3.9 Admin 后台业务设计

## 3.9.1 业务目标

Admin 后台用于支撑 MVP 阶段基础运营管理。

MVP 阶段后台需要支持：

- 管理员登录；
- 商品管理；
- 订单查看；
- 用户查看；
- 库存管理；
- 基础 Dashboard。

MVP 阶段后台不要求完整角色权限、操作日志、支付管理、售后管理、对账管理等能力，这些从一期开始建设。

## 3.9.2 商品管理流程

```text
管理员登录后台
  -> 创建商品分类
  -> 创建商品 SPU
  -> 创建商品 SKU
  -> 上传商品图片
  -> 设置价格和状态
  -> 上架商品
```

## 3.9.3 订单查看流程

```text
管理员进入订单管理
  -> 查询订单列表
  -> 按订单号、用户、状态筛选
  -> 查看订单详情
```

MVP 阶段后台订单管理以查看为主，不要求复杂异常处理。

## 3.9.4 用户查看流程

```text
管理员进入用户管理
  -> 查询用户列表
  -> 查看用户基础信息
  -> 查看用户地址或订单概况
```

## 3.9.5 库存管理流程

```text
管理员进入库存管理
  -> 查询 SKU 库存
  -> 调整库存数量
  -> 保存调整结果
```

## 3.9.6 基础 Dashboard

MVP 阶段 Dashboard 只需要展示基础指标，例如：

- 商品数量；
- 用户数量；
- 订单数量；
- 今日订单数；
- 今日支付金额。

复杂财务、仓储、运营多角色看板放到一期增强或三期数据统计阶段。

## 3.12 MVP 数据一致性设计

## 3.12.1 下单一致性

创建订单时涉及订单、订单项、库存、支付单等数据。

MVP 阶段建议使用数据库事务保证以下操作的一致性：

1. 校验商品和 SKU；
2. 校验并处理库存；
3. 创建订单；
4. 创建订单项；
5. 创建支付单。

如果任一环节失败，应回滚整个下单流程。

## 3.12.2 支付一致性

支付成功后需要同步支付单和订单状态。

MVP 阶段可在同一事务中完成：

1. 更新支付单状态为支付成功；
2. 更新订单状态为已支付。

后续一期增强再补充支付回调幂等、支付状态核验和对账。

## 3.12.3 库存一致性

MVP 阶段库存处理可以采用数据库事务和行级更新保证基础一致性。

后续一期增强再补充：

- 库存预占；
- 库存锁定记录；
- 防超卖；
- 超时释放；
- 补偿任务；
- 库存对账。

## 3.13 MVP 异常场景设计

| 异常场景 | MVP 处理策略 |
|---|---|
| 用户未登录访问下单接口 | 返回未登录错误 |
| 商品已下架 | 禁止加购或下单 |
| SKU 不存在 | 返回商品不可用 |
| 库存不足 | 下单失败，提示库存不足 |
| 地址不存在 | 下单失败，提示地址无效 |
| 支付单不存在 | 返回支付单不存在 |
| 订单不是待支付 | 不允许再次支付 |
| 支付失败 | 支付单标记失败，订单保持待支付或关闭 |
| 用户查看他人订单 | 返回无权限 |
| 管理员未登录访问后台 | 返回未登录或无权限 |

MVP 阶段只处理基础异常，复杂异常补偿和人工处理放到一期增强。

## 3.14 MVP 阶段验收标准

MVP 阶段完成后，应满足以下验收标准：

1. 用户可以注册并登录；
2. 用户可以维护个人资料和收货地址；
3. 后台可以创建分类、商品 SPU 和 SKU；
4. C 端可以查看商品列表和商品详情；
5. 用户可以选择 SKU 并加入购物车；
6. 用户可以修改购物车数量和勾选商品；
7. 用户可以进行结算预览；
8. 用户可以提交订单；
9. 系统可以生成订单、订单项和商品快照；
10. 系统可以完成基础库存校验和扣减/占用；
11. 系统可以创建支付单；
12. 用户可以通过 Mock 或沙箱支付完成支付；
13. 支付成功后订单状态变为已支付；
14. 用户可以查看订单列表和订单详情；
15. 后台可以查看订单、用户和库存；
16. C 端和 Admin 端能够完成完整联调；
17. Swagger 或接口文档能够覆盖 MVP 核心接口；
18. MVP 主链路无阻断性异常。

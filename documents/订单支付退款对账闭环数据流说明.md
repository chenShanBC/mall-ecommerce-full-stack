# 订单-支付-退款-对账完整闭环数据流说明

> 本文基于当前代码实现和 `backend/sql/structrue.sql` 表结构梳理，不依赖既有业务文档。重点描述从用户下单、支付、退款，到轻量对账与线上账单勾兑的完整数据流，以及每一步落到哪些表、记录哪些核心信息。

## 1. 总体闭环概览

当前系统的闭环主线可以概括为：

```text
用户创建订单
  -> 写入订单主表/明细表
  -> 预占库存并写库存锁/库存日志
  -> 创建支付单
  -> 渠道提交支付
  -> 支付回调/Mock 支付成功
  -> 记录支付回调
  -> 支付单成功
  -> 订单标记已支付并确认扣减库存
  -> 用户/后台发起退款或售后审核通过
  -> 写订单退款申请与退款明细
  -> MQ 通知支付模块发起退款
  -> 写支付退款单并推进支付单退款状态
  -> 渠道退款同步/退款查询
  -> 记录退款回调、支付退款单成功/失败、订单退款成功/失败
  -> 支付/退款轻量对账或线上对账任务生成账单快照
  -> 本地账单与渠道账单勾兑
  -> 形成差异项并人工处理/挂起/归档
```

其中最关键的业务号链路是：

| 业务号 | 产生位置 | 贯穿表 | 作用 |
| --- | --- | --- | --- |
| `order_no` | 订单模块创建订单 | `oms_order`、`oms_order_item`、`pay_order`、`pay_refund_order`、`oms_order_refund`、`aftersale_order`、各对账表 | 业务订单主线标识 |
| `pay_order_no` | 支付模块创建支付单 | `pay_order`、`pay_callback_record`、`pay_refund_order`、`pay_reconciliation_record`、线上对账账单/差异表 | 商户支付单号，也是渠道 `out_trade_no` 的核心匹配键 |
| `refund_no` | 订单退款申请创建 | `oms_order_refund`、`oms_order_refund_item`、`pay_refund_order`、`pay_callback_record`、`pay_reconciliation_record`、线上对账账单/差异表、`aftersale_order` | 本地退款单号，串起订单退款和支付退款 |
| `transaction_no` | 支付渠道返回 | `pay_order`、`pay_callback_record`、`pay_refund_order`、对账本地/渠道账单 | 第三方支付交易流水 |
| `channel_refund_no` | 退款渠道返回 | `oms_order_refund`、`pay_refund_order`、`pay_callback_record`、渠道账单 | 第三方退款流水 |

## 3. 阶段一：创建订单与库存预占

### 3.1 入口与代码流

用户下单进入 `OrderApplicationService.createOrder`：

1. 生成 `order_no`。
2. 读取当前用户 `user_id`。
3. 根据配置生成订单支付截止时间 `expire_time`。
4. 调用 `OrderDomainService.createPendingOrder` 构建待支付订单聚合。
5. 先调用 `StockFacade.reserve` 预占库存。
6. 库存预占成功后保存订单。
7. 发布订单超时关闭事件。

### 3.2 数据落表

#### 3.2.1 `oms_order`

创建一条订单主记录，核心信息包括：

- `order_no`：订单号，后续所有支付/退款/对账都用它关联。
- `user_id`：下单用户。
- `order_status`：初始为待支付类状态，代码中使用 `pendingPayment()` 判断。
- `total_amount_cent`、`pay_amount_cent`、`freight_amount_cent`、`discount_amount_cent`：订单金额快照。
- `receiver_name`、`receiver_phone`、`receiver_province_name`、`receiver_city_name`、`receiver_district_name`、`receiver_detail_address`：收货地址快照。
- `pay_type`：默认 `MOCK`，也可后续按支付渠道展示。
- `expire_time`：支付超时时间。
- `version`：乐观锁版本。

#### 3.2.2 `oms_order_item`

每个 SKU 一条订单明细快照：

- `order_id`、`order_no`：关联订单。
- `sku_id`、`spu_id`：商品标识。
- `sku_name`、`sku_image_url`：下单时商品快照，避免商品改名/换图影响历史订单。
- `sale_price_cent`、`quantity`、`total_amount_cent`：成交单价、数量、小计。

#### 3.2.3 `ims_stock_lock`

`StockFacade.reserve` 会以业务类型 `ORDER`、业务单号 `order_no` 预占库存，对每个 SKU 形成库存锁：

- `business_type = 'ORDER'`
- `business_no = order_no`
- `sku_id`
- `quantity`
- `status`：锁定/已释放/已扣减等库存锁状态。
- `lock_time`：预占时间。
- `reserved_synced`、`cancelled_synced`、`confirmed_synced`：用于标记库存锁动作是否已同步到 DB 库存表。

#### 3.2.4 `ims_stock` 与 `ims_stock_operation_log`

预占库存会影响 SKU 库存：

- `ims_stock.locked_stock` 增加。
- `ims_stock.available_stock` 减少。
- `ims_stock.total_stock = locked_stock + available_stock` 由约束保证。
- `ims_stock_operation_log` 记录库存变化前后的快照，通常包括：
  - `operation_type`：预占/释放/扣减/恢复等操作类型。
  - `business_type = 'ORDER'`
  - `business_no = order_no`
  - `change_quantity`
  - `before_total_stock`、`before_locked_stock`、`before_available_stock`
  - `after_total_stock`、`after_locked_stock`、`after_available_stock`

## 4. 阶段二：创建支付单并提交渠道支付

### 4.1 入口与代码流

支付入口在 `PayApplicationService.createPayOrder(orderNo, payChannel, returnPath)`：

1. 先调用 `orderFacade.closeIfTimedOut(orderNo)`，防止过期订单继续支付。
2. 读取 `oms_order` 订单。
3. 校验订单是否可支付。
4. 标准化支付渠道，例如 `MOCK`、`ALIPAY_WAP`、`ALIPAY_PC`。
5. 如果同一订单同一渠道存在可复用支付单，则复用。
6. 否则关闭其它待支付支付单，创建新的 `pay_order`。
7. 调渠道客户端生成支付提交信息，如跳转表单或 URL。
8. 将提交展示载荷保存回支付单。

### 4.2 数据落表：`pay_order`

创建或复用支付单，核心字段：

- `pay_order_no`：支付单号，当前实现中支付宝回调的 `out_trade_no` 必须与它一致。
- `order_no`：关联业务订单。
- `user_id`：支付用户。
- `pay_amount_cent`：支付金额，后续回调会和回调金额校验。
- `pay_status`：初始为待支付/支付中类状态；成功后变为成功；退款中会变成 `REFUND_PENDING`、`REFUNDING`、`REFUNDED`、`PARTIALLY_REFUNDED`、`REFUND_FAILED` 等。
- `pay_channel`：支付渠道。
- `transaction_no`：支付成功后写入第三方交易号。
- `callback_payload`：支付提交展示载荷或回调相关信息。
- `notify_time`、`paid_at`：渠道通知时间/支付完成时间。
- `idempotent_key`：支付单幂等键，有唯一索引。

此阶段还没有形成 `pay_callback_record`，因为还未收到支付成功或渠道回调。

## 5. 阶段三：支付成功回调与订单支付状态同步

### 5.1 Mock 支付成功

`PayApplicationService.markMockSuccess(orderNo)` 会构造一条模拟渠道回调：

- `channel = MOCK`
- `payOrderNo = pay_order.pay_order_no`
- `orderNo = oms_order.order_no`
- `outTradeNo = pay_order.pay_order_no`
- `transactionNo = MOCK-{orderNo}` 或已有交易号
- `tradeStatus = TRADE_SUCCESS`
- `amountCent = pay_order.pay_amount_cent`

然后统一进入 `handleChannelCallback`。

### 5.2 支付宝回调

`handleAlipayCallback(params)` 从支付宝参数中组装 `PayChannelCallbackRequest`：

- `out_trade_no` -> `pay_order_no`
- `passback_params` -> `order_no`
- `trade_no` -> `transaction_no`
- `trade_status` -> 渠道交易状态
- `total_amount` -> `amount_cent`
- `sign` -> 签名

然后也进入 `handleChannelCallback`。

### 5.3 回调处理数据流

`handleChannelCallback` 的核心步骤：

1. 调渠道客户端验签。
2. 先写入 `pay_callback_record`，记录原始回调。
3. 定位本地 `pay_order`。
4. 校验：验签、订单号/支付单号、金额、交易状态。
5. 如果已成功，标记回调为重复忽略。
6. 如果是支付异常订单，可能只记录支付单成功，等待后台确认。
7. 正常情况下调用 `orderFacade.markPaid(orderNo)`。
8. `orderFacade.markPaid` 内部会确认扣减库存，并把订单状态置为已支付。
9. 更新 `pay_order` 为成功。
10. 更新 `pay_callback_record.process_status` 为已处理。

### 5.4 数据落表

#### 5.4.1 `pay_callback_record`

无论后续是否处理成功，系统都会先记录一条支付回调：

- `callback_type = 'PAY'`
- `channel`：支付渠道。
- `pay_order_no`
- `order_no`
- `out_trade_no`：商户交易号，要求与 `pay_order_no` 一致。
- `transaction_no`：渠道交易号。
- `amount_cent`：渠道通知金额。
- `trade_status`：如 `TRADE_SUCCESS`、`TRADE_FINISHED`。
- `signature`：签名。
- `verified`：验签结果。
- `process_status`：处理状态，例如处理成功、验签失败、业务身份不匹配、金额不一致、重复忽略、非成功状态忽略等。
- `fail_reason`：失败原因。
- `raw_payload`：原始回调报文。
- `callback_time`、`processed_at`：回调时间/处理完成时间。

#### 5.4.2 `pay_order`

支付成功后更新：

- `pay_status`：变为成功状态。
- `transaction_no`：写入渠道交易号。
- `callback_payload`：可能记录回调/提交载荷。
- `notify_time`：通知时间。
- `paid_at`：支付完成时间。
- `version`：乐观锁递增。

如果订单已经进入支付异常状态，则代码会记录成功支付单，但不会直接恢复订单履约，而是留给后台核验。

#### 5.4.3 `oms_order`

`OrderApplicationService.markPaid(orderNo)` 会：

- 如果订单是支付异常状态，则跳过自动流转。
- 如果不是待支付，则直接返回。
- 对待支付订单调用 `stockFacade.confirm` 确认扣减库存。
- 更新订单为已支付状态，记录 `paid_at`。

`oms_order` 更新核心字段：

- `order_status`：从待支付变成已支付/待发货类状态。
- `paid_at`：支付时间。
- `updated_at`、`version`。

#### 5.4.4 `ims_stock_lock`、`ims_stock`、`ims_stock_operation_log`

支付成功确认扣减库存：

- `ims_stock_lock` 中 `business_type = 'ORDER'`、`business_no = order_no` 的锁记录进入已扣减状态，写 `deduct_time`，并设置 `confirmed_synced`。
- `ims_stock.locked_stock` 减少，`total_stock` 可能随确认扣减减少，表示库存真正消耗。
- `ims_stock_operation_log` 记录扣减流水，`business_no = order_no`。

## 6. 阶段四：订单取消/超时关闭与支付单关闭

虽然不是退款主线，但它是支付闭环的重要分支。

### 6.1 用户取消或超时取消

订单取消入口包括：

- 用户主动取消：`OrderApplicationService.cancelOrder`
- 请求守卫发现超时：`cancelTimedOutOrderIfNecessary`
- 定时扫描：`autoCancelAllTimedOutOrders`
- MQ 延迟消息：`closeIfTimedOut`

取消后：

1. `oms_order.order_status` 更新为取消或超时取消。
2. 调用 `stockFacade.cancel` 释放 `ORDER + order_no` 的预占库存。
3. 发布订单取消事件。
4. 支付模块消费后关闭待支付的 `pay_order`。

### 6.2 表变化

- `oms_order`：`order_status`、`cancelled_at` 更新。
- `ims_stock_lock`：对应库存锁写 `release_time`，状态改为释放。
- `ims_stock`：释放锁定库存，`locked_stock` 减少、`available_stock` 增加。
- `ims_stock_operation_log`：记录释放流水。
- `pay_order`：待支付支付单被关闭，`pay_status` 变成关闭类状态。

支付模块在关闭支付单前还会查询渠道，如果渠道已支付，会阻止关闭并进入补偿/对账逻辑，避免“用户已付款但本地关闭订单”的资损。

## 7. 阶段五：发起退款申请

### 7.1 用户直接退款

用户退款入口为 `OrderApplicationService.applyRefund(orderId, request)`：

1. 读取当前用户订单。
2. 检查订单是否超时。
3. 调用 `createRefund`。
4. `order.ensureRefundable()` 校验订单可退款。
5. `ensureNoRefundInProgress(orderNo)` 防止同一订单存在进行中的退款。
6. 生成 `refund_no`。
7. 根据请求构造 `OrderRefundItem` 明细。
8. 计算退款金额。
9. 保存 `oms_order_refund`。
10. 保存 `oms_order_refund_item`。
11. 将退款申请状态推进为退款中。
12. 发布 `OrderRefundRequestedEvent`，由支付模块发起支付退款。

### 7.2 后台/售后退款

后台发起退款有两个路径：

- `applyRefundByAdmin`：后台直接提交到支付渠道。
- `createRefundApplicationByAdmin`：先创建退款申请，等待售后审核。
- `approveRefundApplicationByAdmin(refundNo)`：审核通过后把退款单推进到退款中，并发布退款请求事件。

售后场景会额外写入 `aftersale_order`：

- `aftersale_no`：售后单号。
- `order_no`、`user_id`。
- `aftersale_type`：售后类型。
- `status`：待审核、已通过、已驳回、退款中、退款成功、退款失败等状态。
- `origin_order_status`：申请售后前订单状态。
- `refund_amount_cent`。
- `reason`、`reject_reason`。
- `refund_no`：审核通过后关联订单退款单。
- `fail_reason`。
- `reviewed_at`。

### 7.3 数据落表

#### 7.3.1 `oms_order_refund`

保存订单侧退款主记录：

- `order_id`、`order_no`、`user_id`
- `refund_no`：后续支付退款、退款回调、对账都用它关联。
- `refund_amount_cent`：退款金额。
- `refund_status`：初始申请态，提交支付后变为退款中，最后成功/失败。
- `refund_reason`。
- `channel_refund_no`：支付渠道退款成功后回写。
- `fail_reason`：退款失败原因。

#### 7.3.2 `oms_order_refund_item`

保存退款明细：

- `refund_id`：关联 `oms_order_refund.id`。
- `refund_no`。
- `order_item_id`：关联被退的订单明细。
- `sku_id`。
- `quantity`：退款数量。
- `refund_amount_cent`：该 SKU 明细退款金额。

这张表后续用于：

- 判断剩余可退数量。
- 退款成功后恢复库存。
- 退款成功后回滚 SKU 销量。

## 8. 阶段六：支付模块消费退款请求并创建支付退款单

### 8.1 MQ 消费

`PayRefundEventConsumer.onRefundRequested` 监听订单退款请求队列，收到 `OrderRefundRequestedEvent` 后调用：

```text
PayApplicationService.refund(orderNo, refundAmountCent, reason, refundNo)
```

事件中包含：

- `orderNo`
- `refundNo`
- `refundAmountCent`
- `reason`

### 8.2 支付退款处理

`PayApplicationService.refund` 的核心逻辑：

1. 根据 `order_no` 读取 `pay_order`。
2. 校验支付单必须成功或处于退款失败可重试状态。
3. 校验退款金额必须大于 0 且不超过支付金额。
4. 使用订单侧传来的 `refund_no`，如果未传则生成新退款号。
5. 查询是否已有 `pay_refund_order`，没有则创建。
6. 将 `pay_order` 标记为 `REFUND_PENDING`。
7. 再将 `pay_order` 标记为 `REFUNDING`。
8. 将 `pay_refund_order` 标记为退款中。

### 8.3 数据落表

#### 8.3.1 `pay_refund_order`

创建支付侧退款单：

- `refund_no`：与 `oms_order_refund.refund_no` 一致。
- `order_no`。
- `pay_order_no`：原支付单号。
- `user_id`。
- `pay_channel`。
- `refund_amount_cent`。
- `refund_status`：`REFUND_PENDING` / `REFUNDING` / `REFUND_SUCCESS` / `REFUND_FAILED`。
- `transaction_no`：原支付三方流水。
- `channel_refund_no`：渠道退款流水，成功后写入。
- `request_payload`：退款请求摘要。
- `response_payload`：退款响应摘要。
- `fail_reason`。
- `success_at`。

#### 8.3.2 `pay_order`

支付单进入退款流程：

- `pay_status`：从成功变为 `REFUND_PENDING`，再变为 `REFUNDING`。
- `updated_at`、`version` 更新。
- 支付单仍保留原 `transaction_no`，用于渠道退款。

此时订单侧 `oms_order_refund` 已经是退款中，但订单主表 `oms_order` 通常还未变为退款成功，等待支付退款结果。

## 9. 阶段七：退款渠道同步/查询与订单侧回写

### 9.1 退款同步入口

退款真正与渠道交互的入口在 `PayApplicationService.reconcileRefund(orderNo, refundNo, refundAmountCent)`。

这个方法既承担“向渠道发起/同步退款”的作用，也承担“查询渠道退款状态并补偿本地”的作用。

### 9.2 支付侧退款成功

当 `pay_refund_order.refund_status = REFUNDING` 时，代码调用：

```text
payChannelClientRouter.route(payOrder.payChannel()).refund(new PayRefundRequest(...))
```

如果渠道返回成功：

1. `pay_order` 标记退款成功。
2. `pay_refund_order` 标记成功，写 `channel_refund_no`、`response_payload`、`success_at`。
3. 写入 `pay_callback_record`，`callback_type = REFUND`。
4. 写入 `pay_reconciliation_record`，记录退款对账快照。
5. 发布退款成功事件给订单模块。

#### 9.2.1 `pay_order`

更新：

- `pay_status`：退款成功/已退款，部分退款时可能是部分退款状态。
- 退款相关时间与备注通过模型方法推进。

#### 9.2.2 `pay_refund_order`

更新：

- `refund_status = REFUND_SUCCESS`
- `channel_refund_no`
- `response_payload`
- `success_at`
- `updated_at`、`version`

#### 9.2.3 `pay_callback_record`

记录一条退款同步/回调记录：

- `callback_type = 'REFUND'`
- `channel`
- `pay_order_no`
- `refund_no`
- `order_no`
- `out_trade_no`：代码中退款记录使用 `refund_no`。
- `transaction_no`：此处通常写渠道退款号。
- `amount_cent`
- `trade_status = REFUND_SUCCESS`
- `signature = REFUND-SYNC`
- `verified = true`
- `raw_payload`：渠道退款响应。
- `process_status`：处理成功。

#### 9.2.4 `pay_reconciliation_record`

保存退款轻量对账记录：

- `batch_no = RCREFUND...`
- `biz_type = REFUND`
- `order_no`
- `pay_order_no`
- `refund_no`
- `local_status`：本地退款状态。
- `channel_status`：渠道退款状态。
- `local_amount_cent`
- `channel_amount_cent`
- `consistent`：金额和状态是否一致。
- `diff_type`：无差异或具体差异。
- `repair_status`
- `remark`：包含批次号和渠道返回信息。

### 9.3 支付侧退款失败

如果渠道退款返回失败：

- `pay_order.pay_status` 标记为 `REFUND_FAILED`。
- `pay_refund_order.refund_status` 标记为 `REFUND_FAILED`，写 `fail_reason` 和 `response_payload`。
- `pay_callback_record.callback_type = REFUND`，`process_status` 失败。
- `pay_reconciliation_record` 记录退款失败快照。
- 发布退款失败事件给订单模块。

### 9.4 订单模块消费退款结果

订单模块监听：

- `ORDER_REFUND_SUCCEEDED_QUEUE` -> `OrderApplicationService.onRefundSucceeded`
- `ORDER_REFUND_FAILED_QUEUE` -> `OrderApplicationService.onRefundFailed`

#### 9.4.1 退款成功回写订单

`onRefundSucceeded`：

1. 读取 `oms_order`。
2. 防重复：订单已退款则忽略。
3. 读取最新 `oms_order_refund`。
4. 校验事件中的 `refund_no` 和 `refund_amount_cent` 是否匹配。
5. 读取 `oms_order_refund_item`。
6. 判断全额退款还是部分退款。
7. 更新 `oms_order.order_status`：全额退款或部分退款。
8. 如果订单完成后曾累计销量，则按退款明细回滚 SKU 销量。
9. 调用 `stockFacade.restore`，按退款明细恢复库存。
10. 更新 `oms_order_refund` 为成功，写 `channel_refund_no`。

落表变化：

- `oms_order`
  - `order_status`：变为已退款或部分退款。
  - `updated_at`、`version`。
- `oms_order_refund`
  - `refund_status`：成功。
  - `channel_refund_no`：渠道退款流水。
  - `updated_at`。
- `ims_stock`
  - 退货恢复可售或总库存，取决于库存领域实现。
- `ims_stock_operation_log`
  - `business_type = 'ORDER_REFUND'`
  - `business_no = refund_no`
  - 记录退款恢复库存的前后快照。
- `pms_sku`
  - 如果需要回滚销量，`sales_count` 减少。

#### 9.4.2 退款失败回写订单

`onRefundFailed`：

1. 读取最新 `oms_order_refund`。
2. 校验 `refund_no` 和金额。
3. 如果退款已成功则忽略失败事件。
4. 更新 `oms_order_refund.refund_status` 为失败，写 `fail_reason`。

落表变化：

- `oms_order_refund.refund_status = REFUND_FAILED`
- `oms_order_refund.fail_reason`
- `oms_order` 主状态通常不会变为退款成功。

## 10. 阶段八：支付轻量对账

轻量对账由 `PayApplicationService.reconcile(orderNo)` 和退款同步中的 `recordRefundReconciliation` 写入 `pay_reconciliation_record`。

### 10.1 支付轻量对账

`reconcile(orderNo)` 流程：

1. 生成批次号 `RCPAY...`。
2. 读取 `oms_order`。
3. 读取 `pay_order`。
4. 查询支付渠道状态。
5. 如果渠道已支付但本地支付单未成功，尝试补偿 `pay_order` 成功。
6. 如果支付单成功但订单仍待支付，尝试补偿 `oms_order` 为已支付。
7. 比较金额一致性。
8. 比较订单、支付单、渠道状态一致性。
9. 写 `pay_reconciliation_record`。

### 10.2 支付对账差异类型

代码中根据场景判定差异，例如：

| 场景 | 差异含义 |
| --- | --- |
| 订单已支付但缺少支付单 | 平台订单状态异常或支付单缺失 |
| 本地支付成功但渠道未支付 | 平台长款/状态误置风险 |
| 渠道已支付但本地支付单或订单未成功 | 短款/漏同步风险，需要补偿 |
| 支付单成功但订单未支付 | 订单状态未同步 |
| 金额不一致 | 订单金额和支付金额不一致 |
| 状态不一致 | 订单、支付单、渠道状态不一致 |

### 10.3 `pay_reconciliation_record` 记录内容

支付对账写入：

- `batch_no = RCPAY...`
- `biz_type = PAY`
- `order_no`
- `pay_order_no`
- `local_status`：通常是订单状态/支付状态组合或支付单状态。
- `channel_status`：渠道查询状态。
- `local_amount_cent`：本地金额。
- `channel_amount_cent`：渠道或订单金额。
- `consistent`：是否一致。
- `diff_type`：差异类型，无差异为 `NONE`。
- `repair_status`：对异常记录标记待处理、已修复、忽略等。
- `remark`：批次、状态、金额、渠道查询结果摘要。

### 10.4 退款轻量对账

退款对账在 `reconcileRefund` 内部触发。无论退款成功、失败、未知，都会尽量写 `pay_reconciliation_record`：

- `batch_no = RCREFUND...`
- `biz_type = REFUND`
- `order_no`
- `pay_order_no`
- `refund_no`
- `local_status`：`pay_refund_order.refund_status` 或 `pay_order.pay_status`。
- `channel_status`：渠道退款状态。
- `local_amount_cent`
- `channel_amount_cent`
- `consistent`
- `diff_type`
- `repair_status`
- `remark`

这张轻量对账表适合按单查询、支付异常转待处理、退款异常补偿记录，不是完整日终账单勾兑。

## 11. 阶段九：线上人工对账任务与账单勾兑

线上对账由 `AdminOnlineReconciliationApplicationService` 直接使用 `JdbcTemplate` 操作以下表：

- `pay_reconcile_task`
- `pay_reconcile_local_bill_item`
- `pay_reconcile_channel_bill_item`
- `pay_reconcile_diff_item`
- `pay_reconcile_operation_log`

### 11.1 创建对账任务：`pay_reconcile_task`

`createTask` 创建任务时：

1. 校验账期不能是未来日期。
2. 同一账期不能重复创建任务。
3. 按账期和渠道统计本地是否存在支付/退款流水。
4. 生成 `task_no`。
5. 插入 `pay_reconcile_task`。

记录字段：

- `task_no`：对账任务号。
- `reconcile_date`：账期日期。
- `channel`：渠道，如 `MOCK`、`ALIPAY`。
- `status = CREATED`
- `local_bill_status = EMPTY`
- `channel_bill_status = EMPTY`
- `match_status = NOT_MATCHED`
- `remark`

### 11.2 生成本地账单：`pay_reconcile_local_bill_item`

`generateLocalBills(taskId)` 会从本地支付和退款表抽快照。

#### 11.2.1 支付本地账单来源

来源 SQL 逻辑：

```text
pay_order p
LEFT JOIN oms_order o ON p.order_no = o.order_no
WHERE p.pay_channel 匹配任务渠道
  AND p.created_at 在账期范围内
```

写入 `pay_reconcile_local_bill_item`：

- `biz_type = PAY`
- `order_no = p.order_no`
- `pay_order_no = p.pay_order_no`
- `refund_no = NULL`
- `user_id = p.user_id`
- `local_status = p.pay_status`
- `order_status = o.order_status`
- `amount_cent = p.pay_amount_cent`
- `channel = p.pay_channel`
- `transaction_no = p.transaction_no`
- `trade_time = p.created_at`
- `raw_snapshot`：支付单号和状态 JSON 摘要。

#### 11.2.2 退款本地账单来源

来源 SQL 逻辑：

```text
pay_refund_order r
LEFT JOIN oms_order o ON r.order_no = o.order_no
WHERE r.pay_channel 匹配任务渠道
  AND COALESCE(r.success_at, r.created_at) 在账期范围内
```

写入 `pay_reconcile_local_bill_item`：

- `biz_type = REFUND`
- `order_no = r.order_no`
- `pay_order_no = r.pay_order_no`
- `refund_no = r.refund_no`
- `user_id = r.user_id`
- `local_status = r.refund_status`
- `order_status = o.order_status`
- `amount_cent = r.refund_amount_cent`
- `channel = r.pay_channel`
- `transaction_no = r.transaction_no`
- `trade_time = COALESCE(r.success_at, r.created_at)`
- `raw_snapshot`：退款单号和状态 JSON 摘要。

生成后更新 `pay_reconcile_task`：

- `local_bill_status = READY`
- `status` 进入本地账单已就绪或渠道账单已就绪阶段。
- 统计 `local_total_count`、`local_total_amount_cent`。

### 11.3 生成渠道账单：`pay_reconcile_channel_bill_item`

当前代码中有 `generateMockChannelBills(taskId, mode)` 用本地账单模拟渠道账单，也支持构造全量测试差异。

写入字段：

- `task_id`
- `biz_type = PAY / REFUND`
- `channel`
- `out_trade_no`：通常对应 `pay_order_no`。
- `order_no`
- `pay_order_no`
- `refund_no`
- `channel_trade_no`
- `channel_refund_no`
- `channel_status`
- `amount_cent`
- `fee_cent`
- `trade_time`
- `raw_line`：渠道账单原始行或 Mock 信息。

生成后更新 `pay_reconcile_task`：

- `channel_bill_status = READY`
- 统计 `channel_total_count`、`channel_total_amount_cent`。

### 11.4 自动勾兑：`pay_reconcile_diff_item`

`match(taskId)` 会读取本地账单和渠道账单：

- 本地账单：`pay_reconcile_local_bill_item`
- 渠道账单：`pay_reconcile_channel_bill_item`

匹配键：

```text
biz_type + pay_order_no + refund_no
```

对每条本地账单：

1. 找不到渠道账单：插入差异 `LOCAL_EXISTS_CHANNEL_MISSING`。
2. 找到渠道账单：比较金额和状态。
3. 完全一致：插入或计数为 `MATCHED`。
4. 金额/状态不一致：插入对应差异类型。

剩余渠道账单没有本地账单：

- 插入 `CHANNEL_EXISTS_LOCAL_MISSING`。

`pay_reconcile_diff_item` 记录：

- `task_id`
- `biz_type`
- `diff_type`
- `diff_level`
- `order_no`
- `pay_order_no`
- `refund_no`
- `local_item_id`
- `channel_item_id`
- `local_status`
- `channel_status`
- `local_amount_cent`
- `channel_amount_cent`
- `diff_amount_cent`
- `suggested_action`
- `process_status = PENDING`

任务表同步更新：

- `status = MATCHED`
- `match_status = MATCHED`
- `matched_count`
- `diff_count`
- `pending_count`
- `done_count`
- `hang_count`
- `matched_at`

### 11.5 差异处理日志：`pay_reconcile_operation_log`

当后台对差异项进行处理、挂起、忽略、归档等操作时，会写操作日志：

- `task_id`
- `diff_item_id`
- `operation_type`
- `operation_content`
- `before_snapshot`
- `after_snapshot`
- `operator_id`
- `operator_name`
- `remark`

同时更新 `pay_reconcile_diff_item`：

- `process_status`：`PENDING`、`HANGING`、`DONE`、`IGNORED`。
- `process_remark`
- `processed_by`
- `processed_at`

并回写 `pay_reconcile_task` 的：

- `pending_count`
- `done_count`
- `hang_count`
- `completed_by`
- `completed_at`

## 12. 关键状态流转汇总

### 12.1 订单状态流

```text
待支付
  -> 已支付/待发货：支付回调成功，支付单成功，库存确认扣减
  -> 已取消/超时取消：用户取消、超时扫描、延迟消息关闭，库存释放，支付单关闭
  -> 已发货：后台发货
  -> 已完成：用户确认收货或自动确认
  -> 已退款/部分退款：支付退款成功事件回写订单
  -> 支付异常：支付回调、关闭、发货等边界异常进入后台核验
```

落表主线：`oms_order.order_status`。

### 12.2 支付单状态流

```text
待支付/支付中
  -> 成功：渠道回调成功或渠道查询补偿成功
  -> 关闭：订单取消/超时且渠道未支付
  -> 退款待处理
  -> 退款中
  -> 已退款/部分退款
  -> 退款失败
```

落表主线：`pay_order.pay_status`。

### 12.3 订单退款状态流

```text
退款申请创建
  -> 退款中：提交给支付模块
  -> 退款成功：支付模块发布成功事件
  -> 退款失败：支付模块发布失败事件
```

落表主线：`oms_order_refund.refund_status`。

### 12.4 支付退款单状态流

```text
REFUND_PENDING
  -> REFUNDING
  -> REFUND_SUCCESS
  -> REFUND_FAILED
```

落表主线：`pay_refund_order.refund_status`。

### 12.5 对账任务状态流

```text
CREATED
  -> LOCAL_BILL_READY：本地账单已生成
  -> CHANNEL_BILL_READY：渠道账单已生成
  -> MATCHED：完成自动勾兑
  -> COMPLETED/归档类状态：差异处理完毕
```

落表主线：`pay_reconcile_task.status`、`local_bill_status`、`channel_bill_status`、`match_status`。

## 13. 按业务动作查看数据写入路径

### 13.1 用户下单

```text
OrderApplicationService.createOrder
  -> ims_stock_lock：写 ORDER + order_no 库存锁
  -> ims_stock：锁定库存、减少可售
  -> ims_stock_operation_log：记录预占流水
  -> oms_order：写订单主表
  -> oms_order_item：写订单明细快照
```

### 13.2 创建支付单

```text
PayApplicationService.createPayOrder
  -> pay_order：写支付单，关联 order_no
```

### 13.3 支付成功

```text
PayApplicationService.handleChannelCallback
  -> pay_callback_record：先记支付回调原文和处理状态
  -> pay_order：成功，写 transaction_no / paid_at
  -> oms_order：标记已支付，写 paid_at
  -> ims_stock_lock：确认扣减
  -> ims_stock：锁定库存转实际扣减
  -> ims_stock_operation_log：记录扣减流水
```

### 13.4 支付异常或支付补偿

```text
PayApplicationService.verifyOrderPayment / reconcile / syncOrderStatus / repairPaidOrder
  -> pay_reconciliation_record：记录待同步/待修复/金额差异
  -> pay_order：必要时从渠道查询补偿成功
  -> oms_order：必要时补偿为已支付
```

### 13.5 发起退款

```text
OrderApplicationService.applyRefund / applyRefundByAdmin / approveRefundApplicationByAdmin
  -> oms_order_refund：写订单退款申请
  -> oms_order_refund_item：写退款 SKU 明细
  -> MQ：发布 OrderRefundRequestedEvent
```

### 13.6 支付侧接收退款请求

```text
PayRefundEventConsumer.onRefundRequested
  -> PayApplicationService.refund
  -> pay_refund_order：写支付退款单
  -> pay_order：标记 REFUND_PENDING / REFUNDING
```

### 13.7 退款成功

```text
PayApplicationService.reconcileRefund
  -> pay_order：退款成功/部分退款
  -> pay_refund_order：REFUND_SUCCESS，写 channel_refund_no / response_payload / success_at
  -> pay_callback_record：写 REFUND 类型记录
  -> pay_reconciliation_record：写 REFUND 对账快照
  -> MQ：发布退款成功事件
  -> OrderApplicationService.onRefundSucceeded
  -> oms_order：已退款/部分退款
  -> oms_order_refund：成功，写 channel_refund_no
  -> ims_stock：恢复库存
  -> ims_stock_operation_log：写 ORDER_REFUND + refund_no 库存恢复流水
```

### 13.8 退款失败

```text
PayApplicationService.reconcileRefund
  -> pay_order：REFUND_FAILED
  -> pay_refund_order：REFUND_FAILED，写 fail_reason / response_payload
  -> pay_callback_record：写 REFUND 失败记录
  -> pay_reconciliation_record：写 REFUND 对账快照
  -> MQ：发布退款失败事件
  -> OrderApplicationService.onRefundFailed
  -> oms_order_refund：失败，写 fail_reason
```

### 13.9 日终/线上对账

```text
AdminOnlineReconciliationApplicationService.createTask
  -> pay_reconcile_task：创建任务

AdminOnlineReconciliationApplicationService.generateLocalBills
  -> 从 pay_order + oms_order 抽 PAY 本地账单
  -> 从 pay_refund_order + oms_order 抽 REFUND 本地账单
  -> pay_reconcile_local_bill_item：写本地账单快照
  -> pay_reconcile_task：更新本地账单统计

AdminOnlineReconciliationApplicationService.generateMockChannelBills / 导入渠道账单
  -> pay_reconcile_channel_bill_item：写渠道账单快照
  -> pay_reconcile_task：更新渠道账单统计

AdminOnlineReconciliationApplicationService.match
  -> pay_reconcile_diff_item：写匹配结果和差异项
  -> pay_reconcile_task：更新 matched/diff/pending 统计

后台处理差异
  -> pay_reconcile_diff_item：更新处理状态
  -> pay_reconcile_operation_log：写处理日志
  -> pay_reconcile_task：更新 pending/done/hang 统计
```

## 14. 闭环中的一致性与幂等设计

### 14.1 业务号唯一约束

- `oms_order.uk_oms_order_order_no`：保证订单号唯一。
- `pay_order.uk_pay_order_pay_order_no`：保证支付单号唯一。
- `pay_order.uk_pay_order_idempotent_key`：保证支付单幂等创建。
- `oms_order_refund.uk_oms_order_refund_refund_no`：保证订单退款号唯一。
- `pay_refund_order.uk_pay_refund_order_refund_no`：保证支付退款号唯一。
- `ims_stock_lock.uk_ims_stock_lock_business`：保证同一业务单、同一 SKU 的库存锁唯一。

### 14.2 先记录后处理

支付回调的关键设计是：

```text
先写 pay_callback_record
再校验验签/金额/身份/状态
再推进 pay_order 和 oms_order
最后更新 pay_callback_record.process_status
```

这样即使回调处理失败，也能追踪原始报文和失败原因。

### 14.3 退款号跨域复用

订单侧生成 `refund_no`，支付侧沿用同一个 `refund_no`：

```text
oms_order_refund.refund_no
  = oms_order_refund_item.refund_no
  = pay_refund_order.refund_no
  = pay_callback_record.refund_no
  = pay_reconciliation_record.refund_no
  = pay_reconcile_local_bill_item.refund_no
  = pay_reconcile_channel_bill_item.refund_no
  = pay_reconcile_diff_item.refund_no
```

这保证退款在订单、支付、对账三条链路中可以直接串联。

### 14.4 支付关闭前查询渠道

订单取消/超时关闭会触发支付单关闭，但支付模块在关闭前查询渠道：

- 如果渠道未支付，关闭本地支付单。
- 如果渠道已支付，则阻止关闭，尝试补偿支付单和订单，并记录支付对账。

这避免了渠道已收款但本地释放库存/关闭订单的风险。

### 14.5 对账补偿

轻量对账和线上账单对账都不只是展示差异，还会推动补偿：

- 渠道已支付，本地未成功：补偿 `pay_order` 成功，必要时补偿 `oms_order` 已支付。
- 支付单成功，订单未支付：后台可执行 `repairPaidOrder`。
- 退款渠道成功，本地未成功：补偿 `pay_order`、`pay_refund_order`、`oms_order_refund` 和 `oms_order`。
- 金额不一致或身份不匹配：进入 `pay_reconciliation_record` 或 `pay_reconcile_diff_item`，等待人工处理。

## 16. 总结

当前实现里，订单-支付-退款-对账闭环的核心是三类记录：

1. **业务事实表**：`oms_order`、`oms_order_item`、`pay_order`、`oms_order_refund`、`oms_order_refund_item`、`pay_refund_order`。
2. **过程审计表**：`pay_callback_record`、`ims_stock_operation_log`、`pay_reconcile_operation_log`。
3. **一致性校验/对账表**：`pay_reconciliation_record`、`pay_reconcile_task`、`pay_reconcile_local_bill_item`、`pay_reconcile_channel_bill_item`、`pay_reconcile_diff_item`、`ims_stock_reconciliation_record`。

主数据流通过 `order_no` 串起订单和支付，通过 `pay_order_no` 串起支付渠道交易，通过 `refund_no` 串起订单退款、支付退款和退款对账。支付回调采用“先落回调记录，再校验处理”的方式保证可追踪；退款采用“订单侧申请，支付侧执行，订单侧消费结果回写”的事件驱动方式；对账则分为按单轻量对账和按账期线上账单勾兑两层，分别解决实时补偿和批量差异处理问题。

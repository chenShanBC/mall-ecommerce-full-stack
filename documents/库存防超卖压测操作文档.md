# 库存防超卖压测操作文档

## 1. 文档目的

本文档用于指导 `mallFei` 项目库存模块进行防超卖压测，重点验证在高并发场景下，同一 SKU 的库存预占、确认、取消等操作是否满足库存安全要求。

压测目标：

1. 初始库存有限时，并发预占成功数量不能超过可售库存。
2. Redis 实时库存不能出现负数。
3. MySQL 库存不能出现负数。
4. DB 中锁定记录数量不能超过实际库存。
5. 重复业务单不能重复扣减库存。
6. 重复确认不能重复扣减总库存。
7. 重复取消不能重复释放库存。
8. Redis 与 DB 最终库存状态应保持一致。

---

## 2. 测试前提

### 2.1 环境要求

压测前确认以下服务已经正常启动：

- MySQL
- Redis
- RabbitMQ
- 后端服务
- 认证服务
- 网关服务，如果当前环境通过网关访问接口

### 2.2 接口认证

库存初始化、库存调整、库存查询等接口通常需要登录态或管理员权限。压测前需要先通过登录接口获取 token。

后续请求根据项目实际认证方式添加请求头，例如：

```http
Authorization: Bearer {{token}}
```

或：

```http
satoken: {{token}}
```

具体 header 名称以当前项目登录后返回和 Sa-Token 配置为准。

### 2.3 测试 SKU

建议使用独立测试 SKU，避免污染真实商品库存。

示例：

```text
skuId = 999001
```

如果该 SKU 不存在，需要先通过商品模块创建测试商品和测试 SKU；如果只是验证库存接口，也可以选择数据库中已有的测试 SKU。

## 3. 测试数据准备

### 3.1 初始化库存

将测试 SKU 初始化为指定库存。

接口：

```http
POST {{baseUrl}}/api/internal/stocks/999001/init?initialStock=10
```

请求头：

```http
Authorization: Bearer {{token}}
```

预期结果：

```text
totalStock = 10
lockedStock = 0
availableStock = 10
```

如果该 SKU 已经初始化过，可以使用库存调整接口重置库存。

接口：

```http
PUT {{baseUrl}}/api/internal/stocks/999001/adjust
```

请求体：

```json
{
  "totalStock": 10,
  "lockedStock": 0,
  "availableStock": 10,
  "reason": "库存防超卖压测初始化"
}
```

预期结果：

```text
totalStock = 10
lockedStock = 0
availableStock = 10
```

### 3.2 同步 Redis 库存

为了确保 Redis 实时库存与 DB 库存一致，初始化或调整库存后调用同步接口。

接口：

```http
POST {{baseUrl}}/api/internal/stocks/999001/sync
```

预期 Redis 中库存为：

```text
stock:total:999001 = 10
stock:locked:999001 = 0
stock:available:999001 = 10
```

## 4. Apipost 接口配置

### 4.1 新建库存预占接口

接口名称：

```text
库存预占-并发测试
```

请求方法：

```http
POST
```

请求地址：

```http
{{baseUrl}}/api/internal/stocks/reserve
```

请求头：

```http
Content-Type: application/json
Authorization: Bearer {{token}}
```

如果项目使用 Sa-Token 自定义 header，则按实际情况配置。

### 4.2 请求 Body

```json
{
  "businessType": "ORDER_CREATE",
  "businessNo": "{{businessNo}}",
  "items": [
    {
      "skuId": 999001,
      "quantity": 1
    }
  ]
}
```

字段说明：

| 字段 | 说明 |
|---|---|
| `businessType` | 业务类型，压测时可使用 `ORDER_CREATE` |
| `businessNo` | 业务单号，每个请求必须不同 |
| `skuId` | 测试 SKU ID |
| `quantity` | 本次预占数量 |

### 4.3 动态生成业务单号

并发压测时，每个请求必须使用不同的 `businessNo`。如果所有请求使用同一个业务单号，会被库存幂等逻辑识别为重复请求，无法真实模拟并发抢库存。

Apipost 可以使用内置动态变量，例如：

```text
OS_TEST_{{$timestamp}}_{{$randomInt}}
```

请求体可写为：

```json
{
  "businessType": "ORDER_CREATE",
  "businessNo": "OS_TEST_{{$timestamp}}_{{$randomInt}}",
  "items": [
    {
      "skuId": 999001,
      "quantity": 1
    }
  ]
}
```

如果当前 Apipost 版本支持前置脚本，也可以在前置脚本中生成变量：

```javascript
const businessNo = 'OS_TEST_' + Date.now() + '_' + Math.floor(Math.random() * 1000000);
apt.variables.set('businessNo', businessNo);
```

然后请求体使用：

```json
{
  "businessType": "ORDER_CREATE",
  "businessNo": "{{businessNo}}",
  "items": [
    {
      "skuId": 999001,
      "quantity": 1
    }
  ]
}
```

## 7. 压测结果统计

压测完成后，需要统计：

```text
成功预占数量
库存不足失败数量
其他异常数量
平均响应时间
最大响应时间
错误率
```

### 7.1 成功响应

成功预占通常返回业务成功，示例：

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "businessType": "ORDER_CREATE",
    "businessNo": "OS_TEST_xxx",
    "status": "RESERVED"
  }
}
```

实际字段以当前项目返回为准。

### 7.2 库存不足响应

库存不足属于预期内业务失败，示例：

```json
{
  "code": 400,
  "message": "SKU库存不足: 999001"
}
```

只要失败原因是库存不足，就属于正常结果。

### 7.3 其他异常

以下情况需要单独排查：

- HTTP 500
- 请求超时
- 连接失败
- Redis 异常
- RabbitMQ 异常
- 数据库死锁
- DB 库存出现负数
- 锁记录数量超过库存

## 8. 压测后校验

### 8.1 查询库存接口

接口：

```http
GET {{baseUrl}}/api/internal/stocks/999001
```

只预占不确认时，预期：

```text
totalStock = 初始库存
lockedStock = 成功预占数量 * 每单数量
availableStock = 初始库存 - lockedStock
```

例如初始库存 10、每单 1 件、成功 10 个：

```text
totalStock = 10
lockedStock = 10
availableStock = 0
```

### 8.2 MySQL 校验库存表

执行 SQL：

```sql
SELECT sku_id, total_stock, locked_stock, available_stock
FROM ims_stock
WHERE sku_id = 999001;
```

预期满足：

```text
total_stock = locked_stock + available_stock
available_stock >= 0
locked_stock >= 0
total_stock >= 0
```

只预占不确认时，如果初始库存为 10 且成功预占 10 件，预期：

```text
total_stock = 10
locked_stock = 10
available_stock = 0
```

### 8.3 MySQL 校验锁定记录

执行 SQL：

```sql
SELECT status, COUNT(*) AS record_count, SUM(quantity) AS total_quantity
FROM ims_stock_lock
WHERE sku_id = 999001
  AND business_type = 'ORDER_CREATE'
  AND business_no LIKE 'OS_TEST_%'
GROUP BY status;
```

只预占不确认时，预期：

```text
RESERVED  10  10
```

要求：

```text
SUM(quantity) <= 初始库存
```

### 8.4 Redis 校验

如果可以连接 Redis，可以检查以下 key：

```text
GET stock:total:999001
GET stock:locked:999001
GET stock:available:999001
```

预期示例：

```text
stock:total:999001 = 10
stock:locked:999001 = 10
stock:available:999001 = 0
```

## 9. 通过标准

一次压测通过必须满足：

1. 成功请求数量不超过初始库存可支持的最大数量。
2. 库存不足失败属于预期结果。
3. Redis 可用库存不小于 0。
4. DB 可用库存不小于 0。
5. DB 满足 `total_stock = locked_stock + available_stock`。
6. 锁定记录的 `SUM(quantity)` 不超过初始库存。
7. 重复业务单不会重复扣库存。
8. 重复确认不会重复扣总库存。
9. 重复取消不会重复释放库存。
10. Redis 与 DB 最终库存一致。

公式：

```text
成功数 <= floor(初始库存 / 每单购买数量)
```

只预占阶段：

```text
lockedStock = 成功数 * 每单购买数量
availableStock = 初始库存 - lockedStock
totalStock = 初始库存
```

确认扣减后：

```text
totalStock = 初始库存 - 确认数量
lockedStock = 预占数量 - 确认数量
availableStock = 初始库存 - 预占数量
```

取消释放后：

```text
totalStock = 初始库存
lockedStock = 未取消的预占数量
availableStock = 初始库存 - 未取消的预占数量
```

## 10. 常见问题排查

### 10.1 成功数超过库存

可能原因：

1. 请求没有打到同一个 SKU。
2. `businessNo` 生成逻辑异常，导致压测结果统计错误。
3. Redis Lua 脚本没有实际执行。
4. 库存服务被绕过。
5. Redis key 命名不一致。

排查方向：

1. 检查所有请求的 `skuId` 是否一致。
2. 检查每个请求的 `quantity` 是否正确。
3. 检查 Redis 中 `stock:available:{skuId}` 的变化。
4. 检查后端日志中库存预占结果。

### 10.2 Redis 正确但 DB 错误

可能原因：

1. MQ 重复消费导致 DB 重复扣减。
2. MQ 消费失败导致 DB 未同步。
3. DB 同步逻辑缺少幂等控制。
4. 锁记录状态流转异常。

排查方向：

1. 查看 RabbitMQ 消费日志。
2. 查看 `ims_stock_lock` 锁记录。
3. 查看 `ims_stock` 库存表。
4. 检查 MQ 消费端幂等逻辑。

### 10.3 DB 正确但 Redis 被恢复成旧值

可能原因：

1. 定时对账任务使用旧 DB 库存覆盖了 Redis。
2. MQ 同步 DB 延迟期间触发了对账。
3. Redis 和 DB 的主从数据源定义不清晰。

排查方向：

1. 查看定时对账任务执行时间。
2. 查看库存对账日志。
3. 检查对账策略是否是无条件 DB 覆盖 Redis。

### 10.4 重复业务单仍然重复扣库存

可能原因：

1. `businessType` 不一致。
2. `businessNo` 不一致。
3. `skuId` 不一致。
4. Redis 预占 key 生成规则不一致。
5. DB 唯一索引未生效。

排查方向：

1. 确认重复请求三元组完全一致：`businessType + businessNo + skuId`。
2. 查看 Redis reservation key。
3. 查看 `ims_stock_lock` 是否存在重复业务记录。

# mallFei 项目已实现业务功能阶段梳理

> 本文基于当前项目代码与既有阶段规划，对已实现业务能力进行重新归类梳理。  
> 归类口径：
>
> - **MVP 阶段**：支撑最小可用商城交易闭环。
> - **一期阶段**：在 MVP 基础上补齐企业级交易闭环所需的售后、退款、库存预占、订单超时、基础对账、后台权限与审计能力。
> - **一期增强阶段**：强化交易可靠性、库存一致性、支付核验、对账闭环、权限审计、Dashboard 与认证体验。
>
> 说明：本文不展开接口和页面路径，只描述业务功能、业务流程和实现流程。

---

## 一、总体实现结论

当前项目已经完成了电商系统从 C 端用户交易到 Admin 后台管理的完整主链路。

整体完成度可以概括为：

| 阶段 | 当前实现情况 | 结论 |
|---|---|---|
| MVP 阶段 | 用户、商品、购物车、订单、库存、支付、后台基础管理均已具备 | 已基本完成 |
| 一期阶段 | 订单超时、库存预占、售后退款、支付同步、基础对账、后台权限、操作日志均已具备 | 已实现大部分核心能力 |
| 一期增强阶段 | 库存补偿、库存对账、支付异常处理、退款对账、线上人工对账、挂账跟进、Dashboard 增强、认证增强已具备 | 已实现较多增强能力 |

# 二、MVP 阶段已实现业务功能

## 2.1 用户注册、登录与登录态

### 业务目标

让 C 端用户能够进入商城、完成身份认证，并在登录后使用购物车、下单、支付、订单查询等功能。

### 已实现功能

- 用户注册。
- 用户密码登录。
- 用户退出登录。
- 当前登录用户信息查询。
- 用户登录态维护。
- 登录后访问受保护业务功能。

### 业务流程

```text
用户进入登录/注册页
  -> 输入手机号、密码等信息
  -> 系统校验用户身份或创建新用户
  -> 登录成功后生成登录态
  -> 前端保存登录信息
  -> 用户进入首页、购物车、订单、个人中心等页面
```

### 实现流程

```text
前端提交注册/登录信息
  -> 用户服务校验手机号、密码和账号状态
  -> 生成用户身份上下文
  -> 返回登录结果
  -> 后续请求通过登录态识别当前用户
```

### 阶段归属

**MVP 阶段**

# 三、一期阶段已实现业务功能

## 3.1 待支付订单超时取消

### 业务目标

解决用户下单后长期不支付导致库存被占用、订单堆积的问题。

### 已实现功能

- 待支付订单超时扫描。
- 订单超时事件发布。
- 订单超时事件消费。
- 超时订单自动取消。
- 超时取消后释放库存。

### 业务流程

```text
用户创建订单
  -> 订单进入待支付状态
  -> 超过支付有效期仍未支付
  -> 系统定时识别超时订单
  -> 自动取消订单
  -> 释放订单占用库存
  -> 订单状态变更为已取消
```

### 实现流程

```text
定时任务扫描待支付订单
  -> 判断订单是否超过支付截止时间
  -> 发布订单超时事件
  -> 消费端处理订单关闭
  -> 调用库存释放能力
  -> 记录订单状态变更
```

### 阶段归属

**一期阶段**

### 业务流程

```text
  -> 系统同步订单状态
```

```text
  -> 拼图验证码用于登录前风险校验
  -> 黑名单校验用于限制异常账号登录
  -> 登录成功后返回统一登录态
```

### 阶段归属

**一期增强阶段**

## 4.12 文件上传能力

### 业务目标

为商品图片、用户头像、后台资料等业务提供基础文件上传支撑。

### 已实现功能

- 文件上传。
- 文件访问地址返回。
- 业务侧可引用上传后的文件资源。

### 业务流程

```text
用户或管理员选择文件
  -> 上传文件
  -> 系统保存文件
  -> 返回文件访问地址
  -> 商品、头像或其他业务保存该文件地址
```

### 实现流程

```text
文件服务接收上传文件
  -> 校验并保存文件
  -> 生成可访问路径
  -> 返回给业务前端
  -> 业务数据保存文件地址引用
```

### 阶段归属

**MVP 阶段 / 一期增强支撑能力**

说明：作为基础上传能力，它本身可归入 MVP 支撑能力；如果用于更完整的运营后台资料管理，则属于后续增强支撑能力。

# 五、已提前实现但更偏二期的能力

## 5.1 后台发货

### 业务目标

让后台能够对已支付订单执行发货操作。

### 已实现功能

- 后台发货。
- 订单进入发货或待收货状态。
- 后续可配合自动确认收货流程。

### 业务流程

```text
用户支付订单
  -> 后台查看待发货订单
  -> 运营或仓储人员执行发货
  -> 订单状态进入待收货
  -> 用户确认收货或系统自动确认收货
```

### 实现流程

```text
后台提交发货操作
  -> 订单服务校验订单是否已支付且可发货
  -> 更新订单履约状态
  -> 后续进入确认收货或自动确认收货流程
```

### 阶段归属

**二期能力提前实现**

说明：设计规划中物流履约、发货管理属于二期，但当前项目已经具备后台发货的基础能力。

# 订单支付售后对账闭环改造说明

## 1. 背景

本次改造围绕商城交易链路中的订单、支付、库存、退款、售后和对账能力进行闭环补齐，目标是让系统从“能完成交易”升级为“交易状态可追踪、异常可发现、差异可沉淀、问题可处置”的生产级闭环。

改造重点包括：

- 支付回调安全与幂等处理。
- 订单超时、库存释放与支付状态同步。
- 整单退款与部分退款。
- 支付侧独立退款账本。
- 支付/退款轻量对账记录落库。
- 后台对账异常展示与处置。

本期范围覆盖《订单支付售后对账闭环实现方案》中的一期与一期增强内容，不包含二期的渠道账单自动下载、完整人工工单系统、完整退货质检系统等能力。

## 2. 本次闭环改造做了什么

### 2.1 正向交易链路增强

正向交易链路围绕“下单 -> 库存预占 -> 支付 -> 回调 -> 订单支付成功 -> 库存确认扣减”形成闭环。

主要改造内容：

1. 下单时通过库存模块进行库存预占，避免超卖。
2. 订单保存支付过期时间，用于订单倒计时和超时判断。
3. 支付成功后由支付模块驱动订单状态变更。
4. 订单支付成功后确认库存扣减。
5. 支持支付状态主动同步，避免回调丢失导致本地状态长期不一致。
6. 支持订单超时自动取消，并释放预占库存。

### 2.2 支付回调安全闭环

支付回调不再直接信任渠道请求，而是先记录、再校验、再处理。

主要改造内容：

1. 支付回调统一落库到 `pay_callback_record`。
2. 回调处理前进行渠道验签。
3. 校验支付单号、订单号、金额是否与本地一致。
4. 只处理支付成功类状态。
5. 重复成功回调走幂等忽略。
6. 异常回调记录失败原因和处理状态。

当前支持的回调处理状态包括：

- `PROCESSED`：处理成功。
- `VERIFY_FAILED`：验签失败。
- `BUSINESS_MISMATCH`：订单号或支付单号不匹配。
- `AMOUNT_MISMATCH`：金额不一致。
- `IGNORED_DUPLICATE`：重复回调忽略。
- `IGNORED_NON_SUCCESS`：非成功交易状态忽略。

### 2.3 退款业务闭环

退款链路从原来的简单状态变更，升级为“业务退款单 + 退款明细 + 支付退款账本 + 订单状态 + 库存回补”的多对象闭环。

主要改造内容：

1. 增强 `OrderRefund`，承载业务退款申请。
2. 新增 `OrderRefundItem`，承载退款明细，支持部分退款。
3. 新增 `PayRefundOrder`，承载支付侧退款账本。
4. 退款申请支持整单退款和部分退款。
5. 部分退款时校验退款数量不能超过可退数量。
6. 退款金额基于订单明细快照计算，避免依赖商品当前价格。
7. 退款成功后更新订单退款状态。
8. 退款成功后更新业务退款单状态。
9. 退款成功后更新支付侧退款单状态。
10. 退款成功后按退款明细回补库存。
11. 退款失败时记录失败原因和渠道响应。

### 2.4 支付侧独立退款账本

新增支付侧退款单表 `pay_refund_order`，使支付侧退款事实独立沉淀。

该账本记录：

- 本地退款单号。
- 业务订单号。
- 原支付单号。
- 用户 ID。
- 支付渠道。
- 退款金额。
- 退款状态。
- 原渠道交易号。
- 渠道退款流水号。
- 退款请求摘要。
- 退款响应摘要。
- 失败原因。
- 成功时间。

退款流程当前会：

1. 创建或复用 `pay_refund_order`。
2. 标记支付退款单为 `REFUNDING`。
3. 调用渠道退款。
4. 渠道成功时写入：
   - `REFUND_SUCCESS`
   - `channel_refund_no`
   - `response_payload`
   - `success_at`
5. 渠道失败时写入：
   - `REFUND_FAILED`
   - `fail_reason`
   - `response_payload`

这样支付侧退款不再只是修改 `pay_order` 状态，而是具备独立、可追踪、可对账的退款账本。

### 2.5 支付/退款轻量对账记录

新增并接入 `pay_reconciliation_record`，用于沉淀支付和退款对账结果。

该表记录：

- 业务类型：支付 / 退款。
- 订单号。
- 支付单号。
- 退款单号。
- 本地状态。
- 渠道状态。
- 本地金额。
- 渠道金额。
- 是否一致。
- 差异类型。
- 修复状态。
- 备注。
- 修复时间。
- 创建时间。

本次补齐了完整代码链路：

1. `PayReconciliationRecord` 领域模型。
2. `PayReconciliationRecordRepository` 仓储接口。
3. `PayReconciliationRecordDO` 持久化对象。
4. `PayReconciliationRecordMapper` MyBatis-Plus Mapper。
5. `MybatisPayReconciliationRecordRepository` 仓储实现。
6. `PayApplicationService.reconcile()` 支付对账落库。
7. `PayApplicationService.reconcileRefund()` 退款对账落库。

差异类型包括：

- `NONE`：无差异。
- `LOCAL_MISSING`：本地记录缺失。
- `CHANNEL_MISSING`：渠道结果缺失或未知。
- `STATUS_MISMATCH`：状态不一致。
- `AMOUNT_MISMATCH`：金额不一致。
- `STATUS_AND_AMOUNT_MISMATCH`：状态和金额均不一致。

修复状态包括：

- `NONE`：无需处理。
- `PENDING`：待处理。
- `DONE`：已处理。
- `IGNORED`：已忽略。

### 2.6 支付对账和安全补偿

支付对账从单纯本地计算增强为“本地订单 + 本地支付单 + 渠道查询 + 安全补偿 + 记录落库”。

支付对账流程现在会：

1. 查询业务订单。
2. 查询本地支付单。
3. 对支付宝类渠道查询渠道交易状态。
4. 如果渠道已支付、本地支付单未成功，则补偿支付单为成功。
5. 如果支付单成功、订单仍待支付，则补偿订单为已支付，并触发库存确认扣减。
6. 如果金额不一致，则只记录异常，不自动修复。
7. 如果订单已取消但渠道显示已支付，则作为高危异常记录，不盲目自动修复。
8. 将本次对账结果写入 `pay_reconciliation_record`。

该策略遵循“能安全自动修复的自动修复，高风险场景只记录并交由后台处置”的原则。

### 2.7 退款对账和补偿

退款对账从“查询渠道后返回结果”增强为“渠道查询 + 本地补偿 + 支付退款账本更新 + 对账记录落库”。

退款对账流程现在会：

1. 查询本地支付单。
2. 查询本地支付退款单。
3. 调用渠道退款查询。
4. 渠道退款成功时：
   - 补偿 `pay_order` 为退款成功或部分退款。
   - 更新 `pay_refund_order` 为 `REFUND_SUCCESS`。
   - 记录渠道退款流水号。
   - 发布退款成功事件，驱动订单和库存侧处理。
   - 写入退款对账记录。
5. 渠道退款失败时：
   - 补偿 `pay_order` 为退款失败。
   - 更新 `pay_refund_order` 为 `REFUND_FAILED`。
   - 记录失败原因。
   - 发布退款失败事件。
   - 写入退款对账记录。
6. 渠道结果未知时：
   - 不修改本地最终状态。
   - 写入待处理对账异常。

### 2.8 库存退款回补

库存侧遵循“支付确认扣减不回滚，退款回补产生新的库存事实”的思路。

退款成功后会：

1. 根据退款明细构造库存回补请求。
2. 使用退款单号作为业务幂等键。
3. 调用库存模块执行 `RESTORE` 类型库存回补。
4. 记录库存操作日志。

这样可以避免直接篡改原支付扣减流水，使库存流水保持可追溯。

### 2.9 后台运营与异常处置

后台管理端补齐了对支付、退款、对账异常的观察和处置入口。

已支持：

1. 支付单列表和详情。
2. 支付回调记录查询。
3. 订单退款单查询。
4. 退款明细查询。
5. 支付对账列表。
6. 库存对账列表。
7. 支付/退款对账记录查询。
8. 单笔订单重新对账。
9. 支付状态同步。
10. 已成功支付订单补偿。
11. 退款状态同步。
12. 对账记录标记已处理。
13. 对账记录忽略。

后台新增的支付/退款对账记录接口包括：

- `GET /api/admin/reconciliations/pay-records`
- `GET /api/admin/reconciliations/pay-records/{id}`
- `POST /api/admin/reconciliations/pay-records/{id}/done`
- `POST /api/admin/reconciliations/pay-records/{id}/ignore`

## 3. 简要流程步骤

### 3.1 下单支付流程

1. 用户提交订单。
2. 订单模块校验商品和金额。
3. 库存模块预占库存。
4. 订单模块创建待支付订单。
5. 支付模块创建支付单。
6. 用户跳转支付渠道完成支付。
7. 支付渠道回调系统。
8. 系统保存支付回调记录。
9. 系统验签并校验订单号、支付单号、金额。
10. 校验通过后标记支付单成功。
11. 订单模块标记订单已支付。
12. 库存模块确认扣减库存。
13. 交易正向链路完成。

### 3.2 支付超时流程

1. 系统定时或事件触发检查待支付订单。
2. 判断订单是否超过支付截止时间。
3. 若订单未超时，则不处理。
4. 若订单已超时，则关闭订单。
5. 释放预占库存。
6. 发布订单取消事件。
7. 支付模块关闭仍处于待支付状态的支付单。

### 3.3 支付主动同步/对账流程

1. 后台或系统触发订单支付状态同步。
2. 查询本地订单和本地支付单。
3. 对支付宝类支付单查询渠道交易状态。
4. 若渠道已支付但本地支付单未成功，则补偿支付单成功。
5. 若支付单成功但订单仍待支付，则补偿订单已支付。
6. 若存在金额差异或高风险状态差异，则只记录异常，不自动修复。
7. 保存本次对账结果到 `pay_reconciliation_record`。
8. 后台可查看对账记录并进行处理。

### 3.4 整单退款流程

1. 用户或后台发起整单退款。
2. 订单模块校验订单是否允许退款。
3. 生成业务退款单 `OrderRefund`。
4. 根据订单明细生成退款明细 `OrderRefundItem`。
5. 支付模块创建或复用 `PayRefundOrder`。
6. 支付退款单进入 `REFUNDING`。
7. 调用支付渠道退款。
8. 渠道退款成功后更新支付退款账本。
9. 发布退款成功事件。
10. 订单模块更新业务退款单和订单退款状态。
11. 库存模块按退款明细回补库存。

### 3.5 部分退款流程

1. 用户选择部分订单明细发起退款。
2. 系统校验退款明细是否属于当前订单。
3. 系统校验退款数量是否超过剩余可退数量。
4. 根据订单项快照计算退款金额。
5. 创建业务退款单和退款明细。
6. 支付模块创建支付退款单。
7. 调用渠道退款。
8. 退款成功后更新支付侧、订单侧和库存侧状态。
9. 后续再次退款时，会基于已成功退款明细计算剩余可退数量。

### 3.6 退款对账流程

1. 后台或系统触发退款同步。
2. 查询本地支付单和支付退款单。
3. 调用渠道退款查询。
4. 若渠道显示退款成功：
   - 补偿支付单为退款成功或部分退款。
   - 更新支付退款单为 `REFUND_SUCCESS`。
   - 发布退款成功事件。
   - 写入退款对账记录。
5. 若渠道显示退款失败：
   - 补偿支付单为 `REFUND_FAILED`。
   - 更新支付退款单为 `REFUND_FAILED`。
   - 发布退款失败事件。
   - 写入退款对账记录。
6. 若渠道结果未知：
   - 保持本地状态。
   - 写入待处理对账异常。

### 3.7 后台异常处置流程

1. 运营人员进入后台对账运营页面。
2. 查看支付对账列表、库存对账列表、支付/退款对账记录。
3. 对异常订单执行重新对账。
4. 对支付状态异常执行支付同步或补偿。
5. 对退款异常执行退款同步。
6. 对已确认处理完成的对账记录标记为 `DONE`。
7. 对确认无需处理的记录标记为 `IGNORED`。
8. 系统保留处理状态和备注，便于后续追踪。

## 4. 关键表说明

### 4.1 `pay_callback_record`

用于保存支付/退款渠道回调原始信息和处理结果。

核心作用：

- 回调留痕。
- 验签结果记录。
- 金额和业务身份校验记录。
- 幂等处理记录。
- 异常原因记录。

### 4.2 `pay_refund_order`

用于保存支付侧退款账本。

核心作用：

- 独立记录支付退款事实。
- 保存渠道退款流水号。
- 保存退款请求和响应摘要。
- 保存退款成功时间和失败原因。
- 支撑退款对账。

### 4.3 `pay_reconciliation_record`

用于保存支付/退款对账结果。

核心作用：

- 沉淀每次对账结果。
- 保存本地状态和渠道状态。
- 保存本地金额和渠道金额。
- 标记是否一致。
- 标记差异类型。
- 支持后台人工处理状态流转。

### 4.4 `oms_order_refund`

用于保存业务退款申请。

核心作用：

- 表达用户或后台的退款诉求。
- 保存退款原因、退款金额、退款状态。
- 关联订单侧退款处理流程。

### 4.5 `oms_order_refund_item`

用于保存退款明细。

核心作用：

- 支撑部分退款。
- 记录每个订单项的退款数量和退款金额。
- 支撑库存按明细回补。
- 支撑剩余可退数量计算。

## 5. 设计原则

### 5.1 非贫血 DDD

本次改造没有把所有规则都堆在应用服务里，而是让领域对象承载必要规则。

例如：

- `PayRefundOrder` 负责表达退款单状态流转。
- `PayReconciliationRecord` 负责根据快照计算差异类型和修复状态。
- `PayOrder` 负责表达支付单、退款中、退款成功、退款失败等状态转移。

应用服务主要负责流程编排，领域对象负责业务规则表达。

### 5.2 不过度设计

本期没有引入复杂工作流引擎、分布式事务框架或完整工单系统，而是使用现有模块能力完成闭环。

这样做的好处是：

- 改造范围可控。
- 对现有业务影响小。
- 便于验收。
- 后续可以平滑扩展到二期。

### 5.3 安全补偿优先

对账补偿遵循以下原则：

- 明确安全的场景自动补偿。
- 金额不一致不自动修复。
- 订单已取消但渠道已支付等高风险场景不自动修复。
- 高风险差异进入对账记录，由后台人工确认。

### 5.4 可追溯

关键动作都尽量保留记录：

- 支付回调有回调记录。
- 支付退款有退款账本。
- 支付/退款对账有对账记录。
- 库存变化有库存流水。
- 后台异常处理有修复状态。

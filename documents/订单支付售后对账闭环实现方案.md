# 订单支付售后对账闭环 MVP + 一期增强实现方案

> 定位：本方案不是做完整支付中台、售后中台、物流履约系统，而是贴近当前手绘状态图，把商城交易链路做到“状态完整、真实接入支付宝沙箱、支持整单/部分退款、异常可补偿、面试能讲清楚”的程度。
>
> 核心目标：围绕 `订单 -> 支付 -> 库存 -> 退款 -> 对账/补偿` 做一个相对完整但不过度复杂的交易闭环。重点体现交易状态机、支付宝沙箱接入、库存一致性、支付回调幂等、退款回补和轻量对账能力。

---

## 一、整体判断

根据当前目标，项目不应该只停留在“简化 MVP”，而是建议做到：

```text
MVP 阶段：下单、预占库存、支付宝沙箱支付、支付成功确认扣减、超时取消释放库存
一期阶段：真实支付宝沙箱退款、整单退款、部分退款、退款库存回补、支付/退款轻量对账
二期以后：完整物流、复杂售后审核、每日渠道账单下载、人工工单系统
```

也就是说：

- **真实渠道退款需要做**：因为你计划真实接支付宝，而且支付宝沙箱退款能明显提升项目含金量。
- **部分退款可以做**：但只做“按订单明细部分退款”，不做复杂退货、质检、售后举证。
- **状态图里的状态尽量保留**：订单、支付、退款、库存这些状态都要能在代码里出现并被基本使用到。
- **复杂度要收住**：不做完整物流，不做多级审核，不做工单流转，不做日账单自动清分。

适合面试的表达是：

```text
我没有只做普通下单支付，而是把订单、支付、库存、退款和对账做成了一个交易闭环。
MVP 阶段先保证正向链路可靠；一期阶段接入支付宝沙箱真实支付和退款，支持整单退款和部分退款，并通过库存回补、回调幂等、轻量对账保证最终一致。
```

## 二、当前已有能力

当前项目已经具备较好的基础：

```text
用户下单
  -> 创建待支付订单
  -> Redis Lua 预占库存
  -> 创建支付单
  -> 支付宝沙箱 / Mock 支付成功
  -> 支付回调或主动同步
  -> 确认扣减库存
  -> 订单标记已支付
  -> 订单超时取消
  -> 释放预占库存
```

已有亮点：

- Redis Lua 原子预占、取消、确认扣减库存。
- DB 库存锁记录 `ims_stock_lock`。
- 库存操作流水。
- Redis/DB 库存一致性检查与修复记录。
- 订单创建失败后的库存释放补偿。
- 延迟 MQ + 定时任务 + 请求兜底的订单超时取消。
- 订单表已有 `expire_time`，可作为支付截止时间事实字段。
- 支付侧已有支付单、支付回调、支付同步、支付修复雏形。
- 支付渠道已有支付宝沙箱接入基础。

后续重点不是推倒重来，而是沿着手绘状态图补齐：

```text
支付状态完整化
退款状态完整化
库存退款回补
支付/退款异常对账
后台可观察
```

## 三、阶段边界

### 1. MVP 阶段：正向交易主链路

MVP 的目标是：用户从下单到支付成功，或者超时未支付取消，形成完整闭环。

必须包含：

| 能力 | 说明 |
| --- | --- |
| 下单预占库存 | 下单成功后库存从可售变为锁定 |
| 订单支付超时 | 超时未支付订单自动取消 |
| 库存释放 | 订单取消/超时后释放预占库存 |
| 支付宝沙箱支付 | 支持支付宝沙箱 PC/WAP 至少一种真实支付链路 |
| 支付成功确认库存 | 支付成功后预占库存转为真实扣减 |
| 支付回调幂等 | 重复回调不重复扣库存、不重复加销量 |
| 订单详情倒计时 | 基于订单表 `expire_time` 展示剩余支付时间 |
| 支付结果同步 | 前端支付返回后可同步订单支付状态 |

MVP 暂不要求：

- 退款。
- 部分退款。
- 每日渠道账单下载。
- 物流发货、退货入库。
- 人工工单系统。

MVP 面试表达：

```text
MVP 阶段我先把交易正向链路做稳定：下单时 Redis Lua 预占库存，支付宝沙箱支付成功后确认扣减，未支付超时则自动取消并释放库存。为了避免超卖和库存锁死，我做了库存锁记录、操作流水、订单保存失败补偿，以及延迟消息、定时任务、请求兜底三层超时关闭机制。
```

---

### 2. 一期阶段：贴近状态图的支付退款闭环

一期目标是：在 MVP 基础上补齐支付宝沙箱退款、整单/部分退款、轻量对账和异常补偿。

一期建议完成：

| 能力 | 价值 |
| --- | --- |
| 支付回调持久化 | 支付通知可追踪、可审计 |
| 支付金额校验 | 防止错单、金额篡改、串单 |
| 超时取消前查支付状态 | 避免用户已付款但订单被取消 |
| 支付宝沙箱退款 | 真实调用支付宝退款接口，项目含金量更高 |
| 整单退款 | 覆盖最常见售后场景 |
| 部分退款 | 贴近真实电商状态图，体现订单明细级处理能力 |
| 退款成功后库存回补 | 退款和库存形成闭环 |
| 支付/退款轻量对账 | 支持按订单维度发现状态不一致 |
| 后台展示异常状态 | 能看到回调、退款、对账结果 |

一期仍然不做：

- 完整物流发货。
- 退货寄回和质检入库。
- 多级人工审核。
- 售后举证、图片上传、客服仲裁。
- 每日支付宝账单自动下载和全量清分。
- 完整人工工单系统。

一期面试表达：

```text
一期我重点补齐资金闭环：支付回调先落库，再做验签、金额校验、订单号校验和幂等处理；订单超时取消前主动查支付宝状态，避免回调延迟造成误取消；售后接入支付宝沙箱退款，支持整单退款和部分退款，退款成功后按订单明细回补库存；最后通过轻量对账识别支付成功但订单未成功、订单取消但渠道已支付、退款状态不一致等异常。
```

---

### 3. 二期以后：只作为演进方向

二期以后可以写进文档，但不要作为当前交付目标。

包括：

- 支付宝/微信日账单下载和自动清分。
- 完整退货物流、买家寄回、商家收货、质检入库。
- 多级售后审核。
- 异常工单系统。
- 资金流水台账。
- 多支付渠道生产级适配。

## 四、MVP 正向链路设计

### 1. 下单流程

```text
校验商品和库存
  -> 生成 orderNo
  -> 写入订单 expire_time
  -> 创建 PENDING_PAYMENT 订单
  -> Redis Lua 预占库存
  -> 保存库存锁记录
  -> 发送订单超时延迟消息
  -> 返回订单详情和支付倒计时
```

关键点：

- 订单表已有 `expire_time`，支付倒计时和超时判断都以它为准。
- `effectiveExpireTime(fallbackTimeoutMinutes)` 只作为历史兼容兜底。
- 下单预占库存成功后，如果订单保存失败，必须释放预占库存。

### 2. 支付流程

```text
创建支付单
  -> 生成支付宝沙箱支付表单/跳转链接
  -> 用户完成支付
  -> 支付宝异步回调 / 前端返回主动同步
  -> 验签和幂等校验
  -> 支付单 SUCCESS
  -> 订单 PENDING_PAYMENT -> PAID
  -> 库存 RESERVED -> CONFIRMED
  -> 商品销量增加
```

MVP 要求：

- 支付成功处理可以重复执行。
- 重复执行不会重复确认库存。
- 重复执行不会重复增加销量。
- 订单如果已经不是待支付，不重复改状态。

### 3. 超时取消流程

MVP：

```text
延迟 MQ / 定时任务 / 请求兜底触发
  -> 查询订单
  -> 判断 PENDING_PAYMENT 且 now >= expire_time
  -> 订单改 TIMEOUT_CANCELLED
  -> 释放库存锁
  -> 关闭本地待支付支付单
```

一期增强：

```text
超时取消前先查支付宝支付状态
  -> 渠道已支付：走支付成功补偿
  -> 渠道未支付：关闭支付单，取消订单，释放库存
  -> 渠道未知：暂不取消，等待下一轮重试
```

## 五、一期支付安全设计

### 1. 支付回调持久化

建议新增 `pms_pay_callback_record` 表。

一期字段控制在够用即可：

| 字段 | 说明 |
| --- | --- |
| `id` | 主键 |
| `channel` | 支付渠道，如 `ALIPAY_PC`、`ALIPAY_WAP` |
| `callback_type` | `PAY` / `REFUND` |
| `pay_order_no` | 支付单号 |
| `refund_no` | 退款单号，可空 |
| `order_no` | 订单号 |
| `transaction_no` | 支付宝交易号 |
| `amount_cent` | 回调金额，单位分 |
| `trade_status` | 支付宝交易状态 |
| `verified` | 是否验签通过 |
| `process_status` | 处理状态 |
| `fail_reason` | 失败原因 |
| `raw_payload` | 原始报文 |
| `created_at` | 创建时间 |
| `processed_at` | 处理时间 |

一期状态：

```text
RECEIVED
VERIFY_FAILED
BUSINESS_MISMATCH
AMOUNT_MISMATCH
IGNORED_DUPLICATE
IGNORED_NON_SUCCESS
PROCESSED
PROCESS_FAILED
```

### 2. 支付成功校验

支付成功前必须校验：

| 校验项 | 一期要求 |
| --- | --- |
| 验签 | 使用支付宝 RSA2 验签 |
| 支付单号 | `out_trade_no` 等于本地 `pay_order_no` |
| 订单号 | `passback_params` 或本地支付单关联订单号一致 |
| 金额 | 支付宝 `total_amount` 换算分后等于本地支付金额 |
| 支付状态 | 只接受 `TRADE_SUCCESS` / `TRADE_FINISHED` |
| 幂等 | 支付单已成功则回调标记重复忽略 |

面试讲法：

```text
支付宝回调不是收到成功就改订单，我会先落库保留原始报文，再做验签、金额校验、单号校验和幂等判断。金额不一致或订单号不一致的回调只记录异常，不会把订单置为已支付。
```

## 六、一期退款闭环设计

### 1. 退款目标

一期退款要做到：

```text
真实调用支付宝沙箱退款
支持整单退款
支持部分退款
退款成功后回补库存
退款状态可查询、可展示、可对账
```

但不做：

```text
退货物流
商家收货质检
多级人工审核
复杂客服工单
```

### 2. 退款对象设计

建议保留三类对象：

| 对象 | 作用 |
| --- | --- |
| `Order` | 订单主状态，体现是否退款中、部分退款、已退款 |
| `OrderRefund` | 售后/退款申请单，体现用户申请、退款原因、退款明细 |
| `PayRefundOrder` | 支付退款单，体现支付宝退款请求、渠道退款流水、退款结果 |

为什么要有 `PayRefundOrder`：

- `OrderRefund` 是业务售后事实。
- `PayRefundOrder` 是资金退款事实。
- 两者分开后，能表达“售后申请成功但支付宝退款失败”这种真实场景。

### 3. 退款状态设计

结合你的状态图，一期建议保留这些状态。

订单状态：

```text
PENDING_PAYMENT   待支付
PAID              已支付
CANCELLED         已取消
TIMEOUT_CANCELLED 超时取消
COMPLETED         已完成
REFUND_PENDING    退款申请中
REFUNDING         退款处理中
PARTIALLY_REFUNDED 部分退款
REFUNDED          全部退款
REFUND_FAILED     退款失败
REFUND_CLOSED     退款关闭
```

支付单状态：

```text
PENDING           待支付
PAYING            支付中
SUCCESS           支付成功
FAILED            支付失败
CLOSED            已关闭
REFUND_PENDING    退款申请中
PARTIALLY_REFUNDED 部分退款
REFUNDED          全部退款
REFUND_FAILED     退款失败
```

退款单状态：

```text
REFUND_PENDING    退款申请已创建
REFUNDING         已调用支付宝退款，等待结果
REFUND_SUCCESS    退款成功
REFUND_FAILED     退款失败
REFUND_CLOSED     退款关闭
```

### 4. 整单退款流程

```text
用户申请整单退款
  -> 校验订单 PAID / COMPLETED / PARTIALLY_REFUNDED 可退
  -> 计算剩余可退金额 = payAmount - 已成功退款金额
  -> 创建 OrderRefund，refundAmount = 剩余可退金额
  -> 创建 PayRefundOrder
  -> 调用支付宝沙箱退款接口
  -> 支付宝返回退款成功
  -> PayRefundOrder -> REFUND_SUCCESS
  -> OrderRefund -> REFUND_SUCCESS
  -> 订单 -> REFUNDED
  -> 按订单全部商品回补库存
  -> 回滚对应销量
```

关键点：

- 只有支付宝退款成功后，订单才能变成 `REFUNDED`。
- 只有退款成功后，库存才能回补。
- 重复退款请求不能重复回补库存。

### 5. 部分退款流程

部分退款建议控制复杂度，只做“按订单明细选择 SKU 和数量退款”。

```text
用户选择部分商品/数量申请退款
  -> 校验订单已支付且商品可退
  -> 校验退款数量 <= 可退数量
  -> 根据订单商品快照计算退款金额
  -> 创建 OrderRefund 和退款明细
  -> 创建 PayRefundOrder
  -> 调用支付宝沙箱退款接口
  -> 退款成功
  -> 订单状态：
       - 还有未退款金额：PARTIALLY_REFUNDED
       - 全部金额都已退：REFUNDED
  -> 按退款明细回补库存
  -> 按退款明细回滚销量
```

部分退款要控制的规则：

| 规则 | 说明 |
| --- | --- |
| 按订单明细退款 | 不允许随意输入 SKU，只能退订单里的商品 |
| 按数量退款 | 每个订单项记录已退数量，不能超过购买数量 |
| 金额按快照计算 | 使用下单时的商品单价快照，不受后续改价影响 |
| 运费暂不退或整单退款才退 | 避免部分退款金额计算过复杂 |
| 优惠暂时按简单比例分摊或不做复杂优惠 | 一期不要引入复杂营销分摊 |
| 退款成功才回补库存 | 资金成功后再动库存 |

推荐部分退款金额算法：

```text
退款商品金额 = 订单项成交单价 * 退款数量
如果整单全部退，则退剩余可退金额
如果部分退，暂不退运费，优惠可按订单项成交价快照已包含后的价格处理
```

这样可以避免复杂的优惠券、满减、运费分摊。

面试讲法：

```text
部分退款我没有做复杂售后系统，而是按订单明细维度控制退款数量和退款金额。退款金额基于下单快照计算，退款数量不能超过购买数量；支付宝退款成功后，再按退款明细回补库存和回滚销量。这样既贴近真实电商，又控制了一期复杂度。
```

### 6. 退款库存回补

退款成功后执行：

```text
根据退款明细生成库存回补明细
  -> stockFacade.restore("ORDER_REFUND", refundNo, items)
  -> 记录 RESTORE 库存流水
  -> 商品销量按退款数量回滚
```

关键点：

- 回补业务号使用 `refundNo`。
- 同一个 `refundNo + skuId` 只能回补一次。
- 原支付成功时的库存锁 `CONFIRMED` 不回滚，退款作为新的 `RESTORE` 事实。

## 七、轻量对账与补偿设计

一期不做每日渠道账单下载，但要做轻量对账。

### 1. 支付对账

```text
按 orderNo / payOrderNo 触发
  -> 查询本地订单
  -> 查询本地支付单
  -> 查询支付宝交易状态
  -> 比对订单状态、支付状态、金额、支付宝交易号
  -> 返回一致/不一致
  -> 对可自动修复的差异执行补偿
```

重点差异：

| 差异 | 处理 |
| --- | --- |
| 支付宝已支付，本地支付单未成功 | 补偿支付单 SUCCESS |
| 支付单 SUCCESS，订单仍 PENDING_PAYMENT | 补偿订单 PAID，确认库存 |
| 订单已取消，但支付宝已支付 | 高危异常，建议走退款 |
| 金额不一致 | 禁止自动修复，记录异常 |

### 2. 退款对账

```text
按 refundNo 触发
  -> 查询本地退款单
  -> 查询支付宝退款状态
  -> 比对退款金额、退款状态、渠道退款号
  -> 不一致则记录差异
```

重点差异：

| 差异 | 处理 |
| --- | --- |
| 支付宝退款成功，本地退款中 | 补偿本地退款成功，回补库存 |
| 本地退款成功，支付宝未退款 | 高危异常，人工处理 |
| 退款金额不一致 | 禁止自动修复 |

### 3. 对账记录

建议新增轻量表 `pms_pay_reconciliation_record`。

字段够用即可：

| 字段 | 说明 |
| --- | --- |
| `id` | 主键 |
| `biz_type` | `PAY` / `REFUND` |

## 十、验收用例

### 1. MVP 验收

#### 正常支付

```text
下单 -> 锁库存 -> 支付宝沙箱支付 -> 回调成功 -> 订单 PAID -> 库存 CONFIRMED
```

验收点：

- 支付单为 `SUCCESS`。
- 订单为 `PAID`。
- 库存锁为 `CONFIRMED`。
- 重复回调不重复扣库存。

#### 超时未支付

```text
下单 -> 锁库存 -> 不支付 -> 订单超时 -> 订单 TIMEOUT_CANCELLED -> 库存 CANCELLED
```

验收点：

- 订单自动取消。
- 库存释放。
- 支付单关闭。

### 2. 一期验收

#### 支付金额异常

```text
支付宝回调金额 != 本地支付单金额
```

验收点：

- 回调记录为 `AMOUNT_MISMATCH`。
- 订单不变 `PAID`。
- 支付单不变 `SUCCESS`。

#### 支付成功但回调延迟

```text
支付宝已支付 -> 本地未收到回调 -> 超时任务触发 -> 主动查支付宝 -> 补偿支付成功
```

验收点：

- 订单不能被误取消。
- 库存不能被误释放。

#### 整单退款

```text
订单 PAID -> 申请整单退款 -> 调支付宝退款 -> 退款成功 -> 订单 REFUNDED -> 全部商品库存 RESTORE
```

验收点：

- 支付退款单为 `REFUND_SUCCESS`。
- 订单退款单为 `REFUND_SUCCESS`。
- 订单为 `REFUNDED`。
- 库存按全部商品回补。

#### 部分退款

```text
订单 PAID -> 选择部分订单项退款 -> 调支付宝退款 -> 退款成功 -> 订单 PARTIALLY_REFUNDED -> 部分商品库存 RESTORE
```

验收点：

- 退款数量不能超过购买数量。
- 退款金额按订单快照计算。
- 只回补退款明细里的 SKU 和数量。
- 再次退款只能退剩余可退数量和金额。

#### 退款对账

```text
支付宝退款成功，本地退款单仍 REFUNDING
```

验收点：

- 对账识别差异。
- 可以补偿本地退款成功。
- 补偿后只回补一次库存。

## 十一、最终推荐落地顺序

### MVP 顺序

```text
1. 稳定下单预占库存
2. 稳定支付宝沙箱支付成功链路
3. 支付成功确认库存扣减
4. 订单超时取消释放库存
5. 支付成功、取消、库存释放全部幂等
6. 前端展示支付倒计时和支付结果
```

### 一期顺序

```text
1. 支付回调持久化
2. 支付回调金额校验和单号校验
3. 超时取消前主动查支付宝支付状态
4. 新增支付宝沙箱退款接口
5. 新增退款单和退款明细
6. 实现整单退款
7. 实现部分退款
8. 退款成功后库存回补和销量回滚
9. 支付/退款轻量对账
10. 后台展示回调、退款、对账异常
```

## 十二、面试讲法总结

可以这样介绍：

```text
我这个商城项目不是只做 CRUD 和普通支付，而是围绕订单、支付、库存、退款和对账做了一个交易闭环。

MVP 阶段，我先做正向交易链路：下单时通过 Redis Lua 预占库存，支付宝沙箱支付成功后确认扣减库存，超时未支付则通过延迟消息、定时任务和请求兜底自动取消订单并释放库存。

一期阶段，我进一步贴近真实电商状态图：支付回调先落库，再做支付宝验签、金额校验、单号校验和幂等处理；订单超时取消前主动查询支付宝状态，避免回调延迟导致已付款订单被取消；退款接入支付宝沙箱，支持整单退款和部分退款，退款成功后按退款明细回补库存；最后通过轻量对账识别支付成功但订单未成功、退款成功但本地未更新等异常，并提供补偿能力。

我没有做完整物流、复杂售后审核和每日渠道账单下载，因为这些属于二期以后。但在一期范围内，订单、支付、退款、库存这些核心状态都能完整流转，已经能体现交易系统的状态机设计、幂等设计和最终一致性处理能力。
```

一句话总结：

```text
这个闭环的目标不是做重型支付中台，而是用支付宝沙箱真实支付/退款，把订单、库存、支付、整单退款、部分退款和轻量对账做成一个状态完整、逻辑闭环、面试能讲清楚的一期交易系统。
```

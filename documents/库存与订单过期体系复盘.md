# 库存与订单过期体系复盘

## 一、本次问题与修改复盘

### 1. 最初发现的问题

最开始关注的是：

> 库存能不能防住超卖？还要不要分布式锁 / 库存预扣 / 事务隔离？

后来进一步发现：

> 压测下单后，如果不打开订单页面，订单似乎不会自动过期；打开订单页面后才触发过期取消。

本质上这是两个核心问题：

1. 库存防超卖主链路是否可靠。
2. 订单过期、库存释放、支付超时是否形成自动闭环。

---

## 二、库存侧发现的问题

### 1. Redis 预占成功后，DB 锁记录保存失败时当前 SKU 可能回滚不到

原逻辑大概是：

```text
Redis Lua 预占成功
保存 DB 锁记录
记库存流水
发 MQ
最后才把当前 item 加入 reservedItems
```

问题是，如果 Redis 已扣库存，但 DB 保存锁记录或 MQ 发布失败，当前 SKU 还没加入回滚列表，catch 里释放不到。

#### 修改

改成：

```text
Redis Lua 预占成功
立即加入回滚列表
后续任何异常都能回滚当前 SKU
```

#### 收获

> 做补偿时，必须在“外部状态已改变”的第一时间记录补偿上下文。

Redis 已经扣库存，就应该立刻具备回滚能力，不能等后续步骤都成功后再记录。

---

### 2. 订单保存失败导致库存锁死的问题

#### 问题

创建订单流程原来是：

```text
创建订单对象
预占库存
保存订单
```

如果库存预占成功，但订单保存失败，就会出现：

```text
库存已锁定
订单不存在
后续订单超时任务扫不到它
库存可能长期占用
```

#### 修改

给 `createOrder` 增加补偿：

```text
库存预占成功 -> stockReserved = true
订单保存失败 -> 调用 stockFacade.cancel("ORDER", orderNo)
```

#### 收获

> 跨资源操作时，只要前一个资源已成功，后一个资源失败就必须补偿。

这里订单和库存不是一个本地事务，必须通过业务补偿保证最终一致。

---

### 3. DB 库存异步同步存在并发丢更新风险

#### 问题

原来 MQ 同步 DB 库存是：

```text
读取 ims_stock
读取 ims_stock_lock
Java 计算新库存
updateById 覆盖库存行
```

多个 MQ 线程同时同步同一个 SKU 时，可能出现：

```text
线程 A 读 available=100
线程 B 读 available=100
A 写 99
B 也写 99
```

正确应该是 98，但结果是 99，这就是丢更新。

#### 修改

改成 DB 原子 SQL：

```text
预占：available - quantity, locked + quantity
取消：available + quantity, locked - quantity
确认：total - quantity, locked - quantity
```

并保留：

```text
reserved_synced
cancelled_synced
confirmed_synced
```

做幂等。

#### 收获

> 幂等字段解决“重复消息”，原子 SQL/行锁解决“并发更新”。两者不是一回事。

`*_synced` 字段不能替代行级并发控制。

---

### 4. Redis 库存重建不应该只用 DB 库存

#### 问题

Redis key 缺失时，原逻辑直接：

```text
从 DB 读取库存
写回 Redis
```

如果此时有 Redis 预占成功但 DB 尚未同步，DB 可用库存会偏大，Redis 重建后可能把已预占库存重新放出来。

#### 修改

Redis 重建时改为：

```text
读取 DB 库存
查询 ims_stock_lock 中 reserved_synced = 0 的 RESERVED 数量
用 DB + 未同步预占量计算恢复库存
写回 Redis
```

#### 收获

> 缓存重建不能只看 DB 快照，还要看异步链路中的“未落库事实”。

异步架构里，DB 不是任何时刻都代表最新状态。

---

### 5. MQ 消费失败缺少完整死信/重试

#### 问题

之前定义了死信队列，但主队列没有真正绑定死信参数，失败消息可能没有形成完整闭环。

#### 修改

为库存同步队列加：

```text
x-dead-letter-exchange
x-dead-letter-routing-key
```

并配置 Spring Rabbit 重试。

#### 收获

> 有队列不等于有可靠消息链路。主队列必须明确配置重试、死信、消费幂等。

---

## 三、订单过期与支付超时问题复盘

### 1. 页面打开才触发过期的问题

#### 问题

压测下单后不打开页面，订单没有按预期自动关闭。打开订单页面后才触发。

说明实际生效的是：

```text
页面查询时的懒触发过期检查
```

而不是稳定的后台自动关闭。

#### 修改

改成三层机制：

```text
MQ 单消息 TTL 主动关闭
定时任务按 expire_time 补偿扫描
页面访问/支付前兜底
```

#### 收获

> 页面访问只能做兜底，不能作为订单过期关闭的主链路。

后台业务必须能在无人访问时自动推进状态。

---

### 2. 订单过期时间没有统一事实来源

#### 问题

原来订单过期更多依赖：

```text
createdAt + timeoutMinutes
```

或订单号解析时间。

这会带来几个问题：

- 配置变更后历史订单判断不稳定；
- 支付超时和业务订单超时不完全一致；
- 后续特殊订单过期时间不好扩展；
- 排查时不直观。

#### 修改

新增订单字段：

```text
expire_time
```

订单创建时持久化：

```text
expire_time = now + timeoutMinutes
```

所有过期判断统一读它。

#### 收获

> 复杂业务时间必须持久化，不能只靠运行时计算。

`expire_time` 是订单过期的唯一事实来源。

---

### 3. MQ 延迟关闭从队列级 TTL 改为单消息 TTL

#### 问题

队列级 TTL 有局限：

```text
x-message-ttl 固定在队列上
```

如果修改配置，旧队列参数不会自动变化，还可能启动报错。

#### 修改

延迟队列不再设置固定 TTL，而是发布消息时设置：

```text
message.expiration = expireTime - now
```

#### 收获

> 订单过期时间应该跟着订单走，而不是跟着队列走。

单消息 TTL 更适合支持不同订单不同过期时间。

---

### 4. 订单关闭用 DB 条件更新保证幂等

#### 问题

多个入口可能同时关闭订单：

- MQ；
- 定时任务；
- 页面访问；
- 支付前检查；
- 支付回调前检查。

如果只是“查出来判断再 update”，容易并发重复执行后续动作。

#### 修改

改成 DB 条件更新：

```sql
UPDATE oms_order
SET order_status = 'TIMEOUT_CANCELLED',
    cancelled_at = ?,
    version = version + 1
WHERE order_no = ?
  AND order_status = 'PENDING_PAYMENT'
  AND expire_time <= ?
```

只有更新成功的线程才释放库存、关闭支付单。

#### 收获

> 状态流转的幂等，最好落到数据库条件更新上。

业务判断可以在领域模型里做，但最终落库必须能抗并发。

---

### 5. 支付宝超时时间要按订单剩余时间生成

#### 问题

之前支付宝 `timeout_express` 是固定：

```text
mall.order.timeout-minutes + "m"
```

如果用户下单后过了一段时间才支付，就可能出现：

```text
业务订单即将过期
但支付宝仍给完整 15 分钟
```

#### 修改

改成：

```text
remaining = order.expire_time - now
支付宝 timeout_express = floor(remaining / 60) + "m"
```

剩余时间小于 90 秒时，拒绝创建支付单。

#### 收获

> 第三方支付有效期不能超过业务订单有效期。

否则会出现订单已关闭但支付成功的资金风险。

---

## 四、本次整体设计收获

### 1. 防超卖不是只靠“锁”

真正防超卖靠的是：

```text
库存判断 + 库存扣减 在同一个原子操作内完成
```

当前方案使用 Redis Lua，所以不需要再额外加分布式锁作为主防线。

---

### 2. 分布式系统不能只看主流程，还要看失败分支

本次修复的大部分问题不是主流程跑不通，而是：

- Redis 成功，DB 失败；
- 库存成功，订单失败；
- MQ 失败；
- MQ 乱序；
- 页面不访问；
- 支付回调晚到；
- Redis 丢 key；
- DB 异步落后。

#### 收获

> 大厂交付标准不是主流程能跑，而是异常路径也能收敛。

---

### 3. 幂等、补偿、对账是电商系统的底层能力

当前系统已经开始具备：

- 业务幂等键；
- 库存锁记录；
- 库存同步标记；
- MQ 重试/死信；
- 定时补偿；
- DB 条件更新；
- Redis 恢复计划。

这些都是从 MVP 走向生产级的关键能力。

## 五、库存系统当前能力全景

### 1. 库存数据模型能力

当前库存主要有三层数据。

#### 库存主表 `ims_stock`

保存 SKU 当前库存汇总：

```text
total_stock
locked_stock
available_stock
stock_status
warning_status
version
```

并有约束：

```text
total_stock = locked_stock + available_stock
```

#### 库存锁表 `ims_stock_lock`

记录订单级预占：

```text
business_type
business_no
sku_id
quantity
status
reserved_synced
cancelled_synced
confirmed_synced
```

它承担：

- 业务幂等；
- 状态追踪；
- MQ 同步幂等；
- 补偿扫描依据。

#### 库存流水表 `ims_stock_operation_log`

记录库存变更过程，具备业务唯一约束，防重复流水。

---

### 2. 防超卖能力

当前防超卖主链路是：

```text
Redis Lua 原子预占
```

Lua 内部完成：

```text
读取 available
判断库存是否足够
扣减 available
增加 locked
写 reservation key
```

这一套是原子的，因此并发下能防止多个请求同时扣同一份库存。

当前能力等级：

```text
正常高并发下防超卖：已具备
```

---

### 3. 库存预占能力

下单时不是直接扣总库存，而是：

```text
available_stock -> locked_stock
```

支付成功后：

```text
locked_stock -> total_stock 扣减
```

订单取消/超时：

```text
locked_stock -> available_stock
```

这是标准电商库存预占模型。

当前能力等级：

```text
库存预占/确认/释放：已具备
```

---

### 4. 库存幂等能力

通过：

```text
business_type + business_no + sku_id
```

标识一次业务库存操作。

例如：

```text
ORDER + orderNo + skuId
```

同一订单同一 SKU 重复预占、重复取消、重复确认，都能识别。

同时流水表也有业务唯一约束。

当前能力等级：

```text
业务幂等：已具备
```

---

### 5. Redis 与 DB 异步同步能力

当前设计是：

```text
Redis 做实时库存主闸门
DB 做最终库存持久化
MQ 做异步同步
```

这适合高并发场景，因为下单不用直接等 DB 库存行锁。

已增强内容：

```text
DB 原子 SQL 同步库存
```

并结合：

```text
reserved_synced
cancelled_synced
confirmed_synced
```

确保 MQ 重复消费不重复落库。

当前能力等级：

```text
Redis 主库存 + DB 最终一致：已具备
```

---

### 6. MQ 乱序收敛能力

现在如果 `CANCELLED` 或 `CONFIRMED` 先于 `RESERVED` 落库到达，会先补执行 `RESERVED` 同步，再执行终态同步。

例如：

```text
CANCELLED 到达
发现 reserved_synced = false
先补 reserved 落库
再执行 cancel 落库
```

当前能力等级：

```text
基础乱序收敛：已具备
```

---

### 7. Redis 恢复能力

Redis key 缺失时，不再直接用 DB 覆盖，而是：

```text
DB 库存
+ 未同步 RESERVED 锁记录
= Redis 恢复库存
```

避免 Redis 重建时把已预占但未落库库存重新放出来。

当前能力等级：

```text
Redis 异常恢复：具备基础生产能力
```

---

### 8. 库存补偿能力

当前已经有：

- 未同步库存预占补偿任务；
- 超时订单释放库存；
- 库存同步 MQ 重试/死信；
- Redis 恢复计划；
- 订单过期定时补偿。

当前能力等级：

```text
补偿机制：已具备基础闭环
```

---

### 9. 库存与订单协同能力

现在下单链路是：

```text
创建订单领域对象
预占库存
保存订单
发布超时消息
```

并且修复了：

```text
订单保存失败 -> 自动释放已预占库存
```

订单超时关闭后：

```text
DB 条件关闭订单成功
-> 释放库存
-> 关闭支付单
```

当前能力等级：

```text
订单-库存一致性：比之前明显增强，具备业务补偿闭环
```

---

### 10. 库存与支付协同能力

支付成功时：

```text
订单标记已支付
库存确认扣减
销量增加
支付单成功
```

支付前/支付回调前会先校验订单是否超时。

如果订单超时：

```text
拒绝继续支付成功
```

当前能力等级：

```text
支付-订单-库存状态联动：已具备主链路一致性
```

## 六、当前库存系统可以怎么描述

当前库存系统已经不是简单 CRUD 库存，可以描述为：

> 基于 Redis Lua 原子预占的高并发库存防超卖系统，结合库存锁记录、业务幂等、MQ 异步落库、DB 原子同步、Redis 恢复、订单超时释放和补偿任务，实现订单维度库存预占、确认、释放与最终一致。

更简单地说：

```text
Redis 防超卖
DB 做持久
锁记录做幂等
MQ 做同步
任务做补偿
订单过期释放库存
支付成功确认扣减
```

---

## 七、当前还有哪些未达到最高标准的地方

当前系统已经接近“中大型业务可交付”的基础形态，但如果继续往更高标准演进，还有以下方向。

### 1. Outbox 事件表还没有

现在订单关闭后释放库存、发布支付关闭事件等，还是直接调用。

更高标准是：

```text
本地事务更新订单
写 outbox 事件表
异步投递库存释放/支付关闭事件
消费幂等
```

这个能进一步增强事务消息可靠性。

但目前阶段可以暂不做。

---

### 2. 完整订单状态机还没有

现在订单状态迁移分布在 `Order` 领域模型方法中，已经比贫血模型好。

但还不是完整状态机，例如：

```text
PENDING_PAYMENT + PAY_SUCCESS
PENDING_PAYMENT + TIMEOUT
PENDING_PAYMENT + USER_CANCEL
PAID + REFUND
```

如果后续订单状态越来越复杂，可以抽状态机。

当前暂不需要过度设计。

---

### 3. 库存对账还可以更强

可以增强对账：

```text
DB库存 = 锁记录聚合结果
Redis库存 = DB库存 + 未同步锁记录
订单状态与库存锁状态一致
```

并提供自动修复/人工处理面板。

---

### 4. 库存释放失败补偿还可以更明确

订单已经 `TIMEOUT_CANCELLED`，但库存锁仍是 `RESERVED` 的情况，可以增加专门补偿任务：

```text
扫描已取消/超时订单
如果库存锁仍 RESERVED
重新释放库存
```

这会让订单-库存闭环更强。

---

### 5. 支付成功但订单已超时的资金补偿还可以完善

现在已经避免超时订单继续标记支付成功。

后续可以增加：

```text
异常支付单状态
自动退款任务
人工处理入口
```

防止三方支付极端延迟带来的资金问题。

---

## 八、后续分析系统的推荐方法

以后分析系统时，不要只看“正常流程”，而是按下面模板检查。

### 1. 主链路是否原子？

例如库存：

```text
判断库存够不够 + 扣减库存
是否在同一个原子操作中？
```

### 2. 重复请求是否幂等？

例如：

```text
同一个订单重复下单/重复取消/重复支付回调会怎样？
```

### 3. 中间一步失败是否能补偿？

例如：

```text
库存成功，订单失败怎么办？
订单关闭成功，库存释放失败怎么办？
```

### 4. MQ 消息会不会重复、丢失、乱序？

都要问：

```text
重复消费怎么办？
消息丢了怎么办？
后发先到怎么办？
```

### 5. 缓存丢失怎么恢复？

问：

```text
Redis 没了，能不能从 DB 恢复？
DB 是否包含最新事实？
有没有未同步中间态？
```

### 6. 并发更新是否会丢数据？

问：

```text
多个线程同时更新同一行，靠什么防丢更新？
```

### 7. 是否有统一事实来源？

例如：

```text
订单过期时间到底以谁为准？
订单状态到底以谁为准？
库存可售到底以谁为准？
```

### 8. 有没有兜底和对账？

问：

```text
主链路挂了，谁补？
补偿失败，谁发现？
发现不一致，怎么修？
```

## 九、最终总结

本次最大的进步是：

> 从“功能能跑”开始转向“异常可收敛、并发可幂等、状态有事实来源、链路有补偿”的工程化思维。

当前库存系统已经具备：

```text
防超卖
预占库存
支付确认扣减
取消/超时释放
业务幂等
MQ 异步同步
DB 原子落库
Redis 恢复
订单过期释放
支付超时一致
补偿任务
死信/重试
```

这已经是一个比较完整的电商库存闭环了。

后续如果继续提升，重点不是再加分布式锁，而是加强：


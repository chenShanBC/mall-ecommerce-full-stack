# mallFei 线上人工对账模块重构方案

## 1. 背景与目标

当前 mallFei 项目中已经具备支付单、退款单、订单异常、售后审核、支付异常核验、支付/退款状态同步等能力，但这些能力目前分散在多个页面和服务中：

- 订单管理：支付异常标记、支付异常处理、订单补偿、关闭释放库存。
- 支付管理：支付单同步、补偿订单、关闭支付单、退款单执行。
- 售后管理：退款申请审核、驳回、生成退款单。
- 对账管理：支付/退款对账记录、异常标记、人工处理。

这些能力本身有价值，但当前模块边界偏混乱，更像“支付异常处理工具集 + 状态修复工具集”，而不是一个真正的日终账务对账模块。

真正的线上人工对账模块应以“对账任务”为核心，围绕某个对账日期生成本地账单、获取渠道账单、自动勾兑、生成差异明细、人工处理差异、挂账跟进、审核归档，形成完整闭环。

本方案目标是将现有“不伦不类”的对账功能重构为一个更清晰、更接近真实业务的线上人工对账模块。

---

## 2. 核心设计原则

### 2.1 对账任务是核心入口

所有对账数据都应该归属于某个对账任务。

示例：

```text
2026-06-10 MOCK 渠道对账任务
  - 本地账单
  - 渠道账单
  - 自动勾兑结果
  - 差异明细
  - 人工处理记录
  - 挂账记录
  - 归档状态
```

对账页面不应只是展示散落的支付/退款异常记录，而应围绕任务展开。

### 2.2 本地账单从数据库自动生成

本地账单不建议手工上传，而应由系统从数据库自动生成快照。

原因：

- 本地业务数据本来就在系统数据库中。
- 自动生成更稳定，避免运营导出再上传的重复操作。
- 生成快照后可保证历史对账一致性。
- 后续订单/退款状态变化，不应影响已经生成的历史账单。

因此，文档中的“上传本地账单”在 mallFei 中可调整为“生成本地账单”。

### 2.3 渠道账单支持 Mock 生成和真实文件导入

真实线上场景下，支付宝/微信渠道账单通常来自商户平台下载文件。由于当前项目是学习/作品项目，可以先支持 Mock 渠道账单生成，后续再扩展渠道账单文件上传。

建议支持两种模式：

1. **生成 Mock 渠道账单**
   - 用于本地演示和测试。
   - 可模拟正常账单、漏单、单边账、金额不一致、退款状态异常。

2. **导入渠道账单文件**
   - 后续扩展。
   - 支持支付宝/微信 CSV 或 Excel 文件解析。

### 2.4 自动勾兑只发现问题，不直接改业务状态

自动勾兑阶段只做：

```text
本地账单 + 渠道账单 -> 匹配 -> 生成差异明细
```

不应直接修改订单、支付单、退款单。

业务修复必须由人工在差异单上明确点击处理动作，并记录操作日志。

### 2.5 原始账单快照不可随意修改

本地账单和渠道账单一旦生成/导入，应作为对账快照保存。

处理差异时可以修改业务表和差异单处理状态，但不应修改原始账单快照。

### 2.6 归档后只读

对账任务完成归档后，应禁止：

- 重新生成本地账单。
- 重新生成/导入渠道账单。
- 重新自动勾兑。
- 修改差异处理结果。

归档后只能查看、导出、审计。

## 3. 模块边界重构

### 3.1 现有能力重新归类

| 当前能力 | 建议归属 |
|---|---|
| 订单支付异常标记 | 订单异常处理模块 |
| 支付异常核验 | 单笔支付异常处理模块 |
| 支付单同步状态 | 支付单运维能力 |
| 补偿订单 | 支付单运维/对账差异处理动作 |
| 关闭支付单 | 支付单运维能力 |
| 退款单执行退款 | 售后退款/退款执行模块 |
| 支付/退款对账记录 | 未来可迁移为对账差异明细 |
| 对账异常处理按钮 | 应迁移到真正的差异单处理动作 |

### 3.2 新模块定位

新增或重构为：

```text
线上人工对账模块
```

职责：

- 创建对账任务。
- 生成本地账单快照。
- 生成或导入渠道账单。
- 自动勾兑。
- 生成差异明细。
- 人工处理差异。
- 挂账跟进。
- 操作审计。
- 完成归档。

## 4. 核心业务流程

### 4.1 主流程

```text
创建对账任务
  -> 生成本地账单
  -> 生成 Mock 渠道账单 / 导入渠道账单
  -> 开始自动勾兑
  -> 生成差异明细
  -> 人工处理差异
  -> 必要时挂账
  -> 全部处理完成
  -> 任务归档
```

### 4.2 页面按钮设计

在对账任务详情页提供：

```text
生成本地账单
生成 Mock 渠道账单
导入渠道账单
开始自动勾兑
导出差异
完成归档
```

根据任务状态控制按钮是否可用。

## 5. 本地账单自动生成设计

### 5.1 本地支付账单来源

来源表建议包括：

- 支付单表。
- 订单表。
- 支付回调记录表。

生成范围：

- 按对账日期筛选。
- 可按支付成功时间、支付单创建时间或交易时间进行筛选。
- 第一版建议以支付单创建时间或成功时间落在对账日期内为准。

本地支付账单字段：

```text
taskId
bizType = PAY
orderNo
payOrderNo
userId
localStatus
orderStatus
amountCent
payChannel
transactionNo
tradeTime
rawSnapshot
createdAt
```

其中：

- `localStatus`：支付单状态，如 `PENDING`、`PAYING`、`SUCCESS`、`CLOSED`、`REFUNDED`。
- `orderStatus`：订单状态，如 `PENDING_PAYMENT`、`PAID`、`SHIPPED`。
- `tradeTime`：优先取支付成功时间；没有则取支付单创建时间。
- `rawSnapshot`：保存生成账单时的原始 JSON 快照。

### 5.2 本地退款账单来源

来源表建议包括：

- 退款单表。
- 支付单表。
- 订单表。
- 售后单表。
- 退款回调记录表。

本地退款账单字段：

```text
taskId
bizType = REFUND
orderNo
payOrderNo
refundNo
userId
localStatus
orderStatus
amountCent
payChannel
transactionNo
channelRefundNo
tradeTime
rawSnapshot
createdAt
```

其中：

- `localStatus`：退款单状态，如 `REFUND_PENDING`、`REFUNDING`、`REFUND_SUCCESS`、`REFUND_FAILED`。
- `amountCent`：退款金额。
- `tradeTime`：优先取退款成功时间；没有则取退款单创建时间。

### 5.3 本地账单生成规则

点击“生成本地账单”后：

```text
1. 校验任务未归档。
2. 清理当前任务下旧的本地账单快照，或禁止重复生成。
3. 从数据库查询支付单和退款单。
4. 转换为本地账单明细。
5. 保存账单快照。
6. 更新任务的本地账单统计信息。
7. 页面展示本地账单明细。
```

统计字段：

```text
localTotalCount
localTotalAmountCent
localPayCount
localPayAmountCent
localRefundCount
localRefundAmountCent
```

## 6. Mock 渠道账单设计

### 6.1 Mock 渠道账单目标

Mock 渠道账单用于模拟支付宝/微信官方账单，使对账模块可在本地完整演示。

它不应简单等于本地账单，否则无法体现对账差异。

Mock 账单应支持以下差异模式：

```text
无差异
随机少量差异
漏单
单边账
金额不一致
退款状态异常
全量测试样例
```

### 6.2 Mock 渠道账单生成弹窗

点击“生成 Mock 渠道账单”时，可弹出参数选择：

```text
渠道：MOCK / ALIPAY_PC / ALIPAY_WAP
差异模式：无差异 / 随机少量差异 / 漏单 / 单边账 / 金额不一致 / 退款状态异常 / 全量测试样例
差异比例：例如 5%
是否覆盖已生成渠道账单：是 / 否
```

第一版可以简化为：

```text
渠道：MOCK
差异模式：无差异 / 全量测试样例
```

### 6.3 Mock 支付渠道账单结构

示例：

```json
{
  "bizType": "PAY",
  "channel": "MOCK",
  "outTradeNo": "PAY202606100001",
  "orderNo": "ORD202606100001",
  "payOrderNo": "PAY202606100001",
  "channelTradeNo": "MOCK-TX-202606100001",
  "channelStatus": "TRADE_SUCCESS",
  "amountCent": 9900,
  "feeCent": 0,
  "tradeTime": "2026-06-10 10:15:30",
  "buyerAccount": "mock_user_001",
  "rawLine": "{...}"
}
```

### 6.4 Mock 退款渠道账单结构

示例：

```json
{
  "bizType": "REFUND",
  "channel": "MOCK",
  "outTradeNo": "PAY202606100001",
  "orderNo": "ORD202606100001",
  "payOrderNo": "PAY202606100001",
  "refundNo": "RF202606100001",
  "channelTradeNo": "MOCK-TX-202606100001",
  "channelRefundNo": "MOCK-RF-202606100001",
  "channelStatus": "REFUND_SUCCESS",
  "amountCent": 9900,
  "feeCent": 0,
  "tradeTime": "2026-06-10 11:20:30",
  "rawLine": "{...}"
}
```

### 6.5 Mock 差异生成规则

#### 6.5.1 正常支付流水

本地支付单：

```text
payStatus = SUCCESS
```

生成渠道流水：

```text
bizType = PAY
channelStatus = TRADE_SUCCESS
amountCent = payAmountCent
```

勾兑结果：

```text
MATCHED
```

#### 6.5.2 我方有单，渠道无流水

本地有支付成功账单，但 Mock 渠道账单故意不生成对应流水。

勾兑结果：

```text
LOCAL_EXISTS_CHANNEL_MISSING
```

含义：

```text
我方记账成功，但渠道无成功流水。
```

#### 6.5.3 渠道有流水，我方无单

Mock 渠道账单生成一条不存在于本地的流水。

示例：

```text
outTradeNo = MOCK-UNKNOWN-202606100001
channelStatus = TRADE_SUCCESS
amountCent = 9900
```

勾兑结果：

```text
CHANNEL_EXISTS_LOCAL_MISSING
```

#### 6.5.4 金额不一致

本地：

```text
amountCent = 9900
```

渠道：

```text
amountCent = 9800
```

勾兑结果：

```text
AMOUNT_MISMATCH
```

#### 6.5.5 退款状态异常

本地：

```text
localStatus = REFUNDING
```

渠道：

```text
channelStatus = REFUND_SUCCESS
```

勾兑结果：

```text
REFUND_STATUS_MISMATCH
```

建议动作：

```text
SYNC_REFUND_RESULT
```

## 7. 自动勾兑设计

### 7.1 匹配键

支付账单匹配：

```text
本地 payOrderNo = 渠道 outTradeNo
```

退款账单匹配优先级：

```text
1. 本地 refundNo = 渠道 refundNo
2. 本地 channelRefundNo = 渠道 channelRefundNo
3. 本地 payOrderNo + refundNo 与渠道 outTradeNo + refundNo 匹配
```

第一版可以先使用：

```text
bizType + payOrderNo + refundNo
```

### 7.2 差异类型

建议定义：

```text
MATCHED                         完全匹配
LOCAL_EXISTS_CHANNEL_MISSING    我方有单，渠道无流水
CHANNEL_EXISTS_LOCAL_MISSING    渠道有流水，我方无单
AMOUNT_MISMATCH                 金额不一致
STATUS_MISMATCH                 状态不一致
REFUND_STATUS_MISMATCH          退款状态不一致
STATUS_AND_AMOUNT_MISMATCH      状态和金额均不一致
UNKNOWN                         未知差异
```

### 7.3 差异等级

建议定义：

```text
LOW
MEDIUM
HIGH
```

推荐规则：

| 差异类型 | 等级 |
|---|---|
| LOCAL_EXISTS_CHANNEL_MISSING | HIGH |
| CHANNEL_EXISTS_LOCAL_MISSING | HIGH |
| AMOUNT_MISMATCH | HIGH |
| REFUND_STATUS_MISMATCH | MEDIUM |
| STATUS_MISMATCH | MEDIUM |
| MATCHED | LOW |

### 7.4 建议处理动作

```text
NONE
SYNC_PAY_STATUS
REPAIR_ORDER_STATUS
CLOSE_ORDER_AND_RELEASE_STOCK
SYNC_REFUND_RESULT
MARK_REFUND_FAILED
MANUAL_REVIEW
HANG
IGNORE
```

## 8. 差异处理设计

### 8.1 支付类差异处理

#### 8.1.1 渠道已支付，本地未成功

差异：

```text
STATUS_MISMATCH
```

建议动作：

```text
SYNC_PAY_STATUS
```

处理效果：

```text
支付单 -> SUCCESS
订单 -> 根据原始履约状态恢复：PAID / PROCESSING / SHIPPED
差异单 -> DONE
记录操作日志
```

#### 8.1.2 支付单成功，订单未补偿

建议动作：

```text
REPAIR_ORDER_STATUS
```

处理效果：

```text
订单状态补偿
差异单 -> DONE
记录操作日志
```

注意：

如果订单是从已发货状态进入支付异常，补偿后应恢复为 `SHIPPED`，不能退回 `PAID`。

#### 8.1.3 我方有单，渠道无流水

建议动作：

```text
MANUAL_REVIEW / HANG / CLOSE_ORDER_AND_RELEASE_STOCK
```

处理策略：

- 若确认未支付，关闭订单并释放库存。
- 若无法确认，挂账等待后续跟进。
- 若是测试数据，标记忽略。

#### 8.1.4 渠道有流水，我方无单

建议动作：

```text
MANUAL_REVIEW / HANG
```

处理策略：

- 标记单边账。
- 人工核验渠道流水。
- 必要时补录业务记录或挂账。

### 8.2 退款类差异处理

#### 8.2.1 渠道退款成功，本地退款中

建议动作：

```text
SYNC_REFUND_RESULT
```

处理效果：

```text
退款单 -> REFUND_SUCCESS
支付单 -> REFUNDED / PARTIALLY_REFUNDED
订单 -> REFUNDED / PARTIALLY_REFUNDED
库存回补
售后单更新
差异单 -> DONE
记录操作日志
```

#### 8.2.2 渠道退款失败，本地退款中

建议动作：

```text
MARK_REFUND_FAILED
```

处理效果：

```text
退款单 -> REFUND_FAILED
订单恢复原履约状态或售后待处理
差异单 -> DONE / MANUAL_REVIEW
记录操作日志
```

#### 8.2.3 退款金额不一致

建议动作：

```text
MANUAL_REVIEW / HANG
```

处理策略：

- 人工核验退款金额。
- 必要时挂账。
- 不允许自动修改为成功。

## 14. 与现有功能的衔接

### 14.1 支付异常处理保留

订单管理中的支付异常处理应保留，用于单笔实时异常处理。

定位：

```text
单笔订单支付异常核验和履约冻结工具
```

不再把它包装成正式对账模块。

### 14.2 支付管理保留

支付管理中的支付单查询、关闭、同步、补偿仍然保留。

定位：

```text
支付单运维工具
```

对账模块可以调用这些能力作为差异处理动作。

### 14.3 售后退款保留

售后管理继续负责退款申请审核和退款单创建。

对账模块只处理账务差异，不直接替代售后审核流程。

### 14.4 现有 PayReconciliationRecord 处理

现有 `PayReconciliationRecord` 可以暂时保留，后续有两种处理方式：

1. 作为旧版支付异常待处理记录继续保留。
2. 逐步迁移到新的 `pay_reconcile_diff_item`。

建议第一阶段不要急着删除旧表，避免影响现有功能。

## 16. 总结

本地账单建议从数据库自动生成，而不是手工上传。

渠道账单在真实场景下来自商户平台文件导入；在 mallFei 项目中可以先实现 Mock 渠道账单生成，用于完整演示对账流程。

本次重构的关键不是简单新增几个按钮，而是将对账模块从当前的“支付/退款状态修复工具集”升级为：

```text
对账任务
  -> 本地账单快照
  -> 渠道账单快照
  -> 自动勾兑
  -> 差异明细
  -> 人工处理
  -> 挂账
  -> 归档
```

按该方案重构后，mallFei 的对账模块会更加清晰、真实，也更适合作为项目亮点展示。

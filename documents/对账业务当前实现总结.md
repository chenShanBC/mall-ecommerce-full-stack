# 对账业务当前实现总结

> 本文基于当前项目代码与《对账当前决定的实现.txt》梳理，用于后续绘制流程图、状态机图和技术实现图。

## 1. 当前对账业务定位

当前项目实现的是后台运营视角的一期增强版对账体系，重点不是完整财务级账单对账，而是围绕商城核心交易链路做一致性核验和异常处置闭环。

当前对账覆盖四类核心对象：

1. 订单：订单号、订单状态、应付金额、支付金额、履约状态。
2. 支付单：支付单号、支付状态、支付渠道、支付金额、渠道流水号、支付成功时间。
3. 退款/售后：退款单、退款状态、本地退款金额、渠道退款结果。
4. 库存：库存主表、锁定库存、可售库存、理论库存快照、库存对账记录。

当前对账的核心目标是：

- 发现订单、支付单、渠道查询结果之间的不一致。
- 识别疑似长款、疑似短款、金额不一致、状态不一致等业务风险。
- 生成可跟踪的对账记录。
- 根据差异类型给出处理建议和允许动作。
- 让后台运营可以执行真实业务动作或进入人工处理状态。
- 最终把异常流转到已完成或已忽略，形成闭环。

当前没有做渠道账单文件导入，因此它不是“渠道全量流水账单系统”。它更准确的定位是：

> 基于平台订单、支付、退款、库存数据，并结合支付渠道单笔查询能力的一致性核验与业务处置工作台。

## 2. 当前已经做到的范围

### 2.1 支付对账

支付对账已经支持以订单号为入口执行单笔对账。后端会查询业务订单、平台支付单，并在有支付单时调用支付渠道单笔查询能力，然后比较本地与渠道之间的状态和金额。

已覆盖的核验点包括：

- 订单是否存在。
- 支付单是否存在。
- 订单是否已经支付或进入履约后状态。
- 支付单是否成功。
- 渠道是否查询成功。
- 渠道是否显示已支付。
- 订单金额与支付单金额是否一致。
- 支付单成功但订单未同步。
- 渠道成功但本地未成功。
- 本地成功但渠道未确认。
- 订单已支付但缺少支付单。

支付对账会生成支付类对账记录，记录业务类型、批次号、订单号、支付单号、本地状态、渠道状态、本地金额、渠道金额、差异类型、处置状态和备注。

### 2.2 退款对账

退款对账已经有基础链路。后台可以对订单退款单触发退款状态同步/对账，系统会记录退款业务类型的对账记录。

已覆盖的核验点包括：

- 退款单是否存在。
- 本地退款状态。
- 渠道退款状态。
- 本地退款金额。
- 渠道退款金额。
- 退款状态是否一致。
- 退款金额是否一致。

退款记录和支付记录共用 `PayReconciliationRecord` 对账记录模型，通过 `bizType` 区分 `PAY` 和 `REFUND`。

### 2.3 支付异常核验

订单支付异常场景已经接入后台核验能力。后台可以从订单异常入口进入支付核验，系统会查询：

- 订单状态。
- 支付单状态。
- 支付回调记录。
- 渠道支付查询结果。
- 金额一致性。

核验结果会给出：

- 核验结论。
- 风险等级。
- 建议动作。
- 允许动作。
- 说明文案。

这部分用于把订单异常处理与对账处置连接起来，例如将支付异常转为支付同步待处理、订单修复待处理、关闭释放库存待处理或金额对账待处理。

### 2.4 长短款识别

当前实现的是“渠道单笔查询型”的疑似长短款识别。

疑似长款/长单主要指：

> 平台侧支付单成功或订单已支付，但渠道查询没有确认成功，存在平台多记账、未真实收款却继续履约的风险。

疑似短款/短单主要指：

> 渠道查询显示已支付，但本地支付单或订单没有正确入账，存在用户已付款但平台漏记账或未履约的风险。

因为当前没有渠道账单文件导入，所以系统不能发现“渠道存在一笔平台完全不知道的流水”。当前能够识别的是平台已知订单/支付单上的渠道状态差异。

### 2.5 对账差异处置闭环

当前对账记录不是简单标记“已处理”，而是已经根据差异类型提供业务化动作。

后台处置动作包括：

- 重新对账。
- 同步支付状态。
- 补偿订单已支付。
- 恢复订单待支付。
- 关闭订单并释放库存。
- 待渠道调单。
- 待用户补款/追缴。
- 待退款。
- 人工审核。
- 确认已解决。
- 忽略归档。

其中，同步支付状态、补偿订单已支付、恢复待支付、关闭并释放库存属于真实业务动作；待调单、待补款、待退款、人工审核属于状态流转；确认已解决和忽略归档属于归档动作。

### 2.6 库存对账

库存对账已经支持后台查看和处理库存一致性记录。

已覆盖能力包括：

- 发起单 SKU 库存一致性校验。
- 生成库存对账记录。
- 查询库存对账列表。
- 查看库存对账详情。
- 对比库存表快照与理论库存快照。
- 修复库存差异。
- 批量修复库存差异。
- 忽略库存差异。

库存修复会重新计算最新理论库存，如果仍不一致，会将库存主表校准到理论快照，并同步 Redis 库存。

## 3. 当前对账数据来源

### 3.1 平台订单数据

订单模块为对账提供业务订单状态和订单金额基线。当前对账依赖订单侧判断：

- 订单是否待支付。
- 订单是否已支付或进入履约后状态。
- 订单是否支付异常。
- 订单应付金额。
- 订单关闭、恢复待支付、补偿已支付等后台修复能力。

对账处置中，订单侧承接了两个关键修复动作：

1. 补偿订单已支付：当支付单或渠道结果证明已收款，但订单仍未支付时，将订单补偿到已支付/可履约状态。
2. 关闭订单并释放库存：当确认无有效收款证据且不再追缴时，关闭订单并释放库存。

### 3.2 平台支付数据

支付模块为对账提供支付单、回调记录、退款单和对账记录。

支付侧当前承担：

- 创建支付单。
- 处理渠道回调。
- 保存支付回调记录。
- 查询支付单。
- 查询退款单。
- 同步支付状态。
- 执行支付对账。
- 执行退款对账。
- 生成或刷新支付/退款对账记录。

支付对账记录由 `PayReconciliationRecord` 表达，核心字段包括：

- `batchNo`：对账批次号。
- `bizType`：业务类型，支付为 `PAY`，退款为 `REFUND`。
- `orderNo`：订单号。
- `payOrderNo`：支付单号。
- `refundNo`：退款单号。
- `localStatus`：本地状态。
- `channelStatus`：渠道状态。
- `localAmountCent`：本地金额。
- `channelAmountCent`：渠道金额。
- `consistent`：是否对平。
- `diffType`：差异类型。
- `repairStatus`：处置状态。
- `remark`：备注。

### 3.3 渠道单笔查询数据

当前没有导入支付宝、微信或银行的渠道账单文件，因此渠道侧数据来源主要是支付渠道单笔查询。

渠道查询用于判断：

- 渠道是否能查到该支付单。
- 渠道交易状态是否成功。
- 渠道交易号。
- 本地支付状态与渠道状态是否一致。

当前对账能发现的是“平台已知订单/支付单”的渠道状态异常。它不能发现“渠道存在流水，但平台完全没有订单或支付单”的情况，因为缺少渠道全量账单明细。

### 3.4 库存数据

库存对账依赖两类快照：

1. 库存表当前快照：总库存、锁定库存、可售库存、库存状态、预警状态。
2. 理论库存快照：根据业务记录、锁定记录、操作日志等计算出的应有库存结果。

库存一致性校验会比较当前库存与理论库存。当存在差异时，生成库存对账记录，后台可以修复或忽略。

## 4. 单笔支付对账主流程

单笔支付对账是当前支付对账的核心流程，入口通常来自后台对账工作台的“重新对账/重跑”。

流程可以画成：

```text
后台点击重新对账
        ↓
调用后台对账接口
        ↓
PayApplicationService.reconcile(orderNo)
        ↓
生成 PAY 对账批次号
        ↓
查询业务订单
        ↓
查询平台支付单
        ↓
如果存在支付单，则调用渠道单笔查询
        ↓
根据渠道结果尝试补偿支付单状态
        ↓
根据支付单/渠道结果尝试补偿订单状态
        ↓
比较金额是否一致
        ↓
比较状态是否一致
        ↓
解析差异类型 diffType
        ↓
生成 PayReconciliationRecord
        ↓
如果是对平记录：直接保存
        ↓
如果是异常记录：检查同订单 + 同差异类型 + PENDING 是否存在
        ↓
存在则刷新原待处理记录，不存在则新增记录
        ↓
返回前端对账结果并刷新工作台
```

这个流程里有两个重要设计点。

### 4.1 对账时会尝试自动补偿

对账不是只做静态比较。代码中会在查询订单、支付单、渠道结果后，先执行支付单和订单的补偿判断，再计算一致性。

业务含义是：

- 如果渠道确认已支付，但本地支付单还未成功，可以尝试把支付单同步为成功。
- 如果支付单已经成功，但订单仍然待支付或支付异常，可以尝试补偿订单为已支付。

这样可以减少“已知可修复问题”长期停留在异常列表中。

### 4.2 待处理异常记录防重复

异常记录保存时，会判断是否已存在同订单、同差异类型、仍处于 `PENDING` 的记录。

```text
同订单号 + 同差异类型 + PENDING
        ↓
重新对账时刷新原记录
        ↓
不重复新增异常记录
```

如果原记录已经完成、忽略或进入其他归档/处理中状态，再次出现新异常时，会保留新的记录，形成历史轨迹。

## 5. 差异类型识别逻辑

支付对账最终会把异常归类为具体 `diffType`。当前主要逻辑如下。

### 5.1 对平

当金额一致且状态一致时，差异类型为：

- `NONE`

这类记录的 `consistent` 为 `true`，`repairStatus` 为 `NONE`，无需业务处置。

### 5.2 订单已支付但缺少支付单

触发条件：

```text
订单已支付或进入履约后状态
        +
平台没有支付单
```

差异类型：

- `SHORT_PAID_ORDER_PAY_MISSING`

业务含义：订单已经表现为收款或可履约，但平台缺少支付凭据，存在疑似短款或历史脏数据风险。

建议处置：人工审核、恢复待支付、关闭并释放库存、确认已解决或忽略。

### 5.3 平台成功但渠道未确认

触发条件：

```text
平台支付单 SUCCESS
        +
渠道查询结果明确可知
        +
渠道未支付或未确认成功
```

差异类型：

- `LONG_PLATFORM_SUCCESS_CHANNEL_MISSING`

业务含义：平台侧显示已收款，但渠道没有确认真实收款，存在平台多记账、未收款履约或资损风险。

建议处置：重新对账、待渠道调单、恢复待支付、关闭并释放库存、待用户补款。

### 5.4 渠道成功但平台未正确入账

触发条件：

```text
渠道查询已支付
        +
本地支付单未成功 或 订单未支付
```

差异类型：

- `SHORT_CHANNEL_SUCCESS_PLATFORM_UNPAID`

业务含义：用户可能已经付款，但平台支付单或订单未正确入账，存在平台少记收入、用户已付款无法履约的风险。

建议处置：同步支付状态、补偿订单已支付、待渠道调单、确认已解决或忽略。

### 5.5 支付成功但订单状态未同步

触发条件：

```text
支付单 SUCCESS
        +
订单仍未支付
```

差异类型：

- `ORDER_STATUS_NOT_SYNCED`

业务含义：支付成功事实已经成立，但订单状态没有同步，影响发货、售后和后续履约。

建议处置：补偿订单已支付、重新对账、确认已解决或忽略。

### 5.6 金额或状态不一致

当不存在上述长短款特征，但金额或状态比较不一致时，会进入通用差异：

- `AMOUNT_MISMATCH`：金额不一致。
- `STATUS_MISMATCH`：状态不一致。
- `STATUS_AND_AMOUNT_MISMATCH`：状态和金额都不一致。
- `UNKNOWN`：无法明确归类的异常。

金额差异通常需要人工审核、待退款或待补款；状态差异通常需要同步支付状态或补偿订单状态。

## 6. 对账记录与处置状态

支付/退款对账记录使用统一的处置状态 `repairStatus` 表达从发现异常到处理归档的过程。

当前状态包括：

| 状态 | 含义 |
|---|---|
| `NONE` | 无需处理，通常表示对平记录 |
| `PENDING` | 待处理，异常刚生成或待运营处置 |
| `RECHECKING` | 复核中，已经重新发起对账 |
| `WAIT_CHANNEL_CONFIRM` | 待渠道调单，需要去支付渠道后台核查原始流水/报文 |
| `WAIT_USER_REPAY` | 待用户补款/追缴 |
| `WAIT_REFUND` | 待退款 |
| `MANUAL_REVIEW` | 人工审核 |
| `DONE` | 已完成 |
| `IGNORED` | 已忽略 |

推荐状态机可以画成：

```text
PENDING
  ├─ 重新对账 → RECHECKING
  ├─ 同步支付状态 → DONE
  ├─ 补偿订单已支付 → DONE
  ├─ 恢复待支付 → DONE
  ├─ 关闭并释放库存 → DONE
  ├─ 待渠道调单 → WAIT_CHANNEL_CONFIRM
  ├─ 待用户补款 → WAIT_USER_REPAY
  ├─ 待退款 → WAIT_REFUND
  ├─ 人工审核 → MANUAL_REVIEW
  ├─ 确认已解决 → DONE
  └─ 忽略归档 → IGNORED
```

需要注意：`WAIT_CHANNEL_CONFIRM`、`WAIT_USER_REPAY`、`WAIT_REFUND`、`MANUAL_REVIEW` 不是最终完成态，它们表示该对账差异已经从待处理进入具体人工/业务处理队列，后续仍需要人工确认完成或忽略归档。

## 7. 后台业务化处置闭环

### 7.1 处置入口

当前后台主要处置接口是：

```text
POST /api/admin/reconciliations/pay-records/{id}/handle
```

请求参数核心是：

- `action`：处置动作。
- `reason`：处理原因或说明。

后端处理流程可以画成：

```text
后台选择业务处置动作
        ↓
提交 action + reason
        ↓
根据 id 查询 PayReconciliationRecord
        ↓
校验记录是否可处置
        ↓
根据 diffType 获取允许动作列表
        ↓
校验 action 是否允许
        ↓
执行真实业务动作或状态流转
        ↓
写入管理员操作日志
        ↓
更新对账记录 repairStatus
        ↓
返回最新对账记录视图
        ↓
前端刷新工作台
```

### 7.2 处置策略

后台使用 `AdminReconciliationPolicy` 统一负责差异策略，包括：

- 根据 `diffType` 计算差异分类。
- 生成中文分类名称。
- 生成处理建议。
- 返回可执行动作。
- 控制不同差异类型允许哪些动作。

差异分类包括：

| 分类 | 含义 |
|---|---|
| `NORMAL` | 对平 |
| `LONG` | 长款/长单 |
| `SHORT` | 短款/短单 |
| `AMOUNT` | 金额不一致 |
| `STATUS` | 状态不一致 |
| `MANUAL` | 人工核查 |

### 7.3 支持的业务动作

| 动作 | 业务含义 | 处理方式 |
|---|---|---|
| `RETRY_RECONCILE` | 重新对账 | 重新调用单笔对账，并把当前记录标记为复核中 |
| `SYNC_PAY_STATUS` | 同步支付状态 | 调用支付同步逻辑，渠道已支付时同步支付单并补偿订单 |
| `REPAIR_ORDER_PAID` | 补偿订单已支付 | 支付单成功但订单未支付时补偿订单状态 |
| `RESTORE_PENDING_PAYMENT` | 恢复待支付 | 调用订单侧恢复待支付能力 |
| `CLOSE_AND_RELEASE_STOCK` | 关闭并释放库存 | 调用订单侧后台关闭能力，并释放库存 |
| `WAIT_CHANNEL_CONFIRM` | 待渠道调单 | 将记录状态流转到待渠道确认 |
| `WAIT_USER_REPAY` | 待用户补款 | 将记录状态流转到待用户补款/追缴 |
| `WAIT_REFUND` | 待退款 | 将记录状态流转到待退款处理 |
| `MANUAL_REVIEW` | 人工审核 | 将记录状态流转到人工审核 |
| `MARK_DONE` | 确认已解决 | 记录归档为已完成 |
| `IGNORE` | 忽略归档 | 记录归档为已忽略 |

### 7.4 真实业务动作说明

#### 同步支付状态

适用场景：

```text
渠道查询已支付
        +
本地支付单未成功 或 订单未支付
```

处理结果：

- 查询渠道。
- 如果渠道已支付，则把本地支付单标记为成功。
- 如果订单仍待支付或支付异常，则补偿订单为已支付。
- 对账记录标记完成。

#### 补偿订单已支付

适用场景：

```text
支付单 SUCCESS
        +
订单仍为待支付或支付异常
```

处理结果：

- 调用订单侧后台补偿能力。
- 订单进入已支付/可履约状态。
- 对账记录标记完成。

#### 恢复待支付

适用场景：

```text
确认渠道未收款
        +
订单不应保持已支付或支付异常状态
        +
仍允许用户重新支付
```

处理结果：

- 调用订单侧恢复待支付能力。
- 用户可重新发起支付。
- 对账记录标记完成。

#### 关闭并释放库存

适用场景：

```text
确认无有效收款证据
        +
不再继续补款或追缴
```

处理结果：

- 调用订单侧后台关闭能力。
- 订单关闭。
- 释放预占库存。
- 对账记录标记完成。

## 8. 退款对账流程

退款对账入口主要来自后台退款状态同步接口。流程可以画成：

```text
后台对某订单退款单执行同步状态
        ↓
查询支付单
        ↓
查询退款单
        ↓
按退款单号和金额构造渠道退款查询请求
        ↓
调用渠道退款查询能力
        ↓
如果渠道退款成功/失败，则同步本地退款状态
        ↓
比较本地退款状态与渠道状态
        ↓
比较本地退款金额与渠道退款金额
        ↓
生成 REFUND 类型对账记录
        ↓
后台对账记录列表展示
```

退款对账使用的差异类型与支付对账共用一部分通用逻辑，重点关注：

- 退款单是否存在。
- 渠道退款结果是否可知。
- 退款状态是否一致。
- 退款金额是否一致。

当前退款对账可以进入对账记录工作台展示和归档，但相比支付对账，自动化业务处置动作更偏基础，主要作为退款状态核验和异常记录留痕。

## 9. 库存对账流程

库存对账与支付对账在同一个后台对账页面下，但业务链路独立。

### 9.1 库存一致性校验

流程可以画成：

```text
后台在库存管理发起 SKU 一致性校验
        ↓
计算库存当前快照
        ↓
计算理论库存快照
        ↓
比较总库存、锁定库存、可售库存
        ↓
生成库存一致性结果
        ↓
保存库存对账记录
        ↓
后台库存对账列表展示
```

库存对账记录支持状态筛选：

- `CONSISTENT`：一致。
- `INCONSISTENT`：不一致。
- `REPAIRED`：已修复。
- `IGNORED`：已忽略。

### 9.2 库存差异修复

单条修复流程可以画成：

```text
后台选择库存对账记录修复
        ↓
重新计算该 SKU 最新理论库存
        ↓
如果最新结果仍不一致
        ↓
将库存主表校准为理论库存快照
        ↓
同步 Redis 库存
        ↓
对账记录标记为已修复
        ↓
写入管理员操作日志
```

批量修复是前端对多条可修复记录逐条调用修复接口。

### 9.3 库存差异忽略

当确认差异无需处理或属于测试/历史数据时，可以忽略库存对账记录。

```text
后台点击忽略
        ↓
提交备注
        ↓
库存对账记录标记为已忽略
        ↓
写入管理员操作日志
```

## 11. 当前闭环总结

当前对账闭环可以拆成五层。

### 11.1 发现闭环

```text
订单 + 支付单 + 退款单 + 库存记录 + 渠道单笔查询
        ↓
执行支付/退款/库存一致性核验
        ↓
发现长款、短款、金额差异、状态差异、库存差异
        ↓
生成对账记录
```

### 11.2 分类闭环

```text
PayReconciliationRecord.diffType
        ↓
AdminReconciliationPolicy
        ↓
差异分类
        ↓
处理建议
        ↓
可执行动作
```

### 11.3 处置闭环

```text
管理员选择业务动作
        ↓
后端校验动作合法性
        ↓
执行同步支付、补偿订单、恢复待支付、关闭释放库存等动作
        ↓
或进入待调单、待补款、待退款、人工审核状态
        ↓
更新对账记录
```

### 11.4 状态闭环

```text
PENDING
        ↓
RECHECKING / WAIT_CHANNEL_CONFIRM / WAIT_USER_REPAY / WAIT_REFUND / MANUAL_REVIEW
        ↓
DONE / IGNORED
```

### 11.5 防重复闭环

```text
同订单 + 同差异类型 + PENDING
        ↓
重跑对账
        ↓
刷新原记录，不重复新增
```

这使得后台重跑对账时不会产生大量重复待处理数据，同时已处理记录仍能保留历史。

## 13. 当前边界与后续扩展

当前尚未实现完整财务级渠道账单对账，主要边界包括：

1. 未支持渠道账单文件导入。
2. 未支持渠道明细全量入库。
3. 未支持平台流水与渠道流水的全量笔数对账。
4. 未支持手续费、结算净额、汇率、多币种。
5. 未支持财务凭证生成。
6. 未支持自动补生成订单。
7. 未支持基于渠道账单发现“平台完全未知流水”。

后续如果继续增强，可以按以下方向扩展：

- 增加支付宝/微信账单上传与解析。
- 增加渠道账单明细表。
- 增加平台流水与渠道流水按交易号、金额、时间窗口匹配。
- 增加渠道有、平台无的未知流水处理队列。
- 增加手续费、结算金额、退款抵扣等财务字段。
- 增加异步批量对账任务和批次报表。
- 增加财务确认、运营确认、系统自动修复的多角色审批流。

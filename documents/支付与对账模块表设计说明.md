# 支付与对账模块表设计说明

本文基于 `backend/sql/structrue.sql` 中的表结构，简要说明商城系统中支付、退款、支付回调与支付对账相关表的设计用意、字段含义以及表之间的业务联系。

## 一、模块整体设计目标

支付与对账模块主要用于承接订单支付、第三方支付渠道回调、售后退款、退款回调以及支付/退款一致性检查等业务。

整体上，这组表解决以下问题：

1. **支付单管理**：记录每笔订单发起支付后的支付单状态、金额、渠道、三方流水等信息。
2. **回调留痕**：保存支付渠道返回的原始通知，便于验签、幂等处理、问题排查和审计。
3. **退款链路管理**：订单售后产生退款申请后，在支付模块生成支付退款单，并记录渠道退款结果。
4. **轻量对账**：将本地支付/退款状态与渠道状态进行比对，发现状态或金额差异，并支持人工或系统修复闭环。
5. **订单售后联动**：支付模块并不是孤立存在的，它需要和订单表、订单退款表共同构成“下单-支付-退款-对账”的完整链路。

## 二、核心表概览

| 表名 | 中文含义 | 主要作用 |
| --- | --- | --- |
| `pay_order` | 支付单表 | 记录业务订单对应的支付单、支付状态、渠道流水和支付回调摘要 |
| `pay_callback_record` | 支付回调记录表 | 记录支付/退款渠道回调的原始报文、验签结果和处理结果 |
| `pay_refund_order` | 支付退款单表 | 记录向支付渠道发起退款的本地退款单、退款状态和渠道退款流水 |
| `pay_reconciliation_record` | 支付退款轻量对账记录表 | 记录支付或退款的本地数据与渠道数据是否一致，以及差异处理状态 |
| `oms_order` | 订单表 | 支付模块关联的业务订单主体 |
| `oms_order_refund` | 订单退款申请表 | 订单售后侧的退款申请主体 |
| `oms_order_refund_item` | 订单退款明细表 | 记录退款涉及哪些订单明细/SKU/数量/金额 |

其中 `pay_` 前缀表属于支付模块自身的数据，`oms_` 前缀表属于订单/售后模块，但退款、对账场景中会与支付模块强关联。

## 三、`pay_order` 支付单表

### 1. 表整体用途

`pay_order` 是支付模块的主表，用来记录一笔业务订单发起支付后的支付单信息。它和 `oms_order` 通过 `order_no` 关联。

一个订单在支付时会生成一个支付单，支付单记录本地支付状态、支付金额、支付渠道、第三方交易流水、回调报文等信息。后续支付成功、支付失败、支付异常核验、订单超时关闭等业务都会依赖该表判断支付状态。

### 2. 字段说明

| 字段 | 含义 | 业务说明 |
| --- | --- | --- |
| `id` | 支付单ID | 数据库主键，自增 |
| `pay_order_no` | 支付单号 | 支付模块内部生成的唯一支付单编号，是支付链路中的核心标识 |
| `order_no` | 业务订单号 | 对应 `oms_order.order_no`，用于将支付单和订单绑定 |
| `user_id` | 用户ID | 发起支付的用户，用于用户维度查询和权限校验 |
| `pay_amount_cent` | 支付金额分 | 本次需要支付的金额，单位为分，避免小数精度问题 |
| `pay_status` | 支付状态 | 本地支付状态，如待支付、支付中、支付成功、支付失败等 |
| `pay_channel` | 支付渠道 | 支付渠道类型，如 `MOCK`、支付宝等 |
| `transaction_no` | 三方流水号 | 支付渠道返回的交易流水号，支付成功后用于对账和退款 |
| `callback_payload` | 回调报文 | 最近一次或核心支付回调内容摘要，便于快速查看支付通知信息 |
| `notify_time` | 通知时间 | 支付渠道通知到达时间 |
| `paid_at` | 支付完成时间 | 本地确认支付成功的时间 |
| `idempotent_key` | 幂等键 | 防止重复创建支付单或重复支付请求 |
| `version` | 乐观锁版本号 | 并发更新支付状态时使用，避免状态覆盖 |
| `created_at` | 创建时间 | 支付单创建时间 |
| `updated_at` | 更新时间 | 支付单最后更新时间 |
| `deleted_at` | 删除时间 | 软删除标识，当前记录被逻辑删除时记录时间 |

### 3. 关键索引设计

| 索引 | 用途 |
| --- | --- |
| `uk_pay_order_pay_order_no` | 保证支付单号唯一 |
| `uk_pay_order_idempotent_key` | 保证同一幂等键只创建一笔支付单 |
| `idx_pay_order_status_id` | 支持按支付状态分页查询，例如后台查看待处理支付单 |
| `idx_pay_order_order_no` | 支持通过订单号查询支付单 |
| `idx_pay_order_pay_order_no` | 支持通过支付单号快速定位记录 |

## 四、`pay_callback_record` 支付回调记录表

### 1. 表整体用途

`pay_callback_record` 用于保存支付渠道的回调通知，包括支付回调和退款回调。它是支付系统中非常重要的审计与排查表。

支付渠道可能出现重复通知、延迟通知、验签失败、金额不一致、状态不一致等情况，因此不能只更新 `pay_order` 或 `pay_refund_order`，还需要把每次回调原始内容落库，方便后续追踪。

### 2. 字段说明

| 字段 | 含义 | 业务说明 |
| --- | --- | --- |
| `id` | 回调记录ID | 数据库主键，自增 |
| `channel` | 支付渠道 | 回调来自哪个渠道，如 `MOCK`、支付宝等 |
| `callback_type` | 回调类型 | 区分支付回调和退款回调，典型值为 `PAY`、`REFUND` |
| `pay_order_no` | 支付单号 | 支付回调时关联 `pay_order.pay_order_no`；退款回调时也可用于追溯原支付单 |
| `refund_no` | 退款单号 | 退款回调时关联 `pay_refund_order.refund_no` 或 `oms_order_refund.refund_no` |
| `order_no` | 订单号 | 关联业务订单 `oms_order.order_no` |
| `out_trade_no` | 商户订单号 | 传给支付渠道的商户侧交易号，通常可对应支付单号或业务订单号 |
| `transaction_no` | 第三方交易号 | 渠道侧交易流水号，用于和渠道账单核对 |
| `amount_cent` | 金额分 | 回调中携带的支付或退款金额 |
| `trade_status` | 渠道交易状态 | 渠道返回的交易状态，例如成功、关闭、退款成功等 |
| `signature` | 签名信息 | 渠道回调携带的签名内容，用于验签 |
| `verified` | 是否验签通过 | 标识当前回调是否通过签名校验 |
| `process_status` | 处理状态 | 当前回调处理结果，如初始化、处理成功、处理失败等 |
| `fail_reason` | 失败原因 | 验签失败、金额不匹配、状态流转异常等失败说明 |
| `raw_payload` | 原始回调报文 | 保存渠道通知的完整原始内容，方便审计和排查 |
| `callback_time` | 回调时间 | 渠道通知时间或系统接收回调时间 |
| `processed_at` | 处理时间 | 本地完成回调处理的时间 |
| `created_at` | 创建时间 | 回调记录入库时间 |
| `updated_at` | 更新时间 | 回调记录最后更新时间 |

### 3. 关键索引设计

| 索引 | 用途 |
| --- | --- |
| `idx_pay_callback_record_channel` | 按支付渠道筛选回调 |
| `idx_pay_callback_record_pay_order_no` | 通过支付单号查看相关回调记录 |
| `idx_pay_callback_record_order_no` | 通过订单号查看支付/退款通知链路 |
| `idx_pay_callback_record_out_trade_no` | 通过商户交易号定位回调 |
| `idx_pay_callback_record_refund_no` | 通过退款单号查看退款回调 |
| `idx_pay_callback_record_created_at` | 按时间查询回调记录，适合后台分页和排查 |

## 五、`pay_refund_order` 支付退款单表

### 1. 表整体用途

`pay_refund_order` 是支付模块中的退款执行表。订单售后模块产生退款申请后，支付模块会基于该申请创建支付退款单，并向支付渠道发起退款请求。

它和 `oms_order_refund` 都有 `refund_no`，但二者职责不同：

- `oms_order_refund` 偏订单售后视角，记录用户为什么退款、退哪些商品、售后状态如何。
- `pay_refund_order` 偏支付执行视角，记录这笔退款是否已经提交到支付渠道、渠道返回什么结果、退款流水是什么。

### 2. 字段说明

| 字段 | 含义 | 业务说明 |
| --- | --- | --- |
| `id` | 支付退款单ID | 数据库主键，自增 |
| `refund_no` | 本地退款单号 | 退款链路的核心唯一编号，通常与订单退款申请表的 `refund_no` 对应 |
| `order_no` | 业务订单号 | 对应 `oms_order.order_no`，标识退款属于哪笔订单 |
| `pay_order_no` | 原支付单号 | 对应 `pay_order.pay_order_no`，标识基于哪笔支付单退款 |
| `user_id` | 用户ID | 退款所属用户 |
| `pay_channel` | 支付渠道 | 原支付渠道，退款通常需要走原渠道 |
| `refund_amount_cent` | 退款金额分 | 本次退款金额，单位分 |
| `refund_status` | 退款状态 | 如 `REFUND_PENDING`、`REFUNDING`、`REFUND_SUCCESS`、`REFUND_FAILED` |
| `transaction_no` | 原三方交易号 | 原支付成功时的渠道交易流水，发起退款时通常需要使用 |
| `channel_refund_no` | 渠道退款流水号 | 支付渠道返回的退款流水，用于退款查询和对账 |
| `request_payload` | 退款请求摘要 | 本地向支付渠道发起退款时的请求参数摘要 |
| `response_payload` | 退款响应摘要 | 支付渠道对退款请求的响应内容摘要 |
| `fail_reason` | 失败原因 | 退款失败或异常时记录原因 |
| `success_at` | 退款成功时间 | 本地确认退款成功的时间 |
| `version` | 乐观锁版本号 | 并发更新退款状态时使用 |
| `created_at` | 创建时间 | 支付退款单创建时间 |
| `updated_at` | 更新时间 | 支付退款单最后更新时间 |

### 3. 关键索引设计

| 索引 | 用途 |
| --- | --- |
| `uk_pay_refund_order_refund_no` | 保证退款单号唯一 |
| `idx_pay_refund_order_order_no` | 通过订单号查询退款记录 |
| `idx_pay_refund_order_pay_order_no` | 通过支付单号查询退款记录 |
| `idx_pay_refund_order_status_id` | 支持按退款状态分页处理，例如查询退款中、退款失败记录 |

## 六、`pay_reconciliation_record` 支付退款轻量对账记录表

### 1. 表整体用途

`pay_reconciliation_record` 用于保存支付或退款的对账结果。它会把本地系统记录的状态/金额和支付渠道返回的状态/金额进行比较，判断是否一致。

该表的设计目标是形成一个轻量的对账闭环：

1. 本地系统根据 `pay_order` 或 `pay_refund_order` 找到待核验记录。
2. 调用支付渠道查询接口，获取渠道侧状态和金额。
3. 将本地状态、渠道状态、本地金额、渠道金额写入对账记录。
4. 如果一致，则标记 `consistent = 1`。
5. 如果不一致，则记录 `diff_type` 和 `repair_status`，后续由系统或管理员处理。

### 2. 字段说明

| 字段 | 含义 | 业务说明 |
| --- | --- | --- |
| `id` | 支付对账记录ID | 数据库主键，自增 |
| `biz_type` | 业务类型 | 区分对账对象是支付还是退款，典型值为 `PAY`、`REFUND` |
| `order_no` | 业务订单号 | 对应 `oms_order.order_no`，支付和退款都可以回到订单维度查看 |
| `pay_order_no` | 支付单号 | 支付对账时必填；退款对账时用于追溯原支付单 |
| `refund_no` | 退款单号 | 退款对账时使用，对应 `pay_refund_order.refund_no` / `oms_order_refund.refund_no` |
| `local_status` | 本地状态 | 本地 `pay_order.pay_status` 或 `pay_refund_order.refund_status` |
| `channel_status` | 渠道状态 | 从支付渠道查询得到的交易状态或退款状态 |
| `local_amount_cent` | 本地金额分 | 本地记录的支付金额或退款金额 |
| `channel_amount_cent` | 渠道金额分 | 渠道返回的支付金额或退款金额 |
| `consistent` | 是否一致 | 本地状态/金额与渠道状态/金额是否一致 |
| `diff_type` | 差异类型 | 记录差异原因，如状态不一致、金额不一致、本地缺失、渠道缺失等 |
| `repair_status` | 修复状态 | 差异处理进度，如 `NONE`、`PENDING`、`DONE`、`IGNORED` |
| `remark` | 备注 | 对账说明、差异说明或人工处理备注 |
| `repaired_at` | 修复时间 | 差异被修复或忽略的时间 |
| `created_at` | 创建时间 | 对账记录创建时间 |
| `updated_at` | 更新时间 | 对账记录最后更新时间 |

### 3. 关键索引设计

| 索引 | 用途 |
| --- | --- |
| `idx_pay_reconciliation_biz_order` | 按业务类型和订单号查看对账记录 |
| `idx_pay_reconciliation_status_time` | 查询不一致记录或按时间分页展示对账结果 |
| `idx_pay_reconciliation_refund_no` | 根据退款单号定位退款对账记录 |
| `idx_pay_reconciliation_pay_order_no` | 根据支付单号定位支付对账记录 |

## 七、订单售后相关表

支付与对账模块会和订单售后表联动，尤其是在退款场景中。

### 1. `oms_order_refund` 订单退款申请表

#### 表整体用途

`oms_order_refund` 是订单模块中的退款申请主表。用户发起退款时，订单模块先记录退款申请，管理员审核或系统确认后，再驱动支付模块创建 `pay_refund_order` 并发起渠道退款。

#### 字段说明

| 字段 | 含义 | 业务说明 |
| --- | --- | --- |
| `id` | 退款申请ID | 数据库主键，自增 |
| `order_id` | 订单ID | 对应 `oms_order.id` |
| `order_no` | 订单号 | 对应 `oms_order.order_no` |
| `user_id` | 用户ID | 发起退款申请的用户 |
| `refund_no` | 退款单号 | 退款业务唯一标识，会和 `pay_refund_order.refund_no` 关联 |
| `refund_amount_cent` | 退款金额 | 订单售后侧确认的退款金额，单位分 |
| `channel_refund_no` | 渠道退款流水号 | 支付渠道返回的退款流水，方便订单售后侧展示和追踪 |
| `refund_status` | 退款状态 | 订单售后视角下的退款状态，如申请中、退款中、退款成功、退款失败等 |
| `refund_reason` | 退款原因 | 用户或管理员填写的退款原因 |
| `fail_reason` | 失败原因 | 退款失败原因，可来自支付模块或渠道 |
| `created_at` | 创建时间 | 退款申请创建时间 |
| `updated_at` | 更新时间 | 退款申请最后更新时间 |

### 2. `oms_order_refund_item` 订单退款明细表

#### 表整体用途

`oms_order_refund_item` 用于记录一次退款申请中具体退了哪些订单明细、哪些 SKU、退多少数量、退多少钱。

它解决的是“部分退款”的问题：一笔订单可能有多个商品，用户可以只退其中一个 SKU 或一部分数量。

#### 字段说明

| 字段 | 含义 | 业务说明 |
| --- | --- | --- |
| `id` | 主键 | 数据库主键，自增 |
| `refund_id` | 退款记录ID | 对应 `oms_order_refund.id` |
| `refund_no` | 退款单号 | 对应 `oms_order_refund.refund_no` |
| `order_item_id` | 订单明细ID | 对应 `oms_order_item.id` |
| `sku_id` | SKU ID | 被退款的商品 SKU |
| `quantity` | 退款数量 | 本次申请退款的商品数量 |
| `refund_amount_cent` | 退款金额 | 当前明细对应的退款金额，单位分 |

## 八、表之间的关系

### 1. 订单与支付单

- `oms_order.order_no` 对应 `pay_order.order_no`。
- 订单创建后，用户发起支付，支付模块创建 `pay_order`。
- 支付成功后，`pay_order.pay_status` 更新为成功，同时订单侧 `oms_order.order_status` 也会流转为已支付。

业务体现：

- 用户在 H5 订单详情页查看支付状态。
- 后台订单管理查看订单是否已支付。
- 支付异常核验时，通过订单号找到支付单和渠道交易。

### 2. 支付单与支付回调

- `pay_order.pay_order_no` 对应 `pay_callback_record.pay_order_no`。
- 支付渠道通知支付结果时，会先落一条 `pay_callback_record`，验签成功后再更新 `pay_order`。
- 同一支付单可能收到多次回调，因此回调表和支付单是“一对多”的关系。

业务体现：

- 处理支付成功通知。
- 处理重复通知和幂等。
- 支付失败或验签失败时排查原始报文。

### 3. 订单退款申请与支付退款单

- `oms_order_refund.refund_no` 对应 `pay_refund_order.refund_no`。
- 订单售后侧负责产生退款申请，支付侧负责真正请求支付渠道退款。
- 一个退款申请通常对应一个支付退款单。

业务体现：

- 用户申请退款后，订单售后状态进入退款中。
- 支付模块调用渠道退款接口。
- 渠道退款成功后，支付模块回写订单退款状态和渠道退款流水。

### 4. 支付退款单与退款回调

- `pay_refund_order.refund_no` 对应 `pay_callback_record.refund_no`。
- 支付渠道异步通知退款结果时，回调表记录原始通知，再更新 `pay_refund_order.refund_status`。
- 同一退款单也可能收到多次退款通知，因此也是“一对多”的关系。

业务体现：

- 退款成功后订单售后状态变为退款成功。
- 退款失败时记录失败原因，并允许后台查看或重新处理。

### 5. 支付/退款与对账记录

- 支付对账时：`pay_reconciliation_record.biz_type = 'PAY'`，通过 `pay_order_no` 关联 `pay_order`。
- 退款对账时：`pay_reconciliation_record.biz_type = 'REFUND'`，通过 `refund_no` 关联 `pay_refund_order` 和 `oms_order_refund`。
- `order_no` 贯穿支付和退款对账，方便从订单维度查看所有资金链路。

业务体现：

- 本地显示支付成功，但渠道查询未支付：生成一条不一致对账记录。
- 本地退款失败，但渠道查询已退款：生成差异记录，后续修复本地状态。
- 本地金额和渠道金额不同：记录金额差异，进入人工核查。

## 九、典型业务链路

### 1. 下单支付链路

1. 用户创建订单，生成 `oms_order`。
2. 用户发起支付，生成 `pay_order`。
3. 支付渠道返回支付参数或模拟支付结果。
4. 渠道异步通知支付结果，写入 `pay_callback_record`。
5. 验签和金额校验通过后，更新 `pay_order.pay_status`。
6. 支付成功后，订单状态更新为已支付。
7. 后续对账任务可生成 `pay_reconciliation_record`，确认本地和渠道一致。

### 2. 售后退款链路

1. 用户对订单发起退款申请，生成 `oms_order_refund`。
2. 如果是部分退款，同时生成多条 `oms_order_refund_item`。
3. 审核或业务规则通过后，支付模块生成 `pay_refund_order`。
4. 支付模块调用渠道退款接口，并记录请求/响应摘要。
5. 渠道退款成功或失败后，通过退款回调写入 `pay_callback_record`。
6. 本地更新 `pay_refund_order.refund_status`。
7. 同步更新 `oms_order_refund.refund_status`、`channel_refund_no` 和 `fail_reason`。
8. 对账任务可对退款结果生成 `pay_reconciliation_record`。

### 3. 支付退款对账链路

1. 系统扫描支付成功、退款成功、退款失败或异常状态的数据。
2. 根据 `pay_order_no` 或 `refund_no` 调用支付渠道查询接口。
3. 比对本地状态、渠道状态、本地金额、渠道金额。
4. 一致则写入 `consistent = 1`，`diff_type = 'NONE'`。
5. 不一致则写入差异类型，并将 `repair_status` 标记为待处理或未处理。
6. 管理员在后台查看差异记录，选择修复、忽略或进一步核查。

## 十、设计上的关键用意

1. **支付单和订单分离**
   - 订单关心商品、收货、订单状态。
   - 支付单关心支付金额、支付渠道、交易流水和支付状态。
   - 分离后可以让订单域和支付域职责更清晰。

2. **回调记录单独建表**
   - 第三方回调具有重复、延迟、失败、伪造等风险。
   - 单独保存原始报文和验签结果，方便审计、排障和幂等处理。

3. **订单退款和支付退款分离**
   - 订单退款表示“用户要退什么、退多少、原因是什么”。
   - 支付退款表示“是否已经向支付渠道退款、渠道处理结果是什么”。
   - 这样可以支持售后审核、部分退款、退款失败重试和渠道对账。

4. **对账表统一支付与退款差异**
   - 通过 `biz_type` 区分支付和退款。
   - 通过状态、金额和差异类型记录本地与渠道不一致的原因。
   - 通过 `repair_status` 支持后续人工或自动修复闭环。

5. **金额统一使用分**
   - 所有金额字段使用 `xxx_amount_cent`，避免浮点数精度问题。
   - 支付、退款、对账金额口径保持一致。

6. **关键业务编号贯穿全链路**
   - `order_no` 串联订单、支付、退款和对账。
   - `pay_order_no` 串联支付单、支付回调和支付对账。
   - `refund_no` 串联订单退款、支付退款、退款回调和退款对账。

## 十二、总结

当前支付与对账模块的表设计核心是围绕“资金链路可追踪、状态流转可审计、异常差异可修复”展开。

- `pay_order` 负责记录支付主状态。
- `pay_callback_record` 负责记录渠道通知原始证据。
- `pay_refund_order` 负责记录支付渠道退款执行情况。
- `pay_reconciliation_record` 负责记录支付/退款的本地与渠道一致性结果。
- `oms_order_refund` 和 `oms_order_refund_item` 则从订单售后角度描述退款申请与退款明细。

这些表共同支撑了商城系统中的下单支付、支付回调、退款申请、渠道退款、退款回调、支付异常核验和资金对账闭环。

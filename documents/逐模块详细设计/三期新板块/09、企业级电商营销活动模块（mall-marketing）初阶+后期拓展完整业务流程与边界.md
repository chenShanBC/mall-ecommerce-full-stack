# 企业级电商营销活动模块（mall\-marketing）初阶+后期拓展完整业务流程与边界

本文档整合营销活动模块 MVP最小可行版本 与 后期拓展完整版业务，明确各阶段业务流程、职责边界，贴合mallFei微服务架构，兼顾当前落地与后期迭代，可直接用于项目开发、文档归档、技术评审。

# 一、营销活动模块定位（结合微服务拆分·后期完整版）

营销活动模块（mall\-marketing）是电商平台的**流量转化、订单拉升、用户活跃**核心引擎，独立承载全平台促销规则、优惠计算、活动生命周期与权益发放，属于**高并发、强规则、重计算**的核心业务域。

与订单、支付、用户、商品、库存深度协同，但**规则引擎、优惠计算、活动状态、优惠券生命周期**完全自治，不侵入其他微服务业务逻辑。

结合微服务拆分（后期完整版），完整营销模块包含六大核心域：

1. 优惠券域：券的创建、领取、核销、过期、禁用、查询、统计

2. 促销活动域：满减、满折、限时折扣、单品直降、阶梯满减

3. 限时活动域：秒杀、预售、拼团（高并发专属域）

4. 积分营销域：积分获取、积分抵扣、积分兑换、积分过期

5. 优惠计算域：订单级最优优惠组合、叠加规则、互斥控制、价格试算

6. 运营管控域：活动上下线、审核、数据看板、风险控制

# 二、整体架构说明

本营销模块基于微服务拆分设计，核心遵循**优惠计算独立、活动规则自治、数据不跨域**原则：

- mall\-auth：仅做鉴权，不参与优惠规则

- mall\-user：提供用户ID、会员等级、积分余额

- mall\-product：提供商品ID、分类、品牌、价格

- mall\-order：调用营销试算、确认核销、优惠回滚

- mall\-payment：同步优惠金额、实付金额、抵扣明细

- mall\-marketing：**唯一负责优惠计算、券状态、活动规则、叠加互斥**

核心边界：

- 只做“优惠规则与计算”，不修改订单、不扣库存、不处理支付

- 所有优惠结果以**试算接口**输出，订单确认后才做**核销**

- 高并发活动（秒杀）独立限流、队列、防超抢

# 三、初阶阶段业务流程（当前落地核心）

核心目标：支撑商城基础促销，完成下单优惠闭环，无冗余高并发活动。

## 3\.1 核心业务流程（8大必备）

1. **优惠券创建与管理（运营后台）**：运营配置券类型（满减券/无门槛券）、面额、门槛、有效期、发放总量、适用商品/品类、领取限制（每人限领X张）；创建后待上架，支持手动上下架、编辑（未发放可修改，已发放仅可修改上下架状态）；校验规则：面额不可大于门槛（满减券）、有效期开始时间不可晚于结束时间、发放总量不可为负。

2. **用户领取优惠券（C端）**：用户登录后经mall\-auth鉴权通过，进入优惠券领取页面；系统校验活动状态（已上架、未过期）、用户领取限制（未达每人限领上限）、券库存（未领完）；校验通过后，为用户生成专属优惠券记录，更新券发放量，返回领券成功结果；若校验失败，返回具体失败原因（如“已达领取上限”“活动已过期”）。

3. **我的优惠券（C端）**：用户登录鉴权后，展示个人优惠券列表，按“未使用、已使用、已过期”三个tab分类展示；每个券展示核心信息（券类型、面额、使用门槛、有效期、适用范围），提供“去使用”按钮（跳转至适用商品列表）；支持优惠券搜索（按券类型、面额）、筛选（按适用商品）。

4. **满减活动配置（运营后台）**：运营创建满减活动，设置活动名称、有效期、适用范围（全平台/指定商品/指定品类）、满减规则（可设置多阶梯，如“满100减10、满200减30”）；支持活动预览、手动上下架；校验规则：阶梯满减门槛需递增，减额不可大于对应门槛，适用商品/品类需存在。

5. **优惠试算（核心接口，给订单调用）**：订单模块调用营销模块试算接口，入参为用户ID、商品列表（sku、单价、数量）、收货区域、用户选中的优惠券ID；营销模块根据入参，匹配当前生效的优惠活动（满减、优惠券），计算最优优惠组合、优惠金额、实付金额，判断优惠券/满减是否可用（如商品是否在适用范围、是否满足门槛）；出参返回优惠明细（优惠类型、优惠金额、适用规则）、不可用优惠原因、最终实付金额，供订单模块展示与确认。

6. **优惠核销（订单支付成功回调）**：订单支付成功后，订单模块回调营销模块核销接口，入参为订单ID、用户ID、优惠明细（券ID、满减活动ID）；营销模块校验订单状态（已支付）、优惠状态（未使用），校验通过后，更新优惠券状态为“已使用”、满减活动使用次数，写入优惠核销流水（关联订单ID、用户ID、核销时间）；若校验失败（如券已过期、已核销），返回核销失败，触发订单异常处理流程。

7. **优惠券过期定时任务**：依托定时任务模块（mall\-job），每日固定时间扫描系统内未使用的优惠券，筛选出已过有效期的券，自动更新其状态为“已过期”；同步向用户推送券过期提醒（通过消息通知模块），并更新优惠券统计数据。

8. **运营后台活动查询与统计**：运营人员经mall\-auth鉴权\+管理员角色校验通过后，可查看所有营销活动列表（支持按活动类型、状态、时间筛选）；查看单活动详情（配置信息、发放量、已使用量、核销率、优惠总金额）；支持导出活动统计数据（Excel格式），仅提供查询、导出能力，不修改业务数据。

# 四、后期拓展完整版业务流程（企业级升级）

在MVP基础上，补齐高并发活动、积分营销、复杂规则、风控、叠加互斥、智能推荐，形成营销中台，支撑大规模电商业务运营与高并发场景。

## 4\.1 新增拓展业务流程（10大核心，覆盖六大核心域）

1. **秒杀活动（限时活动域）**：运营配置秒杀活动，设置秒杀商品（指定sku）、秒杀价格、秒杀库存、活动开始/结束时间、每人限购数量、预热时间；预热期间展示秒杀商品、倒计时、秒杀价格，吸引用户关注；活动开始后，启用限流机制（防止高并发压垮系统）、防机器抢购（校验用户设备、IP，拦截异常请求）；用户抢购成功后，扣减秒杀活动库存，生成抢购记录，同步通知订单模块创建订单，超时未支付则释放秒杀库存；支持秒杀活动暂停、重启，异常情况可手动结束活动。

2. **预售活动（限时活动域）**：采用“定金\+尾款”模式，运营配置预售商品、定金金额、尾款金额、定金期（支付定金时间段）、尾款期（支付尾款时间段）、定金规则（可退/不可退）、预售库存；用户在定金期支付定金，系统锁定预售库存，生成预售订单；定金可退场景下，用户未支付尾款可申请定金退款，退款后释放库存；尾款期内，用户支付尾款，完成订单创建，触发发货流程；尾款逾期未支付，系统自动取消预售订单，定金不退（不可退场景）或原路退回（可退场景），释放预售库存。

3. **拼团活动（限时活动域）**：运营配置拼团活动，设置拼团商品、拼团价格、拼团人数（N人成团）、拼团有效期、每人限购数量；用户可发起开团，也可参与他人已开的团；开团后，系统生成拼团订单，记录拼团状态（拼团中、成团、失败）；拼团有效期内，达到规定人数则成团，触发订单支付与发货流程；未达到人数则拼团失败，系统自动原路退还用户支付金额，释放商品库存；支持拼团分享（微信、支付宝等渠道），吸引更多用户参团。

4. **积分营销体系（积分营销域）**：整合积分获取与消耗全流程，运营配置积分规则（下单支付按金额比例获取积分、每日签到获取积分、完善个人资料获取积分、邀请好友获取积分等）；用户完成对应行为后，系统自动发放积分，记录积分变动明细；积分可用于订单抵扣（设置抵扣比例、最低抵扣金额）、积分兑换商品/优惠券（配置兑换所需积分、兑换库存）；设置积分有效期，定时清理过期积分，同步向用户推送积分过期提醒；支持积分查询（C端、运营后台）、积分明细导出（运营后台）。

5. **优惠叠加与互斥规则引擎（优惠计算域）**：完善优惠叠加与互斥逻辑，制定明确规则：优惠券可与满减活动叠加（无门槛券可叠加所有满减，满减券仅可叠加高于自身门槛的满减）；秒杀、预售、拼团活动不可使用任何优惠券、满减活动（独立优惠）；单品直降与满减不可叠加（取最优优惠）；同一订单仅可使用一张优惠券（用户自行选择或系统推荐最优）；系统自动计算订单最优优惠组合，无需用户手动选择，提升用户体验。

6. **活动准入规则（运营管控域）**：运营可配置活动准入条件，按用户属性精准管控活动可见与可用范围；支持按会员等级（如仅钻石会员可参与）、用户标签（如新用户、复购用户）、用户地域（指定省份/城市可参与）、渠道来源（如仅小程序用户可参与）设置准入规则；用户登录后，系统校验其是否满足准入条件，不满足则隐藏活动或提示不可参与。

7. **活动风险控制（运营管控域）**：新增全方位风控能力，防范活动异常与作弊行为；设置防刷机制（校验用户IP、设备、账号，拦截高频请求、异常账号）；建立活动黑名单（禁止作弊用户参与所有营销活动）；监控异常领券、异常核销行为（如短时间内大量领券、核销），触发人工审核；对秒杀、拼团等高并发活动，设置库存超卖保护（采用分布式锁，防止超卖）、活动流量限流（限制单IP、单账号请求频率）。

8. **智能推荐优惠（优惠计算域\+运营管控域）**：基于用户行为（浏览记录、下单记录、收藏记录）、用户等级、积分余额，为用户智能推荐优惠；C端首页展示“猜你喜欢的优惠券”“为你推荐的活动”，商品详情页展示“凑单提醒”（如“再买XX元可享受满减”），购物车页面展示“最优优惠推荐”；提升优惠使用率与订单转化率。

9. **优惠回滚（优惠计算域）**：针对订单取消、退款场景，实现优惠回滚逻辑；订单未支付取消：回滚优惠券状态（从“已锁定”变为“未使用”）、回滚活动库存（秒杀、拼团、预售库存）；订单已支付退款：支持配置优惠券退回规则（可退回/不可退回），可退回则将优惠券状态改为“未使用”（有效期不变），不可退回则保持“已使用”状态；回滚后更新优惠核销流水、活动统计数据，同步通知相关模块（订单、用户）。

10. **营销数据大屏（运营管控域）**：搭建运营数据大屏，实时展示营销活动核心数据；包含活动GMV、优惠券领取率、核销率、优惠总投入、投入产出比（ROI）、拉新量、复购率、各活动类型转化对比等；支持按时间（今日、昨日、近7天、近30天）、活动类型、商品品类筛选数据；支持数据导出、异常数据预警（如优惠核销异常、活动库存不足），助力运营人员实时监控活动效果、优化活动策略。

# 五、完整业务边界（通用）

## 5\.1 核心职责

负责六大核心域（优惠券域、促销活动域、限时活动域、积分营销域、优惠计算域、运营管控域）的规则制定、业务落地、数据管理；为订单、商品、C端首页提供优惠试算、核销、回滚等能力；独立完成优惠计算、活动生命周期管控、权益发放与回收，不侵入任何外部微服务业务逻辑。

## 5\.2 对内边界

- → mall\-order：提供优惠试算、优惠核销、优惠回滚接口；接收订单模块的回调（支付成功、订单取消、退款），同步处理优惠状态；不修改订单任何数据，仅返回优惠相关结果。

- → mall\-user：读取用户ID、会员等级、积分余额、用户标签等基础信息；不修改用户任何数据，仅用于活动准入校验、积分计算。

- → mall\-product：读取商品ID、sku信息、商品分类、品牌、基础售价等数据；不上下架商品、不修改商品价格，仅用于优惠适用范围校验、优惠金额计算。

- → mall\-inventory：秒杀、预售活动仅占用活动专属库存，不扣减商品真实库存；库存扣减由库存模块独立完成，营销模块仅同步活动库存变动。

- → mall\-message：推送优惠相关通知（券领取、券过期、活动开始、拼团成团/失败等）；仅发送通知请求，不处理通知发送逻辑。

- ← 所有外部模块仅可调用营销模块的查询、试算、核销、回滚接口，营销模块不主动调用外部模块的写操作，仅接收回调通知。

## 5\.3 对外边界

- 对外开放接口：MVP阶段开放优惠券领取、我的优惠券、优惠试算、核销、回滚、活动列表查询接口；后期拓展后，新增秒杀、预售、拼团、积分兑换、优惠推荐等接口。

- 接口访问范围：仅对C端H5、小程序、PC运营后台开放；第三方平台仅开放拼团分享相关接口（不涉及核心规则与数据）。

- 数据安全要求：所有接口返回数据需脱敏（不暴露运营配置详情、优惠规则源码、活动库存具体数值）；优惠计算结果不可篡改，每笔核销、回滚操作均有流水记录，可追溯。

- 鉴权要求：除公开活动列表查询接口外，其余所有接口均需通过mall\-auth模块的Sa\-Token鉴权与角色校验（如运营接口需管理员角色，C端接口需用户登录鉴权）。

## 5\.4 禁止范围

- 不创建订单、不修改订单状态、不处理订单相关业务（仅提供优惠支撑）；

- 不扣减商品真实库存、不管理商品库存（仅管理活动专属库存）；

- 不处理支付业务、不存储支付记录、不接触支付密钥（仅同步优惠金额与实付金额）；

- 不做用户登录鉴权、Token分发、风控规则制定（统一由mall\-auth认证中心承担）；

- 不跨模块修改其他领域数据（如用户积分、商品价格、订单信息），不接收其他领域的业务指令（仅提供数据查询与优惠支撑）；

- 不允许同一优惠重复核销、不允许超发优惠券、不允许突破叠加互斥规则计算优惠；

- 不存储用户密码、完整身份证号、未脱敏手机号等敏感用户数据（仅存储用户ID用于关联）。

# 企业级电商消息通知模块（mall\-message）初阶+后期拓展完整业务流程与边界

本文档整合消息通知模块 MVP最小可行版本 与 后期拓展完整版业务，明确各阶段业务流程、职责边界，贴合mallFei微服务架构，兼顾当前落地与后期迭代，可直接用于项目开发、文档归档、技术评审。

# 一、消息通知模块定位（结合微服务拆分·后期完整版）

消息通知模块（mall\-message）是电商平台的**信息触达、用户留存、业务协同**核心模块，独立承载全平台消息生成、分发、推送、存储、查询、异常处理全流程，属于**高可用、高时效、强协同、多场景**的核心业务域。

与用户、订单、物流、售后、营销、搜索、库存等所有核心微服务深度协同，但**消息模板管理、消息分发调度、推送渠道对接、消息存储查询、推送异常处理**完全自治，不侵入其他微服务业务逻辑，是连接平台与用户、各业务模块间信息同步的关键枢纽，遵循微服务高内聚、低耦合的核心拆分原则，实现消息通知业务的独立部署、扩展与优化，解决全平台信息高效触达、业务协同同步的核心需求，同时通过消息池模式实现消息的统一管理与调度，降低后期维护成本。

结合微服务拆分（后期完整版），完整消息通知模块包含五大核心域，贴合领域驱动设计（DDD）的限界上下文划分，确保各域职责单一、数据独立，同时适配电商消息通知的全场景需求：

1. 消息模板域：消息模板创建、编辑、启用/禁用、分类管理，支持短信、站内信、APP推送、邮箱等多渠道模板适配，统一消息格式规范，同时为不同业务场景配置专属唯一标识，便于开发调用与管理。

2. 消息生成域：接收各业务模块消息请求、消息内容组装、消息优先级判定，基于消息池模式收集所有待发送消息，实现消息生产与发送的解耦，支撑多业务场景消息的快速生成与统一归集。

3. 消息分发域：推送渠道对接（短信平台、APP推送平台、邮箱服务）、消息分发调度、推送时机控制，根据消息类型与接收人偏好选择最优推送渠道，通过异步调度实现消息的高效分发，应对高并发消息推送场景。

4. 消息管理域：消息存储、已读/未读状态管理、消息查询、消息删除、消息归档，留存所有消息记录用于追溯，同时支持用户与运营人员的消息操作与管理，满足合规与业务追溯需求。

5. 异常处理域：推送失败重试、推送异常监控、异常日志记录、异常告警，针对服务器异常、推送渠道故障等场景建立应急处理机制，确保消息触达的可靠性，同时支撑异常问题的快速排查与解决。

# 二、整体架构说明

本消息通知模块基于微服务拆分设计，核心遵循**模板统一、分发高效、数据不跨域、触达可靠**原则，坚守微服务数据私有、接口交互的核心准则，适配高并发消息推送场景，确保消息推送时效与触达率，同时与其他服务解耦，采用消息池模型实现消息的统一管理与调度，提升系统可维护性：

- mall\-auth：仅做鉴权，不参与消息模板配置、消息分发与推送逻辑，仅提供用户与运营人员的身份校验支撑，确保消息操作的安全性。

- mall\-user：提供用户ID、用户联系方式（手机号、邮箱）、用户偏好（推送渠道、消息类型开关）、用户状态等信息，用于消息精准推送、接收人匹配与用户消息偏好控制，同时支撑消息的个性化触达。

- mall\-order：发起订单相关消息请求（如订单创建、支付成功、订单取消、订单发货提醒），接收消息推送状态回调，支撑订单全流程消息触达，确保用户及时了解订单状态变更。

- mall\-logistics：发起物流相关消息请求（如物流发货、物流轨迹更新、物流异常、退货签收提醒），接收消息推送状态回调，实现物流全流程消息同步，提升用户物流体验。

- mall\-aftersale：发起售后相关消息请求（如售后审核通过、退货退款到账、售后异常提醒），接收消息推送状态回调，支撑售后流程的消息协同，提升售后处理透明度。

- mall\-marketing：发起营销相关消息请求（如优惠券到账、活动提醒、限时秒杀通知），接收消息推送状态回调，助力营销活动曝光与转化，提升活动触达效率。

- mall\-search：发起搜索相关消息请求（如搜索历史同步提醒、个性化推荐消息），接收消息推送状态回调，支撑搜索场景的用户触达与体验优化。

- mall\-inventory：发起库存相关消息请求（如商品补货提醒、库存不足预警），接收消息推送状态回调，支撑库存管理的业务协同，及时同步库存异常信息。

- mall\-message：**唯一负责消息模板管理、消息生成、消息分发、消息存储、推送异常处理**，独享自身业务数据（模板数据、消息记录、推送日志、异常记录），不允许其他服务直接访问数据库，仅通过接口提供消息推送与查询服务，基于消息池模式实现消息的统一管理与调度，支撑全平台消息通知场景，同时可适配Serverless架构，降低运维成本、提升扩展性。

核心边界：

- 只做“消息通知相关”，不修改其他业务模块数据、不参与业务逻辑处理，仅接收各模块消息请求、完成消息推送与状态回调，专注于信息触达核心能力。

- 所有消息操作（生成、推送、重试、删除、归档）均有日志记录、可追溯，消息推送时效满足业务需求（MVP阶段实时推送≤5s，拓展阶段实时推送≤2s），适配高并发推送场景（如大促活动批量消息推送），同时通过消息队列实现削峰填谷，应对峰值流量。

- 消息模板统一管理，支持多渠道适配，确保同一消息在不同推送渠道（短信、站内信等）的格式统一、内容一致，同时支持消息紧急程度划分，优先推送紧急消息（如服务器宕机、订单异常）。

- 支持消息推送状态实时回调，推送异常可自动重试，确保消息触达可靠性；同时支持用户自定义消息偏好，提升用户体验，减少无效消息干扰。

# 三、初阶阶段业务流程（当前落地核心）

核心目标：支撑商城基础消息通知场景，完成核心业务消息的生成、推送、查询，覆盖用户端与运营端基础消息需求，实现消息与各核心业务模块的基础协同，确保消息触达及时、可靠，采用简单高效的消息触发方式，降低初期开发成本，同时完成消息池的基础搭建。

## 3\.1 核心业务流程（7大必备）

1. **消息模板基础配置（运营后台）**：运营人员经mall\-auth鉴权\+管理员角色校验通过后，配置核心业务场景的消息模板（如订单支付成功、物流发货、售后审核通过），支持短信、站内信两种基础推送渠道；模板包含固定内容、可变参数（如订单号、商品名称、金额），配置模板状态（启用/禁用）、消息类型（用户通知、运营通知），系统校验模板格式合法性（如短信字数限制），同时为每个模板配置唯一标识，便于各业务模块调用，完成消息模板的基础管理。

2. **消息请求接收与生成（后台自动）**：各业务模块（如mall\-order、mall\-logistics）触发消息场景时，调用mall\-message的消息推送接口，传入消息类型、接收人ID、可变参数（如订单号）；系统根据消息类型匹配对应的消息模板，组装完整消息内容，判定消息基础优先级（默认统一优先级），将消息存入基础消息池，生成消息记录（关联业务模块、消息类型、接收人、消息内容），完成消息的生产与归集。

3. **基础消息分发与推送（后台自动）**：系统从消息池读取待发送消息，根据接收人ID从mall\-user获取用户联系方式（手机号、站内信接收权限），按模板配置的推送渠道推送消息；短信通过对接第三方短信平台发送，站内信直接推送至用户个人中心消息列表；推送完成后，更新消息推送状态（已推送、未推送），同时记录推送时间，实现消息的基础分发与触达。

4. **消息推送异常基础处理（后台自动\+手动）**：当消息推送失败（如手机号无效、短信平台异常）时，系统自动记录异常日志，标记消息状态为“推送失败”；针对短信推送失败，自动重试1次（间隔1分钟），重试失败后不再重试；运营人员可在后台查看推送异常记录，手动触发重新推送，确保核心消息的有效触达，同时记录异常原因，为后续优化提供支撑。

5. **用户端消息查询与操作（C端）**：用户登录经mall\-auth鉴权通过后，可在个人中心查看所有收到的消息（按时间倒序展示），区分已读/未读消息；支持消息标记为已读、删除单个/全部消息，支持按消息类型（订单、物流、售后）筛选消息，满足用户消息管理需求，提升用户体验。

6. **运营端消息管理与统计（运营后台）**：运营人员可查看所有消息记录（按消息类型、推送状态、时间筛选），统计核心数据（消息发送量、推送成功率、异常率）；支持查看消息推送详情（接收人、推送时间、推送渠道、推送状态），导出消息统计数据（Excel格式），为运营优化提供基础数据支撑，同时可手动发送运营通知类消息（如平台公告）。

7. **消息推送状态回调（后台自动）**：消息推送完成后（无论成功/失败），系统自动回调发起消息请求的业务模块，同步消息推送状态（已推送成功、推送失败），确保各业务模块及时掌握消息触达情况，支撑业务流程的闭环协同，同时记录回调日志，便于问题追溯。

# 四、后期拓展完整版业务流程（企业级升级）

在MVP基础上，补齐多渠道适配、精细化推送、智能调度、异常精细化处理、消息数据分析，形成企业级消息中台，遵循微服务独立扩展原则，提升消息触达率、推送效率与用户体验，适配高并发、多场景、精细化运营需求，完善消息池模型，实现消息全生命周期的精细化管理，同时融入智能优化与风控机制，降低运维成本。

## 4\.1 新增拓展业务流程（10大核心，覆盖五大核心域）

1. **消息模板精细化管理（消息模板域）**：新增多推送渠道适配（APP推送、邮箱、企业微信、短信、站内信），同一消息模板可配置不同渠道的展示格式（如APP推送支持图文、短信仅支持文字）；新增消息模板分类（用户通知、运营通知、系统预警、业务协同），支持模板版本管理（修改后保留历史版本，可回滚）；新增模板变量校验、敏感词过滤，避免违规内容推送；支持按业务模块、消息场景批量管理模板，同时优化模板唯一标识规则，适配多环境部署需求，提升模板管理效率与规范性。

2. **消息生成精细化（消息生成域）**：新增消息优先级划分（紧急、普通、低优先级），如服务器宕机、订单支付异常为紧急消息，营销通知为普通消息，系统报表推送为低优先级；支持消息延迟推送（如定时发送活动提醒、凌晨不推送非紧急消息），可配置推送时间窗口，避免打扰用户；新增消息内容个性化组装（结合用户昵称、会员等级、消费偏好），提升消息相关性；优化消息池机制，采用Kafka/RabbitMQ作为消息池存储介质，提升消息处理效率，支撑千万级消息的高效归集与调度。

3. **多渠道消息分发与智能调度（消息分发域）**：对接更多推送渠道（如微信公众号、小程序推送），支持用户自定义推送渠道偏好（如仅接收APP推送、不接收短信）；引入智能调度算法，根据消息类型、用户偏好、推送渠道触达率，自动选择最优推送渠道，提升触达率；支持批量消息推送（如大促活动全员通知），采用分片推送策略，避免高并发推送压垮服务；新增消息推送限流控制，适配高并发场景，同时引入消息队列实现削峰填谷，应对下单峰值等流量冲击。

4. **消息管理精细化（消息管理域）**：新增消息归档功能，对超过一定时间（如3个月）的消息自动归档，支持归档消息查询与恢复；支持用户端消息置顶、星标收藏，方便用户快速查看重要消息；新增运营端消息批量操作（批量推送、批量删除、批量重试）；支持消息全文检索，可通过关键词、接收人、消息类型快速检索消息记录；新增消息已读状态同步（多端同步已读/未读状态），提升跨端用户体验，同时完善消息存储策略，兼顾性能与可追溯性。

5. **推送异常精细化处理（异常处理域）**：新增多维度异常分类（渠道异常、接收人异常、内容异常、系统异常），针对不同异常类型制定差异化处理策略（如渠道异常自动切换备用渠道，接收人异常标记并通知运营）；优化重试机制，支持自定义重试次数、重试间隔，紧急消息可增加重试次数；搭建异常监控预警机制，设置异常阈值（如推送异常率超过5%触发预警），通过企业微信、短信推送预警信息给运营与运维人员，确保异常及时处理，同时支持异常日志全量留存与分析，快速定位问题根源，对于P0/P1级告警，直接推送至运维人员手机短信与电话，确保紧急问题快速响应。

6. **用户消息偏好精细化（全域联动）**：C端新增消息类型开关（如关闭营销消息、开启物流消息），用户可自主控制接收的消息类型；支持推送时间偏好设置（如仅在9:00\-22:00推送非紧急消息），避免打扰用户；系统记录用户消息点击、删除行为，分析用户偏好，自动优化消息推送频率与内容，减少无效消息推送，提升用户体验，同时尊重用户隐私，支持消息推送权限的全面管控。

7. **系统预警与业务协同消息（全域联动）**：新增系统预警消息（如服务器宕机、数据库异常、推送渠道故障、脚本异常），推送至运维人员与相关负责人，支撑系统稳定运行；新增业务协同消息（如商家订单提醒、库存预警、运营任务通知、订单/派单堆积提醒），实现各角色间的业务协同，提升工作效率；支持自定义预警阈值与协同消息触发条件，适配不同业务场景的需求，同时将预警消息与监控模块联动，实现异常的快速响应与处理。

8. **消息数据精细化统计与分析（消息管理域）**：搭建消息数据看板，实时展示核心数据（消息发送量、推送成功率、触达率、点击转化率、异常率、各渠道使用率、各消息类型占比）；支持按时间、区域、用户等级、消息类型、推送渠道筛选数据；生成消息分析报告，分析消息触达瓶颈（如某渠道异常率高、某类型消息点击低）、用户偏好，为模板优化、渠道选择、推送策略调整提供数据支撑；支持自定义报表导出（PDF/Excel格式），满足不同运营与运维需求，助力精细化运营决策。

9. **消息推送自动化与智能化（消息分发域）**：配置自动化推送规则，实现特定场景消息自动推送（如用户生日自动发送生日祝福、商品补货自动通知关注用户）；引入智能算法，根据用户行为（点击、删除、浏览）精准推送消息，提升消息相关性与点击转化率；支持A/B测试，对不同消息模板、推送渠道、推送时间进行测试，选择最优方案上线；新增消息推送效果跟踪，记录消息点击、跳转行为，形成闭环优化，同时通过算法自动调整消息推送频率，避免消息骚扰。

10. **消息安全与合规管控（异常处理域\+消息模板域）**：新增消息内容合规校验（敏感词、违规内容过滤），对接第三方合规接口，确保消息内容符合监管要求；新增用户隐私保护，推送消息时脱敏用户敏感信息（如手机号隐藏中间4位、邮箱隐藏部分字符）；新增推送权限管控，不同角色仅可推送对应类型的消息（如商家仅可推送自身店铺相关消息）；完善消息日志留存机制，满足合规审计要求，同时记录所有消息操作日志，实现操作可追溯，避免违规推送行为。

# 五、完整业务边界（通用）

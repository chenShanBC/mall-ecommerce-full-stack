# 企业级电商物流模块（mall\-logistics）初阶+后期拓展完整业务流程与边界

本文档整合物流模块 MVP最小可行版本 与 后期拓展完整版业务，明确各阶段业务流程、职责边界，贴合mallFei微服务架构，兼顾当前落地与后期迭代，可直接用于项目开发、文档归档、技术评审。

# 一、物流模块定位（结合微服务拆分·后期完整版）

物流模块（mall\-logistics）是电商平台的**履约交付、物流追踪、体验保障**核心模块，独立承载全平台订单发货、物流对接、轨迹查询、售后退货物流、物流异常处理全流程，属于**强协同、高时效、重体验**的核心业务域。

与订单、用户、商品、售后、库存深度协同，但**物流渠道管理、物流轨迹同步、物流状态管控、退货物流对接**完全自治，不侵入其他微服务业务逻辑，是连接商家与用户、完成电商履约闭环的关键枢纽，遵循微服务高内聚、低耦合的核心拆分原则，实现物流业务的独立部署与扩展。

结合微服务拆分（后期完整版），完整物流模块包含五大核心域，贴合领域驱动设计（DDD）的限界上下文划分，确保各域职责单一、数据独立：

1. 物流渠道域：快递公司对接、物流渠道配置、物流费用设置、渠道权限管理

2. 订单物流域：订单发货、物流单号生成、物流状态同步、物流轨迹查询

3. 售后物流域：退货物流对接、退货轨迹查询、退货签收确认、退货物流异常处理

4. 物流异常域：物流超时、丢件、破损、拒签等异常处理、异常记录、理赔对接

5. 运营管控域：物流数据统计、物流规则配置、物流黑名单、物流时效管控

# 二、整体架构说明

本物流模块基于微服务拆分设计，核心遵循**渠道独立、轨迹同步、数据不跨域**原则，坚守微服务数据私有、接口交互的核心准则，确保物流业务与其他服务解耦：

- mall\-auth：仅做鉴权，不参与物流流程、渠道配置与异常判定

- mall\-user：提供用户ID、收货地址、联系方式（用于物流配送、通知推送）

- mall\-order：提供订单信息、商品信息、收货信息，发起发货请求，接收物流状态回调

- mall\-aftersale：发起退货物流查询、退货签收回调请求，接收物流轨迹与状态同步

- mall\-inventory：接收物流发货/退货回调，同步更新商品库存状态

- mall\-logistics：**唯一负责物流渠道管理、订单发货履约、物流轨迹同步、异常处理、售后物流对接**，独享自身业务数据，不允许其他服务直接访问数据库，仅通过接口提供服务

核心边界：

- 只做“物流履约与轨迹管控”，不修改订单状态（仅回调同步）、不直接操作库存（仅回调通知）、不处理订单支付与售后审核

- 所有物流操作（发货、轨迹同步、异常处理）均有记录、可追溯，物流状态与订单、售后状态实时联动，保障履约透明

- 物流渠道统一对接、集中管理，支持多渠道扩展，适配不同场景物流需求，实现物流资源的灵活调配

- 异常物流有明确处理机制与时效管控，快速响应丢件、破损等问题，降低用户投诉率

# 三、初阶阶段业务流程（当前落地核心）

核心目标：支撑商城基础物流履约场景，完成订单发货、物流轨迹查询、售后退货物流对接，无复杂异常处理与精细化管控，满足用户基础物流需求，实现物流与订单、售后的基础协同。

## 3\.1 核心业务流程（7大必备）

1. **物流渠道配置（运营后台）**：运营人员经mall\-auth鉴权\+管理员角色校验通过后，配置主流快递公司（如中通、圆通、申通）对接信息，录入快递公司API密钥、物流费用模板（按重量/体积/区域计费）、物流渠道状态（启用/禁用）；支持渠道编辑、禁用操作，禁用后不可用于订单发货；系统校验渠道信息合法性，确保API对接正常，符合微服务接口交互规范。

2. **订单发货（商家后台/运营后台）**：订单支付完成后，商家/运营人员进入订单详情页，选择已配置的物流渠道，填写发货商品信息、包裹重量/体积，提交发货请求；系统调用对应快递公司API，生成物流单号，更新订单物流状态为“已发货”，同步回调mall\-order模块更新订单状态，同时生成物流发货记录（关联订单ID、物流单号、渠道、发货时间）；支持批量发货操作，提升发货效率。

3. **物流轨迹查询（C端\+商家后台\+运营后台）**：用户、商家、运营人员可通过订单ID或物流单号，查询物流实时轨迹；系统对接快递公司API，获取物流节点信息（如“已揽收”“运输中”“派件中”“已签收”），按时间倒序展示；支持物流轨迹刷新，确保轨迹信息实时同步，满足用户对物流进度的查询需求。

4. **物流状态同步（自动\+手动）**：系统定时调用快递公司API，同步物流状态，自动更新订单物流状态（如揽收、派件、签收），同步回调mall\-order模块；若自动同步失败（如API异常），运营人员可手动查询并更新物流状态，确保物流状态与实际履约情况一致，保障订单流程顺畅。

5. **售后退货物流对接（C端\+运营后台）**：售后审核通过（退货退款）后，运营后台配置退货地址，系统推送至C端用户；用户填写快递公司、快递单号，提交退货物流信息，系统校验快递公司合法性、快递单号格式，校验通过后关联售后单，同步回调mall\-aftersale模块更新售后物流状态；用户、运营人员可通过售后单ID查询退货物流轨迹，实现退货流程的可视化跟踪。

6. **退货签收确认（运营后台）**：系统同步退货物流轨迹，当检测到退货物流状态为“已签收”时，自动回调mall\-aftersale模块，通知审核人员进行退货验收；运营人员也可手动确认退货签收，同步更新售后物流状态为“已签收”，触发后续验收流程，确保售后与物流的协同闭环。

7. **物流记录查询与统计（运营后台）**：运营人员可查看所有物流记录，按订单ID、物流单号、物流渠道、物流状态、时间筛选；统计核心数据（发货量、签收量、未签收量、退货物流量），支持导出发货与退货物流数据（Excel格式），仅提供查询、统计、导出能力，不修改业务数据，为运营决策提供基础数据支撑。

# 四、后期拓展完整版业务流程（企业级升级）

在MVP基础上，补齐物流异常处理、精细化管控、智能调度、跨境物流、物流数据分析，形成企业级物流中台，遵循微服务独立扩展原则，提升物流履约效率、降低物流成本、优化用户体验，适配高并发、多场景物流需求。

## 4\.1 新增拓展业务流程（10大核心，覆盖五大核心域）

1. **物流渠道精细化管理（物流渠道域）**：新增物流渠道分级（如普通快递、特快快递、冷链快递），配置不同渠道的时效、费用、适用商品（如冷链用于生鲜）；支持物流渠道优先级设置，系统根据订单金额、商品类型、收货区域自动匹配最优物流渠道；对接更多快递公司（含国际快递），支持物流渠道对账，自动核对物流费用，减少人工对账误差，实现物流资源的精细化配置。

2. **智能物流调度（订单物流域）**：引入GIS地图引擎与路径优化算法，结合交通拥堵、天气变化、仓库位置等因素，动态生成最优配送路线，提升配送效率；支持批量订单智能分单，按区域、快递公司运力分配订单，实现物流资源的合理调度；对接仓库管理系统（WMS），实现订单发货与仓库出库的无缝衔接，支持条码/RFID扫码作业，减少发货错误率。

3. **物流时效管控（运营管控域）**：配置各区域、各物流渠道的时效规则（如省内24小时达、跨省48小时达）；系统实时监控物流时效，超时未送达自动触发预警（通过mall\-message推送至运营人员、商家）；统计各物流渠道时效达标率，对时效不达标渠道进行优化或替换，建立SLA服务等级协议，确保系统可用性≥99\.5%，保障物流履约时效。

4. **物流异常全流程处理（物流异常域）**：新增物流异常类型（丢件、破损、拒签、超时、错发），建立异常处理流程；用户、商家可发起物流异常申诉，运营人员审核申诉、对接快递公司核实，处理异常（如补发、理赔）；异常处理完成后，同步更新物流状态与订单/售后状态，生成异常处理记录，可追溯；对接快递公司理赔接口，实现理赔流程自动化，提升异常处理效率。

5. **物流费用精细化管控（物流渠道域）**：配置阶梯物流费用（如重量阶梯、区域阶梯）、优惠物流政策（如满额包邮、商家专属优惠）；支持物流费用预估，用户下单时展示预估物流费用；对接财务模块，同步物流费用明细，支持物流费用对账、报销，实现物流成本的精准管控，降低电商运营成本。

6. **跨境物流支撑（订单物流域）**：新增跨境物流渠道（如国际快递、海外仓配送），配置跨境物流规则（关税计算、清关流程、时效配置）；支持跨境物流轨迹查询、清关状态同步，对接海外仓系统，实现海外仓发货、退货对接；处理跨境物流异常（清关失败、海外丢件），满足跨境电商业务需求，拓展电商业务边界。

7. **物流监控与预警（运营管控域）**：搭建物流监控大屏，实时展示物流核心数据（发货量、签收率、时效达标率、异常率、各渠道运力）；设置异常预警阈值（如异常率超过5%触发预警），自动推送预警信息；监控快递公司API状态，API异常时自动切换备用渠道，确保物流服务不中断，提升物流系统的稳定性。

8. **物流自动化处理（运营管控域）**：配置自动化规则，实现简单物流场景自动处理（如订单支付后自动匹配物流渠道、生成物流单号，物流签收后自动回调订单/售后模块，异常物流自动触发预警）；对接自动化仓储设备，实现订单自动出库、自动打包、自动对接物流，减少人工操作，提升物流履约效率，适配高并发发货场景。

9. **物流数据统计与分析（运营管控域）**：搭建物流数据看板，实时展示核心数据（物流履约率、时效达标率、异常率、各渠道使用率、物流成本）；支持按时间、区域、商品品类、商家筛选数据；生成物流分析报告，分析物流瓶颈（如某区域时效低、某渠道异常率高），优化物流渠道配置与调度策略，助力运营人员降本增效；支持自定义报表导出PDF或Excel格式，满足不同运营需求。

10. **物流通知精细化（全域联动）**：整合物流全流程通知，支持短信、APP推送、站内消息三种形式；通知类型包括：物流发货提醒、物流轨迹更新、物流异常提醒、退货物流签收提醒、物流时效预警等；用户可在C端设置物流通知开关，系统记录通知发送状态（已发送、未发送、已读、未读），确保用户及时了解物流进度，提升用户体验。

# 五、完整业务边界（通用）

## 5\.1 核心职责

负责五大核心域（物流渠道域、订单物流域、售后物流域、物流异常域、运营管控域）的规则制定、业务落地、数据管理；为订单、售后、库存提供物流状态同步、发货回调、物流查询等能力；独立完成物流渠道对接、订单发货履约、物流轨迹同步、异常处理、售后物流对接，不侵入任何外部微服务业务逻辑，坚守微服务高内聚、低耦合原则，实现物流业务的独立自治。

## 5\.2 对内边界

- → mall\-order：提供订单信息、商品信息、收货信息；接收物流状态回调（如已发货、已揽收、已签收、物流异常），同步更新订单状态；不修改订单其他数据，仅接收回调通知，通过接口实现数据交互，不直接访问订单数据库。

- → mall\-user：读取用户ID、收货地址、联系方式；不修改用户任何数据，仅用于物流配送、物流通知推送、收货信息校验，确保用户信息安全。

- → mall\-aftersale：提供售后单信息、退货地址；接收退货物流状态回调（如退货已揽收、已签收），同步更新售后状态；不修改售后其他数据，仅接收回调通知，支撑售后退货流程闭环。

- → mall\-inventory：接收物流发货/退货回调（如订单发货、退货签收），同步更新商品库存状态；不修改库存其他数据，仅发起回调通知，确保库存与物流履约状态一致。

- → mall\-message：推送物流相关通知（发货提醒、轨迹更新、异常提醒等）；仅发送通知请求，不处理通知发送逻辑，依托消息模块完成通知触达。

- ← 所有外部模块仅可调用物流模块的查询、发货、物流状态回调接口，物流模块不主动调用外部模块的写操作，仅接收回调通知并同步自身状态，确保服务间低耦合。

## 5\.3 对外边界

- 对外开放接口：MVP阶段开放物流渠道配置、订单发货、物流轨迹查询、物流状态同步、退货物流对接接口；后期拓展后，新增智能调度、异常申诉、跨境物流、物流费用预估等接口。

- 接口访问范围：仅对C端H5、小程序、PC运营后台、商家后台开放；第三方快递公司仅开放API对接接口（用于物流轨迹、物流状态同步）；第三方平台不开放任何核心接口，仅可查询公开物流状态（需鉴权）。

- 数据安全要求：所有接口返回数据需脱敏（不暴露用户完整收货地址、联系方式、商家敏感信息、物流API密钥）；物流记录、异常处理记录、轨迹信息全程留存，可追溯，满足合规要求；坚守微服务数据私有原则，不泄露自身业务数据。

- 鉴权要求：除物流轨迹公开查询接口外，其余所有接口均需通过mall\-auth模块的Sa\-Token鉴权与角色校验（如运营接口需管理员角色，商家接口需商家角色，C端接口需用户登录鉴权），确保接口访问安全。

## 5\.4 禁止范围

- 不创建订单、不修改订单状态、不处理订单相关业务（仅同步物流状态）；

- 不直接操作商品库存、不管理库存业务（仅回调通知库存模块）；

- 不处理售后审核、退款、纠纷等业务（仅对接售后退货物流，同步物流状态）；

- 不做用户登录鉴权、Token分发、风控规则制定（统一由mall\-auth认证中心承担）；

- 不跨模块修改其他领域数据（如用户信息、商品信息、订单金额、售后状态），不接收其他领域的业务指令（仅提供数据查询与物流支撑）；

- 不允许虚假发货、不允许篡改物流轨迹与状态、不允许违规对接未配置的物流渠道；

- 不存储用户密码、完整身份证号、未脱敏手机号等敏感用户数据（仅存储用户ID用于关联），不存储订单支付信息、售后退款信息等非物流类数据。

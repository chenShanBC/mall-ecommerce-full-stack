# 企业级电商售后模块（mall\-aftersale）初阶+后期拓展完整业务流程与边界

本文档整合售后模块 MVP最小可行版本 与 后期拓展完整版业务，明确各阶段业务流程、职责边界，贴合mallFei微服务架构，兼顾当前落地与后期迭代，可直接用于项目开发、文档归档、技术评审。

# 一、售后模块定位（结合微服务拆分·后期完整版）

售后模块（mall\-aftersale）是电商平台的**用户信任保障、纠纷处理、口碑维护**核心模块，独立承载全平台售后申请、审核、履约、退款、纠纷介入全流程，属于**强流程、高交互、重合规**的核心业务域。

与订单、支付、用户、商品、物流深度协同，但**售后流程管控、退款规则、纠纷判定、售后状态**完全自治，不侵入其他微服务业务逻辑，是电商闭环不可或缺的关键环节。

结合微服务拆分（后期完整版），完整售后模块包含五大核心域：

1. 售后申请域：仅退款、退货退款申请提交、申请修改、申请撤销、售后进度查询

2. 售后审核域：商家审核、平台审核、审核规则、驳回原因管理、审核记录

3. 售后履约域：退货物流、退货验收、退款处理、退款到账、售后凭证管理

4. 纠纷处理域：平台介入、纠纷判定、举证管理、协商调解、售后仲裁

5. 运营管控域：售后数据统计、售后规则配置、异常售后处理、售后黑名单

# 二、整体架构说明

本售后模块基于微服务拆分设计，核心遵循**流程独立、规则自治、数据不跨域**原则：

- mall\-auth：仅做鉴权，不参与售后流程与规则判定

- mall\-user：提供用户ID、个人信息、收货地址（用于退货地址推送）

- mall\-order：提供订单信息、支付信息、商品信息，接收售后状态回调

- mall\-payment：接收退款请求、处理退款到账、同步退款结果

- mall\-logistics：提供物流轨迹查询、退货物流对接、物流状态同步

- mall\-aftersale：**唯一负责售后流程管控、审核判定、退款发起、纠纷处理**

核心边界：

- 只做“售后全流程管控与退款发起”，不修改订单状态（仅回调同步）、不直接处理支付退款（仅发起请求）

- 所有售后操作均有记录、可追溯，审核与判定遵循预设规则，保障公平合规

- 异常售后（纠纷、超时）有明确处理机制，避免流程卡顿

# 三、初阶阶段业务流程（当前落地核心）

核心目标：支撑商城基础售后场景，完成仅退款、退货退款闭环，无复杂纠纷处理，满足用户基础售后需求。

## 3\.1 核心业务流程（7大必备）

1. **售后申请（C端）**：用户登录经mall\-auth鉴权通过，进入订单详情页发起售后申请；支持两种类型（仅退款、退货退款），需填写申请原因、退款金额（不可超过实付金额）、问题描述，可上传凭证图片；系统校验订单状态（仅支持已支付、已发货、已收货状态的订单）、申请时限（如确认收货后7天内）、退款金额合法性，校验通过后生成售后单，状态为“待审核”，同步通知运营后台与商家（多商户场景）。

2. **售后申请修改与撤销（C端）**：售后单处于“待审核”状态时，用户可修改申请信息（原因、金额、凭证）或撤销申请；修改后需重新提交审核，撤销后售后单状态变为“已撤销”，不可再次发起同一订单同一商品的售后申请（可重新发起新申请）；若售后单已进入审核或后续流程，禁止修改与撤销。

3. **售后审核（运营后台/商家后台）**：审核人员（运营/商家）经mall\-auth鉴权\+对应角色校验通过后，查看售后单详情（申请信息、订单信息、凭证）；支持“审核通过”“审核驳回”操作，驳回需填写明确驳回原因（如“凭证不足”“超出售后时限”）；审核通过后，根据售后类型跳转至对应流程（仅退款→发起退款，退货退款→通知用户退货）；审核驳回后，售后单状态变为“已驳回”，同步通知用户，允许用户补充凭证后重新申请。

4. **退货物流提交（C端，仅退货退款）**：售后审核通过后，系统向用户推送退货地址（运营/商家配置）；用户填写快递公司、快递单号，提交退货物流信息；系统校验快递公司合法性、快递单号格式，校验通过后更新售后单状态为“待退货验收”，同步通知审核人员；用户可修改物流信息（仅未验收前），可查询物流轨迹（对接mall\-logistics）。

5. **退货验收（运营后台/商家后台）**：审核人员收到退货后，核对商品完整性、是否符合退货规则（如无破损、不影响二次销售）；支持“验收通过”“验收驳回”操作，驳回需填写原因（如“商品破损”“已影响二次销售”）；验收通过后，售后单状态变为“待退款”；验收驳回后，状态变为“验收驳回”，同步通知用户，用户可发起申诉（MVP阶段仅支持重新提交售后申请）。

6. **退款处理（运营后台\+支付模块）**：售后单处于“待退款”状态时，审核人员发起退款操作；系统调用mall\-payment退款接口，入参为售后单ID、用户ID、退款金额、退款原因；支付模块处理退款，同步退款进度与结果；退款到账后，售后单状态变为“已完成”；若退款失败（如支付账户异常），售后单状态变为“退款失败”，审核人员可重新发起退款，同步通知用户。

7. **售后进度查询与记录（C端\+运营后台）**：C端用户可查看所有售后单列表，按“待审核、待退货、待退款、已完成、已驳回、已撤销”分类展示，点击详情可查看售后全流程进度、操作记录、审核意见；运营后台可查看所有售后单，支持按订单号、用户ID、售后状态、时间筛选，查看售后详情与操作日志，支持导出售后数据（Excel格式）。

# 四、后期拓展完整版业务流程（企业级升级）

在MVP基础上，补齐纠纷处理、复杂售后场景、智能风控、精细化运营、合规管控，形成企业级售后中台，提升用户体验与运营效率，降低纠纷率。

## 4\.1 新增拓展业务流程（9大核心，覆盖五大核心域）

1. **售后类型拓展（售后申请域）**：新增换货、维修、补发三种售后类型，覆盖全场景售后需求；换货流程：申请换货→审核通过→用户退货→验收通过→商家发货→用户确认收货→售后完成；维修流程：申请维修→审核通过→用户寄回→维修完成→寄回用户→售后完成；补发流程：申请补发→审核通过→商家补发→物流签收→售后完成；每种类型配置独立流程与规则。

2. **售后时效管控（售后审核域）**：配置各环节时效规则（如审核时效24小时、退货时效7天、退款时效48小时）；超时未操作自动触发提醒（通过mall\-message推送），超时严重则自动按预设规则处理（如审核超时自动通过、退款超时自动发起退款）；运营后台可查看时效预警列表，及时处理超时售后。

3. **售后凭证精细化管理（售后履约域）**：区分用户凭证（问题商品图片、物流面单）与商家/运营凭证（验收图片、维修记录、补发物流单）；支持凭证在线预览、下载、打印；凭证上传有格式与大小限制，系统自动校验凭证清晰度；售后单归档时，凭证与售后记录同步留存，用于后续纠纷追溯与合规检查。

4. **平台介入与纠纷仲裁（纠纷处理域）**：用户对售后审核、验收结果有异议时，可发起平台介入申请；平台介入人员接收申请后，调取售后全流程记录、双方凭证，进行协商调解；调解不成则启动仲裁流程，依据平台规则、双方举证做出仲裁结果，仲裁结果不可更改；支持双方在线举证、答辩，仲裁过程全程记录，确保公平公正。

5. **售后退款精细化配置（售后履约域）**：支持多种退款方式（原路退回、余额退款、优惠券补偿），可配置退款规则（如仅退款退全款、退货退款退商品金额\+运费/仅退商品金额）；支持部分退款（如订单多商品仅退其中一件），退款金额可精准拆分；退款到账后，同步更新订单支付状态、优惠核销状态（如优惠券退回），对接mall\-payment完成退款闭环。

6. **售后风控与黑名单（运营管控域）**：建立售后风控机制，监控异常售后行为（如频繁发起售后、恶意退货、虚假凭证）；对异常用户标记为售后黑名单，限制其售后申请权限（如限制次数、禁止申请）；运营后台可查看风控预警、黑名单列表，手动添加/移除黑名单，配置风控规则（如异常行为判定标准）。

7. **售后自动化处理（运营管控域）**：配置自动化规则，实现简单售后场景自动处理（如未发货订单仅退款自动审核通过、小额退款自动发起、无凭证且符合规则的售后自动通过）；减少人工操作，提升售后效率；支持自动化规则编辑、启用/禁用，可根据运营需求调整。

8. **售后数据统计与分析（运营管控域）**：搭建售后数据看板，实时展示核心数据（售后率、审核通过率、退货率、退款时效、纠纷率、各售后类型占比）；支持按时间、商品品类、用户等级、商家（多商户）筛选数据；生成售后分析报告，助力运营人员发现售后高频问题（如某类商品退货率高），优化商品与服务。

9. **售后通知精细化（全域联动）**：整合售后全流程通知，支持短信、APP推送、站内消息三种形式；通知类型包括：售后申请提交成功、审核结果、退货地址推送、验收结果、退款进度、平台介入提醒、纠纷仲裁结果等；用户可在C端设置售后通知开关，系统记录通知发送状态（已发送、未发送、已读、未读），确保用户及时了解售后进度。

# 五、完整业务边界（通用）

## 5\.1 核心职责

负责五大核心域（售后申请域、售后审核域、售后履约域、纠纷处理域、运营管控域）的规则制定、业务落地、数据管理；为订单、支付、物流提供售后状态同步、退款发起、物流对接等能力；独立完成售后全流程管控、纠纷调解、退款协调，不侵入任何外部微服务业务逻辑。

## 5\.2 对内边界

- → mall\-order：提供订单信息、支付信息、商品信息；接收售后状态回调（如售后发起、审核通过、退款完成），同步更新订单售后状态；不修改订单其他数据，仅接收回调通知。

- → mall\-user：读取用户ID、个人信息、收货地址；不修改用户任何数据，仅用于售后通知推送、退货地址关联。

- → mall\-payment：发起退款请求、查询退款进度；接收支付模块的退款结果回调，同步更新售后单退款状态；不处理支付业务、不存储支付密钥，仅发起退款请求。

- → mall\-logistics：查询物流轨迹、对接退货物流；接收物流模块的物流状态回调（如退货签收），同步更新售后单物流状态；不管理物流渠道、不修改物流信息，仅查询与接收回调。

- → mall\-message：推送售后相关通知（申请状态、审核结果、退款进度等）；仅发送通知请求，不处理通知发送逻辑。

- ← 所有外部模块仅可调用售后模块的查询、状态回调接口，售后模块不主动调用外部模块的写操作，仅接收回调通知并同步自身状态。

## 5\.3 对外边界

- 对外开放接口：MVP阶段开放售后申请、申请修改/撤销、售后进度查询、审核回调、退款发起接口；后期拓展后，新增换货/维修/补发申请、平台介入、举证、售后数据查询等接口。

- 接口访问范围：仅对C端H5、小程序、PC运营后台、商家后台开放；第三方平台不开放任何核心接口，仅可查询公开售后状态（需鉴权）。

- 数据安全要求：所有接口返回数据需脱敏（不暴露用户完整信息、商家敏感信息、退款明细密钥）；售后记录、凭证、仲裁结果全程留存，可追溯，满足合规要求。

- 鉴权要求：除售后进度公开查询接口外，其余所有接口均需通过mall\-auth模块的Sa\-Token鉴权与角色校验（如运营接口需管理员角色，商家接口需商家角色，C端接口需用户登录鉴权）。

## 5\.4 禁止范围

- 不创建订单、不修改订单状态、不处理订单相关业务（仅同步售后状态）；

- 不直接处理支付退款、不存储支付记录、不接触支付密钥（仅发起退款请求）；

- 不管理物流渠道、不修改物流信息、不承担物流履约责任（仅对接物流查询）；

- 不做用户登录鉴权、Token分发、风控规则制定（统一由mall\-auth认证中心承担）；

- 不跨模块修改其他领域数据（如用户信息、商品信息、订单金额），不接收其他领域的业务指令（仅提供数据查询与售后支撑）；

- 不允许无规则审核、不允许虚假退款、不允许篡改售后记录与凭证；

- 不存储用户密码、完整身份证号、未脱敏手机号等敏感用户数据（仅存储用户ID用于关联）。
